# Анализ и оптимизация запросов 1С

Запрос оптимизируют **по плану и измерению**, а не по наитию. Алгоритм: найти тяжёлый запрос → воспроизвести →
снять план → найти неоптимальность → исправить → **повторно замерить**. Версии СУБД/платформы — НАСТРОЙ ПОД
ПРОЕКТ. Где не уверен в поведении оптимизатора конкретной версии — [проверить] на реальном плане.

## 1. Найти тяжёлый запрос (откуда брать кандидатов)
- **Технологический журнал**: события `DBMSSQL`/`DBPOSTGRS` (SQL к СУБД) и `SDBL` (запрос к модели 1С) с
  большой `duration` или большой суммарной длительностью (top-5 — см. `tech-journal.md`). Поля `Sql`,
  `Context` (откуда вызван), `Rows`/`RowsAffected`.
- **ЦУП/КИП**: список долгих запросов по ключевым операциям.
- **Встроенный замер производительности** (Конфигуратор/EDT) — построчно, локально, на тестовых данных.
- **PostgreSQL**: `pg_stat_statements` (top по `total_exec_time`/`mean_exec_time`/`calls`), `auto_explain`
  для медленных запросов в лог. **MS SQL**: DMV `sys.dm_exec_query_stats` + `sys.dm_exec_sql_text`.

## 2. Воспроизвести запрос (до анализа плана)
Алгоритм воспроизведения (дерево принятия решений из методики 1С):
1. Запрос **служебный** (нет контекста + служебные таблицы)? → это работа платформы (реструктуризация,
   фоновое и т.п.); проверь логи, что за процесс это запустил — оптимизация прикладного кода не нужна.
2. Запрос вернул/записал **много строк**? → проверь, **правильно ли установлены фильтры** (параметры отбора).
   Часто причина — отсутствующий или неселективный отбор, а не отсутствие индекса. Если данных правда много —
   возможно, перенести в ночное/фоновое формирование.
3. Найди **текст запроса в конфигурации** (по `Context`), воспроизведи в консоли запросов **с теми же
   параметрами** и **с RLS** (под тем же пользователем).
4. Медленно **только под конкретным пользователем**? → причина в RLS (ограничения на уровне записей), а не в
   самом запросе; оптимизируй условия RLS / шаблоны ограничений.
5. **Текст запроса большой**? → применяй **метод усечения** (бинарный поиск): убирай половину
   соединений/условий, смотри, пропала ли деградация — так локализуешь проблемный фрагмент.

## 3. Снять план запроса
- **PostgreSQL**: `EXPLAIN (ANALYZE, BUFFERS) <запрос>;` — показывает реальные строки vs план, время по
  узлам, чтения буферов. Ключевое: **Seq Scan** на большой таблице, расхождение `rows` (estimated) и
  `actual rows` (плохая статистика), дорогие соединения (`Nested Loop` с большим числом итераций vs
  `Hash Join`), сортировки/хэши в диск (`external merge Disk`).
- **MS SQL**: фактический план выполнения (Include Actual Execution Plan) или `SET STATISTICS IO, TIME ON`;
  ищи `Index Scan`/`Table Scan` вместо `Index Seek`, `Key Lookup`/`RID Lookup` (нет покрывающего индекса),
  толстые стрелки (фактические vs оценочные строки), `Sort`/`Hash Match` с большим cost.
- **Из ТЖ**: включить `<plansql/>` — план попадёт в свойство `planSQLText` событий DBMSSQL/DBPOSTGRS (удобно,
  когда нельзя воспроизвести вне прода). Снимать точечно.
- 1С генерирует SQL сам — план снимают на **итоговом SQL**, не на тексте запроса 1С. Имена таблиц
  (`_Document123`, `#tt17`) → разворачивай через структуру хранения / временные таблицы менеджера.

## 4. Признаки неоптимальности (что искать в плане)
1. **Сканирование вместо поиска по индексу** (Seq/Table/Index Scan) на большой таблице → нет подходящего
   индекса ИЛИ условие не саргируемо (функция над полем, неявное приведение типов, `ПОДОБНО '%x'`).
2. **Соединения с подзапросами** (вложенный запрос в соединении) → 1С не всегда строит хороший план; вынеси
   подзапрос во **временную таблицу** с индексом.
3. **Чтение лишнего**: `ВЫБРАТЬ *` / лишние поля, соединение таблиц, из которых поля не используются,
   отсутствие `ПЕРВЫЕ N`, чтение всей таблицы ради нескольких строк.
4. **Расхождение оценок и факта** (`rows` ≠ `actual`) → устаревшая статистика → обновить статистику /
   ANALYZE; иногда нужна явная перестройка индекса.
5. **Соединение по неиндексированному полю** / по выражению → нет индекса под условие соединения.
6. **Плохие временные таблицы**: соединение больших ВТ без индекса; ВТ строится дороже, чем экономит.
7. **Параметры-несаргируемость**: условие вида `ГДЕ Поле + 1 = &П` или приведение типов в JOIN.

## 5. Лечение
### Индексы (главный рычаг)
- **Составной (многоколоночный) индекс** — порядок полей: сначала поля с равенством (`=`), затем диапазон/
  сортировка. Селективные поля впереди.
- **Покрывающий индекс** — включает все поля, нужные запросу (в MS SQL — `INCLUDE`), чтобы убрать
  `Key/RID Lookup`. В 1С — через индексирование реквизитов «Индексировать с доп. упорядочиванием» и состав
  измерений регистра.
- В 1С индекс задаётся **в конфигураторе/EDT** (свойство «Индексировать» реквизита/измерения, индексы
  регистров) — это **IDE-операция человека**, AI её не делает (см. `1c-dev`). Эксперт ставит ДИАГНОЗ «нужен
  индекс по (Поле1, Поле2)», разработчик создаёт его в IDE.
- Не плодить индексы бесконтрольно: каждый индекс замедляет запись и занимает место. Проверяй
  востребованность (MS SQL: `sys.dm_db_index_usage_stats`; Postgres: `pg_stat_user_indexes`).

### Временные таблицы
- Разбивай тяжёлый запрос с подзапросами/множественными соединениями на **пакет** с временными таблицами
  (`ПОМЕСТИТЬ`), **индексируй** ВТ по полям соединения (`ИНДЕКСИРОВАТЬ ПО`). Используй
  `МенеджерВременныхТаблиц` для переиспользования.
- ВТ материализует промежуток с известной статистикой → СУБД строит лучший план соединения.

### Переписывание
- Убрать соединения с подзапросами → ВТ. Заменить `ВЫБОР КОГДА` в соединениях на предрасчёт. Не соединять
  ради проверки существования — `В (ВЫБРАТЬ ...)` или `ОБЪЕДИНИТЬ`/полусоединение. Минимизировать читаемые
  поля и таблицы. Добавить недостающие фильтры/`ПЕРВЫЕ N`. Сделать условия саргируемыми (не оборачивать поле
  функцией).
- Правильно установить **отбор** — самая частая причина «много строк»: проверь параметры до индексов.

### Статистика
- PostgreSQL: `ANALYZE <таблица>;` / автовакуум; MS SQL: `UPDATE STATISTICS` / автообновление + трейс-флаги
  (2371 для крупных таблиц). Регламентное обслуживание статистики/индексов — это зона `1c-dba`; эксперт
  фиксирует «план плохой из-за устаревшей статистики», dba настраивает регламент.

## 6. Пошаговый алгоритм (свод)
1. Симптом → найти кандидата (ТЖ/ЦУП/pg_stat_statements/замер).
2. Классифицировать: служебный? много строк из-за фильтра? только под пользователем (RLS)?
3. Воспроизвести с теми же параметрами и RLS; большой текст → метод усечения.
4. Снять план (EXPLAIN ANALYZE / план MS SQL / planSQLText).
5. Найти неоптимальность (скан / подзапрос в соединении / чтение лишнего / плохая статистика / нет индекса).
6. Гипотеза фикса (индекс / ВТ / переписать / отбор / статистика).
7. Применить (индекс/метаданные — человек в IDE; код — `1c-dev`).
8. **Повторно замерить** план и длительность — деградация ушла? Иначе вернуться к п.4.

## 7. Типичные провалы
- Оптимизируют по тексту запроса 1С, а не по итоговому SQL-плану → мимо.
- Добавляют индекс «на всякий случай» без плана → замедляют запись, проблема остаётся.
- Не воспроизводят под RLS/тем же пользователем → не видят настоящую причину (RLS).
- Не обновили статистику → план «не чинится» индексами.
- Нет замера «после» → нельзя утверждать, что починили.
