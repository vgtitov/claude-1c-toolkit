# Производительность контура PROD — экспертный отчёт и план реализации (01.07.20XX)

**Единый рабочий документ для команды.** Данные Zabbix за сутки: сервер 1С (дашборд 101) + СУБД
(дашборд 102) + все хосты `srv-1c-*`, окно 24 ч, 818 элементов. SQL-таблицы переведены в объекты 1С
по карте структуры хранения ERP_Main. Ниже: диагноз, затем план по направлениям — что делать, где,
как снять данные и как проверить результат.

## Инвентарь контура

| Хост | Роль | ОС / СУБД | Ресурсы |
| --- | --- | --- | --- |
| `srv-1c-app-01` | сервер 1С (кластер) | Windows Server 2022 | 14 vCPU / **потолок 12 ядер rphost (лицензия ПРОФ)**, 85 GiB RAM |
| `srv-1c-db-01` | СУБД | PostgreSQL 15 / Linux | 14 vCPU, 41 GiB RAM, /data 676 GB |

Боевые базы: ERP_Main (~96 GB), UT_Main, BU_Main, ЗУПы. Архивные: `*_Old`.

## Главный вывод

**Узкое место — не железо, а несколько механизмов кода и регламента.** С поправкой на лицензию ПРОФ
(rphost ограничен 12 ядрами из 14) CPU app-сервера за сутки ≈39%, потолок лицензии не упирается;
утренний пик 90% был сцеплен с длинной транзакцией и разошёлся вместе с ней. Кэш СУБД попадает,
подключения PG далеко от предела. Теряем на пяти механизмах — по каждому есть конкретный SQL или модуль.

# План реализации по направлениям

Порядок — по эффекту (сначала дешёвые победы с широким охватом). У каждого направления: **где · шаги ·
как снять данные · критерий готовности · исполнитель**.

## Направление 1. Выключить регламентные задания в архивных базах  ⏱ самая дешёвая победа

- **Где:** базы `*_Old` (в топе CALL и держателях сессий только `SystemBackgroundJob`, живых
  пользователей нет). В первую очередь — полнотекстовый поиск (`ОбновлениеИндексаППД`) и извлечение
  текста (подсистема ИзвлечениеТекста): именно они в топе CALL по CPU.
- **Шаги:** 1) убедиться по консоли кластера, что активных сеансов кроме фоновых нет;
  2) заблокировать регламентные задания на уровне ИБ (для целиком архивной базы — предпочтительно),
  либо снять «Включено» у конкретных РЗ.
- **Как проверить:** через сутки повторный `zabbix_perf` — контексты `*_Old` уходят из топа CALL.
- **Критерий готовности:** в топ-15 CALL-контекстов ТЖ нет строк из `*_Old`.
- **Исполнитель:** тех-лид (≈15 мин).

## Направление 2. Очередь обмена с кассами — блокировки и дисковое чтение  ⏱ главный источник по СУБД

- **Где:** РС.ОчередьПолученныхСообщенийКасс (`_InfoRg32784`); модули постановки/обработки очереди
  (`ПолучениеСообщенийСлужебный.ОбработатьСообщения`, запись набора — стр. по коду контура).
- **Шаг 2.1 — подтвердить блокировки (ТЖ на app-сервере), только на пик обмена:**

  ```xml
  <log location="C:\1C_TJ\lock" history="4">
    <event><eq property="name" value="TLOCK"/></event>
    <event><eq property="name" value="TDEADLOCK"/></event>
    <event><eq property="name" value="TTIMEOUT"/></event>
    <property name="all"/>
  </log>
  ```

  Разобрать: какие `Regions` в ожиданиях, кто держит (`WaitConnections`) — подтвердить, что ожидания
  идут по таблице очереди. Конфиг снять после расследования.
- **Шаг 2.2 — снять полный SELECT выборки очереди и план (на СУБД).** Zabbix режет текст запроса,
  полный — из `pg_stat_statements`:

  ```sql
  SELECT query, calls, round(mean_exec_time::numeric,1) AS avg_ms, rows
  FROM pg_stat_statements
  WHERE query LIKE '%\_InfoRg32784%' ESCAPE '\'
  ORDER BY total_exec_time DESC LIMIT 20;
  ```

  По самому дорогому — `EXPLAIN (ANALYZE, BUFFERS)` с реальными параметрами; существующие индексы —
  `pg_indexes`.
- **Шаг 2.3 — код (по итогам 2.1–2.2):** частота опроса и размер порции; индекс под условие выборки
  необработанных (индексировать ИЗМЕРЕНИЕ в конфигурации, не руками в PG); пакетное удаление
  обработанных — вынести из пиков.
- **Критерий готовности:** чтение `_InfoRg32784` уходит из топа СУБД по диску; TLOCK-persistence падает.
- **Исполнитель:** тех-лид + разработчик; ТЖ и снапшоты — DBA.

## Направление 3. Транзакция на 1 ч 42 мин — разблокировать vacuum

- **Где:** ERP_Main, фоновое задание, `INSERT INTO pg_temp.tt694` (большая временная таблица в одной
  транзакции).
- **Шаг 3.1 — поймать виновника в момент алерта** (снапшот на СУБД):

  ```sql
  SELECT pid, usename, application_name,
         now()-xact_start AS xact_age, wait_event_type, left(query,200) AS query
  FROM pg_stat_activity
  WHERE xact_start IS NOT NULL AND state <> 'idle'
  ORDER BY xact_age DESC;
  ```

  `application_name` у 1С несёт имя фонового задания. Альтернатива без СУБД: ТЖ `CALL`/`DBPOSTGRS`
  за окно пика с `min_duration`.
- **Шаг 3.2 — код:** порционная обработка с промежуточной фиксацией; не держать транзакцию на время
  долгого чтения.
- **Критерий готовности:** пик `pg.long.transactions` за сутки < 60 сек; `pg_bloating_tables` снижается.
- **Исполнитель:** разработчик + DBA (снапшот в момент алерта).

## Направление 4. Индексы — приоритетные кандидаты

Общий метод: полный SQL из `pg_stat_statements` → `EXPLAIN (ANALYZE, BUFFERS)` → проверить `pg_indexes` →
**индекс добавлять индексированием реквизита в конфигурации 1С, НЕ в PostgreSQL** (ручной снесёт
реструктуризация). По каждому кандидату: время × вызовы/сутки = вклад; определить реквизит по коду,
гипотезы помечать `[проверить]`.

**4.1. Партнёры — поиск по контактной информации.** ТЧ КонтактнаяИнформация ⋈ Партнёры,
**214 мс × ~5200 вызовов/сутки — №2 по времени СУБД**. Похоже на поиск по телефону/email прямым
перебором ТЧ; штатно решается регистром КонтактнаяИнформация — проверить, используется ли. `[проверить]`

**4.2. СделкиСКлиентами — фильтр по строковому реквизиту** (`_Fld45176`), 80 мс × ~5000. Определить
реквизит по метаданным, индексировать или искать по ссылке.

## Направление 5. EXCP — вскрыть источник исключений

- **Шаг:** ТЖ на app с событием EXCP за сутки → топ-5 по `Context` (тысячи исключений почти всегда —
  1–2 механизма обмена, частая причина — ретраи).
- **Критерий готовности:** названы топ-2 источника; по ним заведены задачи.
- **Исполнитель:** тех-лид.

## Направление 6. Autovacuum и bloat (после 2–3)

Для горячих таблиц снизить `autovacuum_vacuum_scale_factor`, разово `pg_repack` распухших. **Только
вместе с №2–3** — пока живёт длинная транзакция, vacuum бесполезен. Проверка: `n_dead_tup` чистятся,
`pg_bloating_tables` < 10. Исполнитель: DBA.

## Направление 7. work_mem / temp-файлы

`log_temp_files = 0` → найти запросы с сортировкой на диск → поднять `work_mem` посессионно (не
глобально — умножается на ~530 соединений). Проверка: `pg_temp_files_per_sec` снижается. Исполнитель: DBA.

## Направление 8. Своп app-сервера — подтвердить или снять

Добавить счётчики `\Paging File(_Total)\% Usage` и `\Memory\Available MBytes`: Pages/sec высок при
Available в норме и %Usage≈0 — это memory-mapped I/O, не своп. Исполнитель: сисадмин.

## Направление 9. Apdex — доказать ускорение бизнесу

Включить «Оценку производительности», 5–15 ключевых операций с целевым временем (кандидаты — из топа
CALL), выгрузка → `apdex_to_zabbix.py` → trapper-item'ы. Исполнитель: тех-лид + аналитик; планировщик — сисадмин.

## Как докажем ускорение (общий критерий закрытия)

- Повторный прогон **тем же пресетом и окном**: `python scripts/zabbix_perf.py --contour prod` —
  сравнение impact-строк: TLOCK persistence, bloat, temp-файлы, пик длинных транзакций, дисковое чтение очереди.
- `EXPLAIN ANALYZE` до/после по каждому индексному кандидату (seq scan → index scan).
- Apdex по ключевым операциям «до/после».

## Открытые заявки админу / DBA

1. **Полный текст SQL + `EXPLAIN ANALYZE`** по индексным кандидатам (Zabbix режет запрос).

3. **Apdex** — включить константу и завести операции.
4. **Paging file %Usage + Available MBytes** в мониторинг app.
5. **История текстовых item'ов ≥7 дней** — разбор инцидентов задним числом.
