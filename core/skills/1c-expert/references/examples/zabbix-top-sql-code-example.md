# ЭТАЛОН среза «код» — разбор ТОП-N тяжёлых SQL до конкретных правок (обезличенный образец)

Планка глубины для среза «код» (адресат — тех-лид/разработчик): каждый запрос из топа доведён до
модуля:строки с ЦИТАТОЙ кода, диагнозом и правкой с оценкой эффекта. Никаких инфраструктурных
направлений и заявок админам — только «что править». Реальный отчёт по боевому контуру, имена заменены.
Все цитаты кода — из реального репозитория (MCP onec-code), ничего не сочинено.

_Дата анализа: 01.07.20XX. Источник: pg.top.sql.time/disk за окно 24 ч (пресет `prod`)._

## 🔑 Главное: почти весь ТОП порождён одним горячим путём — обработкой сообщений касс из Kafka

Строки **1, 2, 3, 4** ТОП-10 — звенья одной цепочки: поток сообщений от касс → на каждое сообщение
код ищет партнёра по телефону, находит/создаёт сделку и пишет записи. Отсюда миллионы вызовов.

## Детальный разбор

### 🥇 №2 — самый дорогой (214 мс × 5419 вызовов = 1162 сек/сут). `ПартнерыСервер`

```bsl
// ТелефонСуществуетВБазе (стр. 141) — проверка дубля при регистрации клиента, горячий путь касс
| Справочник.Партнеры.КонтактнаяИнформация КАК КИ
| ГДЕ КИ.НомерТелефонаБезКодов ПОДОБНО &ТелефонПоиск   // &ТелефонПоиск = "%" + цифры + "%"
```

**Проблема:** ведущий `%` в `ПОДОБНО` не даёт использовать индекс — Seq Scan на каждый вызов.

**Что делать:** искать по нормализованному номеру с префиксным шаблоном `"цифры%"` (работает обычный
индекс); либо GIN-индекс `pg_trgm` по нормализованному полю (уровень СУБД, [проверить] с DBA).

### 🥈 №4 — 80 мс × 5285 = 422 сек/сут. `CRMСервер.ПолучитьИлиСоздатьТекущуюСделку`

```bsl
ВЫБРАТЬ ПЕРВЫЕ 2 Сделки.Ссылка ИЗ Справочник.СделкиСКлиентами КАК Сделки
ГДЕ Сделки.Партнер = &Клиент И Сделки.Подразделение = &Подразделение И НЕ Сделки.Закрыта
```

Совпадает с SQL из топа (`WHERE _Fld1466=$1 AND _Fld45176=$2`). Индекса под `Партнер` нет → скан.

**Что делать:** индексировать реквизит `Партнер` (метаданные, IDE); при необходимости — составной
с `Подразделение`.

### №3 и №1 — объём, а не единичная медленность

Очередь пишется по одной записи на сообщение (1,02 млн INSERT/сут; сам INSERT 0,48 мс — штатно);
FASTTRUNCATE — следствие пакетов с `ПОМЕСТИТЬ`, сократится сам после правки №2.
