# Анализ производительности по Zabbix → экспертный отчёт (методика)

**Разделение ролей — жёсткое, чтобы не было случайного выбора:** СКИЛЛ для любого анализа
Zabbix/производительности — этот, `1c-expert` (он определяет методику и синтез).
ИНСТРУМЕНТ — всегда onec-ops: CLI `scripts/zabbix_perf.py` или MCP `zabbix_perf_report_tool`
(это не скилл, а исполнитель съёма). Других путей (свои скрипты к API, ручной обход графиков,
другие скиллы) не изобретать.

Когда метрики контура собраны в Zabbix (ОС + СУБД + кластер 1С + события ТЖ), полный анализ
делается инструментами `onec-ops` без ручного хождения по графикам. Твоя работа делится на два
такта: **снять данные инструментом → синтезировать экспертный отчёт**. Пересказ сырого вывода
инструмента отчётом НЕ является.

## Такт 1 — снять данные (полный охват, не эконом)

**Канонический путь — пресет контура** (config/zabbix-contours.toml, образец
`zabbix-contours.example.toml`; env `ZABBIX_CONTOURS_CONFIG`):

```bash
python scripts/zabbix_perf.py --contour <имя>     # дашборды + хосты + окно 24ч + лимит ядер одним словом
```
или MCP: `zabbix_perf_report_tool(contour="<имя>")`. Пользователь сказал «проанализируй контур X» →
сперва ищи пресет `X`; он гарантирует полный охват и одинаковое «до/после». Пресета нет — собери
охват вручную по правилам ниже И ПРЕДЛОЖИ записать его пресетом (однократная работа, дальше — одно слово):

```bash
python scripts/zabbix_perf.py --dashboard <id[,id2]> --hosts "<маска-всех-хостов-контура>" --window 24
```

Правила охвата (их же обязан покрывать пресет) — нарушение любого делает отчёт «бедным»:
- **И `--dashboard` (ВСЕ дашборды контура через запятую: app-сервер + СУБД), И `--hosts`.**
  Дашборд — витрина одного сервера; хосты добирают всё остальное (СУБД, соседние узлы).
  Без `--hosts` не увидишь базу данных вообще. Сперва `dashboard.get` без фильтра — не появился ли
  новый дашборд (админы добавляют их без предупреждения).
- **Окно 24 часа** (или период инцидента). Короткое окно рушит статистику «% времени за порогом».
- **Инвентарь хостов обязателен**: `zabbix_api_tool("item.get", {"hostids": […]})` — ядра, RAM
  total/used, диски и их заполненность, версии ОС/СУБД, uptime. Без «14 vCPU / 85 GiB» цифры
  находок не интерпретируются.
- Активные триггеры (`active_problems` в отчёте) — прочитай: админские пороги часто подтверждают
  или уточняют твои находки.
- `unclassified` не игнорируй: там могут лежать метрики, которых классификатор ещё не знает —
  специфичные для контура. Упомяни их и предложи расширить правила.
- **Лимит ядер лицензии 1С** передавай всегда, если известен (`--licensed-cores "app=12"` —
  ПРОФ ограничивает rphost 12 ядрами на рабочий сервер): системный CPU «зелёный» ≠ 1С не в потолке.
- **Карты структуры хранения** (env `ONEC_STORAGE_MAPS_DIR`, `scripts/storage_map.py status`) —
  проверь покрытие БАЗ контура до съёма: без карты SQL-улики останутся `_InfoRg…` без имён;
  чего не хватает (env, токен, карта базы) — попроси, не выдумывай.

## Срез отчёта — СНАЧАЛА пойми, для кого анализ (реальный провал: отчёт не попал в адресата)

Один и тот же съём даёт РАЗНЫЕ отчёты под разных читателей. «Проанализируй производительность
контура X» без адресата — повод уточнить ОДНИМ вопросом («для кого/что нужно: план работ по контуру,
конкретные правки кода или разбор инцидента?»), а не угадывать. Спросить нельзя (автономный запуск) —
делай «план» и в конце добавь: какая область самая горячая и что даст срез «код» по ней.

| Срез | Кому | Что внутри | Эталон |
|---|---|---|---|
| **план** (дефолт) | руководитель, архитектор | весь контур: диагноз + план реализации по направлениям, заявки админу/DBA | `examples/zabbix-perf-report-example.md` |
| **код** | тех-лид, разработчик | ТОЛЬКО топ-N тяжёлых SQL (pg.top.sql.*), КАЖДЫЙ доведён до модуля:строки с цитатой кода, диагнозом и правкой; приоритеты с эффектом «мс × вызовы = сек/сутки»; без инфраструктуры и заявок | `examples/zabbix-top-sql-code-example.md` |
| **инцидент** | дежурный, тех-лид | окно инцидента: что горело именно тогда (пики, длинные транзакции, TLOCK, EXCP, держатели сессий), таймлайн → виновник | формат «план», но только затронутые направления |

Признаки в запросе: «что править / топ запросов / покажи код» → **код**; «отчёт по контуру / план
работ / для руководства» → **план**; «в HH:MM всё встало / пользователи жалуются» → **инцидент**
(окно = период инцидента, не сутки). Для среза «код» помни: топ волатилен — сегодня один лидер,
завтра другой; повторять выборочно и в моменты проблем, «до/после» сравнивать тем же охватом.

## Такт 2 — синтез (шесть шагов)

1. **Коррелируй находки в проблемы.** Несколько строк часто — одна болезнь: Pages/sec за порогом
   + триггер про своп + системный диск >80% = «пейджинг бьёт по диску ОС». Формулируй болезни,
   а не список метрик.
2. **Интерпретируй через инвентарь.** «31 GiB rphost» — это много или норм? Только относительно
   общей RAM и соседей по серверу. Каждая цифра в отчёте обязана иметь знаменатель.
3. **Места кода и SQL — главная ценность.** Секция «Указатели на места кода и SQL»
   (`evidence_texts` + `evidence_peaks`): контексты CALL из ТЖ — назови механизм 1С словами
   (обмен, полнотекстовый поиск, курсы валют…), модуль:строку оставь как есть; вызовы в
   архивных/старых базах — всегда кандидат «выключить регламентные задания», самая дешёвая победа.
   SQL-улики СУБД (топ по времени/диску, длинные транзакции) читай так: имена `_InfoRg…`/`_Reference…`
   переведи в объекты 1С по структуре хранения (или по совпадению полей с кодом — тогда пометь
   [проверить]); **пик за окно** у длинных транзакций важнее среднего — транзакция на тысячи секунд
   держит xmin (bloat) и провоцирует TLOCK у 1С; связывай SQL-улику с CALL-контекстом того же
   механизма — это и есть корреляция «ГДЕ болит (Zabbix) → ЧТО именно (код)».
4. **Дойди до кода: объект 1С → модуль:строка.** Расшифрованный объект (`ОчередьСообщений…`,
   справочник из SQL-улики) прогони по КОДУ контура — MCP `onec-code` с профилем контура
   (`onec_profile` из пресета) или grep по клонам: найди модуль записи/выборки, частоту опроса,
   размер порции, индексируемые измерения. Улика без места в коде — полдиагноза; строка кода
   превращает «таблица горячая» в «вот эта выборка в этом модуле». Не досягаемо (нет клона,
   код в данных БД) — скажи явно, чего не хватает, не выдумывай.
5. **Каждой рекомендации — исполнитель.** Железо/ОС/планировщик — администратор; регламентные
   задания и код — тех-лид/разработчик; ключевые операции Apdex — тех-лид + аналитик. Рекомендация
   без адресата не выполняется.
6. **Приоритизируй по эффекту, а не по severity.** Порядок «что делать» может отличаться от
   порядка находок: дешёвая победа с широким эффектом (убрать паразитную нагрузку) идёт раньше
   дорогой стройки (докупить железо). Явно скажи, что даст каждый пункт.

## Формат экспертного отчёта — «диагноз + план реализации», не список находок

Планка глубины — **эталон `examples/zabbix-perf-report-example.md`** (обезличенный боевой отчёт):
прочитай его перед синтезом и сдавай ТАКОЙ уровень. Скелет:

```
# <Контур> — экспертный отчёт и план реализации (<дата>)
<шапка: источник, охват (пресет/дашборды/хосты), окно, элементов>
## Инвентарь контура            — таблица хостов (роль, ОС/СУБД, ресурсы, лицензионный потолок) + список баз
## Главный вывод                — 2–4 предложения: где узкое место и что НЕ проблема (снятые гипотезы)
## Диагноз                      — таблица по убыванию вклада: болезнь · чем подтверждено · объект/место
## Расшифровка SQL-улик в объекты 1С — таблица (карта хранения; _Fld без карты полей → [проверить])
# План реализации по направлениям    — по убыванию эффекта, у КАЖДОГО направления:
##   Направление N. <название>  — Где · Шаги (готовые сниппеты: logcfg.xml — tech-journal.md,
                                  pg_stat_* — query-optimization.md) · Как проверить ·
                                  Критерий готовности · Исполнитель
## Как докажем ускорение        — повторный прогон ТЕМ ЖЕ пресетом/окном (сравнение impact-строк),
                                  EXPLAIN до/после, Apdex до/после
## Открытые заявки админу/DBA   — чего не хватило для полного анализа (полный SQL, карта полей,
                                  счётчики, история item'ов) — превращай пробелы в заявки, не в молчание
```

Сравнение «до/после» автоматизировано: первый прогон сохрани с `--out base.json`, после работ —
`--contour <имя> --baseline base.json` (или MCP `zabbix_perf_diff_tool`) — решённые/новые находки и
динамика impact считаются сами; расхождение охвата инструмент бракует (scope_mismatch).

Сниппеты ТЖ и SQL не сочиняй — бери проверенные из references этого скилла (tech-journal.md,
query-optimization.md, locks-and-deadlocks.md) и подставляй имена таблиц/окна из данных.

## Анти-паттерны (по ним отчёт бракуется)

- Пересказ сырого markdown инструмента без корреляции и знаменателей.
- Отчёт не под того читателя: тех-лид спросил «что править» — получил план по направлениям
  (или наоборот). Срез выбирается ДО синтеза, при сомнении — уточнить.
- Ручная сборка охвата при СУЩЕСТВУЮЩЕМ пресете контура (потеря дашборда/лимита ядер → бедный отчёт).
- Запуск только по дашборду / коротким окном → выводы «у СУБД всё хорошо», хотя СУБД не смотрели.
- Рекомендации без исполнителя или без ожидаемого эффекта.
- Диагноз без плана: направление без шагов съёма и критерия готовности — не сдаётся.
- SQL-улика осталась `_InfoRg…` без объекта 1С и места в коде, хотя карта и клон доступны.
- Сравнение «до/после» разным охватом или разным окном.
- Игнор `unclassified` и активных триггеров.
