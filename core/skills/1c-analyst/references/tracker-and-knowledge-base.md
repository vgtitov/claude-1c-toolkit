# Трекер задач и база знаний — достать контекст задачи и связать с кодом

Часть функционала обычно НЕ лежит в git: внешние обработки, обмены, настройки в данных базы, регламенты. Поэтому
вывод «в коде не нашёл» без проверки базы знаний — неполный. Этот слой даёт достать постановку задачи из трекера,
пройти по ссылкам в базу знаний и связать задачу + код + документацию в один анализ.

Инструмент обезличенный: CLI `scripts/atlassian.py` (только stdlib, кроссплатформенно, без внешних библиотек и без
MCP). Работает с трекером и вики в стиле Jira/Confluence — как Server/DC, так и Cloud. Свою организацию подключаешь
через переменные окружения, ничего в коде не меняя.

## Подключение (свой домен и токен)

Задать в `.env` (в корне репо или CWD; уже заданное окружение не перетирается) или в окружении:

    TRACKER (Jira-совместимый):     JIRA_URL=https://<твой-домен>       JIRA_PAT=<токен>
    БАЗА ЗНАНИЙ (Confluence-совм.):  CONFLUENCE_URL=https://<твой-домен>  CONFLUENCE_PAT=<токен>

- **Server/DC** — персональный токен идёт как `Bearer` (задай только `*_PAT`).
- **Cloud** — задай ещё `JIRA_USER` / `CONFLUENCE_USER` (email), тогда аутентификация `Basic user:api_token`.
- Чтение работает и с read-only токеном; публикация страниц (`conf publish`) требует токен с правом записи.
- Из корп-сети — обычно напрямую; извне — через VPN организации. Это org-специфика, в ядре её нет.

Секреты и реальный домен — только в `.env` (вне git) или в Team-слое локализации, НИКОГДА в публичное ядро.

## Команды

    # Трекер задач
    python scripts/atlassian.py jira issue <KEY> --comments      # карточка задачи: постановка, статус, оценка/факт, ссылки
    python scripts/atlassian.py jira search "<JQL>" --limit 20    # поиск по JQL (напр. закрытые задачи того же типа — аналоги)

    # База знаний
    python scripts/atlassian.py conf page <id|URL|точный заголовок>   # текст страницы (--storage — исходный XHTML)
    python scripts/atlassian.py conf tree <id|URL> --depth 2          # дерево дочерних страниц раздела
    python scripts/atlassian.py conf search "<текст>"                 # поиск по тексту (или "cql:<выражение>")
    python scripts/atlassian.py conf publish <file.md> --parent <id>  # md → страница (создать/обновить по заголовку)

Добавь `--json` к `jira issue`/`jira search`/`conf page`, если результат нужен машинно.

## Паттерн «связать воедино»: задача → код → база знаний

1. **Начни с задачи.** `jira issue <KEY>` — прочитай постановку, статус, оценку/факт, исполнителя. Внутри карточки
   часто есть ссылки на ТЗ/спеку в базе знаний — иди по ним.
2. **Проверь по коду.** По сущностям из постановки — `find_object` / `search` / чтение модулей по ОБОИМ слоям контура
   (конфигурация + расширение). Вопрос всегда один: то, что требует задача, уже есть в системе или нет?
3. **Сверься с базой знаний.** `conf search` / `conf page` по теме — там ТЗ смежных доработок, описания обменов,
   регламенты. Особенно важно для того, чего нет в git (обмены, внешние обработки, настройки в данных).
4. **Свяжи и сделай вывод.** Три источника — постановка, код, документация — в один ответ: что требуется, что уже
   готово, чего не хватает, где риск. Это снимает часы переписки и ловит «доработку меньше/больше, чем кажется» до
   старта работ.

## Слепые зоны — честно

- Что зашито в ДАННЫХ базы (настройки обмена, код в реквизитах, регламенты по расписанию) — грепом по git не найдёшь.
  Ищи описание в базе знаний, детали сверяй с автором/тех-лидом, не выдавай «в коде нет» за «функционала нет».
- Незагруженные слои/контуры — это «не смотрел», а не «нет». Не делай вывод об отсутствии по непросмотренному слою.

## Что обезличено, а что — в локализации

- **В ядре (обезличено):** сам CLI, эти команды, паттерн связки, имена переменных окружения.
- **В Team-слое организации (НЕ в публичное ядро):** реальный домен, токены, карта разделов базы знаний с id
  «раздел → когда смотреть», префиксы проектов трекера, правила доступа/VPN. Каркас карты — `knowledge-base-map-template.md`.
