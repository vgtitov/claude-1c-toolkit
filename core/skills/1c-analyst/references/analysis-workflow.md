# Рабочий цикл аналитика/архитектора: подготовить вход для разработки

Поток: **аналитик готовит ЧТЗ → (опц.) проходит через ваш конвейер подготовки → тех-лид дожимает технику и пишет
тех-проект → разработчик + `1c-dev` превращают артефакт в атомарные задачи и код.** Этот скилл — про левую часть:
понять задачу и отдать дальше качественный, однозначный вход. Чем точнее вход, тем чище атомы и код на выходе.

## Операционный цикл аналитика
0. **Собрать вход:** задача + материалы + сама запись в трекере (контур, база, смежное). Контур и базу определить
   сразу, искать по ОБОИМ слоям через MCP.
1. **Понять бизнес-смысл:** проблема/потребность, зачем, ожидаемый результат, инициатор.
2. **/clarify — gap-анализ (до передачи):** зафиксировать, что НЕ ясно, конкретными вопросами. Минимум: границы
   (что входит/НЕ входит); какие данные и откуда; сложные/краевые кейсы; бизнес-ошибки и запреты операций; критерии
   приёмки; влияние на интеграции/отчётность/поддержку. Нет ответа от бизнеса — блокер/открытый вопрос, не домысел.
3. **Логическая модель данных:** какие сущности/реквизиты участвуют, как меняются, какой результат. На уровне СМЫСЛА,
   не таблиц БД (физика — тех-проект).
4. **Ожидаемое поведение (ЧТЗ):** TO-BE по сценариям.
5. **Критерии приёмки:** измеримые проверки «сделано правильно» — основа тестов на этапе разработки.
6. **Классифицировать:** тип задачи, размер S/M/L/XL, workflow (Spec-Driven / Light / Context-Driven / Direct /
   Research), нужен ли архитектор. Заполнить обязательные поля трекера.
7. **Передать:** тех-лиду — на тех-проект; в трекере виден следующий шаг и ответственный.

## Правило «поведение, не реализация» — железно
ЧТЗ описывает ОЖИДАЕМОЕ БИЗНЕС-ПОВЕДЕНИЕ, а не способ реализации.
- **МОЖНО в ЧТЗ:** цель/бизнес-смысл; область применения и контуры; границы с типовой (что можно/нельзя менять по
  смыслу); логическая модель данных; жизненный цикл данных; сложные бизнес-кейсы; бизнес-ошибки и запреты операций;
  подсистемы и роли; критерии приёмки.
- **НЕЛЬЗЯ в ЧТЗ (это тех-проект тех-лида):** DTO/структуры, сигнатуры методов, имена модулей/объектов метаданных,
  серверный/клиентский контекст, директивы компиляции, стиль кода, конкретные запросы. Тянет описать «как» — сигнал
  передать тех-лиду.

## Чек-лист «хорошего ЧТЗ для dev-цикла»
- [ ] Границы чёткие (что входит и ЯВНО что не входит).
- [ ] Логическая модель данных понятна (сущности, реквизиты по смыслу, связи).
- [ ] Поведение по сценариям, включая краевые/ошибочные.
- [ ] Бизнес-ошибки и запреты операций перечислены.
- [ ] Критерии приёмки измеримы (по ним можно написать тесты).
- [ ] Контур/базы и влияние указаны.
- [ ] НЕТ деталей реализации (если есть — вынести в тех-проект).
- [ ] Открытые вопросы зафиксированы, а не «замолчаны».

## Анализ влияния
`find_object` + `read_module` + `search` «где используется» по ОБОИМ слоям контура. Учесть слепые зоны (что в
данных/настройках, не в коде — верить автору/тех-лиду) и незагруженные слои.

## Архитектор: дерево решения и роль
Подключать, если: меняются архитектурные принципы/домены/слои; влияние на несколько подсистем/контуров; новый/
нетиповой паттерн; правила не дают ответа; риск нарушения архитектуры; решение станет повторяемым правилом. При
сомнении — в пользу архитектора.
- **Архитектурное согласование** — укладывается в архитектуру, нужно подтверждение (статус/комментарий).
- **Архитектурное решение** — меняет/уточняет правила; «решение = статус принятия тех-проекта». ADR фиксирует
  вопрос/варианты/риски/выбор и долгосрочные последствия.

## Связь с разработкой
Выход этого скилла (ЧТЗ + тех-проект) = вход dev-цикла (`1c-dev` → `references/dev-workflow.md`): сверка по коду →
атомарные задачи [Разработчик/IDE]|[AI/код] → инструкции по не-коду → код → проверки/тесты → передача. Неоднозначный
ЧТЗ = грязные атомы; вкладывай точность здесь.
