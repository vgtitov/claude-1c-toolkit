# Карта базы знаний и трекера — ШАБЛОН (заполни под свою организацию)

Это каркас. Заполни СВОИМИ данными и держи в Team-слое локализации (корп-репо), НЕ в публичном ядре — здесь будут
реальный домен и id разделов. Заполненная карта позволяет AI сразу идти в нужный раздел, а не искать вслепую.

Как работать с трекером и базой знаний через CLI — `tracker-and-knowledge-base.md`. Не угадывай id и разделы:
получи их через `conf tree <корневой-id> --depth 2` и `conf search "<тема>"`, впиши подтверждённое.

## Подключение

    Трекер (Jira-совм.):     JIRA_URL = https://<...>        JIRA_PAT = <в .env, вне git>
    База знаний (Confl.-совм.): CONFLUENCE_URL = https://<...>  CONFLUENCE_PAT = <в .env, вне git>
    Cloud — добавить JIRA_USER / CONFLUENCE_USER (email). Доступ из сети: <напрямую / VPN организации>.

## База знаний — корень и ключевые разделы

Корневой раздел (space/ветка): `<название>`, id = `<...>` (проверь: `conf tree <id> --depth 2`).

Заполни таблицу под свои разделы — что где лежит и когда туда смотреть:

| Раздел | id | Когда смотреть |
|---|---|---|
| ТЗ доработок конфигурации | `<id>` | смежные доработки перед новой задачей |
| Внешние обработки / отчёты / ПФ | `<id>` | объектов НЕТ в git — прежде чем заключить «функционала нет» |
| Интеграции / обмены | `<id>` | обмены с внешними системами, форматы, потоки данных |
| Очередь сообщений / шина (если есть) | `<id>` | событийные обмены — код часто в данных, не в git |
| Инфраструктура | `<id>` | базы, серверы, хранилище, регламенты, окружения |
| Стандарты разработки | `<id>` | конвенции команды, «грабли», глобальные соглашения |
| Процесс подготовки задач | `<id>` | паспорт/шаблоны/путь задачи в разработку |

⚠ Отметь разделы, где функционал зашит в ДАННЫХ базы (обмены, настройки через UI) — там код грепом не найдёшь,
описание бери из базы знаний, детали — у автора/тех-лида.

## Трекер — что откуда брать

- Префикс(ы) проектов и что они значат: `<PROJ>` = `<доработки / объекты / ...>`.
- Тип задачи и поля оценки/факта (`timetracking`) — источник аналогов при оценке (закрытые задачи того же типа).
- Связь с базой знаний: карточка задачи обычно ссылается на ТЗ/спеку — начинай с `jira issue <KEY>`, иди по ссылкам.

## Правило

Локализация под организацию = заполнить ЭТИ данные, а не править скилл или CLI. Обновилась структура базы знаний —
поправь карту, поведение инструмента не меняется.
