# Резервное копирование и восстановление PostgreSQL под 1С

Главный принцип: **бэкап без проверенного восстановления — не бэкап**. Регламент обязан включать
регулярную тренировку restore на отдельном стенде. Хранить копии **на отдельном сервере** (не на том же
диске/хосте, что прод).

Два независимых уровня защиты, дополняющих друг друга:
- **физический** (`pg_basebackup` + архив WAL) — даёт восстановление всего кластера и **PITR** (на любой
  момент времени);
- **логический** (`pg_dump`/`pg_dumpall`) — переносимый дамп конкретной базы; удобен как «веха» и для
  частичного/межверсионного восстановления.

---

## 1. Логический бэкап — pg_dump / pg_dumpall

- **pg_dump** — одна база. Форматы: `-Fc` (custom, сжатый, для `pg_restore`, выборочное восстановление,
  параллельный restore `-j`), `-Fd` (directory, **параллельный дамп** `-j`), `-Fp` (plain SQL).
  ```bash
  pg_dump -Fc -d your_1c_database_name -f /backup/db_$(date +%F).dump
  pg_restore -d your_1c_database_name -j 4 /backup/db_2026-06-28.dump
  ```
- **pg_dumpall** — весь кластер, включая **глобальные объекты** (роли, табличные пространства):
  `pg_dumpall --globals-only` минимум нужно снимать вместе с per-database `pg_dump`, иначе при restore не
  будет ролей.
- Дамп консистентен на момент старта (снимок транзакции), но при больших базах 1С идёт долго и грузит
  систему → ночное окно, формат с `-j` для параллелизма.
- Пароль — из `~/.pgpass` (chmod 0600), не `PGPASSWORD`.

Когда логический: «веха» перед закрытием периода, миграция/перенос базы, частичное восстановление,
быстрый откат отдельной базы. Когда мало: для больших баз как единственный механизм восстановления «на
точно сейчас» — нет (нет PITR); для этого нужен физический + WAL.

---

## 2. Физический бэкап и PITR — pg_basebackup + архив WAL

- **Полный бэкап кластера** — `pg_basebackup` (или штатные средства Postgres Pro):
  ```bash
  pg_basebackup -D /backup/base_$(date +%F) -Ft -z -X stream -P
  ```
- **Архивирование WAL** — в `postgresql.conf`:
  ```
  wal_level = replica          # минимум для архива/репликации
  archive_mode = on
  archive_command = 'test ! -f /backup/wal/%f && cp %p /backup/wal/%f'
  ```
  `archive_command` копирует завершённые сегменты WAL на сервер бэкапов; проверка `test ! -f` не даёт
  перезаписать существующий. (В реальном регламенте WAL архивируются «на лету» на отдельный сервер.)
- **PITR (восстановление на момент времени):** развернуть базовый бэкап → подложить архив WAL → задать
  `restore_command` и `recovery_target_time` → создать `recovery.signal` → старт; СУБД накатит WAL до
  нужной точки.

> Профессиональные инструменты под Postgres Pro — `pg_probackup` (инкрементальные бэкапы по WAL,
> валидация, retention, проверка целостности; в полном объёме — **Postgres Pro Enterprise**) и
> `pgBackRest`. Бесплатные WAL-инструменты под Linux (без GUI): **WAL-G**, **WAL-E**, **Barman**,
> **pgBackRest**. Базовый бесплатный бинарный способ — встроенный `pg_basebackup`. Для прод-регламента 1С
> они предпочтительнее самописного `cp`-архивирования; команды — по справке своей версии, помечай
> **[проверить]**.

### pgBackRest — рекомендуемый прод-инструмент (стратегии full/diff/incr)
Даёт **собственные** типы бэкапа независимо от версии PostgreSQL (на уровне файлов/блоков), проверку
целостности и PITR:
- Типы: **`full`** (полный) / **`diff`** (разностный — от последнего полного) / **`incr`**
  (инкрементальный — от последнего любого бэкапа). Diff крупнее incr, но восстанавливается короче
  (полный + один diff), incr экономнее по объёму, но цепочка длиннее.
- Конфиг `/etc/pgbackrest/pgbackrest.conf`, репозиторий — локально / **S3** / Azure / GCS (часто на
  отдельном сервере или в объектном хранилище).
- Команды: `pgbackrest --stanza=<имя> stanza-create` (инициализация), `... backup --type=full|diff|incr`,
  `... restore` (PITR: `--type=time --target='2026-06-29 12:00:00'`), `... check`, `... verify` (проверка
  целостности репозитория), `... info` (состав/цепочки бэкапов).
- Архивирование WAL — через `archive_command = 'pgbackrest --stanza=<имя> archive-push %p'` и
  `restore_command = 'pgbackrest --stanza=<имя> archive-get %f %p'`; асинхронный режим `archive-async=y`
  не тормозит мастер на пиках WAL.
- Retention: `repo1-retention-full` / `repo1-retention-diff` (авто-удаление старых цепочек). Параллелизм
  `--process-max`. Дельта-восстановление `--delta` (досинхронизирует существующий каталог данных, не
  выкачивая всё заново). Версии/ключи — помечай **[проверить]**.

### Особенности бэкапа PostgreSQL именно под 1С (учитывать)
- **`pg_dump` НЕ годится для PITR** — это логический дамп одной базы, восстановление долгое, нет «на точно
  сейчас». PITR/инкремент даёт только WAL-архивирование (base + WAL) или потоковая репликация.
- Кроме продуктивной базы **обязательно бэкапить системный каталог СУБД** (роли, таблич. пространства,
  кластерные данные) — `pg_dumpall` или `pg_dumpall --globals-only` вместе с per-database дампами.
- Для каждой базы 1С предпочтителен **отдельный экземпляр/кластер** PG (PITR и многие инструменты работают
  на уровне всего сервера).
- Бинарный бэкап привязан к **версии СУБД и архитектуре CPU**; при восстановлении `block_size` и
  `wal_block_size` исходного и целевого сервера должны **совпадать**. Храни копию дистрибутива СУБД
  отдельно, чтобы поднять идентичную версию.
- После `pg_restore` выполнить **`ANALYZE`** (статистика в дамп не входит — без неё первые планы плохие).

## 3. Типовой регламент (пример, адаптируй сроки/окна)

| Объект | Тип | Периодичность | Время | Хранение |
|---|---|---|---|---|
| Весь кластер | полный (`pg_basebackup`) | еженедельно | напр. вс 23:00 | 2 недели |
| WAL-логи | онлайн-архив (`archive_command`) | непрерывно | — | 2 недели |
| База 1С | дамп (`pg_dump`) | ежедневно | напр. 05:00 | 2 недели |
| База 1С (веха) | дамп в отдельный каталог | еженедельно/перед закрытием периода | напр. пн 05:30 | 1 год |
| Журнал регистрации 1С | копия (rsync) `~/.1cv8` с сервера приложений | по регламенту | — | по регламенту |
| Снимки ВМ (snapshot) | средствами виртуализации | еженедельно | — | 2 недели |

Дополнительно: **глобальные объекты** (`pg_dumpall --globals-only`) — вместе с дампами; **журналы
регистрации 1С** живут в домашнем каталоге пользователя кластера 1С (`/home/usr1cv8/.1cv8`), их тоже
копировать на сервер бэкапов. «Веха» перед закрытием расчёта месяца/года — обязательна для возможности
анализа на ключевых точках.

## 4. Runbook восстановления (потеря сервера СУБД, Postgres Pro)

1. Развернуть чистую ОС.
2. Установить ту же редакцию/версию Postgres Pro (репозиторий → `yum install postgrespro-ent-1x-contrib`)
   и инициализировать кластер с тюнингом 1С:
   `pg-setup initdb --pgdata=<data> --tune=1c --locale=ru_RU.UTF-8`, затем `pg-wrapper links update`.
3. Восстановить базовый бэкап кластера: остановить сервер, очистить каталог данных, развернуть
   `tar xzf <бэкап> -C <data>`.
4. Подложить архив WAL: `restore_command = 'cp <каталог_wal>/%f %p'` в `postgresql.conf`. Для PITR — также
   `recovery_target_time`.
5. Создать файл `recovery.signal` в каталоге данных.
6. Запустить сервер (`systemctl start postgrespro-ent-1x`) — пойдёт накат WAL до целевой точки.
7. Проверить: подключение, целостность ключевых таблиц, что 1С запускается и видит данные.

Потеря сервера приложений 1С — развернуть из snapshot и вернуть журналы регистрации с сервера бэкапов.
Потеря сервера лицензирования — развернуть из snapshot.

## 5. Проверка восстановимости (обязательно, регулярно)

- По расписанию (напр. ежемесячно) разворачивать последний бэкап на **отдельном стенде** и убеждаться,
  что база поднимается и 1С с ней работает.
- Проверять валидность бэкапов средствами инструмента (`pg_probackup validate`, контроль контрольных сумм).
- Хронометрировать restore — знать реальное **RTO** (за сколько поднимется) и **RPO** (сколько данных
  теряется = глубина архива WAL/частота дампов). Сверять с требованиями бизнеса.
- Документировать runbook и держать его актуальным (версии, пути, пароли — в секрет-хранилище, не в
  runbook).
- **Тест DRP** — не реже 1 раза в 6 месяцев (реальный прогон восстановления на чистую ВМ, замер RTO/RPO).

> **Ориентир целевых показателей (реальный DRP, пример заказчика; подставь свои):** PostgreSQL с базой 1С
> — критичность 1, целевые **RPO = 5 мин**, **RTO = 2 ч** (часто стартуют от AS-IS «RPO 1 сутки»).
> 1С:ERP — RPO 60 мин / RTO 2 ч; Хранилище конфигурации 1С — RPO 60 мин / RTO 4 ч. Стратегия под RPO 5
> мин: base backup + непрерывный WAL (`pgBackRest`/`barman`), инкремент раз в 5 мин, полный раз в сутки,
> хранилище S3/NAS/NFS/DR-площадка.

### Хранение бэкапов: правило 3-2-1
- **3** копии данных, **2** разных типа хранилища, **1** копия — вне площадки (off-site).
  Напр.: облако (S3) + хранилище в РФ + офлайн-лента. Регулярно проверять валидность всех копий.
- Типовая процедура восстановления PG(1С): чистая ВМ → подключить репозиторий бэкапов → PITR до нужного
  момента → запуск 1С и проверка данных (см. §4).

## 6. Согласование с выгрузкой ИБ 1С (.dt)

- Выгрузка ИБ через Конфигуратор/`.dt` — это **прикладной** бэкап (логически согласованная копия
  информационной базы 1С), он не заменяет бэкап СУБД: `.dt` тяжёл, требует монопольного доступа (все
  пользователи выйти), долог на больших базах. Использовать для переноса/архива ИБ, не как ежедневный
  оперативный бэкап.
- Оперативный регламент строй на уровне СУБД (физический + WAL + дампы) — он быстрее, не требует
  останова и даёт PITR. `.dt` — реже, как дополнительная «вешка» или для миграции платформенными
  средствами.
- Точка согласованности: дамп/`pg_basebackup` снимают консистентное состояние СУБД на момент времени; для
  1С этого достаточно (платформа на старте проверит и при необходимости пересоздаст сессионные данные).
  Останавливать кластер 1С под `pg_dump`/`pg_basebackup` не требуется.

> Дампы/бэкапы баз 1С содержат **персональные данные** → хранить по регламенту ИБ, доступ ограничивать,
> во внешние сервисы/LLM не передавать. Пароли подключения — только в `~/.pgpass`/секрет-хранилище.

## 7. PostgreSQL 17: встроенные инкрементальные (разностные) бэкапы

**Новое в PG 17:** `pg_basebackup` умеет снимать инкрементальный бэкап — только блоки, изменённые с момента
опорного (reference) бэкапа. Для больших, но слабоменяющихся баз 1С это резко сокращает объём и время. Раньше
такую нишу закрывали только сторонние инструменты (`pg_probackup`/`pgBackRest`/WAL-G) — теперь есть штатное
средство. Доступно **только с PG 17** (и совместимых сборок) — под Postgres Pro проверяй отдельно, помечай
**[проверить]**.

### Предпосылка — WAL summarizer
- Включить `summarize_wal = on` в `postgresql.conf` (по умолчанию **`off`**, **требует рестарта** сервера).
  Фоновый процесс *WAL summarizer* пишет сводки в `pg_wal/summaries/`; по ним сервер вычисляет, какие блоки
  изменились с опорного бэкапа.
- `wal_summary_keep_time` (по умолчанию **10 дней**) — срок хранения сводок. Должен быть **больше** интервала
  между полным бэкапом и зависящими от него инкрементами, иначе инкремент не построится (сводок уже нет). При
  `summarize_wal = off` старые сводки автоматически не удаляются.
- Нельзя стартовать с `summarize_wal = on` при `wal_level = minimal` (для 1С и так минимум `replica` — не
  проблема). Summarizer можно включать и на реплике.
- Диагностика — утилита **`pg_walsummary`** (просмотр содержимого summary-файлов).

### Снятие инкремента
```bash
# опорный полный бэкап (в каталоге появится backup_manifest)
pg_basebackup -D /backup/full -Ft -z -X stream -P
# инкремент относительно манифеста ПРЕДЫДУЩЕГО бэкапа
pg_basebackup --incremental=/backup/full/backup_manifest -D /backup/incr1 -Ft -z -P
# следующий инкремент — относительно манифеста предыдущего инкремента (цепочка)
pg_basebackup --incremental=/backup/incr1/backup_manifest -D /backup/incr2 -Ft -z -P
```
- Опора — файл `backup_manifest` из каталога предыдущего бэкапа (его сервер анализирует и шлёт только
  изменённые блоки). Non-relation файлы кладутся целиком, relation-файлы — в «инкрементальном» виде.
- Инкремент возможен, только если воспроизведение начнётся с **более позднего checkpoint**, чем у опорного
  бэкапа. На **мастере** выполняется всегда (каждый бэкап триггерит checkpoint); на **реплике** может не
  сняться, если активности с прошлого бэкапа почти не было (нет нового restartpoint).
- Если нужных summary-файлов нет/неполны — `pg_basebackup` ждёт их появления; при недоступности падает.

### Восстановление — `pg_combinebackup`
Инкремент **нельзя развернуть напрямую**. Сначала собирается синтетический полный бэкап из цепочки
(полный + все инкременты, **строго от старого к новому**):
```bash
pg_combinebackup /backup/full /backup/incr1 /backup/incr2 -o /restore/synthetic
```
Дальше — обычный физический restore: подложить WAL, создать `recovery.signal`, старт; сервер накатит WAL
(PITR — см. §4). Опции: `--clone` / `--copy-file-range` (быстрое создание через reflink/оптимизированный
syscall, если ФС поддерживает), `-n/--dry-run`, `-T` (перенос табличных пространств).

### Ограничения (важно)
- **Нет встроенного учёта зависимостей цепочки.** PostgreSQL сам не отслеживает, какой полный/инкремент нужен
  для восстановления; **удаление любого звена ломает restore** → веди учёт зависимостей вручную (или доверь
  это `pgBackRest`/WAL-G, где цепочки и retention управляются инструментом).
- `pg_combinebackup` **не пересчитывает контрольные суммы страниц**: при смене статуса checksum между бэкапами
  в выходе возможны невалидные суммы → после смены checksum (`pg_checksums`) снять новый **полный** бэкап.
- Проверять отдельные бэкапы — `pg_verifybackup` (combinebackup сам их не валидирует).
- Целевой смысл — **большие базы со слабоменяющимися данными**. Для малых баз или баз с массовыми изменениями
  инкремент почти не экономит → проще полный.

### Место в регламенте 1С
Типовая ротация: **полный раз в неделю + инкременты ежедневно** (или чаще) + непрерывный WAL-архив для PITR
между точками. Не отменяет проверку восстановимости (§5): регулярно прогоняй `pg_combinebackup` + подъём на
отдельном стенде и замеряй RTO/RPO.
