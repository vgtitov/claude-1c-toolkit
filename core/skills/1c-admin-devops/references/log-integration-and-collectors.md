# Логирование 1С во внешние системы и коллекторы

Правило раздела: **логи 1С вытесняй из оперативной базы наружу**. 1С отдаёт логи двумя штатными
механизмами (журнал регистрации и технологический журнал); долго хранить и быстро искать их нужно во
внешней системе (ClickHouse / ELK / Loki), а мониторить состояние — через Zabbix или Prometheus+Grafana.
Версии платформы/библиотек и пороги — **НАСТРОЙ ПОД СВОЙ ПРОЕКТ**, версионно-зависимое помечено `[проверить]`.

Этот файл — про **вывод логов и интеграцию с коллекторами**. Что мерить на сервере и чем снимать живое
состояние кластера — в `monitoring-and-backup.md`; каталог готовых OSS-инструментов — `docs/ops-tools-catalog.md`.

## 1. Два механизма вывода логов из 1С

| Механизм | Слой | Назначение | Запись из кода | Где лежит |
| --- | --- | --- | --- | --- |
| **Журнал регистрации (ЖР)** | прикладной | бизнес-события, обмены, ошибки сеансов, действия пользователей | `ЗаписьЖурналаРегистрации()` + БСП | файлы ИБ (новый формат `.lgd` SQLite / старый `.lgf`) `[проверить]` |
| **Технологический журнал (ТЖ)** | ядро платформы | низкоуровневые события: запросы к СУБД, исключения, блокировки, загрузка компонент | настраивается, не пишется из кода | текстовые файлы в каталоге из `logcfg.xml` |

Эти механизмы дополняют друг друга: ЖР — «что сделал прикладной код», ТЖ — «что делало ядро и почему упало».
Оба используются как **транспорт** — данные забираются наружу и хранятся/анализируются в коллекторе.

## 2. Журнал регистрации (ЖР) как источник для коллектора

### Антипаттерн — НЕ хранить логи в регистрах сведений / справочниках
Запись интеграционных и отладочных логов в регистр сведений 1С — тупик: таблица растёт на порядки
(на нагруженном ритейле логи-регистры доходят до четверти объёма базы), 2/3 объёма — индексы,
запись/чтение деградируют, а удалить накопленное средствами 1С практически нереально (DELETE по огромной
таблице идёт сутками). Плюс транзакционность: логи привязываются к транзакции бизнес-операции.
**Вывод: логи в оперативной базе не держать**, писать в ЖР (вне транзакции) и выносить наружу.

### Запись структурированного лога в ЖР
Пиши в ЖР **плоский JSON** в комментарий события — так коллектор/парсер разложит его по колонкам:
```bsl
ЛогDTO = Новый Структура;
ЛогDTO.Вставить("correlation_id",   CorrelationID);   // сквозной ID для связки событий
ЛогDTO.Вставить("event_type",       "API_Успех");
ЛогDTO.Вставить("flow_id",          ПотокУИД);
ЛогDTO.Вставить("execution_time_ms", ВремяМС);        // длительность операции
ЛогDTO.Вставить("retention_days",   ДнейХранения);    // срок жизни строки → TTL на стороне СУБД
ЛогDTO.Вставить("request_body",     ТелоЗапроса);
ЛогDTO.Вставить("response_body",    ТелоОтвета);
ЗаписьЖурналаРегистрации("Интеграция.API_Успех", УровеньЖурналаРегистрации.Информация,
    , , ПолучитьОтформатированныйJSON(ЛогDTO));
```
- `correlation_id` — связывает цепочку событий одного запроса (1С → шина → ответ).
- `execution_time_ms` — для APDEX/поиска долгих операций уже на стороне коллектора.
- `retention_days` — индивидуальный срок хранения потока, отдаётся в SQL-движок (см. §3, TTL).
- Запись в ЖР идёт **вне транзакции БД** → не тормозит проведение документов.

### Чтение ЖР из кода (БСП)
- `ЖурналРегистрацииКлиент.ДобавитьСообщениеДляЖурналаРегистрации(ИмяСобытия, Уровень, Комментарий, …)`
  — простая запись с клиента/сервера.
- `ЖурналРегистрации.ЗаписатьСобытияВЖурналРегистрации(СписокЗначений)` — пакетная запись структурами.
- `ВыгрузитьЖурналРегистрации(ТаблицаЗначений, Отбор, , , КоличествоСобытий)` — платформенная выгрузка по отбору.
- БСП-обёртка `ЖурналРегистрации.ПрочитатьСобытияЖурналаРегистрации(Параметры, АдресХранилища)` — запускать
  в фоне через `ДлительныеОперации.ВыполнитьВФоне(...)` (не блокировать сеанс на больших выборках).
- Отбор — структура с ключами: `Пользователь`, `СобытиеЖурналаРегистрации`, `ДатаНачала`, `ДатаОкончания`,
  `Данные`, `Сеанс`, `Уровень` (`"Информация"/"Ошибка"/"Предупреждение"/"Примечание"`), `ИмяПриложения`.

### Характеристики записи в ЖР (порядок величин — методика)
- Запись в ЖР — **очень быстрая** (десятки–сотни тысяч событий/сек на адекватном железе); как точка
  логирования всей системы тянет реальный продакшен. Конкретные цифры замеряй на своём стенде.
- Параллельная запись **упирается примерно в 16 потоков** — сильнее параллелить смысла нет (ограничение
  сервиса журналов `rmngr`) `[проверить]`. Нагрузка делится между `rmngr` и `rphost`.
- Тип диска под ЖР слабо влияет на скорость записи (SATA-SSD/NVMe сопоставимы) — дорогой диск ради ЖР не нужен.
- Формат **`.lgd` (SQLite) деградирует при параллельной записи** (упирается уже на ~4 потоках);
  последовательная запись быстрее. Для тяжёлых баз ЖР выноси на отдельный диск/сервер (см. cluster).
- Ротация: файлы бьются по времени (новый файл каждый час) — удобно архивировать/перекладывать кусками,
  длительность хранения задаётся настройкой сервиса журналов `[проверить]`.

## 3. ELT-пайплайн: ЖР → коллектор → хранилище → просмотр из 1С

Индустриальная схема: **1С → ЖР (JSON) → агент-парсер → ClickHouse/ELK → просмотр (HTTP из 1С / Kibana)**.
1С не парсит и не чистит логи — это делает внешний агент и СУБД.

### Агент-транспорт
- **`akpaevj/OneSTools.EventLog`** — служба (C#, Win/Linux): читает файлы ЖР (`1Cv8.lgd`), парсит на лету и
  пишет в ClickHouse/Elastic. **Нагрузки на сервер 1С не создаёт** — работает рядом, по файлам.
- Альтернатива транспорта — `Logstash`/`Filebeat`/`Fluentd`: читают файлы и парсят JSON-комментарий ЖР
  фильтром (в Logstash — `json { source => "comment" }`), кладут в целевое хранилище.

### Хранилище для долгого хранения — ClickHouse
- Сильное сжатие колоночного формата (логи ужимаются в разы — порядка единиц–десятка процентов от исходного
  объёма ЖР `[проверить]`) → дёшево хранить большой ретеншн (месяцы).
- **TTL на стороне СУБД** вместо регламента очистки в 1С. Передаём `retention_days` из лога в колонку и:
  ```sql
  ALTER TABLE EventLog MODIFY TTL Date + INTERVAL RetentionDays DAY;
  ```
  ClickHouse сам удаляет просроченные строки в фоне — у 1С нет тяжёлого регламентного `DELETE`.
- Доступ без ODBC-драйверов на каждом сервере 1С — через **HTTP REST API** (порт `8123` `[проверить]`,
  `FORMAT JSON` в конце запроса).

### Хранилище для «горячего» хвоста — ELK / OpenSearch
- Kibana-дашборды, мощный полнотекст и визуализация; минус — **раздув объёма** (на ноду уходит больше,
  чем исходный ЖР `[проверить]`) → дорого для долгого хранения.
- Практика: **горячий хвост (1–2 недели) в Elastic/OpenSearch, долгий архив (месяцы) в ClickHouse**.
- `OpenSearch` — бесплатный форк ElasticSearch (Apache-2.0), без проблем с лицензией Elastic.
- `Loki` (Grafana) — лёгкая альтернатива для логов рядом с Prometheus-метриками в одной Grafana.

### Просмотр логов из 1С без ODBC
HTTP-запрос к ClickHouse прямо из кода + разбор JSON:
```bsl
Соединение = Новый HTTPСоединение(СерверCH, 8123, Логин, Пароль,,, Неопределено);
Запрос = Новый HTTPЗапрос("/");
Запрос.УстановитьТелоИзСтроки(
    "SELECT Date, Event, Severity, Comment FROM default.EventLog"
    + " WHERE Comment LIKE '%""correlation_id"": """ + CorrelationID + """%'"
    + " ORDER BY Date DESC LIMIT 50 FORMAT JSON", КодировкаТекста.UTF8);
Ответ = Соединение.ОтправитьДляОбработки(Запрос);
Если Ответ.КодСостояния = 200 Тогда
    Данные = ДесериализоватьJSON(Ответ.ПолучитьТелоКакСтроку()).data; // массив строк лога
КонецЕсли;
```
Адрес/порт/учётку ClickHouse храни в константах/env, **не в коде** (см. CLAUDE.md → Безопасность).

## 4. Технологический журнал (ТЖ) — настройка и фильтры

ТЖ включается файлом **`logcfg.xml`**. Собирай **точечно** — ТЖ генерирует много данных, широкий сбор на
проде держать нельзя (раздувает диск и тормозит). Глубже про разбор ТЖ — см. `1c-expert`.

### Расположение `logcfg.xml`
- **Windows:** каталог `conf\` платформы (`...\1cv8\conf\`).
- **Linux, серверные процессы** (`rphost`/`rmngr`): `/opt/1C/v8.3/x86_64/conf/logcfg.xml` `[проверить путь версии]`.
- **Linux, клиент под пользователем:** `~/.1cv8/1C/1cv8/conf/logcfg.xml`.
Регистр имён каталогов и права на запись в каталог логов критичны (иначе ТЖ молча не пишется).

### Структура и фильтры
```xml
<?xml version="1.0" encoding="UTF-8"?>
<config xmlns="http://v8.1c.ru/v8/tech-log">
    <log location="/var/log/1c/logs" history="24">       <!-- history = часы хранения -->
        <event><eq property="name" value="EXCP"/></event>
        <event>
            <eq  property="name" value="DBMSSQL"/>
            <like property="p:processName" value="%MyInfoBase%"/>  <!-- только нужная база -->
            <ge  property="Durationus" value="20000000"/>          <!-- только дольше 20 c -->
        </event>
        <property name="all"/>
    </log>
    <dump location="/var/log/1c/dumps" create="true" type="3"/>
</config>
```
- Условия фильтра: `eq` (равно), `ne` (не равно — `<ne property="name" value=""/>` = собирать всё),
  `ge`/`gt`/`le`/`lt` (сравнение, напр. по `Durationus` в **микросекундах**), `like` (шаблон с `%`).
- `<property name="all"/>` — писать все свойства события в лог.
- **Дампы:** `<dump create="true" type="3"/>`; на **Linux создание дампов по умолчанию выключено** и
  включается средствами ОС, не через `logcfg.xml` `[проверить]`.

### Ключевые события ТЖ
| Событие | О чём |
| --- | --- |
| `EXCP` / `EXCPCNTX` | исключения, аварийные завершения (поле `Descr` — текст; ищи стек в конце файла) |
| `DBMSSQL` / `DBPOSTGRS` / `SDBL` | запросы к СУБД / к языку запросов 1С (долгие, частые) |
| `TLOCK` / `TTIMEOUT` / `TDEADLOCK` | управляемые блокировки, таймауты, взаимоблокировки (`DeadlockConnectionIntersections`, `t:connectID` участников) |
| `CALL` / `SCALL` | серверные вызовы |
| `ADDIN` | загрузка внешних компонент (`Crashed=1` — компонента упала при загрузке) |
| `CONN`/`PROC`/`SESN`/`CLSTR`/`ADMIN`/`SYSTEM` | соединения, процессы, сеансы, кластер, администрирование |

Минимальный «дежурный» набор для прода: `EXCP+CONN+PROC+ADMIN+SESN+CLSTR` — малый объём, но отвечает
на вопрос «что произошло перед инцидентом».

### Развёртывание каталогов ТЖ на Linux
```bash
mkdir -p /var/log/1c/logs /var/log/1c/dumps
chown -R usr1cv8:grp1cv8 /var/log/1c     # фактический пользователь сервера 1С
chmod -R 775 /var/log/1c
```
Анализ — `grep`/скриптом по файлам (perl-однострочники: группировка `EXCP` по `Descr`, сумма длительности
`DBMSSQL`+`SDBL` по последней строке стека и т.п.). Для машинного разбора — MCP `tech_journal_parse` (§7).

### ТЖ как диагностика внешних компонент (Native) на Linux
Когда «компонента не подключается», платформа даёт общую ошибку — детали только в ТЖ (события `ADDIN`+`EXCP`).
Типовые `Descr`: `libssl.so.1.1: cannot open shared object file` (нет библиотеки), `undefined symbol: …`
(несовместимая версия `.so`). Дальше — `ldd /path/component.so` (ищем `=> not found`), пути через
`LD_LIBRARY_PATH` (в systemd-юните сервиса 1С), лимит дескрипторов `nofile` до 65535 в `/etc/security/limits.conf`.

## 5. Zabbix под 1С

Агент Zabbix на сервере снимает ОС-метрики; под 1С метрики заводят через расширения агента/сервера:
- **`UserParameter`** (agent) — своя метрика командой: bash/python/powershell/консольная утилита (напр. вывод `rac`).
- **External check** (server/proxy) — внешний скрипт на стороне сервера Zabbix.
- **`zabbix_sender`** — **push** нерегулярных данных (события бэкапа, аварийные сигналы): пакетом, с произвольным
  `timestamp`; сбором управляет внешний планировщик (cron), не сам Zabbix.
- **HTTP agent** — опрос API с ответом в JSON/XML/**Prometheus exposition format**; «мастер-метрика» +
  зависимые метрики через препроцессинг (можно тянуть метрики экспортёра 1С напрямую без отдельного скрипта).
- Windows-агент: модули не поддерживаются, но есть `perf_counter` и `wmi.get`.

Готовые шаблоны не изобретать: каталог **`share.zabbix.com`** и **`zabbix.com/integrations`** (раздел 1С —
`zabbix.com/integrations/1c`); шаблоны PostgreSQL — `github.com/lesovsky/zabbix-extensions` и `postgrespro/mamonsu`
(~90 метрик PG, BSD); общий подход — `node`-метрики + RAS-метрики кластера + СУБД на одной панели.

## 6. Prometheus + Grafana под 1С

Компоненты: **Prometheus server** (TSDB + scrape), **экспортёры** (snmp/node/postgres/1С), **Push Gateway**
(push коротких джобов, когда scrape невозможен), **Alert Manager** (группировка/маршрутизация алертов),
**Grafana** (дашборды + алертинг). Источники метрик для 1С: бизнес-данные/интеграции/счётчики, ОС-метрики
окружения, кластер 1С (через `rac`), сервер СУБД, технологический журнал.

OSS-флагманы (детали и лицензии — `docs/ops-tools-catalog.md`):
- **`LazarenkoA/prometheus_1C_exporter`** — метрики кластера 1С через `rac` (сессии, лицензии, память `rphost`,
  длительность вылов) + Grafana-дашборды. **Основной экспортёр кластера 1С.**
- **`ValentinKozlov/1C_PrometheusExporter`** — histogram-метрики прикладного слоя → **APDEX в Grafana**.
- **`prometheus-community/postgres_exporter`** — метрики PostgreSQL.

**APDEX** (целевой показатель отзывчивости): `APDEX = (NS + NT/2) / N`, где `T` — целевое время операции,
`NS` — число выполнений с откликом `0…T`, `NT` — с откликом `T…4T`, `N` — всего. На типовой считается штатно;
на нетиповой — добавляется замер через расширение.

Что класть в алерты (пороги — под проект): своп > 0 на сервере 1С/СУБД; latency диска СУБД выше порога;
дедлоки/мин; недоступность health-точки публикации; рестарт `rphost`; занятость лицензий у лимита.

## 7. Связка с MCP-инструментами эксплуатации

Для машинного разбора логов проекта — MCP `onec-ops` (`mcp/onec_ops_mcp.py`, см. `docs/mcp-ops-tools-design.md`):
- **`tech_journal_parse`** — ✅ реализован: парсинг файлов ТЖ → структурированный JSON (мультикорневой обход,
  фильтры по событию/длительности). Ядро-образец — `medigor/tech-log-parser` (Rust, MIT).
- **`cluster_metrics`** — планируется: снятие живых метрик кластера через `rac` (образец — `feenlace/mcp-1c`,
  у которого есть `get_event_log`/`execute_query` через HTTP-сервис ИБ — кандидат на разбор ЖР).
- Для PromQL-запросов к Prometheus — `prometheus-mcp-server` (`pab1it0`, MIT) как образец `cluster_metrics`.

Принцип скилла: сначала **снять данные** (ТЖ/ЖР/вывод `rac`/метрики экспортёра), потом делать вывод —
не диагностировать «по памяти». 1С-факты и сигнатуры сверять с реальным кодом/справкой, версионное — `[проверить]`.
