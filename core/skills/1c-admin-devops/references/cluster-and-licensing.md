# Кластер серверов 1С и лицензирование

Источник истины — официальная документация администратора клиент-серверного варианта (ИТС, раздел про кластер) и
**реальный вывод `rac`**, а не память. Версии команд/сервисов сверяй со своей платформой — состав сервисов
кластера расширяется от версии к версии (ниже список по 8.3.23, у тебя может быть больше). Имена сервисов и ключи
`rac`, в которых не уверен под свою версию, помечай **[проверить]** и сверяй `rac --help`.

## 1. Анатомия кластера: процессы

| Процесс | Что это | Сколько |
| --- | --- | --- |
| `ragent` | Агент сервера — служба ОС, запускает/держит кластер на рабочем сервере, слушает порт агента (по умолч. 1540). | один на рабочий сервер |
| `rmngr` | Менеджер кластера — несёт сервисы кластера (см. §3), порт по умолч. 1541. | ≥1 на кластер (можно «менеджер под каждый сервис») |
| `rphost` | Рабочий процесс — в нём исполняется прикладной код, держатся клиентские соединения и сеансы. Порты из диапазона (по умолч. 1560–1591). | N, регулируется кластером и РНФ |
| `rdbgs` | Сервер отладки (если включён). | по необходимости |
| `rmstt` | Спец-менеджер обслуживания сервиса распознавания речи (если в кластере есть модель распознавания). | авто |

«Центральный сервер» — рабочий сервер с флагом «Центральный сервер»; на нём живут непереносимые сервисы (состояние
кластера, конфигурация кластера). Рабочих серверов в кластере может быть несколько; их регистрируют в кластере и
им назначают РНФ.

Базовая проверка живости (Linux):
```bash
ps -ef | grep -E 'ragent|rmngr|rphost' | grep -v grep
ss -ltnp | grep -E ':1540|:1541|:15(6|7|8|9)[0-9]'   # агент, менеджер, диапазон rphost
systemctl status srv1cv8-8.3.<версия>@default        # имя юнита НАСТРОЙ ПОД СВОЙ ПРОЕКТ
```
Если процессов нет / порт не слушается — сначала сервис (`journalctl -u srv1cv8...`), см. `linux-server-ops.md`.

## 2. RAS и RAC — администрирование кластера из командной строки

- **RAS** (`ras`) — сервер администрирования: посредник между утилитой `rac` и агентом кластера. Слушает порт
  (по умолч. **1545**). Запускается как `ras cluster --port=1545 <хост-агента>:1540` или как systemd-сервис (см.
  `linux-server-ops.md`).
- **RAC** (`rac`) — консольный клиент. Обращается к RAS. Это основной инструмент скриптуемого администрирования
  (мониторинг, управление сеансами, бэкапные блокировки, РНФ).

Типовой рабочий цикл — сначала узнать идентификаторы, потом действовать по ним:
```bash
RAC="rac localhost:1545"                                  # адрес RAS

# Кластеры
$RAC cluster list                                         # → cluster : <UUID>, host, port, name ...
CL=<UUID-кластера>

# Информационные базы (нужны админ-логин/пароль кластера, если заданы)
$RAC infobase --cluster=$CL summary list \
    --cluster-user=<admin> --cluster-pwd='<пароль>'       # пароль — НЕ хардкодить, из env
IB=<UUID-инфобазы>

# Рабочие серверы и процессы
$RAC server --cluster=$CL list                            # рабочие серверы
$RAC process --cluster=$CL list                           # rphost: pid, memory-size, available-perfomance ...

# Сеансы и соединения
$RAC session --cluster=$CL list                           # кто в системе, app-id, host, hibernate, started-at
$RAC connection --cluster=$CL list

# Завершить «висящий» сеанс (например, держит блокировку перед обновлением)
$RAC session --cluster=$CL terminate --session=<UUID-сеанса>
```
**Как читать вывод:** растущий `memory-size` у `rphost` без спада — утечка/тяжёлая операция; много сеансов с одного
host и `app-id=WebClient` — веб-нагрузка; `available-perfomance` низкий → процесс перегружен (кластер должен начать
балансировать новые соединения на другой `rphost`).

Блокировка установки сеансов (перед бэкапом/обновлением — «выгнать» пользователей):
```bash
$RAC infobase --cluster=$CL update --infobase=$IB \
    --sessions-deny=on --denied-message="Идёт обновление, вход закрыт" \
    --permission-code=<код> --cluster-user=<admin> --cluster-pwd='<пароль>'
# вход разрешён только с параметром /UC <код>; снять: --sessions-deny=off
```
Это штатный способ закрыть базу для регламентных работ без остановки всего кластера.

## 3. Сервисы кластера и их характеристики

Функциональность менеджера разнесена по независимым **сервисам кластера**. У каждого — характеристики, которые
определяют, можно ли его переносить между рабочими серверами и создавать на него РНФ. Обозначения:
- **Память** / **Диск** — ресурсоёмкий по ОЗУ+CPU / по дисковой подсистеме.
- **Репликация** — есть реплика основного/резервного экземпляра (включается при уровне отказоустойчивости > 0).
- **Деление по ИБ** — для каждой ИБ свой экземпляр сервиса.
- **Перенос+** — мигрирует между серверами без потери данных; **Перенос–** — с потерей; **Не переносится** —
  только на центральном сервере (на такие РНФ создать нельзя); **Один сервер** — на любом одном рабочем сервере.

Ключевые сервисы (состав 8.3.23, у тебя сверь):
- **Лицензирования** — выдаёт **программные** лицензии. Если сервис временно недоступен, ранее выданные лицензии
  держатся ещё не дольше **20 секунд**. Для надёжной выдачи процессы `rphost`/`rmngr` должны иметь права на
  создание/чтение/изменение файла **`1cv8conn.pfl`** (см. §7). Перенос+.
- **Сеансовых данных** — хранит/кэширует сеансовую информацию (формы и т.п.) и обеспечивает получение **клиентских**
  лицензий. Диск, Репликация, Перенос+, Деление по ИБ.
- **Транзакционных блокировок** — управляемые блокировки. Память, Перенос+, Деление по ИБ.
- **Блокировок объектов** — пессимистические (не транзакционные) блокировки. Память, Репликация, Перенос+, Деление.
- **Заданий** — фоновые и регламентные задания. Репликация, Перенос+.
- **Журналов регистрации** — доступ к ЖР. Диск, Перенос–, Деление по ИБ.
- **Конфигурации кластера** / **Состояния кластера** — Не переносится (центральный сервер).
- **Обновления конфигурации базы данных** — фоновая реструктуризация; перенос требует остановки всех `rphost`,
  после чего фоновое обновление встаёт на паузу. Репликация, Перенос+, Деление по ИБ.
- **Полнотекстового поиска (в.2)** — Диск, Перенос–, Деление по ИБ.
- **Повторного использования сеансов** — пул сеансов для web/http-сервисов и OData. Перенос+.

Полный список и характеристики — официальная таблица сервисов кластера (она же в исходнике курса). Практический
вывод: «тяжёлые» по диску/памяти сервисы (полнотекстовый поиск, ЖР, журналы) разводи РНФ на отдельные рабочие
серверы, чтобы не конкурировали с прикладной нагрузкой `rphost`.

## 4. Требования назначения функциональности (РНФ)

РНФ привязывают объект требования (сервисы, соединения с ИБ, клиентские соединения определённого типа) к рабочему
серверу: **Назначать** / **Не назначать** / **Авто**. Зачем:
- развести нагрузку (например, фоновые задания — на отдельный рабочий сервер; интерактивные клиенты — на другой);
- изолировать тяжёлый сервис (полнотекстовый поиск, ЖР);
- организовать отказоустойчивость и «тёплый» резерв.

```bash
$RAC server --cluster=$CL list                            # узнать UUID серверов
$RAC rule --cluster=$CL --server=<UUID-сервера> list      # текущие РНФ сервера
$RAC rule --cluster=$CL --server=<UUID-сервера> insert \
    --position=0 --object-type=BackgroundJob --rule-type=auto \
    --application-ext=...                                  # ключи сверь по rac rule --help [проверить]
```
Порядок правил важен (сверху вниз). На сервисы с характеристикой «Не переносится» РНФ не создаются. После правки
РНФ проверь по `process list`/`session list`, что нагрузка реально легла куда задумано — **не верь на слово,
измерь**.

### Как читать и проектировать ТНФ
- **Правила читаются сверху вниз, срабатывает первое подходящее** — конкретные правила ставь выше, замыкающее
  «Любая другая работа → Не назначать» — **последним**. Без замыкающего на «лишнем» сервере осядет нагрузка по
  правилу «Авто».
- **Тип требования:** `Назначать` / `Не назначать` / `Авто`.
- **Приоритет** — число, больше = важнее/первоочерёдно. Равный высокий приоритет на двух серверах = делят нагрузку;
  меньший приоритет = «тёплый» резерв (подхватит при падении основного).
- **Объекты требований** (object-type): `Сервис лицензирования`, `Сервис сеансовых данных`, `Сервис журнала
  регистрации`, `Сервис полнотекстового поиска`, `Сервис работы с внешними источниками данных`, `Фоновое задание`
  (BackgroundJob), `Клиентское соединение`.
- **Паттерн выделенного сервера:** `нужный сервис → Назначать` + замыкающее `Любая другая → Не назначать`.
- **С 8.3.27** через ТНФ можно распределять и **сеансы пользователей** по серверам (раньше — только сервисы/задания).

**Боевая ТНФ-схема кластера из 5 узлов** (пример, имена/UUID — под свой проект; идея — выделить лицензирование и
тяжёлые сервисы):
- `srv-1c-central-1` / `srv-1c-central-2` — пользователи; страхуют друг друга по сеансовым данным / журналу
  регистрации / внешним источникам с приоритетами **100 / 90** (основной/резерв).
- `srv-1c-worker-1` — только `Фоновое задание` (всё фоновое сюда, с интерактива снято).
- `srv-1c-worker-2` — полнотекстовый поиск + клиентские соединения; фоновое **запрещено**.
- `srv-1c-lic-1` — только `Сервис лицензирования` + замыкающий запрет на остальное.
Сервер лицензирования вынесен отдельно: при его падении **не выдаются только новые лицензии**, текущие сеансы
продолжают работать. **Проверка схемы:** поочерёдно гаси узлы (`kill ragent`) и смотри, что нагрузка подхватывается
по резервным приоритетам — без теста ТНФ существует только на бумаге.

### Регламентный перезапуск рабочих процессов (с 8.3.27 — иначе)
**Критично при апгрейде на 8.3.27+:** параметр кластера «Интервал перезапуска» рабочих процессов **удалён**, старый
интервал больше не действует — перенастрой перезапуск, иначе он молча перестанет происходить.
- Было: `IClusterInfo.LifeTimeLimit` (интервал в секундах).
- Стало: cron-расписание через `rac`:
  ```bash
  $RAC cluster update --cluster=$CL --restart-schedule=...    # для кластера в целом
  $RAC server  update --cluster=$CL --server=<UUID> --restart-schedule=...   # по рабочему серверу
  ```
  Свойства объектной модели: `IClusterInfo.RestartSchedule` / `АдминистрированиеКластер.РасписаниеПерезапуска`.
  Сопутствующе переименовано `IWorkingProcessInfo.IsEnable()` → `TurnedOn`.

## 5. Отказоустойчивость и резервирование

- **Уровень отказоустойчивости** кластера (`fault tolerance level`, 0..N) — сколько резервных рабочих серверов
  держать «горячими» для непрерывности при падении. При уровне > 0 включается репликация реплицируемых сервисов.
- **Несколько центральных серверов** — для репликации сервисов конфигурации/состояния кластера.
- **Резервирование рабочих серверов** — `rphost` поднимаются на нескольких серверах; при отказе одного сеансы
  перераспределяются (поведение зависит от сервиса: Перенос+ без потери, Перенос– с потерей сеансовых данных).
- На уровне СУБД отказоустойчивость — отдельно (репликация PostgreSQL/Always On MS SQL), это не функция кластера 1С.

```bash
$RAC cluster info --cluster=$CL                           # текущие параметры, в т.ч. отказоустойчивость
$RAC cluster update --cluster=$CL --lifetime-limit=... --max-memory-size=... \
    --security-level=... --cluster-user=<admin> --cluster-pwd='<пароль>'   # параметры сверь по --help
```
**Тестируй отказ:** убей `ragent` на одном рабочем сервере и проверь, что сеансы переехали и пользователи
продолжили работу. Без теста отказоустойчивость — на бумаге.

## 6. Автономный сервер (`ibsrv` / `ibcmd`)

Лёгкая альтернатива полному кластеру: один процесс `ibsrv` обслуживает **одну ИБ** + сам поднимает веб-публикацию
(веб-клиент, тонкий через веб, интернет-сервисы, OData) — отдельный веб-сервер не нужен. `ibcmd` — оффлайн-утилита
создания/выгрузки баз и генерации конфигурации сервера. Версии ключей сверяй по `ibsrv --help` / `ibcmd --help`.

**Публикация серверной базы с удалённым доступом** (`--address` + `--port` HTTP + `--direct-regport` для прямого
подключения тонкого клиента/конфигуратора):
```bash
ibsrv.exe --db-path="C:\bases\<имя-ИБ>_1CD" --address=<хост-сервера> \
    --data="C:\bases\<имя-ИБ>" --name=<имя> --port=<порт> --direct-regport=<порт-прямого>
```
Подключение к нему:
```bash
1cv8c.exe enterprise /S <хост-сервера>:<порт-прямого>\<имя-ИБ>   # тонкий клиент
1cv8.exe  designer   /S <хост-сервера>:<порт-конф>\<имя-ИБ>      # конфигуратор (нужна лицензия)
```

**Параметры публикации** (веб-слой поднимается автоматически при старте `ibsrv`):
- `--http-base` (по умолч. `/`), `--http-port` (по умолч. **8314**), `--http-address` (по умолч. `localhost`;
  `any` = все интерфейсы, либо конкретный IPv4/IPv6).
- В одном из доков встречается опечатка `8341` — **каноничный дефолт `8314`**.

**Запуск как служба** (сам `ibsrv` в службу Windows не регистрируется — через `sc`; одна служба = одна ИБ):
```bash
ibsrv.exe --service --config="...yml" --data="..."      # запуск в режиме службы
# регистрация: sc create / sc start / sc stop / sc delete (есть готовый register-ss.bat)
ibsrv --daemon --config=... --data=...                  # Linux: фоновый демон
```
- **Приоритет параметров:** командная строка > конфиг-файл > умолчания.
- **Эксклюзивность:** с одним `--data` одновременно — только один `ibsrv` ИЛИ один `ibcmd`; с одной БД — только
  одно из {`ibsrv`, `ibcmd` offline, кластер, файловый клиент с прямым подключением}.
- **Каталог файловой ИБ по умолчанию:** Windows `%LOCALAPPDATA%\1C\1cv8\ss-data\db-data`,
  Linux `~/.1cv8/ss-data/db-data`.
- **С 8.3.24** параметры `--debug-server-password` (`ibsrv`) и `password` в секции `debug-server` **исключены и
  игнорируются** (были нерабочими) — удали их из скриптов/конфигов: ошибки не будет, но и эффекта тоже.

**`ibcmd` — создание/выгрузка баз и генерация `conf.yml`:**
```bash
ibcmd infobase create --db-path=... [--restore=...\1cv8.dt]      # создать файловую (опц. из dt)
ibcmd infobase dump   --db-path=... "out.dt"                     # выгрузить в dt

# создать серверную MSSQL-базу из dt:
ibcmd infobase create --dbms=mssqlserver --database-server=<sql-сервер> \
    --db-user=sa --database-password='<пароль>' --database-name=<имя-БД> \
    --create-database --restore=...dt

# сгенерировать конфиг сервера и работать по нему:
ibcmd server config init --db-server=... --dbms=MSSQLServer --db-name=... \
    --db-user=... --db-pwd='<пароль>' "c:\conf.yml"
ibcmd infobase dump --config="c:\conf.yml" ...
```

## 7. Лицензирование

### Типы лицензий
- **Программные** — привязка к параметрам компьютера (pin-код активации). Выдаются сервисом лицензирования
  кластера (для серверных и клиентских лицензий, размещённых на сервере).
- **Аппаратные (HASP)** — USB-ключ. На сервере читается через сетевой менеджер лицензий (HASP LM) и файл
  **`nethasp.ini`** (адрес сервера HASP). В контейнере ключ пробрасывают (см. `containers-docker-k8s.md`).
- **Комьюнити-лицензия** — бесплатная для некоммерческих/небольших проектов, ограничения по числу пользователей и
  функциональности. Активация через клиент 1С (Конфигуратор → Администрирование → Лицензирование → Получить
  лицензию) или из CLI на сервере без GUI:
  ```bash
  # ВАЖНО: reg_lic лежит ОТДЕЛЬНО от платформенных бинарников (webinst/1cv8/ras — в /opt/1cv8/x86_64/<версия>/).
  # Проверенный путь утилиты активации community-лицензии на Linux:
  /opt/1c/v8.3/x86_64/reg_lic --community \
      --user=<логин-1С> --password='<пароль>' --serial=<серийный-номер>
  # на части сборок путь версионный: /opt/1c/v8.3.<сборка>/x86_64/reg_lic — сверь `ls /opt/1c/`
  ```
- **СЛК** (Система лицензирования и защиты конфигураций) — для защищённых сторонних решений; отдельная служба
  (часто разворачивается своим контейнером, см. containers).

### Программные лицензии и версия платформы (важно при обновлении)
- Файл `*.lic` привязан к **железу + регистрационным данным**, **не к минорной версии платформы**. Лицензии
  8.0–8.3 совместимы между собой. Простое обновление `8.3.x` на том же сервере **не требует переактивации**.
- «Слёт» лицензии — только при смене **железа / ОС / гипервизора-виртуализации / диска** → восстановление по
  **резервному PIN-коду**.
- **Все бинарники одного кластера** (`ragent`/`rmngr`/`rphost`) обязаны быть **одной версии**. Сервер лицензирования
  для кластеров **разных** версий: поднять несколько `ragent` разных версий на разных портах/каталогах; при
  включении сервера в кластер указывать порт агента нужной версии; **совпадение разрядности обязательно**.

### Восстановление лицензии и дополнительный PIN-код
- Доп./резервный PIN-код и восстановление — письмом на **`lic@1c.ru`** с **зарегистрированного** email (тема
  «Запрос дополнительного пинкода»; указать рег. номер ПП, организацию, действующий PIN-код, причину). Альтернатива —
  портал `portal.1c.ru/support/license/get-pin-code`.
- Community-лицензия на Linux без GUI:
  ```bash
  /opt/1c/v8.3/x86_64/reg_lic --community \
      --user=<логин-1С> --password='<пароль>' --serial=<серийный-номер>
  ```
- Расшифровать содержимое файла лицензии — утилита **LicDataViewer** (jakondo.ru).

### Где выдаётся что
- **Серверная** лицензия — нужна самому серверу 1С (на каждый рабочий сервер кластера, по правилам лицензирования).
- **Клиентская** лицензия — на сеанс пользователя; выдаётся либо локально (ключ/файл на клиенте), либо **сервером**
  через сервис «Сеансовых данных» (серверные клиентские лицензии). Один из двух механизмов — сверь, как лицензирован
  твой проект.
- **Сервер лицензирования** — отдельная роль: один сервер раздаёт программные клиентские лицензии всему кластеру
  (тогда лицензии активируются на нём, а не на каждом клиенте).

### Файл `1cv8conn.pfl`
Сервис лицензирования пишет в `1cv8conn.pfl` (в каталоге кластера). Если у пользователя, под которым работают
`rphost`/`rmngr` (на Linux обычно `usr1cv8`), нет прав на создание/чтение/изменение этого файла — лицензии не
выдаются стабильно. Это частая причина «то есть лицензия, то нет» после миграции/смены прав.

### Диагностика «не хватает лицензий» (дерево решений)
1. **Какая лицензия?** Клиентская (пускает пользователя) или серверная (поднимается ли сам сервер)? Текст ошибки
   и где она возникает (старт сервера vs вход пользователя) различают случаи.
2. **Снять факт:** в Конфигураторе → Администрирование → Лицензирование (что реально активно/сколько занято).
   На сервере — лог `1cv8`/события сервиса лицензирования. Для HASP — доступен ли менеджер ключей с сервера.
3. **HASP:** проверь `nethasp.ini` (адрес HASP LM), сетевую доступность HASP-сервера, что ключ виден (`hasp_srm` /
   утилита Sentinel). В контейнере — проброшен ли ключ.
4. **Программные:** не «слетела» ли активация после смены железа/гипервизора/MAC (программная лицензия чувствительна
   к параметрам машины) → переактивация.
5. **Сервер лицензирования / сервис лицензирования:** жив ли, не вылетал ли, права на `1cv8conn.pfl` (§ выше).
   Помни про окно 20 секунд удержания при кратком сбое сервиса.
6. **Лимит:** реально упёрлись в число лицензий (часы пик, не закрытые сеансы) → `rac session list` покажет, кто
   держит сеансы; завершить «зависшие».

**Никогда** не вставляй pin-коды активации, серийные номера и пароли в чат/репозиторий/`Dockerfile` — это секреты.

## 8. Лицензирование 1С в CI-контейнерах (community + динамическая активация)

В CI 1С лицензируют **Community-лицензией с активацией заново при каждом старте контейнера**, а не пробросом `*.lic`.
Статическая привязка `*.lic` с фиксацией MAC/Hostname в CI **неверна**: при параллельных задачах — коллизии
(ARP-конфликты в сети Docker) и нарушение идемпотентности.

В `entrypoint.sh` динамическая активация через `ring`:
```bash
/opt/1C/v8.3/x86_64/ring license activate --community \
    --user="$DEV_LOGIN" --password="$DEV_PASSWORD" \
    --name="CI-Job-$CI_JOB_ID"            # имя из ID задачи GitLab
```
- Требуется **исходящий доступ** к `*.1c.ru:443` (`login.1c.ru`). Учётка — из **CI Secrets**, не в репо.
- `--hostname` генерировать **динамически** из ID задачи; `--mac-address` **НЕ фиксировать** (иначе ARP-конфликты
  при параллельных задачах в сети Docker).
- Митигация недоступности `1c.ru`: кэш лицензий на постоянном Docker Volume / резервный USB-ключ / bypass для hotfix.

## 9. Версионные изменения 8.3.25–8.3.27 (что перенастроить при апгрейде)

Сверяй с релиз-ноутами своей версии. Частые «молчаливые» поломки после обновления:

- **Уникальность email пользователей ИБ (с 8.3.25).** Email пользователя ИБ ≤ 128 символов и **уникален по всей ИБ**
  (даже при разном регистре). После обновления выполни **ТИИБ «Проверка логической целостности ИБ»** (категория
  «Прочее») в режиме исправления — создаёт индекс таблицы пользователей и ускоряет проверку уникальности email.
- **Хеширование паролей (с 8.3.26).** Выбор алгоритма: SHA-1 / SHA-256 / SHA-512 / PBKDF2-HMAC-SHA256. При выборе
  алгоритма **≠ SHA-1** нужны клиенты и серверы **8.3.26+**.
- **Запрет локального распознавания речи.** С 8.3.27 ключ `/DisableLocalSpeechToText` и флажок в диалоге добавления
  файловой ИБ **удалены**. Теперь: клиент-сервер — флажок «Запретить локальное распознавание речи» в диалоге
  создания ИБ (Конфигуратор / Управление серверами); файловая — параметр `DisableLocalSpeechToText` в `conf.cfg`
  каталога ИБ. С 8.3.25 свойство `КаталогМоделейРаспознаванияРечи` и `rac --speech-to-text-model-directory`
  устарели → альтернативные каталоги задают для сервиса `SpeechRecognitionService` (`rac service-setting` / консоль).
- **КОРП-функции кластера** (требуют лицензию **КОРП**): OpenMetrics-мониторинг (8.3.25); завершение сеансов по
  бездействию (8.3.26); альтернативные каталоги для сервисов кластера и `DataAcceleratorService` (8.3.25);
  управление местом хранения БД Дата-акселератора (память / память+диск, 8.3.27); произвольные индексы СУБД через
  метаданные (8.3.26).
- **ТЖ:** `history="0"` в `logcfg.xml` **отключает автоудаление** журнала — тогда чистка только вручную.
