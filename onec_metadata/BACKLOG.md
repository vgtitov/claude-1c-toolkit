# onec_metadata — задачи на дальнейшие улучшения

Приоритезированный бэклог. Инвариант для КАЖДОЙ задачи (не повторяется в пунктах):
TDD (сначала падающий тест), byte-perfect минимальный дифф, оба формата
(Конфигуратор + EDT) где применимо, обезличенные фикстуры из реальной структуры,
применение только в ТЕСТ-базу + round-trip, adversarial-ревью перед merge.

## P1 — Редактирование существующего (главный пробел)

Сейчас операции почти все на ДОБАВЛЕНИЕ; правка существующего — только текст
запроса СКД и значение права. Это закрывает основной разрыв.

- [x] **`set-property`** ✅ — изменить скалярное свойство существующего объекта или
  реквизита по адресу (объект → [реквизит] → свойство → значение): синоним,
  комментарий, флаг (`Indexing`/`Индексирование`, `FullTextSearch`, `PasswordMode`,
  `Use`/`Использование` и т.п.). Приёмка: значение меняется, соседние узлы не
  тронуты, идемпотентно, оба формата.
- [x] **`edit-type`** ✅ — сменить/расширить тип существующего реквизита: длина/
  точность строки/числа (квалификаторы), составной тип (несколько `Type`),
  ссылка. Приёмка: корректная перестройка блока `Type` + квалификаторов.
- [ ] **`rename`** — переименование объекта/реквизита (рискованно: ломает ссылки).
  Только с явной проверкой отсутствия внешних ссылок или пометкой «небезопасно».

## P3 — Создание объектов с нуля

- [x] **Создание реквизита без образца — через ДОНОР** ✅ (`add_child --donor`):
  образец берётся из однотипного объекта конфигурации + регенерация id (версия-
  корректно). Донор — корневой уровень, элементы с порождаемым типом (ТЧ) не
  поддержаны. Пустой ChildObjects (совсем нет детей) — ещё отдельная операция.

## P4 — Формы (крупный блок)

- [ ] **Правка управляемых форм** — добавить реквизит формы, элемент (поле/группу/

  требует отдельной спеки и, вероятно, поэтапной поддержки видов элементов.

## P5 — Надёжность и инфраструктура

- [ ] **Рантайм-энфорс реестра `catalog.SUPPORTED`** — сейчас реестр только
  документирует; сделать его предохранителем в CLI/ops (операция вне реестра для
  типа объекта → отказ), как заявлено в `CLAUDE.md`.
- [ ] **Version pinning — режим `assert`** — опционально блокировать применение
  при расхождении версии/совместимости (сейчас только «предупреждать»); читать
  версию платформы с ХОСТА apply, а не только из пути `ONEC_1CV8_BIN`.
- [x] **Старые фикстуры → реализм** ✅ — `task.xml`, `subsystem.xml` переведены на
  BOM+CRLF (как реальные объектные файлы Конфигуратора); CRLF-путь этих операций
  теперь покрыт.
- [ ] **Семантические проверки типа** — валидировать формат ссылки типа
  (`type_ref`) перед подстановкой (не пропускать заведомо битый тип).
- [ ] **Golden-эталоны результата** — для ключевых операций сравнивать не только
  round-trip-стабильность, но и с записанным эталонным выводом (страховка от
  «вакуумных» byte-perfect тестов).

## Аудит 2026-07 (по итогам сравнения с отраслью)

Полное заключение — `docs/architecture-review-2026-07.md`. Вердикт: фундамент
подхода верен и совпадает с best practice (file-first, платформа=истина,
round-trip, deepcopy-образец); byte-perfect — наше преимущество. Что перенять:

- [x] **Scope-граница** — зафиксировать в CLAUDE.md/README: инструмент делает
  только БЕЗОПАСНЫЕ аддитивные и скалярные правки; деструктив с каскадом ссылок
  (rename/delete/move/смена вида) — вне scope, отдаётся EDT-плагину или человеку.
  (Сделано аудитом.)
- [x] **Канонизация артефакта платформой** ✅ (зафиксировано): в git идёт `after`
  из round-trip — платформенно-перевыгруженное дерево (пересчитанный ConfigDumpInfo/
  порядок), не пред-apply правка. Docstring `dump_extension` + аудит. Наш поток уже
  выгружает канонический результат; upload исключает ConfigDumpInfo (полная загрузка).
- [ ] **Решение по `catalog.SUPPORTED`** — либо рантайм-энфорс (гейт по виду
  объекта), либо честно оставить как «проверенное покрытие» (tested coverage) и

  scope-граница). Выбрано второе + добавлен потребляемый API `is_supported`.
- [ ] **Оценить `1cedtcli`** (новый headless CLI EDT, вместо `ring`) как
