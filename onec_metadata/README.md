# onec_metadata — программные структурные правки метаданных 1С

Модуль toolkit: программно **создавать и редактировать объекты метаданных,
формы и СКД** 1С:Предприятие 8.3 в форматах Конфигуратора (`DumpConfigToFiles`
XML) и EDT (`.mdo`/`.dcs`), с **byte-perfect round-trip** (минимальный дифф,
«как человек в IDE»), загрузкой в **тестовую** ИБ и верификацией round-trip.

**ОС-независимо.** Сам движок (разбор/правка XML) — чистый Python (`pathlib`,
`lxml`), работает одинаково на **macOS / Linux / Windows**. Apply-слой вызывает
`ssh`/`scp`/`tar` (есть на всех трёх; Windows 10+ — из коробки) и общается с
СЕРВЕРОМ 1С (всегда Windows) — путь к платформе там задаётся строкой Windows,
это единственная платформо-специфичная часть, и она про удалённый сервер, а не
про машину, где запускается toolkit.

Снимает прежний жёсткий IDE-гейт («структуру создаёт только человек») — но
строго под гарантией: платформа приняла загрузку в тест-базе + повторная
выгрузка дала diff ровно целевых файлов.

## Что умеет (Фаза 1, формат Конфигуратора)
- `template add-column` — колонка в макет табличного документа (печатная форма):
  копия соседней ячейки (наследует формат), сдвиг индексов, гард на пересечение
  объединённых ячеек, минимальный дифф.
- `scd add-field` / `get-query` / `set-query` — поле и запрос набора данных СКД.
- `attribute add` — реквизит объекта метаданных (deepcopy-образец + новый uuid).
- `detect` — автоопределение формата дерева (Конфигуратор / EDT).
- apply-слой: `upload_tree` → `load_extension` (+`UpdateDBCfg -Extension`) →
  `dump_extension` → `fetch_tree` → `roundtrip_verify`.

## Как устроено
```
onec_metadata/
  formats/configurator.py  # byte-perfect чтение/запись XML (BOM/CRLF/табы)
  formats/detect.py        # формат дерева исходников
  ops/                     # чистые операции над XML (lxml)
  apply/runner.py          # SSH-раннер + designer_cmd
  apply/dumpload.py        # load/dump расширения, roundtrip_verify
  catalog.py               # реестр покрытия (что чем поддержано)
bin/1c-meta                # headless CLI
skills/1c-metadata/        # skill-обёртка для агента
```

## CLI
```
1c-meta detect <root>
1c-meta template add-column <Макет.xml> --after "<Якорь>" --header "<Новый>" --parameter <Пар>
1c-meta scd add-field <Схема.xml> --dataset <Набор> --field X --data-path X --title "..."
1c-meta scd get-query|set-query <Схема.xml> --dataset <Набор> [--from-file q.sql]
1c-meta attr add <Объект.xml> --name X --type xs:string --synonym "..."
```
Exit 2 = ошибка предусловия (якорь не найден / дубль) — файлы не изменены.

## Настройки (обезличено, без хардкода под организацию)
- SSH-хост сервера 1С — аргумент `Runner(host)`; алиас в `~/.ssh/config`.
- Путь к платформе — env `ONEC_1CV8_BIN` (иначе типовой путь 8.3.x).
- Рабочий каталог на сервере — env `ONEC_REMOTE_WORKDIR` (по умолчанию `D:\src`).
- Пароль ИБ — не в коде/логах: передаётся аргументом, логирование маскирует `/P`.
- Конкретные настройки организации (хост, базы, версия платформы, имена
  расширений) держите в ОТДЕЛЬНОМ приватном репозитории настроек, не здесь.

## Гарантия корректности
- **Unit** (без платформы): фикстуры синтетические обезличенные; каждая операция
  проверяется на края (дубль→ошибка, нет якоря→ошибка) + byte-perfect после op.
- **Integration-гейт** (нужна платформа): загрузка в тест-базу → повторная
  выгрузка → `roundtrip_verify` == ровно целевые файлы. Прод не трогаем.

## Дорожная карта / статус
- ✅ **Фаза 2** — формат-слой сохраняет стиль файла (BOM/LF/CRLF); операции
  `attribute add` и СКД работают и для **EDT** (`.mdo`, `.dcs`).
- ✅ **Фаза 3** — generic `child add`: Attribute/Dimension/Resource/EnumValue/
  TabularSection/Command в обоих форматах; реквизиты табличной части — через
  `--parent <ИмяТЧ>`.
- ✅ **Фаза 4** — внешние `.epf`/`.erf` (`apply.external`: dump/build) и
  бинарные расширения `.cfe` (dump/load + UpdateDBCfg).
- ✅ **Фаза 5** — регистрация внешних обработок в БСП «Дополнительные отчёты и
  обработки» (`apply.bsp.register_external` + BSL-шаблон регистратора в
  `onec_metadata/templates/`).
- Далее — расширение видов объектов и семантические проверки по мере задач.

## Известные ограничения Фазы 1
- Формат — только Конфигуратор (EDT — Фаза 2).
- Реальная ссылка `&#13;` во входном XML не byte-perfect (на типовых выгрузках
  не встречается; см. docstring `configurator.py`).
- XPath в операциях чувствителен к одинарной кавычке в произвольном тексте
  заголовка (1С-идентификаторы кавычек не содержат).
- `template add-column`: merge-гард best-effort (проверяет стартовую строку
  объединения); подтверждается числом merge-узлов + round-trip.
