# Как внести вклад в bsl-ai-toolkit

Рады и людям, и AI-агентам. Проект — про качественную, производительную и безопасную AI-разработку на 1С:Предприятие,
поэтому вклад ценим конкретный: правила, скиллы, проверки, движок метаданных, адаптеры под новых агентов.

## Быстрый старт
```bash
git clone https://github.com/vgtitov/bsl-ai-toolkit && cd bsl-ai-toolkit
sh onboard/onboard.sh                 # хуки + скиллы + профиль + самотест (см. флаги внутри)
uv run --with mcp --with pytest pytest tests/ -q   # тесты должны быть зелёными
sh build.sh                           # пересобрать конфиги агентов из core/
```

## Архитектура (важно понять до PR)
- **Правь `core/`, не сгенерированное.** Единый источник — `core/` (`AGENTS.md`, `skills/`, `mcp/servers.json`). Файлы в
  корне (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`, `.mcp.json`, `.claude/`) **генерируются** командой `sh build.sh` — руками их не редактируют.
- Три оси переносимости: **AGENTS.md** (правила) + **SKILL.md** (навыки) + **MCP** (инструменты). Новый агент —
  тонкий адаптер в `adapters/<agent>/`, не форк ядра.
- Claude Code — эталонная реализация; при добавлении фич сначала работает в ней, потом раскладывается адаптерами.
- Движок правки метаданных — `onec_metadata/` (byte-perfect, round-trip verify); CLI `bin/1c-meta`; скилл `1c-metadata`.

## Что приветствуется
- Новые/уточнённые **правила** по реальному коду 1С (со ссылкой на справку платформы/БСП или исходники — не «по памяти»).
- **Проверки** на анти-паттерны 1С (запрос в цикле, поломка СКД, дрейф свойств…) — с фикстурой в `tests/` и тестом.
- **Скиллы** и references (разработка, анализ, DBA, эксплуатация, метаданные) — формат SKILL.md.
- Расширение покрытия **движка метаданных** (`onec_metadata/`) с round-trip-тестом.
- **Адаптеры** под новых AI-агентов; улучшения `onboard`/`build.sh`, документации, CI.

## Правила для PR
1. **Обезличенность.** Никаких внутренних данных организаций, реальных хостов/баз/доменов, ПДн, секретов. Проект намеренно обезличен.
2. **Проверяемость, не «по памяти».** Факты о 1С — со ссылкой на справку/исходники. Непроверенное помечай `[проверить]`.
3. **Тесты зелёные.** Новая проверка = фикстура + тест. `pytest tests/ -q` → всё зелёное (и твои новые тесты).
4. **Хуки проходят.** commit-msg (чистые сообщения) + pre-commit (`bsl_guard.py` — запрос в цикле). Тест-фикстуры с
   намеренным анти-паттерном коммить с `git commit --no-verify` и пометкой в сообщении.
5. **`sh build.sh` без ошибок**, сгенерированные файлы соответствуют `core/`.
6. Стиль: без жаргона, «заголовок = вывод», честные оговорки. По-русски (проект русскоязычный).

## Процесс
Форк → ветка `feat/...` или `fix/...` → изменения в `core/` (или движке/скриптах) → `sh build.sh` → тесты → PR по шаблону.
Мелкие правки доков — можно напрямую PR. Крупные изменения архитектуры — сначала issue на обсуждение.

## Для AI-агентов
Репозиторий **agent-friendly**: читай `AGENTS.md` (правила) и `docs/AI_INSTALL_GUIDE.md`. Можешь открывать issue и PR.
Соблюдай те же правила: правь `core/`, добавляй тест на каждую проверку, не коммить секреты/ПДн, факты — по коду/справке 1С.
