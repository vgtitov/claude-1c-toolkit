# Публикация ИБ на веб-сервере (Apache / nginx / IIS)

Публикация даёт: **web-клиент** (1С в браузере), **тонкий клиент через http(s)**, **web-сервисы (SOAP)**,
**http-сервисы**, стандартный интерфейс **OData**. Источник истины при разборе ошибки — **HTTP-код + лог
веб-сервера + лог сервера 1С**, не догадки. Версии путей/модулей — НАСТРОЙ ПОД СВОЙ ПРОЕКТ.

## 1. Что создаёт публикация

Публикация = каталог на веб-сервере с файлом **`default.vrd`** + подключение модуля 1С к веб-серверу:
- **Linux/Apache** — модуль `wsap24.so` (имя зависит от версии Apache/платформы) из каталога платформы, директива в
  конфиге Apache, указывающая на каталог публикации.
- **Windows/Apache** — `wsap24.dll`.
- **Windows/IIS** — `wsisapi.dll` как ISAPI-расширение + обработчик, плюс права пула приложения (имя модуля сверь
  по своей версии платформы — **[проверить]**).
- **nginx** — нативного модуля 1С нет; nginx ставят **обратным прокси** перед Apache/IIS (или перед сервером
  публикаций), сам по себе nginx ИБ 1С не публикует. [проверить под свою схему]

Публикуют двумя путями:
- из **Конфигуратора** (Администрирование → Публикация на веб-сервере) — интерактивно;
- утилитой **`webinst`** — скриптуемо (для CI и серверов без GUI):
```bash
# Linux/Apache, пример — пути/версии ПОД СВОЙ ПРОЕКТ
/opt/1cv8/x86_64/<версия>/webinst -publish -apache24 \
    -wsdir <алиас> -dir /var/www/<алиас> \
    -connstr "Srvr=<сервер>;Ref=<имя-ИБ>;" \
    -confPath /etc/apache2/apache2.conf
systemctl reload apache2
```
После публикации каталог `-dir` должен быть **читаем пользователем веб-сервера** (`www-data`/`apache`), а каталог
временных файлов 1С — записываемым. Это источник половины ошибок 403/500.

## 2. `default.vrd` — что в нём

`default.vrd` — XML, описывает, какая ИБ публикуется и какие точки включены. Ключевое (имена атрибутов сверь по
своей версии, **[проверить]**):
- `base` / строка соединения `Srvr=...;Ref=...` (клиент-серверная) или `File=...` (файловая);
- `enable` web-клиента;
- `<ws>` — список **web-сервисов** (SOAP) с алиасами и `enable`;
- `<httpServices>` — список **http-сервисов**;
- `<standardOdata enable="true"/>` — включение **OData** (по умолчанию выключен; включается отдельно);
- `<point>` для каждого http-сервиса.

Правка `default.vrd` — частая операция: включить/выключить OData, добавить http-сервис, сменить строку соединения.
После правки — **reload веб-сервера** и проверка, что 1С перечитал файл.

**Безопасность:** в `default.vrd` может оказаться строка соединения. Логины/пароли к ИБ в `default.vrd` —
нежелательно; если нужны, файл должен быть закрыт от чтения извне веб-сервером. Не коммить `default.vrd` с
секретами в репозиторий.

## 3. Включение точек: web-сервисы, http-сервисы, OData

- **Web-сервис (SOAP):** объект «Web-сервис» в конфигурации + публикация (галка/`<ws>` в vrd). Проверка — запросить
  WSDL: `GET http://<хост>/<алиас>/ws/<имя-сервиса>?wsdl` → должен вернуться XML WSDL, не 404/500.
- **http-сервис:** объект «HTTP-сервис», корневой URL в свойствах. Проверка — `GET http://<хост>/<алиас>/hs/<корень>/...`.
- **OData (стандартный интерфейс):** включить в `default.vrd` (`standardOdata`) **и** в самой ИБ настроить состав
  публикуемых объектов (через обработку/настройку стандартного интерфейса OData). Проверка списка сущностей:
  `GET http://<хост>/<алиас>/odata/standard.odata/` → метаданные/список. Требует прав и, как правило, явной настройки
  доступа, иначе 401/пусто.

## 4. Диагностика ошибок публикации (дерево решений)

Сначала — **код ответа и лог**, потом гипотеза:
```bash
curl -i http://<хост>/<алиас>/                 # web-клиент: что реально отдаёт сервер
tail -f /var/log/apache2/error.log             # Apache (Linux); IIS — Failed Request Tracing / Event Viewer
journalctl -u srv1cv8-... -f                   # сервер 1С: ошибки соединения с ИБ
```

| Симптом | Частая причина | Куда смотреть / фикс |
| --- | --- | --- |
| **404** на корень публикации | алиас/каталог не подхвачен, директива в конфиге веб-сервера отсутствует или reload не сделан | конфиг Apache/IIS, перезапуск; `webinst` отработал? |
| **403 Forbidden** | прав нет: каталог публикации/временные файлы не читаются/не пишутся пользователем веб-сервера | `chown -R www-data:www-data <dir>`, права на temp 1С |
| **500** при входе | модуль 1С не загрузился / несовместимость версии модуля и платформы / не та разрядность | `error.log`: «cannot load module …»; версия `wsap*.so`/`.dll` = версии платформы |
| **500** «Файл не найден» / нет соединения с ИБ | неверная строка соединения в `default.vrd`, сервер 1С/ИБ недоступны | проверь `Srvr=;Ref=;`, `rac infobase list`, сеть до сервера |
| web-клиент грузится, **web-сервис 404** | сервис не опубликован (нет `<ws>`/галки) или неверный алиас | `?wsdl`-проверка, правка vrd |
| **OData 401/пусто** | OData не включён в vrd или не настроен состав/права | `standardOdata enable`, настройка доступа OData в ИБ |
| кодировка/«кракозябры», кириллица в путях | локаль веб-сервера, кодировка | локаль `ru_RU.UTF-8`, латиница в алиасах |
| **Apache: конфликт модулей / не стартует** | две директивы LoadModule, неверный путь к `.so`, MPM | проверить `apachectl -t`, один модуль 1С |
| первый запрос долгий, далее ок | холодный старт `rphost`/пула повторного использования сеансов | норм; если стабильно — capacity/РНФ |

**IIS-специфика (по ИТС, Руководство администратора гл. 8.10.1 «Особенности настройки IIS»):**
- **Разрядность.** Пул приложения и расширение веб-сервера должны быть одной разрядности с платформой. Для 32‑битного
  `wsisapi.dll` в 64‑битном IIS — в пуле приложения включить **«Разрешены 32‑разрядные приложения» (Enable 32‑Bit
  Applications = True)**. Лучше брать расширение, совпадающее по разрядности с платформой.
- **Права пула.** Идентификатор пула приложения должен иметь права на каталог публикации и каталог temp 1С;
  «Анонимная»/«Windows»-аутентификация — под нужный режим входа в ИБ.
- **Длина URL.** IIS по умолчанию режет длинные запросы — у 1С URL бывают длинными (500/404.15). Поднять
  `requestLimits maxUrl`/`maxQueryString` (web.config) и при необходимости реестровые `UrlSegmentMaxLength`/
  `MaxFieldLength` HTTP.sys. [проверить пороги под свой кейс]
- **Представление ошибок.** Включить подробные ошибки/Failed Request Tracing, иначе IIS отдаёт обобщённый 500 без
  причины.

**Apache однопоточность (Windows):** классический prefork/однопоточный режим под Windows ограничивает параллелизм —
для нагрузки берут многопоточный MPM или nginx-прокс перед несколькими бэкендами. [проверить под свою версию]

## 5. HTTPS и эксплуатация

- HTTPS навешивается на веб-сервере (сертификат Let's Encrypt/корпоративный), 1С работает поверх. Тонкий клиент по
  https — строка подключения `https://...`.
- **Автообновление тонкого клиента** через публикацию — в `default.vrd`/настройках публикации, чтобы клиенты
  тянули новую версию с сервера.
- При смене версии платформы сервера — пере-публикуй (модуль `wsap*` должен соответствовать новой версии), иначе
  500 на загрузке модуля.
- Health-check для мониторинга: `curl -fsS http://<хост>/<алиас>/hs/<health-сервис>` или WSDL-пинг web-сервиса —
  заводи в Zabbix/Prometheus (см. `monitoring-and-backup.md`).

## Канон (ИТС)

Структура и точные диалоги/ключи — официальный источник: **ИТС → Платформа 1С:Предприятие 8.3.x. Документация →
Руководство администратора → Глава 8 «Настройка веб-серверов для работы с 1С:Предприятием»**. Там по каждому виду
(клиентское приложение, Web-сервисы, OData, HTTP-сервисы) отдельно расписаны **IIS / Apache 2.0 / 2.2 / 2.4** под
**Windows и Linux**, утилита **`webinst`** (8.3.3) и «Особенности настройки веб-серверов» (8.10): IIS (пул, длина
URL, представление ошибок, HTTPS, встраивание веб-клиента), Apache (алгоритм поиска веб-сервера, встраивание) и
**Reverse Proxy**. При расхождении версий — сверяйся с ИТС под свою версию платформы, эти страницы первичны.
