# Обновление конфигураций/расширений и CI/CD для 1С

Две связанные темы: как **безопасно обновить** прикладное решение и как **автоматизировать доставку** (CI/CD).
Общее правило: перед любым обновлением прода — свежий проверенный бэкап и план отката; ничего деструктивного без
окна и блокировки сеансов (см. `cluster-and-licensing.md`, §2). Версии инструментов — НАСТРОЙ ПОД СВОЙ ПРОЕКТ.

## 1. Режим поддержки и «замок»

Типовая конфигурация стоит **на поддержке поставщика**. Объекты в дереве помечены замком:
- замок **закрыт** — объект не редактируется (полная поддержка, обновляется автоматически);
- замок открыт «с сохранением поддержки» — можно менять, но при обновлении поставщик объединяет;
- поддержка снята — связь с поставщиком потеряна, обновлять придётся вручную (сравнением-объединением).

**Правило:** меняй типовую как можно меньше; функционал — в **расширение** (`cfe`), типовую держи близко к
поставке, чтобы обновления шли «дважды-кликом». Снятие поддержки — крайняя мера, осознанно.

## 2. Сравнение, объединение, форматы cf/cfu/cfe

- **`cf`** — полный файл конфигурации (выгрузка/поставка целиком).
- **`cfu`** — файл **обновления** (дельта от версии к версии), применяется к конфигурации на поддержке.
- **`cfe`** — файл **расширения** конфигурации.
- **`dt`** — выгрузка всей информационной базы (данные + конфигурация), это бэкап/перенос ИБ, не обновление кода.

Обновление типовой:
1. Получить `cfu`/`cf` нужной версии (релизы поставщика).
2. Конфигуратор → Поддержка → Обновить конфигурацию → выбрать `cfu`/`cf`.
3. Платформа строит таблицу **сравнения-объединения**: что меняет поставщик, что трогали вы; разрешить конфликты
   (правило для дважды-изменённых объектов: «брать из основной конфигурации» / «из файла» — осознанно по каждому).
4. Принять, потом **Обновить конфигурацию базы данных** (реструктуризация).

Скриптуемо (для CI) — пакетный режим конфигуратора:
```bash
# примеры ключей — СВЕРЬ по своей версии платформы [проверить]
1cv8 DESIGNER /IBConnectionString "..." /UpdateCfg <файл>.cfu /UpdateDBCfg
1cv8 DESIGNER /IBConnectionString "..." /LoadCfg <файл>.cf /UpdateDBCfg
1cv8 DESIGNER /IBConnectionString "..." /LoadExtensionCfg <расш>.cfe -Extension <ИмяРасширения>
1cv8 DESIGNER /IBConnectionString "..." /DumpIB <файл>.dt              # бэкап ИБ перед операцией
```
Сборка расширения из исходников и загрузка — также через DESIGNER/`ibcmd`/EDT CLI (см. ниже).

## 3. Динамическое / фоновое обновление и их риски

- **Динамическое обновление** (без выхода пользователей, «обновить динамически») — применяет изменения
  конфигурации на лету. **Когда НЕЛЬЗЯ:** при изменении структуры БД (новые/изменённые реквизиты, таблицы) —
  требуется реструктуризация, динамически её не сделать; при сложных изменениях возможны ошибки у работающих
  сеансов. Безопасно — только для совместимых изменений кода. Не используй как штатный способ выкатки структуры.
- **Фоновое обновление** (сервис «Обновления конфигурации базы данных», см. cluster §3) — реструктуризация идёт в
  фоне без полной остановки. Перенос этого сервиса между серверами требует остановки всех `rphost` и ставит
  фоновое обновление на паузу.
- **Реструктуризация** (изменение структуры таблиц) — тяжёлая операция, на больших базах долгая и блокирующая;
  планируй в окно, делай бэкап, оценивай время на копии.

**Регламент обновления прода (чек-лист):**
1. Бэкап ИБ (dt и/или СУБД) + проверка восстановимости (см. `monitoring-and-backup.md`).
2. Закрыть вход сеансов (`rac ... sessions-deny=on`), дождаться выхода / завершить висящие.
3. Применить обновление (cfu/cf/cfe) → обновить конфигурацию БД (реструктуризация).
4. Прогнать smoke-тесты (см. §6) и ключевые сценарии.
5. Открыть вход. Если провал — откат из бэкапа по плану.

## 4. EDT vs Хранилище конфигурации

Два контура командной разработки:
- **Хранилище конфигурации** — классика: общая база-хранилище, захват/помещение объектов, история. Работает с
  Конфигуратором. Версионирование снаружи (git) — через `gitsync` (см. §5).
- **1C:EDT** — IDE на Eclipse, исходники в **текстовом формате в git** нативно, нормальные ветки/merge/PR,
  CLI (`1cedtcli`/EDT CLI) для скриптов в пайплайне. Современный путь для git-CI/CD.

Выбор: новый проект / нужен git-flow и PR-ревью → EDT. Большой легаси на хранилище → хранилище + gitsync как мост
в git для анализа/CI. Не смешивай источники истины бесконтрольно.

## 5. CI/CD-стек 1С: OneScript, gitsync, GitLab/Jenkins

**OneScript (`oscript`)** — движок 1С-подобного скриптового языка для автоматизации (вне платформы). Менеджер
пакетов `opm`. На нём построены ключевые инструменты экосистемы:
- **`gitsync`** — выгружает историю **хранилища конфигурации** в git (каждая версия хранилища → коммит). Мост
  «хранилище → git» для статического анализа и истории.

  **Практика gitsync:**
  - `gitsync init` запускать **в КОРНЕ репозитория** (напр. `C:\Repo`), а каталог исходников задавать параметром
    **`WORKDIR`** (`src\cf`) — НЕ запускать внутри `src\cf`. `init` создаёт файлы `VERSION` и `AUTHORS`.
  - **`AUTHORS`** (в корне репо) сопоставляет авторов: строки вида
    `ИмяПользователяВХранилище=Имя Пользователя <моя@почта.ру>`. После заполнения **обязательно закоммитить
    `AUTHORS`** в репозиторий — иначе авторство коммитов не сопоставится. Проверь почты пользователей хранилища.
  - **`VERSION`** (`src/cf/VERSION`) хранит номер последней обработанной версии хранилища. После разрушения и
    пересоздания файлового хранилища достаточно **сбросить номер в `VERSION`** (+ сверить `AUTHORS`), чтобы
    восстановить синхронизацию.
  - Расширения — опция `--ext`, исходники в `src/cfe/<Имя>`. Плагины ставятся через `gitsync plugins init`.
  - **Лицензия:** сам `gitsync` лицензию **не требует**, но 1С требует её при работе с авто-создаваемой временной
    файловой базой. В свойствах базы **отключить** получение аппаратной лицензии и **разрешить** программные. Для
    серверной базы — опция `-C/--ibconnection/--ib-connection` или env `$GITSYNC_IB_CONNECTION` при `gitsync sync`.

  **Предупреждение про файловое/сетевое хранилище без сервера хранилища:** на Windows из-за проблемы назначения
  прав при коммитах от **разных пользователей** такое хранилище **разрушается**. С полноценным **сервером хранилища
  1С** проблемы нет — для командной работы используй его.
- **`vanessa-runner` (`vrunner`)** — обвязка для типовых операций: создать/обновить ИБ, загрузить/выгрузить
  конфигурацию и расширения, прогнать тесты, синтаксический контроль. Настройки в `vrunner.json` / `xUnitParams.json`.

**`ibcmd` — сборка/обновление конфигурации headless, без конфигуратора и БЕЗ серверной лицензии 1С.** Ключевой
инструмент CI на Linux/Docker: `ibcmd` собирает и разбирает конфигурацию без графики, лицензия сервера ему не нужна.
```bash
# сборка CF из исходников:
ibcmd infobase create --data=build_temp
ibcmd infobase config import --data=build_temp ./src/cf
ibcmd infobase config save   --data=build_temp ./release.cf
# синтаксконтроль:
ibcmd infobase config check  --data=build_temp
# деплой на серверную базу:
ibcmd infobase config update --db-server=... --db-name=... --db-user=... --db-pwd='<пароль>' \
    --load=./release.cf --force
ibcmd infobase config apply
```
`ras` = сервис администрирования, `rac` = клиент команд кластера, `ring` = лицензии/активация.

**UI-тесты в контейнере headless** — тонкий клиент `1cv8c` + **Xvfb** (виртуальный фреймбуфер). В образ тестера
(Vanessa) добавляют `xvfb`, `dbus`, шрифты. Этап динамтеста: поднять Xvfb → активировать community-лицензию →
создать пустую файловую базу → загрузить `.cf` → smoke-тесты (Vanessa Automation, отчёт Allure).

**Пайплайн (GitLab CI / Jenkins) — типовые стадии:**
1. **prepare** — поднять/обновить ИБ из исходников (vrunner / DESIGNER `/LoadCfg`).
2. **syntax-check** — синтаксический контроль конфигурации/расширения.
3. **static-analysis** — SonarQube + 1С:АПК (см. §7), BSL Language Server.
4. **smoke** — дымовые тесты Vanessa (запуск, открытие ключевых форм).
5. **acceptance** — приёмочные сценарии Vanessa Automation (по фичам).
6. **build/deploy** — собрать `cf`/`cfe`, доставить в тестовую/прод ИБ (в окно).

Агенты CI с платформой 1С удобно держать **в Docker** (образы сервера/клиента/gitsync/vanessa из onec-docker —
см. `containers-docker-k8s.md`). На Jenkins лицензии для агентов раздают через `nethasp.ini` (HASP) — файл кладут
как docker config и монтируют в `…/conf/nethasp.ini`.

Пример (Jenkins, объявление стадии — концептуально):
```groovy
stage('Static analysis') {
  steps { sh 'vrunner syntax-check --ibconnection "/F./build/ib"' 
          sh 'sonar-scanner' }              // sonar-project.properties в корне проекта
}
```

**GitLab Flow для 1С (релизная дисциплина):**
- `main` защищена правами GitLab: **прямой push запрещён**, вход только через **Merge Request**. Рабочие ветки —
  `feature/<задача>-<описание>`, короткоживущие.
- Собранный артефакт `.cf`/`.cfe` версионируется в **GitLab Package Registry** (неизменяемый) и проходит
  **DEV → QA → Prod без пересборки** — один бинарник на все среды. **Rollback** = повторный Deploy предыдущей
  версии артефакта из Registry.
- Стадии `.gitlab-ci.yml`: Syntax Check → Build → SonarQube (параллельно) → Unit → Deploy DEV → Smoke UI →
  Deploy TEST; на стадии Sonar `allow_failure: false`.

**Quality Gates и KPI пайплайна:**
- Статанализ: **SonarQube + BSL Language Server** (ошибки, дублирование, code smells, тех. долг), веб-интерфейс на
  порту **9000**; дополнительно — 1С:АПК. Quality Gate блокирует merge при падении покрытия **нового** кода
  (принцип храповика).
- KPI пайплайна (ЗАДАЙ ПОРОГИ ПОД СВОЙ ПРОЕКТ): время прогона (держать в разумных минутах, иначе им перестают
  пользоваться), покрытие кода (растить от стартового к целевому, ratchet на **новом** коде), доля зелёных сборок
  `main`. Конкретные цифры — твои; смысл в тренде «не хуже предыдущего», а не в абсолютном числе.
- Постоянные сервисы CI: **SonarQube + PostgreSQL** (под Sonar).

## 6. Тесты: Vanessa

- **Дымовые (smoke)** — быстрые проверки «не упало»: открываются формы, запускается конфигурация. Дёшевы, гоняй на
  каждый коммит.
- **Приёмочные (BDD)** — сценарии на русском (Гherkin-подобный синтаксис) в Vanessa Automation, проверяют бизнес-
  поведение. Медленнее, гоняй на ветку/релиз.
- Отчёты — Allure (наглядный результат прогона) для CI. Покрытие кода — через образ EDT/coverage (`COVERAGE41C`).

## 7. Статический анализ: SonarQube + 1С:АПК

- **SonarQube** + плагин для BSL: анализ кода конфигурации/расширения, метрики, code smells, gate.
- **1С:АПК** (Автоматизированная проверка конфигураций) — проверки на соответствие стандартам разработки 1С;
  результат выгружается и подхватывается SonarQube (правила появляются с тегом **`ACC`** и ответственным).

Каркас настройки (по методичке курса):
1. Установить **SonarScanner**, в `sonar-project.properties` задать проект и **версию** (`sonar.projectVersion`).
2. Выгрузить конфигурацию в файлы (исходники) в каталог проекта, закоммитить.
3. Запустить `sonar-scanner` → результат в SonarQube.
4. (Опционально, АПК) Сделать базу с **чистой конфигурацией на поддержке** (через файл поставки `1cv8.cf`),
   добавить в АПК конфигурацию-поставку и конфигурацию проекта, прогнать проверки, выгрузить отчёт
   (`tools/acc-export/acc.bat`), указать путь к выгрузке в `sonar-project.properties`, перезапустить анализ —
   в SonarQube появятся ошибки с тегом `ACC`.

Помни: **не все ошибки ловит SonarQube**, не все — АПК; статический анализ дополняет ревью и тесты, не заменяет.

## 8. Частые провалы и контрприёмы
- Обновили динамически изменение структуры → ошибки у пользователей. **Контр:** структуру — только через
  реструктуризацию в окно.
- Обновление типовой «взяли всё из файла» поверх своих доработок → потеряли правки. **Контр:** разрешать конфликты
  объект-за-объектом; доработки выносить в расширение.
- CI «зелёный», прод упал → smoke без приёмочных. **Контр:** ключевые сценарии в acceptance.
- Хранилище и EDT правят одно и то же без синхронизации → расхождение. **Контр:** один источник истины кода.
- Секреты (пароль ИБ, токен GitLab, pin лицензии) в `sonar-project.properties`/`vrunner.json`/`Jenkinsfile`.
  **Контр:** только переменные CI/секреты, плейсхолдеры в репозитории.
