# Linux-сервер 1С: установка, сервисы, эксплуатация

Сервер 1С под Linux — современный дефолт (PostgreSQL/PostgresPro рядом). Источник истины — официальная инструкция
по установке под твой дистрибутив и **вывод системных утилит** (`systemctl`, `journalctl`, логи `1cv8`), не память.
Версии платформы/дистрибутива/каталоги — НАСТРОЙ ПОД СВОЙ ПРОЕКТ (пути ниже — примеры).

## 1. Установка платформы

- **deb (Ubuntu/Debian):** пакеты `1c-enterprise-<версия>-server`, `-common`, `-client`, веб-расширение
  `-ws`/`-wsap` для публикации. Ставить из каталога дистрибутива:
  ```bash
  sudo dpkg -i 1c-enterprise*<версия>*.deb || sudo apt-get -f install
  ```
- **rpm (RHEL/CentOS/Astra-rpm-варианты):** аналогичные пакеты через `dnf/yum`/`rpm -i`.
- **Зависимости/окружение** (частые): `locales`, `procps` (без него ловятся странные сбои инициализации соседних
  сервисов в минимальных образах — см. ниже про локали), шрифты для отчётов/печатных форм, библиотеки печати.
- Дистрибутивы качаются с `releases.1c.ru`/`users.v8.1c.ru` под учётной записью с правами — не клади учётку в репо.

После установки создаётся пользователь сервиса (обычно **`usr1cv8`** : группа `grp1cv8`); под ним работают
`ragent`/`rmngr`/`rphost`. Каталог кластера — `~usr1cv8/.1cv8/1C/1cv8/` (точный путь зависит от версии/пакета).

## 2. systemd-юниты

Сервер 1С обычно ставится как systemd-сервис (имя зависит от версии/пакета — **СВЕРЬ под себя**):
```bash
systemctl list-units 'srv1cv8*'                          # найти реальное имя юнита
systemctl status  srv1cv8-<версия>@default               # статус
systemctl enable  srv1cv8-<версия>@default               # автозапуск
systemctl restart srv1cv8-<версия>@default               # перезапуск кластера
journalctl -u srv1cv8-<версия>@default -f                # живой лог сервиса
```

**RAS как отдельный сервис** (нужен для `rac` и мониторинга через RAS) — обычно отдельный юнит или его заводят
вручную. Если юнита нет, типовой `ras.service` (НАСТРОЙ путь/версию):
```ini
# /etc/systemd/system/ras-1c.service
[Unit]
Description=1C RAS
After=srv1cv8-<версия>@default.service
[Service]
User=usr1cv8
ExecStart=/opt/1cv8/x86_64/<версия>/ras cluster --port=1545 localhost:1540
Restart=on-failure
[Install]
WantedBy=multi-user.target
```
```bash
sudo systemctl daemon-reload && sudo systemctl enable --now ras-1c
rac localhost:1545 cluster list                          # проверка, что RAS отвечает
```
Права на исполняемые файлы платформы — если ставил руками, проверь `chmod`/владельца (`ragent`/`ras` должны быть
исполняемы пользователем сервиса).

## 3. Лимиты ОС и тюнинг ядра

Сервер 1С под нагрузкой упирается в лимиты по умолчанию. Для пользователя `usr1cv8` (и СУБД) подними:
```bash
# /etc/security/limits.d/99-1c.conf  — НАСТРОЙ значения под нагрузку
usr1cv8  soft  nofile  65536
usr1cv8  hard  nofile  65536
usr1cv8  soft  nproc   65536
usr1cv8  hard  nproc   65536
```
Для systemd-сервиса лимиты задаются ещё и в юните (`LimitNOFILE=`, `LimitNPROC=`) — `limits.conf` сам по себе на
службу не действует. Ядро (`/etc/sysctl.d/*.conf`, под СУБД/сеть):
```
vm.swappiness = 10                   # реже свопить (своп на сервере 1С/СУБД = тормоза)
vm.overcommit_memory = 2             # под PostgreSQL (сверь рекомендации своей СУБД) [проверить]
```
**transparent_hugepage:** для PostgreSQL обычно рекомендуют отключить (`THP=never`) — проверь рекомендацию своей
версии PostgresPro/PostgreSQL. Применить sysctl: `sudo sysctl --system`. После — **измерь**, а не верь, что
«стало быстрее» (см. `monitoring-and-backup.md`).

## 4. Локали, шрифты

1С под Linux требует корректную локаль, иначе — ошибки кодировки, кривая кириллица в отчётах:
```bash
sudo locale-gen ru_RU.UTF-8
sudo update-locale LANG=ru_RU.UTF-8
locale                                # проверить, что LANG/LC_* = ru_RU.UTF-8
```
Для службы локаль задают и в окружении юнита (`Environment=LANG=ru_RU.UTF-8`). Шрифты (для печатных форм/макетов с
картинками) — поставить пакеты шрифтов, иначе пустые/битые печатные формы.

## 5. Диагностика «сервер не стартует / база недоступна»

Дерево:
1. `systemctl status srv1cv8-…` → запущен ли вообще; `journalctl -u … -e` → последняя ошибка.
2. Порты: `ss -ltnp | grep -E ':1540|:1541|:1545'` → слушает ли агент/менеджер/RAS.
3. Логи платформы `1cv8` (каталог логов кластера) + ТЖ (`EXCP`) — конкретная причина (нет лицензии, нет связи с
   СУБД, нет прав на каталог, нет локали).
4. Связь с СУБД: с сервера 1С `psql -h <хост> -U <польз> -d <база>` (PostgreSQL) — доступна ли СУБД и проходят ли
   креды (`.pgpass`, не пароль в команде).
5. Права/владелец каталога кластера и `1cv8conn.pfl` (лицензии, см. `cluster-and-licensing.md`, §6) — `usr1cv8`
   должен владеть/иметь rw.
6. Память/OOM: `dmesg | grep -i oom`, `journalctl -k | grep -i kill` — не убивает ли ядро `rphost` по нехватке ОЗУ.

Не делай вывод «платформа сломана» без п.1–6 — в 9 случаях из 10 это права, локаль, лицензия, СУБД или OOM.

## 6. S3 как объектное хранилище

S3 (MinIO/совместимое/облако) используют для: хранения **томов** (файлы, картинки) вне ИБ (разгрузка СУБД), бэкапов
(см. `monitoring-and-backup.md`, правило 3-2-1, копия вне площадки), стендов Fresh (Fresh подразумевает S3 —
`s3.<домен>` в hosts, см. `containers-docker-k8s.md`). Подходы: внешняя компонента/механизм работы 1С с S3
(например, решения вида `S3_SSL`), либо S3 как backend для файлового хранилища/тома инфраструктуры. Ключи доступа
S3 (`access key`/`secret key`) — **только в env/секретах**, никогда в коде/`default.vrd`/репозитории. [проверить
под свою схему интеграции S3 с 1С]

## 7. Контрприёмы
- Лимиты подняли в `limits.conf`, а служба их «не видит». **Контр:** дублировать `LimitNOFILE/LimitNPROC` в
  systemd-юните.
- Кривая кириллица в печатных формах. **Контр:** `ru_RU.UTF-8` в системе И в окружении юнита + шрифты.
- `rac` не подключается. **Контр:** RAS не запущен/не тот порт — проверь `ras`-сервис и `ss -ltnp | grep 1545`.
- «То работает, то нет лицензия» после переустановки. **Контр:** права `usr1cv8` на `1cv8conn.pfl` и каталог
  кластера.
- Пароль СУБД в командах/скриптах. **Контр:** `.pgpass` (chmod 600), секреты CI.
