# Мониторинг и резервное копирование сервера 1С

Правило раздела: **мерить, а не гадать**. «Тормозит» — это симптом; найди узкое место по данным, потом чини.
И отдельно: бэкап без проверки восстановления — это не бэкап. Версии/пороги — НАСТРОЙ ПОД СВОЙ ПРОЕКТ.

## 1. Что мерить (и где это в стеке)

| Слой | Ключевые метрики | Чем снять |
| --- | --- | --- |
| Сервер 1С (кластер) | число сеансов/соединений, память каждого `rphost`, `available-perfomance`, число `rphost`, рестарты процессов, длительные вызовы | `rac session/connection/process list` (через RAS), ТЖ |
| Прикладной слой | долгие операции, APDEX ключевых операций, ошибки `EXCP`, ожидания на блокировках, дедлоки | технологический журнал, ЦКК/ЦУП, журнал регистрации |
| СУБД (PostgreSQL) | долгие запросы, блокировки, кэш-хит, bloat, autovacuum, активные соединения, размер БД/таблиц | `pg_stat_activity`, `pg_stat_user_tables`, `pg_locks`, лог медленных запросов, `pgwatch`/exporter |
| СУБД (MS SQL) | долгие запросы, ожидания, блокировки, план кэш | DMV (`sys.dm_exec_requests`, `dm_os_wait_stats`), Profiler/XE |
| ОС сервера | CPU, RAM (своп!), диск (IOPS, latency, очередь), сеть | `node_exporter`/Zabbix-агент, `vmstat`, `iostat -x`, `sar` |
| Веб-публикация | HTTP-коды, время ответа, доступность health-точки | curl-проба http-сервиса/WSDL, лог веб-сервера |

Снятие живого состояния кластера для мониторинга:
```bash
RAC="rac localhost:1545"; CL=<UUID>
$RAC session --cluster=$CL list | grep -c session-id      # сколько сеансов
$RAC process --cluster=$CL list                            # memory-size, available-perfomance каждого rphost
```
Рост `memory-size` `rphost` без спада → утечка/тяжёлая операция → ищи по ТЖ, что грузит этот процесс.

## 2. Технологический журнал (ТЖ)

`logcfg.xml` (в каталоге conf платформы) включает сбор событий. Собирай **точечно** (ТЖ генерирует много данных —
не держи широкий сбор постоянно на проде):
- `DBPOSTGRS`/`DBMSSQL`/`SDBL` — обращения к СУБД (долгие/частые запросы);
- `CALL` — серверные вызовы;
- `EXCP` — исключения (в т.ч. падения `rphost` с `mdmp`);
- `TLOCK`/`TTIMEOUT`/`TDEADLOCK` — управляемые блокировки, таймауты, дедлоки;
- `QERR`, `EXCP` — ошибки запросов.

Анализ — `grep`/скриптом по файлам ТЖ. Пример (из практики): сводка падений `rphost` по причине:
```bash
cat rphost*/*.log | grep ',EXCP,.*mdmp' | awk -F',' '{print $NF}' | sort | uniq -c | sort -rn
```
Долгие запросы — по `Durationus`/`Sql=` в событиях DB; топ по длительности → кандидаты на оптимизацию (план запроса,
индексы; см. `1c-dev/references/performance.md` и стандарты запросов).

## 3. Журнал регистрации (ЖР)

Прикладной журнал событий ИБ. Версии хранения — НАСТРОЙ (старый формат / новый `lgd`/`lgf`). Из него: ошибки сеансов,
действия пользователей, системные события. Доступ — сервис «Журналов регистрации» (Диск, Деление по ИБ — см.
cluster). Для тяжёлых баз ЖР выноси РНФ на отдельный сервер, чтобы не конкурировал с прикладной нагрузкой.

## 4. ЦКК / ЦУП

- **ЦКК** (Центр контроля качества) — регламентный мониторинг качества/производительности по сценариям и
  показателям (APDEX, доступность, регламентные проверки). Заводят сценарии регламентного мониторинга.
- **ЦУП** (Центр управления производительностью) — глубокий разбор: долгие запросы, блокировки, дедлоки, показатели
  производительности и анализа. Инструмент эксперта по техвопросам.
Это «1С-родные» инструменты прода поверх ТЖ; ставятся и подключаются к продуктивной системе.

## 5. Zabbix / Prometheus + Grafana

- **Zabbix:** агент на сервере (ОС-метрики) + готовые **шаблоны для 1С** (часто работают через **RAS** — поднимают
  службу RAS, импортируют шаблон, заводят узел сети, элементы данных и триггеры; дашборды по серверу 1С, по СУБД,
  по веб-серверу; уведомления, например в Telegram).
- **Prometheus + Grafana:** exporter'ы — `node_exporter` (ОС), `postgres_exporter` (PostgreSQL), кастомный
  exporter метрик 1С (через RAS/ЖР/ТЖ). Grafana — дашборды и алерты. Подход «Observability для 1С»: ОС + СУБД +
  кластер + публикация на одной панели.
Что класть в алерты (пороги — под проект): своп > 0 на сервере 1С/СУБД; latency диска СУБД выше порога; число дедлоков
в минуту; недоступность health-точки публикации; рестарт `rphost`; занятость лицензий близко к лимиту.

## 6. Резервное копирование: dt vs СУБД

| Способ | Что это | Плюсы | Минусы / когда |
| --- | --- | --- | --- |
| **Выгрузка `dt`** | `1cv8 DESIGNER /DumpIB файл.dt` — выгрузка всей ИБ средствами платформы | переносимо между СУБД/версиями, простая загрузка `/RestoreIB` | МЕДЛЕННО и блокирующе на больших базах; требует монопольного/согласованного состояния; не для частого прод-бэкапа |
| **Бэкап на уровне СУБД** | `pg_dump`/`pg_basebackup`/`pgBackRest` (PostgreSQL), `BACKUP DATABASE` (MS SQL) | быстро, штатный регламент, **PITR** (point-in-time), не блокирует работу | привязка к СУБД/версии; восстановление в ту же СУБД |

**Практика:** регулярный прод-бэкап — на уровне СУБД (быстро, PITR). `dt` — для разовых переносов, архива на смену
версии платформы, или малых баз. Согласованность: бэкап СУБД даёт согласованный снимок; `dt` требует, чтобы ИБ была
в согласованном состоянии (без активных длинных транзакций — лучше после блокировки сеансов).

PostgreSQL — типовой регламент:
```bash
pg_dump -Fc -d <база> -f /backup/<база>_$(date +%F).dump          # логический, переносимый
# или физический + WAL-архив для PITR:
pgbackrest --stanza=<имя> backup                                   # full/diff/incr по расписанию
```
MS SQL — Full + Differential + Log backups в полной модели восстановления (для PITR). Обслуживание (реиндексация,
обновление статистик, `vacuumdb`/autovacuum для PostgreSQL) — отдельный регламент, влияет на производительность.

## 7. Проверка восстановимости (главное)

Бэкап, который ни разу не разворачивали, — лотерея. Регламент проверки:
1. По расписанию (например, еженедельно) **восстанавливать** последний бэкап на отдельный экземпляр СУБД/стенд.
2. Подключить восстановленную ИБ, проверить запуск и ключевые данные (контрольные суммы/счётчики записей,
   открытие ключевых форм/отчётов — те же smoke-сценарии, что в CI).
3. Зафиксировать **RPO** (сколько данных теряем — глубина точки восстановления) и **RTO** (за сколько разворачиваем)
   и проверять, что укладываемся.
4. Хранить копии по правилу 3-2-1 (3 копии, 2 носителя, 1 вне площадки; S3/удалённое хранилище — см.
   `linux-server-ops.md`). Ротация и контроль, что бэкапы реально создаются (алерт на «бэкап не сделан»).

## 8. Контрприёмы
- Мониторинг есть, но без алертов на своп/диск/дедлоки/лицензии → узнаёшь о проблеме от пользователей. **Контр:**
  алерты с порогами.
- Только `dt`-бэкап большой базы по ночам — не успевает/блокирует. **Контр:** СУБД-бэкап + PITR.
- Бэкапы делаются, восстановление не проверяется. **Контр:** регламент тест-восстановления (§7).
- ТЖ включён широко на проде постоянно → диск забит, нагрузка. **Контр:** точечный сбор по событиям и по времени.
- «Тормозит» лечат добавлением `rphost`/памяти вслепую. **Контр:** сначала ТЖ/счётчики/план запроса — корневая
  причина, потом ресурс.
