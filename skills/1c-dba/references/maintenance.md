# Регламентное обслуживание PostgreSQL под 1С

«Чтобы база работала быстро — в ней нужен порядок» (А. Дорошкевич). PostgreSQL версионный: старые версии
строк (от UPDATE/DELETE и каждого перепроведения в 1С) остаются «мёртвыми» строками, индексы пухнут,
статистика устаревает. Регламент = убирать мусор (VACUUM), обновлять статистику (ANALYZE), не дать
переполниться счётчику транзакций (FREEZE), перестраивать распухшие индексы (REINDEX/pg_repack).

**Главное:** регламентные скрипты **дополняют**, а не заменяют правильно настроенный **автовакуум**
(см. `postgres-tuning.md` §4). Сначала агрессивный автовакуум, потом — ночной/периодический скрипт для
планового и специфического (распухшие индексы, заморозка крупной базы).

**Чего НЕ делать в регламенте 1С (типовые ошибки):**
- НЕ отключать автовакуум.
- НЕ ставить `VACUUM FULL` в регламент — он **блокирует** таблицу целиком (для 1С = реструктуризация,
  работать нельзя). Онлайн-замена — `pg_repack`.
- НЕ вызывать `pg_stat_reset()` в регламенте — сбрасывает накопленную статистику обращений
  (`pg_stat_*`/`pg_stat_statements`), теряются данные для диагностики; на сам план запросов не влияет, но
  лишает тебя метрик. (Распространённая ошибка в старых самописных скриптах.)
- НЕ включать FREEZE в **ежедневный** прогон — он I/O-ёмкий (см. §3).

## 1. VACUUM и ANALYZE

- **VACUUM** помечает мёртвые версии строк как свободное место (НЕ возвращает место ОС, в отличие от
  `VACUUM FULL`; место переиспользуется внутри таблицы), обновляет карту видимости/заморозки.
- **ANALYZE** обновляет статистику распределения данных → планировщик строит корректные планы. Под 1С
  критично из-за работы в последнем периоде огромной базы.
- `vacuumdb --analyze` делает оба за один проход. Параллелит по таблицам: `--jobs=N`.

Ночной плановый прогон (что и делает скрипт, см. §5) — параллельный VACUUM+ANALYZE по базе 1С:
```bash
vacuumdb --analyze --verbose --jobs=4 --dbname=your_1c_database_name
```
`--jobs` обычно = числу ядер (или из конфига `VACUUMDB_JOBS`). На сервере с многими равноценными базами
бывает оправдан `-a` (все базы), но для целевого обслуживания 1С лучше указывать конкретную `--dbname`.

Диагностика мёртвых строк и давности обслуживания таблиц:
```sql
SELECT schemaname, relname, n_live_tup, n_dead_tup,
       round(100*n_dead_tup / NULLIF(n_live_tup+n_dead_tup,0),1) AS dead_pct,
       last_autovacuum, last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC LIMIT 20;
```
Большой `dead_pct` при свежем `last_autovacuum` = автовакуум не успевает → усилить настройки/добавить
плановый VACUUM этой таблицы.

## 2. Распухание (bloat): индексы и таблицы, REINDEX CONCURRENTLY, pg_repack

В 1С только **btree**-индексы, и они пухнут особенно: записей в индексах базы 1С примерно вдвое больше,
чем данных (на 13 ТБ базе нередко 8–9 ТБ индексов и ~4 ТБ данных). Обычный `REINDEX` блокирует, обычный
`VACUUM` распухание индекса не убирает.

- **REINDEX INDEX CONCURRENTLY `<idx>`** (PostgreSQL 12+) — перестраивает индекс без блокировки записи.
  Эквивалент ручной связки `CREATE INDEX CONCURRENTLY new; DROP INDEX CONCURRENTLY old; ALTER INDEX new
  RENAME TO old;` (так делали до 12). Аналог `INDEX REBUILD WITH (ONLINE = ON)` в MS SQL.
  - Риск: при сбое остаётся **невалидный** индекс (`indisvalid = false`) — его надо найти и
    пере-/удалить (см. §5, `find_invalid_indexes`).
- **pg_repack** — онлайн-замена `VACUUM FULL`: создаёт копию таблицы, переливает данные, ловит инкремент,
  на 1–2 сек блокирует для переключения, удаляет старую. Возвращает место ОС и заодно перестраивает
  индексы. Требует **отдельной установки** (собирается из исходников под версию) и `CREATE EXTENSION
  pg_repack;` в базе; пользователю нужны права суперпользователя или роль `pg_repack`.

**Как понять, что индекс/таблица распухли** — надёжного штатного инструмента нет. Стратегии:
- Точный замер — расширение **`pgstattuple`**: `SELECT * FROM pgstattuple('schema.table');` и
  `pgstatindex('schema.index');` (поля `dead_tuple_percent`, для индекса `avg_leaf_density` /
  `leaf_fragmentation`). Точно, но тяжело (сканирует объект) — по крупным выборочно.
- Оценочный SQL по `pg_class.relpages/reltuples` (как в скрипте) — быстро, но приблизительно.
- Тренд размера: ежедневно снимать размеры всех таблиц/индексов и ловить аномальный рост относительно
  профиля за месяц-два (надёжнее всего, но требует накопления истории).
- Простое правило: индексы крупнее порога (напр. > 1 ГБ) перестраивать раз в неделю/месяц по расписанию.

Текущий размер индексов/таблиц (для тренда и порогов):
```sql
SELECT n.nspname||'.'||c.relname AS obj, c.relkind,
       pg_size_pretty(pg_relation_size(c.oid)) AS size
FROM pg_class c JOIN pg_namespace n ON n.oid=c.relnamespace
WHERE n.nspname='public' ORDER BY pg_relation_size(c.oid) DESC LIMIT 20;
```

## 3. VACUUM FREEZE — против TXID wraparound, отдельным редким заданием

Счётчик транзакций PostgreSQL 32-битный (~2 млрд «вперёд» / 2 млрд «назад»). Без заморозки старые строки
рискуют переполнить счётчик — **wraparound**, и база встаёт («не могу выделить номер транзакции»). VACUUM
помечает старые строки битом «заморожено», и они перестают зависеть от счётчика.

Побочный, но главный для нас эффект (с 9.6): **карта заморозки** — автовакуум больше не смотрит на
полностью замороженные страницы. Поэтому регулярный FREEZE всей базы делает последующие проходы
автовакуума почти мгновенными и **возобновляемыми** (убили на 99% — при перезапуске обработается только
оставшийся 1%, а не вся таблица заново). Для 1С (15 лет данных + постоянные перепроведения) это сильно
разгружает фоновые процессы.

**FREEZE неблокирующий для DML, но I/O-ёмкий** → НЕ в ежедневный регламент. Запускать **отдельным**
заданием раз в **месяц/квартал** (или чаще, если мониторинг показывает быстрый рост возраста XID):
```bash
vacuumdb --freeze --analyze --verbose --jobs=4 --dbname=your_1c_database_name
```
Контроль необходимости — возраст «замороженности» базы (чем ближе к ~2 млрд, тем срочнее):
```sql
SELECT datname, age(datfrozenxid) FROM pg_database ORDER BY age(datfrozenxid) DESC;
```
Подробнее про интерпретацию `age(datfrozenxid)` — `monitoring-and-locks.md`.

## 4. Расписание (адаптируй под свой профиль)

| Операция | Что | Периодичность | Чем |
|---|---|---|---|
| VACUUM + ANALYZE | убрать мусор, обновить статистику | ежедневно ночью | `pg_1c_daily_maintenance.sh` (`vacuumdb --analyze --jobs`) |
| REINDEX CONCURRENTLY | распухшие индексы (по порогу) | ежедневно/еженедельно | тот же скрипт |
| pg_repack | сильно распухшие таблицы | по необходимости/еженедельно | тот же скрипт (опц.) |
| VACUUM FREEZE | заморозка против wraparound | раз в месяц/квартал | `pg_periodic_freeze.sh` |
| Переаналитика горячих таблиц | свежая статистика часто меняющихся таблиц | по cron (час/несколько часов) | `vacuumdb --analyze`/`ANALYZE` точечно |

Автовакуум при этом работает постоянно в фоне (настроен агрессивно). Скрипты дополняют его.

## 5. Готовые скрипты (`scripts/`) — что делают и как настроить

### `pg_1c_daily_maintenance.sh` (ежедневное)
1. **Захват блокировки (mutex)** — атомарно через `mkdir` каталога-локфайла в `/var/lock/
   pg_maintenance_locks/pg_1c_maintenance.lock` + PID-файл. Если жив процесс с записанным PID — выходит
   (`exit 1`), чтобы прогоны не наложились. Устаревшую блокировку (мёртвый PID) снимает сам. Через
   `trap ... EXIT INT TERM` гарантированно снимает свою блокировку при выходе. **Важно:** имя локфайла
   общее с freeze-скриптом — ежедневное обслуживание и заморозка не запустятся одновременно.
2. **Конфиг и аутентификация** — читает `pg_1c_daily_maintenance.conf` (рядом со скриптом), требует
   `PGDATABASE` и `PGUSER`. Параметры подключения через env (`PGHOST`/`PGPORT`/`PGUSER`/`PGDATABASE`),
   **пароль — из `~/.pgpass`** пользователя ОС, от которого идёт прогон (`PGPASSWORD` намеренно не
   используется).
3. **Проверка невалидных индексов ДО** (`find_invalid_indexes`) — ищет `pg_index.indisvalid = false`
   (остатки от прошлых сбоев REINDEX CONCURRENTLY) и предупреждает (вручную: `DROP INDEX CONCURRENTLY` /
   `REINDEX INDEX CONCURRENTLY`).
4. **VACUUM ANALYZE** параллельно: `vacuumdb --analyze --verbose --jobs="$VACUUMDB_JOBS" --dbname=...`.
5. **REINDEX распухших индексов** (если `ENABLE_REINDEX=true`): оценочный SQL по `pg_class` выбирает
   btree-индексы размером ≥ `MIN_INDEX_SIZE_MB` и оценочным распуханием ≥ `BLOAT_THRESHOLD_PERCENT`; для
   каждого — `REINDEX INDEX CONCURRENTLY "schema"."index"`, затем `ANALYZE` его таблицы (однократно на
   таблицу). При ошибке REINDEX — повторная проверка невалидных индексов. После всех — лёгкий VACUUM.
6. **pg_repack** (опционально, `ENABLE_PG_REPACK=true`): оценочный SQL выбирает распухшие **таблицы**
   (≥ `REPACK_MIN_TABLE_SIZE_MB`, ≥ `REPACK_BLOAT_THRESHOLD_PERCENT`) и гоняет `pg_repack --table=...
   --jobs --no-order --wait-timeout=600`. Требует установленного `pg_repack` (путь — `PG_REPACK_PATH`).
7. **Проверка невалидных индексов ПОСЛЕ** и завершение. Логи с таймингами каждого шага — рядом со
   скриптом (`pg_maintenance_YYYYmmdd_HHMMSS.log`).

> Встроенная оценка распухания в скрипте — **приблизительная** (по `relpages/reltuples`). Для точности
> ставь `pgstattuple` и при желании замени запрос на основанный на нём. Пороги (`*_THRESHOLD_PERCENT`,
> размеры) подбирай под свою базу.

### `pg_periodic_freeze.sh` (периодическое, раз в месяц/квартал)
Берёт **тот же** `pg_1c_daily_maintenance.conf` и **тот же** mutex (не пересечётся с ежедневным),
выполняет `vacuumdb --freeze --analyze --verbose --jobs --dbname=...`. Свой лог
`pg_periodic_freeze_*.log`. Запускать значительно реже ежедневного (см. §3), по cron.

### Настройка окружения
1. Заполни `pg_1c_daily_maintenance.conf`: `PGUSER`, `PGDATABASE` (обязательно), при нужде
   `PGHOST`/`PGPORT`; `VACUUMDB_JOBS`; флаги REINDEX/repack и пороги.
2. Настрой `~/.pgpass` пользователя ОС, от которого пойдёт регламент (см. шаблон
   `scripts/pgpass-example.txt`):
   - путь: домашний каталог этого пользователя (`/home/<user>/.pgpass`; для `postgres` в RED OS обычно
     `/var/lib/pgsql/.pgpass` — уточни `grep postgres /etc/passwd`; для cron-пользователя — его дом.каталог);
   - формат строки: `hostname:port:database:username:password` (можно `*`);
   - **права обязательно `chmod 0600`** — иначе PostgreSQL проигнорирует файл.
3. У `PGUSER` должны быть права на VACUUM/ANALYZE/REINDEX (а для pg_repack — суперпользователь/роль
   `pg_repack`).
4. В postgresql.conf — агрессивный автовакуум и `maintenance_work_mem` (см. `postgres-tuning.md`).
5. Cron: ежедневный — ночью; freeze — раз в месяц/квартал, в другое окно. Логи ротировать/чистить.
6. **Сначала предпрод**, проверить корректность, потом прод.

> Пароли — только в `.pgpass`/секрет-хранилище, НИКОГДА в `.conf`, скрипт, репозиторий или лог. В репо
> лежат только обезличенные шаблоны.
