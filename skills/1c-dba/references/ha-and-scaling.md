# Отказоустойчивость (HA) и масштабирование PostgreSQL под 1С

Цели разделять: **HA/отказоустойчивость** (пережить падение узла без потери данных и с минимальным
простоем) и **масштабирование** (вынести нагрузку — аналитику/чтение — с продуктивного мастера, ускорить
отчёты, расти по объёму). Решения разные, не путать.

---

## 1. Кластер высокой доступности: Patroni + etcd

Типовой отказоустойчивый стек под Postgres Pro:
- **Patroni** — оркестратор: следит за здоровьем узлов, проводит автоматический failover (повышает
  реплику в мастер), управляет ролями.
- **etcd** (кластер из 3+ узлов) — распределённое хранилище конфигурации/состояния (кто сейчас мастер).
  Требует кворума → минимум 3 узла, стабильная сеть между ними.
- **HAProxy** (Proxy1/Proxy2) — единая точка входа: направляет соединения 1С на текущего мастера
  (определяет через REST API Patroni).
- **PgBouncer** (опционально) — пул соединений: 1С открывает много сеансов, пул снижает накладные расходы
  на подключения к PostgreSQL.

Пример состава кластера (имена/числа — под себя):

| Узел | Назначение | Компоненты |
|---|---|---|
| Core | сервер 1С:Предприятия | сервер 1С |
| DBOne | основной сервер БД (мастер) | Postgres Pro, Patroni |
| DBTwo | резервный сервер БД (реплика) | Postgres Pro, Patroni |
| DBFour | сервер копий баз 1С для аналитики | Postgres Pro, postgres_fdw |
| Etcd1/2/3 | хранилище конфигурации Patroni | etcd (кворум) |
| Proxy1/2 | балансировщик | HAProxy |
| PgBouncer1/2 | пул соединений | PgBouncer |

Сайзинг: мастер (DBOne) — максимум CPU/RAM, быстрые NVMe, скоростная сеть; реплика (DBTwo) —
сопоставимо (при синхронной репликации — равноценно; при асинхронной можно чуть скромнее); etcd —
небольшие ресурсы, но SSD и стабильная сеть критичны для кворума.

## 2. Репликация: потоковая и логическая

- **Потоковая (физическая) репликация** — реплика получает поток WAL мастера, бит-в-бит копия кластера.
  - синхронная (`synchronous_commit = on` + `synchronous_standby_names`) — транзакция подтверждается
    после записи на реплику: нулевая потеря данных, но выше задержка коммита;
  - асинхронная — быстрее, но при падении мастера возможна потеря последних транзакций (RPO > 0).
  - Реплики потоковой репликации **только для чтения**.
- **Логическая репликация** (publication/subscription) — реплицирует изменения на уровне строк выбранных
  таблиц; реплика **доступна на запись**, может быть другой версии/структуры. Используется для копий под
  аналитику и частичной репликации (см. §3).

---

## 3. Реплики только для чтения и копии баз 1С для аналитики

**Ограничение для 1С (важно):** реплики PostgreSQL под Patroni доступны **только для чтения, включая
временные таблицы**. А платформа 1С при выполнении многих запросов **создаёт временные таблицы** → на
read-only реплике такие запросы **падают**. Поэтому «направить отчёты 1С прямо на потоковую реплику»
в общем случае **не работает**.

Штатный механизм **«Копии баз данных» 1С** (для ускорения аналитических отчётов и Дата-акселератора) сам
по себе не поддерживает запись в копию → те же проблемы с временными таблицами и механизмом версий данных.

**Рабочая архитектура копий для аналитики:**
- Отдельный сервер PostgreSQL (DBFour), **не входящий в кластер Patroni**, настроенный **на запись**.
- Связь с основным кластером через **`postgres_fdw`** (Foreign Data Wrapper): запросы на **чтение**
  проксируются через FDW к репликам основного кластера (или данные приходят логической репликацией),
  запросы на **запись** (включая временные таблицы 1С) выполняются локально на сервере копий.
- Синхронизацию данных между сервером копий и мастером администратор настраивает асинхронно (логическая
  репликация / триггеры), исходя из требований к консистентности и производительности.

Так аналитика и тяжёлые отчёты уходят с продуктивного мастера, а 1С получает базу, в которую можно писать
временные таблицы. Альтернатива копиям через FDW — **логическая репликация** нужных таблиц на отдельный
писать-способный сервер.

## 4. Масштабирование по объёму: секционирование и шардирование

- **Секционирование (партиционирование)** — одна таблица режется на секции (по диапазону дат/значению)
  внутри одного сервера. Ускоряет работу с крупными таблицами (partition pruning), упрощает обслуживание
  и архивацию старых периодов. В контексте 1С применимо ограниченно (структуру таблиц задаёт платформа),
  чаще для регистров/больших таблиц при глубокой кастомизации — помечай **[проверить]** под конкретную
  конфигурацию.
- **Шардирование** — данные распределяются по нескольким серверам (горизонтальное масштабирование). Для
  очень больших инсталляций 1С нередко приоритетнее секционирования, но это сложное архитектурное
  решение.
- **Citus** (расширение PostgreSQL) — распределённый PostgreSQL (шардирование + распределённые запросы).
  Совместимость с конкретной конфигурацией/нагрузкой 1С нужно проверять отдельно — помечай **[проверить]**.

> Прежде чем шардировать/секционировать, убедись, что исчерпаны более дешёвые пути: тюнинг (`postgres-
> tuning.md`), вынос аналитики на копии (§3), оптимизация тяжёлых запросов в коде 1С (`1c-dev`). Шардинг —
> последний рубеж, дорогой в эксплуатации.

---

## 5. Дерево решений «когда что»

- Нужно пережить падение узла БД с автопереключением → **Patroni + etcd** (+ HAProxy/PgBouncer).
- Нельзя терять ни одной транзакции → **синхронная** потоковая репликация (ценой задержки коммита).
- Тяжёлые отчёты грузят прод → вынести на **сервер копий** (DBFour, postgres_fdw / логическая
  репликация); прямая read-only реплика не подойдёт из-за временных таблиц 1С.
- Нужна копия под отчётность/Дата-акселератор 1С → механизм «Копии баз» поверх отдельного писать-
  способного сервера, не голая потоковая реплика.
- База перерастает один сервер по объёму/нагрузке записи → **секционирование** (внутри сервера) →
  **шардирование/Citus** (между серверами), как крайняя мера.
- Много коротких подключений 1С → **PgBouncer** (пул).

> Все формулы сайзинга и числа узлов — под конкретный профиль (пользователи, объём, RPO/RTO). Конкретные
> версии Patroni/etcd/Postgres Pro/Citus и их совместимость — по справке и тестам на предпроде, помечай
> **[проверить]**.
