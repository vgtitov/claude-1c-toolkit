# Тюнинг PostgreSQL / Postgres Pro под 1С

Цель тюнинга под 1С: снизить нагрузку на дисковую подсистему, эффективно использовать RAM, дать тяжёлым
отчётам параллелизм, держать автовакуум «в голоде» (готов, но не молотит постоянно). База 1С нетипична:
тысячи таблиц/индексов, данные за годы при работе в последнем периоде, частые перепроведения, только
btree-индексы. Поэтому настройки агрессивнее «учебных».

> **Зачем (масштаб эффекта).** Бенчмарк ненастроенный vs настроенный PostgreSQL под 1С (Дорошкевич,
> УТ 11.4 / ЗУП 3.1): начисление ЗП 400 сотр. **1 ч 25 мин → 8 мин**; распределение НДС **5 мин 10 с →
> 25 с**; открытие карточки клиента + CRM **9.6 с → 0.9 с**. Тюнинг СУБД даёт кратное ускорение даже без
> правок кода 1С.

**Дисциплина изменений (соблюдать всегда):**
1. Сними диагностику ДО изменений на характерной нагрузке (тяжёлый отчёт, сложная сортировка
   динамического списка) — зафиксируй время/IO/план.
2. Сначала **предпрод**, потом прод.
3. Каждый сервер — **отдельный пересчёт** параметров (от его RAM/ядер/дисков). Не копируй чужой
   postgresql.conf числами.
4. Перед правкой — копия настроек и базы. Меняй по одному параметру (или одной группе), с планом отката.
5. После изменения — диагностика ПОСЛЕ. Решение: оставить / докрутить / откатить / следующий параметр.

Применение без перезапуска: `SELECT pg_reload_conf();` (или `pg_ctl reload`) — для параметров уровня
sighup. Параметры с пометкой «требует перезапуска» (`shared_buffers`, `max_worker_processes`,
`shared_preload_libraries`, `max_connections`) — только рестарт. Проверить текущее значение и нужен ли
рестарт: `SELECT name, setting, unit, context FROM pg_settings WHERE name = '...';` (`context` =
`postmaster` → нужен рестарт; `sighup` → reload; `user`/`superuser` → можно на сессию).

Быстрый снимок ключевых параметров:
```sql
SELECT name, setting, unit FROM pg_settings
WHERE name IN ('shared_buffers','effective_cache_size','maintenance_work_mem','work_mem',
 'temp_buffers','random_page_cost','effective_io_concurrency','max_parallel_workers',
 'max_worker_processes','max_parallel_workers_per_gather','max_connections')
ORDER BY name;
```

## 1. Память

### Алгоритм расчёта (пример, считай от СВОЕГО железа)
Пусть сервер выделен под СУБД, RAM = 512 ГБ, ~300 одновременных сеансов 1С:
1. **Доступная память:** вычесть то, что съедают ОС и прочие процессы (напр. 64 ГБ). Остаётся
   512 − 64 = **448 ГБ** под Postgres.
2. **shared_buffers** = ~25% доступной: 448 × 0.25 ≈ **112 ГБ**. (Требует перезапуска.) Это кэш страниц
   СУБД; больше — меньше обращений к диску, но не отдавай всё: ОС тоже кэширует файлы.
3. **Память на сеанс** = (448 − 112) / 300 ≈ **1.12 ГБ/сеанс**.
4. **work_mem** и **temp_buffers** = делим память сеанса (грубо пополам): ≈ **512 МБ** на каждый.

> Осторожно с `work_mem`: он выделяется НА КАЖДУЮ операцию сортировки/хеша в запросе, умножается на
> число параллельных воркеров и сеансов. 512 МБ оправдан при контролируемом числе тяжёлых сеансов; при
> сотнях активных — считай пик `work_mem × сорт-операции × сеансы`, чтобы не уйти в своп. Можно держать
> глобально умеренным и поднимать на сессию для отчётов: `SET work_mem = '512MB';`.

### Параметры памяти
- **shared_buffers** — главный кэш СУБД, ~25% RAM. Цель — высокий cache hit ratio (см.
  `monitoring-and-locks.md`), стремиться > 95–99%.
- **effective_cache_size** — НЕ выделяет память, это подсказка планировщику, сколько всего кэша (СУБД +
  ОС) доступно под данные. Обычно ~50–75% RAM. Влияет на выбор индексных планов.
- **work_mem** — память на сортировку/хеш/bitmap до сброса во временные файлы на диск. Мало → запросы
  пишут temp-файлы (видно в `log_temp_files`), много → риск переполнения памяти.
- **temp_buffers** — кэш под **временные таблицы** сеанса. Для 1С важен: платформа активно создаёт
  временные таблицы. Поднятие (напр. 256 → 512 МБ) уменьшает дисковую нагрузку от них. Можно на сессию.
- **maintenance_work_mem** — память под обслуживание (VACUUM, CREATE INDEX/REINDEX, ANALYZE). Под 1С —
  512 МБ … 2 ГБ+ (от RAM): ускоряет регламент и автовакуум.

### Готовая таблица сайзинга памяти (отправная точка, пересчитай под сеансы/нагрузку)
Формулы: `shared_buffers` = RAM/4 (свыше ~40% RAM прироста нет), `effective_cache_size` = RAM −
shared_buffers (≈50–75% RAM), `temp_buffers` ≈ 256 МБ, `work_mem` = RAM/32…64 (32–128 МБ),
`maintenance_work_mem` = RAM/16…32. `huge_pages`: `off` на малой RAM, `try` на крупной.

| RAM | shared_buffers | work_mem | maintenance_work_mem |
|---|---|---|---|
| 32 ГБ | 8 ГБ | 32 МБ | ~420 МБ |
| 64 ГБ | 16 ГБ | 64 МБ | ~620 МБ |
| 128 ГБ | 32 ГБ | 64 МБ | ~920 МБ |
| 256 ГБ | 64 ГБ | 128 МБ | ~1620 МБ |

> `work_mem` — на **каждую** операцию сортировки/`order by`/`distinct`/хеша в **каждой** сессии; считай пик
> по числу активных тяжёлых сеансов, не задирай глобально.

### Диагностика памяти
- cache hit ratio: `pg_stat_database` (см. `monitoring-and-locks.md`), цель > 95%.
- временные файлы: `log_temp_files = 512MB` → искать в логе строки `temporary file`; анализировать
  запросы, рождающие temp-файлы, и корректировать `work_mem`/`temp_buffers`. Сводно по базе (в идеале 0):
  `SELECT sum(temp_files), pg_size_pretty(sum(temp_bytes)) FROM pg_stat_database;`.
- на время загрузки `.dt`/массового импорта — кратно поднять `maintenance_work_mem` (ускоряет создание
  индексов), вернуть после.
- потребление памяти процессом СУБД: `top`, `free` (не уйти в swap).
- откат: вернуть прежние значения, рестарт (для `shared_buffers`).

### pg_stat_temp в tmpfs (снижение дисковой активности от статистики)
Файлы статистики постоянно перезаписываются → дисковая нагрузка. Перенос в RAM-диск:
1. В `postgresql.conf`: `stats_temp_directory = '/run/shm/pg_stat_temp'` (и, где параметр есть,
   `pg_stat_temp_directory = '/run/shm/pg_stat_temp'`).
2. Создать каталог `/run/shm/pg_stat_temp`, дать пользователю `postgres` права на запись.
3. Рестарт Postgres Pro.
4. Проверка: `SHOW stats_temp_directory;` (и `SHOW pg_stat_temp_directory;`); `iostat`/`vmstat`/
   `pg_stat_bgwriter` — убедиться в снижении дисковой активности.
5. Откат: вернуть исходные значения, рестарт.

> **[проверить] по версии:** `stats_temp_directory` существовал до PostgreSQL 14. В PostgreSQL 15+
> сбор статистики переведён в shared memory, и этот параметр **удалён** — приём с tmpfs там не нужен и
> вызовет ошибку. На Postgres Pro 9.6–14 — актуально; на 15+ — пропустить. Сверь со справкой своей версии.

## 2. Ввод/вывод (IO)

- **random_page_cost** — стоимость случайного чтения относительно `seq_page_cost = 1.0` (шкала; сам
  `seq_page_cost` не трогать). По умолчанию **4.0** (расчёт на единичный SATA/HDD). Значения по носителю:
  NVMe **0.4**, SSD **1.1–1.3**, RAID **1.5–2.0**, единичный SATA **4.0**. Это принципиально улучшает
  выбор планов: планировщик перестаёт избегать индексов и не уходит в лишние seq scan. Завышение
  `random_page_cost` → seqscan вместо индексов = плохо. На SSD/NVMe-сервере 1С один из первых параметров.
- **effective_io_concurrency** — число параллельных IO-операций, которые СУБД может инициировать (важно
  для `bitmap heap scan`). По умолчанию часто **1**. Для SSD/NVMe — ставить заметно выше: начать
  со **100**, экспериментально поднимать до 200–300, контролируя утилизацию дисков (`iostat`). На HDD
  оставить низким (= числу шпинделей).
- **seq_page_cost** — обычно 1.0, не трогать без причины.

Диагностика IO: `iostat`/`vmstat` (утилизация, await, KB/s), `EXPLAIN (ANALYZE, BUFFERS)` для тяжёлых
запросов — смотреть, ушёл ли план с seq scan на индекс/bitmap после правки `random_page_cost`.

**Размещение данных на дисках (под нагрузку):** раздельные tablespace под данные и индексы 1С
(`v81c_data` / `v81c_index`) на **разных** дисках; **WAL — на отдельном RAID** (отделить
последовательную запись WAL от случайного IO данных). Высоконагруженный 1С на PostgreSQL — **только
Linux** + асинхронная реплика (см. `ha-and-scaling.md`).

## 3. Параллелизм (тяжёлые отчёты 1С)

Считать от числа **физических** ядер CPU (N_CPU):
- **max_worker_processes** = N_CPU (требует перезапуска) — общий пул фоновых процессов.
- **max_parallel_workers** = N_CPU (требует перезапуска) — сколько из них под параллельные запросы.
- **max_parallel_workers_per_gather** — для OLTP-нагрузки 1С по умолчанию параллелизм скорее **вреден**:
  типовое значение **0**, на высоконагруженных — **2** (только после теста на своих отчётах). Включает
  параллельные планы для тяжёлых отчётов; обязательно протестировать (не всем запросам помогает, а
  `work_mem` множится на воркеров). **Сильное распараллеливание SQL = признак неоптимального кода 1С** —
  корень чаще у разработчика, а не в сервере.
- **max_parallel_maintenance_workers** — `0` при **многопоточной** загрузке `.dt` (потоки конкурируют),
  `10` при однопоточной (ускорить создание индексов). Под обычную нагрузку — умеренно.

Готовая таблица по числу ядер (`per_gather` / `max_parallel_workers` / `max_worker_processes` /
`max_parallel_maintenance_workers`) — отправная точка под высоконагруж. с отчётами:

| Ядра | per_gather | parallel_workers | worker_processes | maintenance_workers |
|---|---|---|---|---|
| 8 | 2 | 8 | 8 | 4 |
| 16 | 4 | 16 | 16 | 8 |
| 32 | 8 | 32 | 32 | 16 |

Смена `max_worker_processes` → **рестарт** службы. Для read-only нагрузки автовакууму хватает 1 воркера.

Диагностика: `EXPLAIN (ANALYZE)` тяжёлого отчёта ДО/ПОСЛЕ — появились ли узлы `Parallel`, сократилось ли
время; следить за CPU при пиковой нагрузке 1С.

## 4. Автовакуум — «держи VACUUM в голоде»

Это **фундамент** здоровья базы 1С; регламентные скрипты его дополняют, но не заменяют. Отключать
автовакуум — типовая фатальная ошибка (через ~2 недели база встаёт колом от мёртвых версий). Настройки
под 1С агрессивнее стандартных, потому что мы работаем в последнем периоде огромной базы:

- **autovacuum = on** — буквально раскомментировать строку (частая ошибка: «включён», но закомментирован).
- **autovacuum_max_workers** ≈ **N_CPU / 2**. Воркерам надо обойти десятки тысяч файлов (таблицы + каждый
  индекс = отдельный файл; даже пустая демо-БП ≈ 40 тыс. файлов, ~6 тыс. таблиц × 3–5 индексов).
- **autovacuum_vacuum_cost_limit** — увеличивать **пропорционально** числу воркеров. Если воркеров стало
  6 (было 3), лимит 400 → **800**. Иначе рост воркеров не ускорит уборку (общий бюджет делится между всеми).
- **autovacuum_vacuum_cost_delay** — пауза, когда воркеры в сумме набрали лимит «стоимости» (чтобы
  система «продышалась»). Балансирует с cost_limit.
- **autovacuum_vacuum_scale_factor** — снизить с 0.2 (20%) до **0.01 (1%)**. Иначе в 15-летней базе 10%
  строк меняются почти год, и автовакуум придёт «через год».
- **autovacuum_analyze_scale_factor** — снизить с 0.1 до **0.005 (0.5%)**: статистику пересчитывать
  часто, иначе планы строятся по устаревшему распределению. (Аналог — FULLSCAN-обновление статистики в
  MS SQL.)
- **autovacuum_naptime** — частота проверок; под 1С типично **20s**. Уменьшать при быстром накоплении
  мёртвых строк, увеличивать при слишком частых пустых запусках.
- Для отдельных крупных таблиц можно задать абсолютные `_threshold` и индивидуальные scale_factor
  (`ALTER TABLE ... SET (autovacuum_vacuum_scale_factor = ...)`).
- **log_autovacuum_min_duration = 0** (логировать все запуски) или, чтобы не зашумлять, 250ms — контроль
  длительности и срабатываний.
- **maintenance_work_mem** влияет и на скорость автовакуума (см. §1).

**Контроль «голода»** — один запрос, среднее число активных воркеров за рабочее время должно быть
**меньше** `autovacuum_max_workers`, а суммарно работа автовакуума — ~1–2% времени БД:
```sql
SELECT count(*) FROM pg_stat_activity WHERE query LIKE 'autovacuum:%';
```
- Высокая нагрузка от автовакуума / CPU > 80% → уменьшить `max_workers` либо смягчить (увеличить delay /
  уменьшить cost_limit).
- Воркеров мало, мёртвые строки копятся, прогресс медленный → увеличить `max_workers` и cost_limit.

> Разделить число воркеров отдельно на VACUUM и ANALYZE в PostgreSQL **нельзя** — один процесс делает и
> то, и другое. Поэтому частую переаналитику отдельных таблиц добавляют отдельным cron-скриптом
> (`vacuumdb --analyze` / `ANALYZE`).

## 5. WAL и контрольные точки

Контрольная точка (checkpoint) сбрасывает грязные страницы на диск; слишком частые — всплески IO,
слишком редкие — долгое восстановление и большой WAL. Параметры (значения подбирать под нагрузку/диски,
помечай **[проверить]** по версии):
- **checkpoint_timeout** — интервал между контрольными точками (по умолчанию 5 мин). Под нагрузочную 1С
  обычно увеличивают (напр. 15 мин), чтобы реже сбрасывать и сгладить IO. Тюнингом 1С-профиля Postgres
  Pro это уже частично сделано.
- **max_wal_size** / **min_wal_size** — целевой объём WAL между контрольными точками (сегмент WAL = 16 МБ);
  занижен → checkpoint по объёму запускается слишком часто. Поднимать для записьёмких 1С: типично
  `min_wal_size` 512 МБ…4 ГБ, `max_wal_size` ≈ 2× от него. По размеру базы (отправная точка):

  | Размер БД | max_wal_size |
  |---|---|
  | 100 ГБ | ~1024 МБ |
  | 1 ТБ | ~10240 МБ |
  | 10 ТБ | ~16384 МБ |
- **checkpoint_completion_target** — «размазать» запись грязных страниц во времени (обычно 0.9), чтобы не
  было пилообразных всплесков IO.
- **wal_buffers** — буфер WAL; обычно авто (−1) достаточно.
- **wal_compression** — сжатие full-page образов; снижает объём WAL ценой CPU, полезно при PITR/архиве.
- **synchronous_commit** — `on` для гарантии сохранности транзакций (по умолчанию). Менять на `off`/
  `local` — осознанный компромисс: риск потери **последних ~0.5–1 с** транзакций при сбое, но
  **целостность БД сохраняется** (это не fsync). Оправдан только при ИБП + мониторинге; для учётной 1С
  обычно НЕ ослаблять. На репликах/синхронной репликации — см. `ha-and-scaling.md`.
- **commit_delay** — группировка коммитов (напр. `1000` мкс) для снижения числа fsync под нагрузкой.
- **fsync** и **full_page_writes** НЕ выключать (риск порчи БД). При RAID-контроллере с кэшем write-back
  проверять **батарейку (BBU)** кэша — без неё write-back теряет данные при потере питания.

> О кэше PostgreSQL «прогревать/чистить вручную» не нужно (в отличие от MS SQL): СУБД сама вытесняет
> страницы, которые больше не нужны транзакциям; этим в т.ч. управляет цикл контрольных точек. «Утренних
> тормозов» от ночного регламента, как в MS SQL, тут нет. Прогрев данных нагрузочным сценарием — отдельная
> и редко нужная история (см. `mssql-for-1c.md` про MS SQL).

## 6. Специфика Postgres Pro для 1С

> **Pre-flight (обязательно при инициализации кластера):** локаль ДОЛЖНА быть задана, иначе база 1С не
> запустится. Linux — `locale=ru_RU.UTF-8`; Windows — `locale=Russian_Russia.1251`. Сверь локаль кластера
> до загрузки базы: `SHOW lc_collate;` / `SHOW server_encoding;`.

- **Установка с тюнингом под 1С** — при инициализации кластера передать `--tune=1c`, тогда часть
  параметров (в т.ч. под профиль 1С) выставляется сразу:
  `pg-setup initdb --pgdata=<data> --tune=1c --locale=ru_RU.UTF-8`. `--tune=1c` авто-считает
  `autovacuum_max_workers` от числа ядер — **при смене числа vCPU виртуалки пересчитать вручную** (§4).
- **shared_preload_libraries** под 1С/мониторинг обычно включает:
  `online_analyze, plantuner, pg_stat_statements` (Postgres Pro), при использовании — добавить
  `pgpro_stats`, `pg_repack` (после установки). Требует перезапуска.
  - `online_analyze` — синхронное обновление статистики по временным таблицам сразу после наполнения
    (важно для 1С: почти все запросы 1С идут через временные таблицы, наполняются и тут же участвуют в
    соединениях; без свежей статистики план плохой — патч многократно ускоряет). Сопутствующие параметры:
    `online_analyze.table_type = 'temporary'` (или `'all'` при проблемах со статистикой на регламентах —
    см. ниже), `online_analyze.threshold = 50`, `online_analyze.scale_factor = 0.1`,
    `online_analyze.min_interval = 10000`.
  - `plantuner` — управление выбором планов (подсказки планировщику). `plantuner.fix_empty_table = on`
    убирает пессимистичность планировщика к пустым (только что созданным) таблицам — для 1С важно.

### 1С-обязательные параметры платформы
Платформа 1С требует от PostgreSQL ряд значений, иначе работа некорректна или медленна:
- `standard_conforming_strings = off`, `escape_string_warning = off` — обязательны для 1С.
- `max_locks_per_transaction = 256` — дефолт 64 мал для 1С (тысячи объектов в транзакции).
- `max_files_per_process = 10000` — дефолт 1000 мал для базы с 40000+ файлов (таблицы + каждый индекс).
- `row_security = off` — у 1С свой механизм RLS, серверный не нужен.
- `ssl = off` — обычно в доверенном контуре СУБД↔сервер 1С.
- `enable_mergejoin = off` — рекомендуется под профиль 1С. Прочие `enable_*` планировщика НЕ выключать
  (см. §7 — точечно только при разборе зависаний).
> Windows-специфика: ошибка `No buffer space available (10055)` под нагрузкой → исчерпание TCP-портов
> (лимит 5000). Поднять `MaxUserPort = 65534` в реестре (`HKLM\...\Tcpip\Parameters`).
- Сопутствующие параметры из типового развёртывания Postgres Pro под 1С:
  `track_activity_query_size = '16kB'` (видеть полные запросы в `pg_stat_activity`),
  `default_statistics_target = 100`, `compute_query_id = auto` (нужно для связки запросов в
  `pg_stat_statements`), `pg_stat_statements.max = 10000`, `pg_stat_statements.track = all`,
  `max_connections` под число сеансов 1С (напр. 1000).
- Мониторинг Postgres Pro Enterprise — расширение **`pgpro_stats`** (надстройка над `pg_stat_statements`
  с планами и ожиданиями `wait_stats`), см. `monitoring-and-locks.md`.

> Точные дефолты и наличие параметров/расширений зависят от версии и редакции Postgres Pro — сверяй по
> `postgrespro.ru/docs` под СВОЮ версию, непроверенное помечай **[проверить]**.

### Зависания на тяжёлых регламентах (закрытие месяца, расчёт себестоимости)
Причина обычно — неактуальная статистика + сложные планы соединений больших таблиц. По нарастающей (с
замером ДО/ПОСЛЕ, осознанно):
- `default_statistics_target` 1000…10000 (детальнее статистика → лучше планы крупных JOIN; дороже ANALYZE).
- `online_analyze.table_type = 'all'` — если плохие планы именно на регламентах через временные таблицы.
- `enable_nestloop = off` — точечно, если планировщик выбирает гибельный nested loop на больших объёмах.
- `join_collapse_limit = 1` — только если уверен в порядке JOIN (фиксирует порядок как в запросе).
- `from_collapse_limit`/`join_collapse_limit` > 8 (16–20) — лишь при быстром CPU (~4 ГГц/ядро), иначе
  планирование само станет тормозом.

## 7. Уровень ОС (Linux) — обязательное окружение под 1С

Тюнинг postgresql.conf не работает в вакууме: ряд настроек ОС критичны.
- **Transparent Huge Pages — выключить:** `transparent_hugepage = never` (THP даёт скачки латентности у
  СУБД). Проверка: `cat /sys/kernel/mm/transparent_hugepage/enabled`.
- **I/O scheduler в виртуалке — NOOP/none** (`elevator=noop`): хост сам управляет очередью, гостевой
  планировщик только мешает.
- **vm.swappiness = 1** — почти не свопить (СУБД сама управляет памятью; своп = провал латентности).
- **net.core.somaxconn = 4096** — очередь входящих соединений под много сеансов 1С.
- **pg_stat_tmp в tmpfs** — см. §1 (на версиях ≤ 14). В `/etc/fstab`:
  `tmpfs /var/lib/pgsql/.../pg_stat_tmp tmpfs size=512M 0 0` (типовые 32–64 МБ/БД, высоконагруж.
  256–512 МБ/БД).
- **Антивирус** — исключить из проверки процессы `postgres`, `ragent`, `rmngr`, `rphost`, `1cv8` и
  каталоги данных СУБД, WAL и бэкапов (иначе деградация IO и ложные блокировки файлов).

## Чек-лист тюнинга под 1С (порядок)
1. Зафиксировать железо (RAM, физ. ядра, диски, сеансы) и снять диагностику ДО.
2. Память: `shared_buffers` ≈ 25% RAM, `effective_cache_size` ≈ 50–75%, `maintenance_work_mem`,
   `work_mem`/`temp_buffers` от памяти на сеанс.
3. IO: `random_page_cost` 1.1–1.5 (SSD), `effective_io_concurrency` от 100 (SSD).
4. Параллелизм: `max_worker_processes`/`max_parallel_workers` = N_CPU,
   `max_parallel_workers_per_gather` = N_CPU/2.
5. Автовакуум агрессивно: `on`, workers ≈ N/2, cost_limit пропорционально, scale_factor 1%/0.5%,
   логирование.
6. `pg_stat_temp` в tmpfs (если версия ≤ 14).
7. WAL/контрольные точки под запись.
8. Postgres Pro: `--tune=1c`, нужные `shared_preload_libraries`.
9. Снять диагностику ПОСЛЕ, сравнить, решить (оставить/докрутить/откатить). Только потом — прод.
