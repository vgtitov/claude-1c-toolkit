# Архитектурное проектирование: документ «Архитектура решения» и нотации

Контекст: этот reference про архитектурный артефакт и нотации, которыми архитектор/аналитик описывает целевую
систему ДО реализации. Общеизвестная методика (C4, BPMN, IDEF, ГОСТ Р 57100). Дополняет блок про архитектора в
SKILL.md и `analysis-workflow.md`. Описывает поведение и границы, не код.

## Документ «Архитектура решения» (АР)
- Делается ДО реализации — вдумчивое проектирование заранее, а не документирование постфактум.
- Цель — договориться ВСЕЙ командой (бизнес-заказчик, разработка, ИБ, архитектура, поставщики/получатели данных) и
  формально согласовать. В итоге в хранилище остаётся документ, отвечающий на вопросы: что хотели заказчики, как
  запланировали сделать, почему именно так, какие были варианты.
- Единый шаблон + фиксация и согласование архитектурных требований. Формат — текстовый документ или презентация.

### Типовые разделы АР
Паспорт проекта → резюме (о проекте и решении) → общая дорожная карта → **место решения в функциональной
бизнес-модели** (изменяемый и затрагиваемые сегменты) → **соответствие корпоративной архитектуре** (новая и
изменяемая функциональность) → **модель функций и потоков данных** → используемые технологии/ИТ-продукты →
**архитектура программного решения** → требования к инфраструктуре → **схема сетевого взаимодействия** → анализ
рисков → оценка стоимости владения → итоговое проектное решение.

## Модель функций и потоков данных
- Верхнеуровневое взаимодействие систем и описание потоков данных, которые между ними передаются.
- ЯВНО подсвечивать (например, цветом) изменяемые проектом области — сразу видно, какие функции и потоки меняются,
  а какие остаются без изменения в целевой архитектуре.

## C4 — визуализация архитектуры ПО
Метод основан на структурной декомпозиции; четыре уровня абстракции (нотация-независим):
1. **Контекст (context)** — моделируемая система + смежные системы + люди (people). Высший уровень.
2. **Контейнеры (containers)** — отдельно развёртываемые/запускаемые единицы: серверное приложение, БД (MS SQL /
   PostgreSQL), мобильное приложение, файловая система. Контейнер — то, что должно быть запущено для работы системы.
3. **Компоненты (components)** — группы связанных функций за чётко определённым интерфейсом внутри контейнера.
4. **Код (code)** — взаимодействие функций. Самый детальный, нужен редко.
Использовать только нужные уровни — для многих команд достаточно контекста + контейнеров. Инструменты: c4model.com,
C4-PlantUML, Structurizr DSL.

## Схема сетевого взаимодействия (системная архитектура)
- Показывает физическое воплощение решения: какие системные компоненты задействованы, расположение по сегментам.
- Нужна, чтобы ИБ оценила доступы и чтобы понять, какие сетевые доступы запрашивать.
- Ключевые элементы: сегменты сети (Интернет, DMZ, внутренняя сеть); протоколы, адаптеры, шины; стрелки потоков
  взаимодействия с указанием портов.
- Это НЕ схема развёртывания — без конкретных устройств и IP-адресов.

## Нотации моделирования — что когда применять
- **BPMN** — бизнес-процессы: потоки работ, исполнители, события, ветвления.
- **IDEF0** — функциональная декомпозиция: вход / выход / управление / механизм для каждой функции.
- **UML State (диаграмма состояний)** — жизненный цикл объекта (переходы состояний).
- **ER / логическая модель данных** — сущности, атрибуты, связи и их мощности (см. `requirements-and-chtz.md`).

## Паттерны и подходы проектирования (когда поднимать)
- **DDD** — выделение домена, ограниченных контекстов и единого языка для сложной предметной области.
- **Паттерны корпоративных приложений / интеграции** — типовые решения слоёв и обмена (см. `integration-and-api-design.md`).
- В 1С эти подходы ложатся на слои контура (конфигурация + расширение) и точки расширения — описывай влияние на оба слоя.

## Когда нужен АР / ADR
Готовить АР и фиксировать ADR, если: затрагивается несколько подсистем/контуров; вводится новый/нетиповой паттерн;
влияние на корпоративную архитектуру; правила не дают ответа; решение станет повторяемым. ADR фиксирует вопрос,
варианты, риски, выбор и долгосрочные последствия. При сомнении — в пользу архитектора (см. SKILL.md).
