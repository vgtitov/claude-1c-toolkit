# Проектирование интеграций и API

Контекст: этот reference про интеграционную часть анализа — перечень потоков данных, контракты, выбор синхронного
REST или событийной модели. Общеизвестная методика (OpenAPI/AsyncAPI, EIP). Описывает поведение обмена, не
реализацию: сигнатуры HTTP-сервисов и структуры планов обмена — тех-проект тех-лида.

## Перечень потоков данных — обязательный артефакт
Для интеграционного анализа составить перечень потоков; по каждому потоку зафиксировать:
- источник и приёмник данных;
- состав передаваемых данных (на уровне смысла);
- способ взаимодействия (REST / файловый / брокер / план обмена);
- синхронность (онлайн / отложенно);
- перенос исторических данных (нужен ли начальный объём);
- выделить, какие потоки изменяются проектом, какие остаются.

## REST / OpenAPI как контракт
Последовательность описания спецификации REST API:
1. определить конечные точки (endpoints);
2. разработать схемы данных запроса/ответа;
3. описать параметры запросов;
4. описать способы аутентификации;
5. детально задокументировать операции;
6. собрать всё в одном YAML-файле спецификации (OpenAPI);
7. проверить в редакторе Swagger, итеративно уточнить.

## Обязательные нефункциональные свойства API
- **Версионирование** — заложить с самого начала.
- **Обратная совместимость** — не ломать существующих потребителей при развитии.
- **Идемпотентность для POST** — ключи идемпотентности, чтобы повторный вызов не создавал дубль.
- **Описание развёртывания по контурам** — на каких контурах опубликован API.

## OpenAPI vs AsyncAPI — выбор модели
- **OpenAPI / REST** — синхронные вызовы «запрос-ответ», когда нужен немедленный результат.
- **AsyncAPI / событийная** — обмен через очереди/брокеры (Kafka, RabbitMQ и т.п.), когда важны слабая связность,
  буферизация и гарантии доставки. Выбор зависит от связности и требований к доставке.

## Привязка к 1С-контурам
- Интеграция реализуется через расширение, обмены/РИБ, брокеры сообщений; правки — в расширение, типовую read-only.
- Контракт = поведение обмена (смысл полей, сценарии), физическая реализация (HTTP-сервис, план обмена, формат
  сообщения) — тех-проект. Проверять существующие механизмы обмена по РЕАЛЬНОМУ коду через MCP, не по памяти [проверить].

## Чек интеграции (что заложить в анализ)
- Поведение при недоступности приёмника (ретраи, таймауты).
- Дедупликация / идемпотентность (защита от повторной обработки).
- Гарантии доставки (at-least-once / exactly-once на уровне смысла).
- Влияние на отчётность и историю данных.
- Безопасность канала и аутентификация.
