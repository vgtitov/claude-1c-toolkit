# Методика расследования проблем производительности 1С

От симптома («тормозит», «висит», «падает», «не тянет нагрузку») к корневой причине — по данным, а не по
догадкам. Дисциплина: **сначала собери данные, потом гипотеза, потом подтверждение, потом фикс, потом
валидация повторным замером**. Версии/пороги — НАСТРОЙ ПОД ПРОЕКТ; неуверенное — [проверить].

## 1. Общий цикл расследования
```
Симптом → Собрать данные → Гипотеза (CPU/IO/память/блокировки/сеть) →
        → Подтвердить причину на данных → Фикс → Повторный замер (валидация) → [не помогло → к гипотезе]
```
Никогда не прыгай от симптома к фиксу. «Перепроведём / перестроим индексы / добавим памяти» без диагноза —
лечение наугад.

## 2. Какие данные собрать (перечень для расследования)
Минимальный набор для технологической проблемы:
- **Технологический журнал** за окно проблемы: `DBMSSQL`/`DBPOSTGRS` (с `duration` фильтром), `SDBL`, `CALL`,
  `EXCP`, `TLOCK`, `TTIMEOUT`, `TDEADLOCK`, `QERR`, `MEM`, `PROC` (см. `tech-journal.md`).
- **Счётчики ОС** на сервере приложений И сервере СУБД: CPU, очередь к CPU, память/своп, диск (очередь,
  латентность, util), сеть (см. `performance-apdex.md`).
- **Счётчики СУБД**: MS SQL (Page life expectancy, buffer cache hit, deadlocks, lock wait, recompilations);
  PostgreSQL (pg_stat_statements, pg_stat_activity, deadlocks, temp_files, cache hit).
- **APDEX / время ключевой операции** — до и (после фикса) после.
- **Контекст инцидента**: когда началось, что менялось (релиз, обновление платформы/СУБД, рост данных/
  пользователей), воспроизводимость (всегда / в пик / у одного пользователя), окно (рабочее/регламентное).
- **Дампы** rphost при падениях (см. `tech-journal.md` про имена и группировку по offset).

Снимай данные **синхронно** (ТЖ и счётчики за одно и то же окно) — иначе не скоррелируешь.

## 3. Дерево гипотез (по доминирующему ресурсу)
Сначала определи, **что насыщено**, по счётчикам:

| Симптом в счётчиках | Гипотеза | Куда копать |
|---|---|---|
| `% Processor Time` сервера **СУБД** высок | CPU СУБД: тяжёлые/частые запросы, перекомпиляции, отсутствие индексов | top-запросы (ТЖ/pg_stat_statements), планы, статистика → `query-optimization.md` |
| `% Processor Time` сервера **приложений** (rphost) высок | CPU приложения: тяжёлый код/циклы, серверные вызовы | top CALL (ТЖ), замер по коду → `1c-dev` |
| `Avg. Disk Queue Length`/`await`/`%util` высокие, `Page life expectancy` низкий | IO/память СУБД: нехватка ОЗУ под кэш, тяжёлое чтение с диска, tempdb/temp_files | память СУБД, индексы (меньше читать), диск; обслуживание → `1c-dba` |
| Своп (`si/so`, `Pages/sec`), `Available Mbytes` мал | Память: нехватка ОЗУ, утечки (MEM), лимиты rphost | MEM-события, лимиты процессов, ОЗУ |
| Рост `TLOCK`-ожиданий, `TTIMEOUT`, `Number of Deadlocks/sec` | Конкурентность: блокировки/дедлоки | → `locks-and-deadlocks.md` |
| Сетевые счётчики у предела, латентность каналов | Сеть: канал клиент↔сервер / сервер↔СУБД | сетевая инфраструктура, число серверных вызовов (CALL) |
| CPU/IO/память в норме, но операция долгая | конкретный запрос/код/блокировка точечно | ТЖ по операции (Context), план запроса, TLOCK |

Частая ловушка: высокий CPU СУБД — это **следствие** плохих запросов/отсутствия индексов, а не «нужно больше
ядер». Сначала запросы, потом железо.

## 4. Подтверждение причины
Гипотеза подтверждается, только если **данные сходятся**:
- Долгий запрос из ТЖ воспроизводится с тем же планом и длительностью → его план показывает скан/подзапрос/
  плохую статистику. (Не «вероятно этот» — воспроизвести.)
- Ожидание на блокировке: корневая транзакция-держатель найдена в TLOCK, её запрос/контекст идентифицирован.
- Память: MEM-события / счётчик растёт у конкретного процесса, коррелирует с операцией.
- Корреляция по времени: всплеск счётчика совпадает с событием ТЖ той же секунды.

Если данные не сходятся — гипотеза неверна, вернись к §3 (а не «всё равно сделаем индекс»).

## 5. Фикс и кто его делает
Эксперт ставит **диагноз и формулирует, ЧТО лечить**; реализацию ведут:
- код / запрос на уровне конфигурации, индексы реквизитов, управляемые блокировки в коде → **`1c-dev`**
  (метаданные/индексы создаёт человек в IDE);
- настройки СУБД, регламентное обслуживание (vacuum/analyze, обновление статистики, индексное обслуживание),
  параметры PostgreSQL/MS SQL, бэкапы → **`1c-dba`**;
- топология кластера, ТНФ, отказоустойчивость, масштабирование → этот скилл +
  `highload-cluster-loadtest.md`.

## 6. Валидация (обязательна)
- Повтори **тот же замер** ключевой операции (APDEX / время / план / счётчики), что снимал в базлайне.
- Подтверди по ТЖ/счётчикам, что узкое место ушло (нет долгого запроса / упали TLOCK-ожидания / вырос Page
  life expectancy / снизился CPU СУБД).
- Под нагрузкой — повторный нагрузочный тест, если проблема была про масштабирование.
- Нет улучшения по данным → фикс не закрыт, цикл повторяется. «Стало лучше на глаз» — не валидация.

## 9. Антипаттерны расследования (свод)
- Фикс без диагноза («перепроведём/добавим памяти/перестроим индексы наугад»).
- Данные ТЖ и счётчиков сняты в разные окна → нет корреляции.
- Лечат следствие (высокий CPU СУБД), а не причину (плохой запрос/нет индекса).
- Берут «вероятно этот запрос» вместо воспроизведения с планом.
- Нет базлайна и нет повторного замера → нельзя доказать, что починили.
- Полный ТЖ на проде на постоянку → сами создали проблему IO/диска.
