# Высоконагруженный кластер серверов 1С и нагрузочное тестирование

Масштабирование и отказоустойчивость 1С строятся на кластере серверов. Узкое место при росте пользователей
ищут нагрузочным тестом и измерениями, а не догадками. Версии платформы/ОС — НАСТРОЙ ПОД ПРОЕКТ; неуверенное —
[проверить] по справке ИТС.

## 1. Архитектура кластера

### Процессы
- **`ragent`** (агент сервера) — служба ОС, запускает и контролирует процессы кластера на рабочем сервере.
  «Рабочий сервер» = сервер, на котором запущен ragent (+ rmngr/rphost).
- **`rmngr`** (менеджер кластера) — управляет кластером: сеансы, соединения, блокировки, лицензии, очереди,
  распределение по rphost. Может быть несколько (с распределением сервисов).
- **`rphost`** (рабочий процесс) — выполняет код прикладного решения и серверные вызовы; в нём «живут» сеансы.
- **Центральный сервер** — рабочий сервер, на котором запущен **главный менеджер кластера** (rmngr) и
  для которого установлено свойство «Центральный». Центральных серверов в кластере может быть несколько (для
  отказоустойчивости менеджера).

### Ключевые файлы (для расследования и настройки)
| Файл | Назначение |
|---|---|
| `logcfg.xml` | настройка технологического журнала |
| `1cv8clst.lst` | реестр кластера |
| `1CV8Reg.lst` | список ИБ, зарегистрированных в кластере |
| `swpuser.ini` | запуск процессов rphost от имени определённого пользователя ОС |

### Число рабочих процессов
Число rphost **устанавливается системой автоматически**. Прямое явное задание числа недоступно; регулируют
**косвенно** через консоль администрирования: «макс. объём памяти на процесс», «количество соединений на
процесс», «количество ИБ на процесс». Для лицензии **ПРОФ** прямое управление числом процессов ограничено
политикой лицензирования (доступны только косвенные методы); для **КОРП** — расширенные возможности
тонкой настройки. Ориентир требований назначения: ~1 rphost на ~128 соединений или ~8 ИБ [проверить под версию].

## 2. Балансировка и требования назначения функциональности (ТНФ)
- **Требования назначения функциональности** (раздел «Рабочие серверы» в консоли) — правила, какие виды
  работы (объекты требования: клиентские соединения с ИБ, фоновые задания, соединения с конкретной ИБ, и т.п.)
  выполнять или НЕ выполнять на конкретном рабочем сервере. Так разносят нагрузку:
  - выделить отдельный сервер под **фоновые/регламентные задания**, чтобы тяжёлые фоновые операции не били по
    интерактивным пользователям;
  - привязать тяжёлую ИБ к мощному серверу;
  - вынести лицензирование/менеджер на отдельный узел.
- **Балансировка** между рабочими серверами — по режиму («производительность» / «память») и ТНФ; rmngr
  распределяет новые соединения. Тип объекта требования + значение «Назначать»/«Не назначать»/«Авто».

## 3. Отказоустойчивость
- **Максимальный уровень отказоустойчивости = (количество центральных серверов кластера) − 1.** Чтобы
  пережить отказ K центральных серверов, нужно K+1 центральных серверов. Резервируются менеджеры кластера и
  рабочие процессы; при падении узла сеансы переподхватываются (с учётом настроек).
- **Доступность системы = Требуемое время работы системы / Общее время (за период)** — целевой показатель
  SLA, от которого считают допустимый простой.
- Резервирование: несколько центральных серверов (резерв rmngr), несколько rphost (переключение при падении
  процесса), резерв на уровне СУБД (зеркалирование/AlwaysOn/репликация — зона `1c-dba`/инфраструктуры).
- Признак проблемы: ТЖ фиксирует **исчезновение rphost** без дампа, нагрузка по PerfMon **снизилась** →
  процесс не «упал» с ошибкой, а был **штатно завершён/перезапущен** (например, по достижению лимита памяти
  или внешним рестартом) — это не авария кода, а срабатывание ограничения/регламента. [проверить] по PROC/MEM.

## 4. Масштабирование
- **Вертикальное** — больше CPU/RAM/быстрый диск на сервер СУБД и сервер приложений (часто первый шаг для 1С,
  т.к. многое упирается в СУБД и в один rphost под сеанс).
- **Горизонтальное** — больше рабочих серверов в кластере + ТНФ для разнесения нагрузки. СУБД масштабируется
  тяжелее (обычно вертикально + вынос на отдельный мощный сервер; шардирование 1С штатно не поддерживает).
- Память: один сеанс «живёт» в одном rphost; контролируй лимит памяти на rphost, иначе процесс перезапускается
  (рвёт сеансы). Балансируй число процессов косвенными настройками.

## 5. Нагрузочное тестирование
Цель — найти, **где система упирается при росте числа пользователей** (bottleneck), до того как это случится
в проде. Инструменты: «1С:Тест-центр» (КИП), сценарии на встроенном языке / эмуляция многопользовательской
работы.

### Метод
1. **Сценарии** — воспроизводят реальные ключевые операции (проведение, отчёты, открытие форм, ввод
   документов) в реалистичных пропорциях, с реалистичными данными (не пустая база).
2. **Ступенчатая нагрузка** — наращивай число виртуальных пользователей ступенями (10 → 50 → 100 → …),
   на каждой ступени держи устойчивый режим и снимай метрики.
3. **Метрики на каждой ступени**: APDEX ключевых операций, время операций, счётчики ОС (CPU/IO/память) на
   сервере приложений и СУБД, счётчики СУБД (Page life expectancy, блокировки, дедлоки), число
   TLOCK-ожиданий/TTIMEOUT/TDEADLOCK по ТЖ.
4. **Точка деградации** — ступень, на которой APDEX/время резко падают. Смотри, **что насытилось первым**:
   - CPU сервера приложений (rphost) → масштабировать приложение (процессы/серверы);
   - CPU/IO/память СУБД → оптимизировать запросы/индексы или масштабировать СУБД;
   - рост ожиданий на блокировках/дедлоков → конкурентность (см. `locks-and-deadlocks.md`);
   - своп/нехватка ОЗУ → память.
5. **Профилирование** на точке деградации — ТЖ (top длительных запросов/вызовов/ожиданий) + счётчики,
   чтобы локализовать узкое место.

### Что считать результатом
Кривая «число пользователей → APDEX/время» и определённый предел масштабирования с указанием **ресурса-
ограничителя**. Не «выдержали N пользователей», а «при N упёрлись в <CPU СУБД / блокировки / память rphost>,
лечится <…>».

## 6. Пошаговый чек-лист (кластер/масштабирование)
1. Снять текущую топологию: центральные/рабочие серверы, rmngr/rphost, ТНФ, привязки ИБ.
2. Снять базлайн нагрузки (счётчики ОС/СУБД, APDEX) в пике.
3. Гипотеза узкого места по базлайну (CPU прил. / CPU СУБД / IO / память / блокировки).
4. Нагрузочный тест со ступенчатым ростом → найти точку деградации и ресурс-ограничитель.
5. Решение: ТНФ (разнести фоновые/тяжёлую ИБ) / +rphost-сервер / вертикаль СУБД / оптимизация запросов /
   конкурентность.
6. Отказоустойчивость: достаточно ли центральных серверов под целевой уровень = (N_центральных − 1)?
7. **Повторный нагрузочный тест** — точка деградации сдвинулась? APDEX под целевой нагрузкой в норме?

## 7. Типичные провалы
- Тест на пустой/нереалистичной базе → не ловит проблемы планов и блокировок.
- Меряют «выдержали N», не называя ресурс-ограничитель → нечего лечить.
- Все фоновые задания на тех же серверах, что интерактив → пользователи страдают от регламентов (нет ТНФ).
- Один центральный сервер при требовании отказоустойчивости > 0.
- Лимит памяти rphost не настроен → процессы перезапускаются, рвут сеансы под нагрузкой.
- Масштабируют приложение, когда упор в СУБД (или наоборот) — без профилирования точки деградации.
