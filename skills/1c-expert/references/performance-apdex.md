# APDEX, замеры производительности и счётчики

Производительность измеряют, а не оценивают на глаз. Базовая метрика пользовательского опыта — **APDEX** по
ключевым операциям; источники истины — встроенный механизм замера, счётчики ОС и СУБД. Пороги ниже —
ориентиры; точные значения НАСТРОЙ ПОД СВОЙ ПРОЕКТ/оборудование. Неуверенное — [проверить].

## 1. APDEX

Формула (N — общее число выполнений операции за период; Nt — уложившиеся в целевое время T; N4t — не
уложившиеся в T, но уложившиеся в 4·T):

```
APDEX = (Nt + N4t/2) / N
```

- Диапазон **0…1** (если получилось > 1 — расчёт ошибочен). Чем ближе к 1, тем лучше.
- Шкала оценки (каноническая, как на сертификации эксперта):

  | APDEX | Оценка |
  |---|---|
  | 0,94–1,00 | отлично |
  | 0,85–0,94 | хорошо |
  | 0,70–0,85 | удовлетворительно |
  | 0,50–0,70 | неудовлетворительно |
  | 0,00–0,50 | неприемлемо |

- Замер ключевой операции хранит: **целевое время** + **приоритет** операции + статистику по фактам
  (среднее / стд. отклонение / мин / макс). APDEX считается на этой статистике.
- Пример: N=16, Nt=8, N4t=8 → APDEX = (8 + 8/2)/16 = 12/16 = **0,75**.
- APDEX считается **по каждой ключевой операции** со своим целевым временем T. Целевое время задаёт бизнес/
  методология, а не «среднее по факту».
- По одному числу APDEX без целевых времён и оценки пользователей вывод о системе делать нельзя — это
  относительная метрика к выбранному T.

**Технологическое качество** (что обеспечиваем, не только APDEX): доступность, стабильность, устойчивость,
работоспособность, производительность, масштабируемость, отсутствие ошибок блокировок, неухудшение
технологических показателей при внесении изменений в ИБ.

## 2. Встроенный механизм замера и ключевые операции
- **Замер производительности** в Конфигураторе/EDT — построчный замер (где именно уходит время в коде):
  первый локальный инструмент.
- **Подсистема БСП «Оценка производительности»** — регистрирует время **ключевых операций** в проде и считает
  APDEX без отдельного нагрузочного теста; позволяет сравнить APDEX **до и после** изменения. Имена методов
  подсистемы — [проверить] по реальной БСП своей версии (сверять через код, не по памяти).
- **Замер проведения документа / формирования отчёта** — обернуть операцию замером времени (ТекущаяДата с
  миллисекундами / встроенный замер) и фиксировать как ключевую операцию.

### Замер открытия управляемой формы (рецепт)
Чтобы измерить полное время открытия формы (включая `ПриСозданииНаСервере` → `ПриОткрытии`):
1. В **модуле менеджера** объекта, в `ОбработкаПолученияФормы`, зафиксировать момент начала (например, в
   параметре/реквизите, передаваемом форме).
2. Добавить реквизит формы `ВремяНачалаОткрытияФормы` (тип Число).
3. В `ПриСозданииНаСервере` записать в него момент старта.
4. В серверной процедуре `ЗавершитьЗамерОткрытия` вычислить разницу и записать замер ключевой операции.
5. В конце `ПриОткрытии` вызвать `ЗавершитьЗамерОткрытия`.

Так замеряется именно пользовательское время открытия, а не отдельные серверные вызовы. Аналогично замеряют
проведение/отчёт, оборачивая операцию началом/концом замера.

## 3. Счётчики Windows (PerfMon) и пороги

### Сервер приложений / общие (ОС)
| Счётчик | Рекомендуемое значение |
|---|---|
| `Memory\Available Mbytes` | >> 0 (есть свободная память) |
| `Memory\Pages/sec` | коррелирует с загрузкой диска (рост = нехватка ОЗУ, своппинг) |
| `Processor\% Processor Time` | не более **70%** длительно |
| `System\Processor Queue Length` | не более **2 × число процессоров** |
| `PhysicalDisk\Avg. Disk Queue Length` | не более **2 × число дисков**, работающих параллельно |
| `PhysicalDisk\Avg. Disk sec/Transfer` | ≤ **0,01 с** в среднем, ≤ 0,03 с в пиках |
| `Network Interface\Bytes Total/sec` | не более **65%** пропускной способности интерфейса |
| `LogicalDisk\Free Megabytes` | >> 0 для системного диска, дисков БД и служебных файлов 1С |

### Сервер СУБД (MS SQL)
| Счётчик | Рекомендуемое значение |
|---|---|
| `Buffer Manager\Page life expectancy` | > **1000 сек** (ниже → нехватка памяти под кэш данных) |
| `Buffer Manager\Buffer cache hit ratio` | как можно ближе к **100** |
| `Plan Cache\Plan cache hit ratio` | как можно ближе к 100 |
| `Buffer Manager\Lazy writes/sec` | < 20 |
| `SQL Statistics`: (Batch req/sec − Compilations/sec − Re-Compilations/sec)/Batch req/sec | > **90%** (мало перекомпиляций) |
| `Transactions\Free space in TempDB` | >> 0 |
| `Latches\Average Latch Wait Time (ms)` | ≤ **10 мс** в среднем |
| `Locks\Average Wait Time (ms)` | как можно ближе к **0** |
| `Locks\Lock Timeouts (timeout > 0)` | **0** |
| `Locks\Number of Deadlocks/sec` | **0** |

Интерпретация связок: низкий `Page life expectancy` + высокий `Avg. Disk Queue Length` + высокий
`Page reads/sec` → **нехватка памяти СУБД → давление на диск**. Высокий `% Processor Time` + много
`SQL Re-Compilations/sec` → проблема компиляции планов/параметризации.

## 4. Счётчики Linux (top/vmstat/iostat/sar)
| Инструмент | Что смотреть |
|---|---|
| `top` / `htop` | %CPU по процессам (rphost/postgres), load average, RES память |
| `uptime` / `w` / `cat /proc/loadavg` | load average (норма ≲ числа ядер) |
| `vmstat 5 -SM -t` | `r` (очередь на CPU), `si/so` (своп — должен быть 0), `wa` (ожидание IO), `free/buff/cache` |
| `iostat -x 5` | `%util` диска (→100% = насыщен), `await` (латентность IO, мс), `r/s w/s` |
| `sar 5 -u -d -r` | CPU/диск/память в динамике; `sar -f /var/log/sa/saNN` — чтение исторических логов (sadc) |
| `atop -w log_atop` / `atop -r log_atop` / `atopsar` | сбор и постфактум-разбор CPU/диск/RAM/процессы |
| `free -h` | свободная память; `df -h` / `du -sh` | место на дисках/размер каталога логов |
| `mpstat` / `dstat -cmdnt` / `nmon` | CPU по ядрам / сводно |

Пороги-ориентиры Linux: своп (`si/so`, `swap used`) при наличии ОЗУ → 0; `iowait` высокий → диск bottleneck;
`%util` диска стабильно ~100% и `await` в десятки мс → диск насыщен; load average >> число ядер при низком CPU
→ ждут IO/блокировки.

## 5. Счётчики PostgreSQL
- `pg_stat_statements` — top-запросы по `total_exec_time` / `mean_exec_time` / `calls` (узкие места по
  суммарной и средней длительности).
- `pg_stat_activity` — активные сессии, `wait_event_type`/`wait_event` (Lock, IO, LWLock), долгие запросы,
  `state='idle in transaction'` (зависшие транзакции, держащие блокировки).
- `pg_stat_database` — `blks_hit`/`blks_read` (cache hit ratio = hit/(hit+read), цель → 1), `deadlocks`,
  `temp_files`/`temp_bytes` (временные файлы = нехватка `work_mem` или тяжёлые сортировки).
- `pg_locks` (+ join с `pg_stat_activity`) — кто кого блокирует; `pg_stat_bgwriter` — checkpoint-нагрузка.
- `pg_stat_user_tables`: `n_dead_tup`, `last_autovacuum` — раздувание/частота вакуума (обслуживание → `1c-dba`).

## 6. Как пользоваться (методика замера)
1. Зафиксируй **базлайн**: APDEX ключевых операций + ключевые счётчики ОС/СУБД в нормальном и пиковом режиме.
2. При инциденте сравни текущие значения с базлайном — что вышло за порог.
3. Связывай метрики: ТЖ (что долго) ↔ счётчики ОС (CPU/IO/память) ↔ счётчики СУБД (кэш/блокировки/компиляции).
4. После фикса — **повторный замер той же операции**: APDEX вырос, счётчики вернулись в норму? (см.
   `investigation-methodology.md`).

## 7. Типичные провалы
- Смотрят среднее время вместо APDEX по целевым временам ключевых операций.
- Один счётчик в вакууме (высокий CPU ≠ причина; смотри связку CPU+IO+память+блокировки).
- Нет базлайна → не с чем сравнивать «стало хуже».
- Замеряют отдельные серверные вызовы вместо полного пользовательского времени операции.
- Игнорируют `Page life expectancy`/`iowait`/своп — пропускают нехватку памяти и диск.
