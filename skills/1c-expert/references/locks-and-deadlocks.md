# Блокировки, взаимоблокировки, транзакции в 1С

Конкурентность — частая причина «то быстро, то висит». Разбирается по ТЖ (TLOCK / TTIMEOUT / TDEADLOCK) и по
данным СУБД, а не по догадкам. Версии платформы/СУБД — НАСТРОЙ ПОД ПРОЕКТ; неуверенное — [проверить].

## 1. Два уровня блокировок

### Управляемые блокировки 1С (в управляемом режиме)
- Накладываются **прикладным кодом / платформой** на уровне объектов 1С (регистры, документы) в **менеджере
  блокировок кластера**, поверх блокировок СУБД. Видны в ТЖ как **TLOCK**.
- Режим транзакции `READ COMMITTED` на стороне СУБД (для MS SQL — с версионированием/без, см. ниже); логику
  целостности обеспечивают именно управляемые блокировки.
- Гранулярность управляемая: блокируются **пространства (Regions)** с конкретными ключами (значениями
  измерений), а не вся таблица — меньше конфликтов.
- Виды: разделяемые (S, `Shared`) и исключительные (X, `Exclusive`). S совместима с S, X несовместима ни с чем.

### Автоматические блокировки СУБД (в автоматическом режиме)
- Платформа полагается на блокировки **самой СУБД** и повышенный уровень изоляции (вплоть до SERIALIZABLE для
  ряда операций). Гранулярность определяет СУБД (строки → страницы → таблица), отсюда **эскалация** и больше
  конфликтов/дедлоков.
- Современный режим для новых конфигураций — **управляемый**; автоматический — легаси.

**Режим управления блокировками** задаётся: на уровне конфигурации (свойство «Режим управления блокировкой
данных» = Автоматический / Управляемый / Автоматический и управляемый) и на уровне конкретного объекта/
транзакции. Цель оптимизации — управляемый режим + точечные управляемые блокировки.

## 2. Ожидания на блокировках (TLOCK)
Признак ожидания в событии **TLOCK** — непустое поле **`WaitConnections`** (перечислены соединения, которых
ждали). Ключевые поля: `Regions` (пространство/таблица), `Locks` (что блокировали), `WaitConnections`, `DBPID`.

`TTIMEOUT` — ожидание превысило **максимальное время ожидания блокировки** (по умолчанию обычно 20 c,
[проверить] под версию) → операция падает с ошибкой «конфликт блокировок».

### Алгоритм анализа ожиданий на блокировках
1. **Выбрать самое верхнее (корневое) ожидание** в цепочке — того, кто держит блокировку и сам никого не
   ждёт (виновник, а не жертвы).
2. Длительность ожидания **маленькая**? → игнорируем (шум).
3. Ожидания **в нерабочее время** (регламентные задания, закрытие периода)? → часто допустимо, отдельно.
4. **Гранулярность = таблица** (заблокирована вся таблица, а не пространство)? → **сократить число блокировок
   или отключить эскалацию**; проверить, не автоматический ли режим, не блокирует ли запрос лишнее.
5. Иначе → разобрать по **таблице причин ожиданий**: неоптимальный запрос внутри транзакции (долго держит
   блокировку → ускорить запрос, см. `query-optimization.md`); избыточная блокировка (блокируется больше
   нужного → сузить отбор/пространство); длинная транзакция (вынести лишнее из транзакции).

### Эскалация и гранулярность
- **Эскалация** (СУБД): много блокировок строк/страниц → СУБД повышает до блокировки **таблицы** → массовые
  ожидания. MS SQL: при ошибке «1204 (невозможно выделить ресурс блокировки) из-за недостатка памяти»
  использовать трейс-флаг **1224** (эскалация по памяти) вместо **1211** (полное отключение эскалации —
  опасно по памяти, включать осторожно).
- **Гранулярность**: чем точнее пространство/ключи в управляемой блокировке, тем меньше конфликтов. Снижают
  конкуренцию: точные отборы, индексы (чтобы блокировался индекс, а не скан таблицы), короткие транзакции.

## 3. Взаимоблокировки / дедлоки (TDEADLOCK)
Дедлок — циклическое ожидание: A держит ресурс1 и ждёт ресурс2, B держит ресурс2 и ждёт ресурс1. СУБД/1С
выбирает жертву и откатывает её транзакцию. В ТЖ — событие **TDEADLOCK** (управляемый режим).

### Алгоритм анализа дедлока
1. **Выбрать самую верхнюю (корневую) взаимоблокировку** в журнале.
2. **Смотрим порядок захвата блокировок** участниками. Две типовые причины:
   - **Недостаточный уровень блокировки** — порядок вида «сначала S, потом X»: транзакция сначала читает (S),
     потом пытается изменить (X) уже прочитанное; параллельная транзакция делает то же → взаимная блокировка
     на эскалации S→X. **Лечение:** сразу брать нужный уровень (исключительную блокировку на чтение тех
     данных, что будут меняться) — `Заблокировать` с режимом Исключительный до чтения.
   - **Разный порядок захвата строк/ресурсов** — порядок «сначала X потом …» у одних и обратный у других:
     транзакции берут одни и те же ресурсы в **разном порядке**. **Лечение:** **упорядочить захват ресурсов**
     — все транзакции берут блокировки в одном и том же порядке (например, сортировать обрабатываемые ключи
     перед блокировкой/записью).
3. После устранения причины — **retry** на уровне прикладной логики: операцию, упавшую по дедлоку/таймауту,
   имеет смысл повторить N раз с небольшой паузой (особенно фоновые/пакетные операции). Retry **не заменяет**
   устранение причины — это страховка от редких остаточных конфликтов.

Механика уровней: при **чтении** набора платформа ставит **Shared**, при **записи** — **Exclusive**; снимаются
**в конце внешней транзакции**. Типовая причина дедлока — повышение уровня в одной транзакции (Shared→Exclusive
крест-накрест у двух параллельных). Лечение — поставить **исключительную блокировку ДО чтения** тех данных,
что будут меняться. Для воспроизведения гонок в тесте: `СделатьПаузу(<мс>)` — метод **КИП:ТестЦентр** (не
штатный язык), искусственно растягивает удержание блокировки.

### Разбор TDEADLOCK по `DeadlockConnectionIntersections`
Граф пересечения соединений лежит в свойстве **`DeadlockConnectionIntersections`** события TDEADLOCK: пары
`<кто ждёт> <кого ждёт> <пространство> <режим Exclusive/Shared> <ключи Fld…>`. Пример:
`DeadlockConnectionIntersections='17 16 InfoRg10.DIMS Exclusive Fld11=...'` — соединение 17 ждёт соединение 16
на исключительной блокировке измерения регистра сведений 10. По номерам соединений сопоставляешь TLOCK
(`t:connectID`), чтобы увидеть, **что именно** каждая транзакция держала и какой запрос/контекст за этим стоял.

### Готовый logcfg.xml для сбора блокировок
Точечный сбор только блокировочных событий по нужной ИБ (фильтр по имени процесса = имени базы):
```xml
<log location="C:\LOGS\LOCKS" history="48">
  <event><eq property="name" value="TLOCK"/><eq property="p:ProcessName" value="<ИБ>"/></event>
  <event><eq property="name" value="TTIMEOUT"/><eq property="p:ProcessName" value="<ИБ>"/></event>
  <event><eq property="name" value="TDEADLOCK"/><eq property="p:ProcessName" value="<ИБ>"/></event>
  <property name="Regions"/><property name="Locks"/><property name="WaitConnections"/>
  <property name="DeadlockConnectionIntersections"/><property name="alreadyLocked"/>
  <property name="escalating"/><property name="Context"/><property name="Usr"/><property name="SessionID"/>
</log>
```
`alreadyLocked`/`escalating` — признаки повторной/эскалирующей блокировки; `Usr`/`SessionID`/`Context` —
кто и из какого кода. Имена свойств регистронезависимы.

### Управляемые блокировки не лечат всё (два канонических случая избыточности)
Перевод на управляемый режим блокирует **реально затронутые транзакцией строки**, но избыточность остаётся:
- **Запрос мимо индекса** внутри транзакции читает и блокирует **всю таблицу** (а не нужные строки). Лечение —
  оптимизация запроса/индекс под условие (см. `query-optimization.md`), а не сам режим блокировок.
- **Партионное списание в транзакции** требует **монопольной блокировки регистра партий** → конкуренты ждут.
  Лечение — **отложенное списание** (списывать партии вне основной транзакции проведения).

### Счётчик дедлоков и ЦУП
`Число взаимоблокировок СУБД` в ЦУП — количество дедлоков **во всех базах сервера СУБД**, где лежит
исследуемая база. Поэтому ненулевой счётчик при пустом анализе по конкретной базе означает, что дедлоки были в
**других базах того же сервера СУБД** (не в исследуемой). PerfMon: `SQLServer:Locks\Number of Deadlocks/sec`
целевое значение **0**.

## 4. Транзакции и версионирование (изоляция, MVCC, снимки)
- **PostgreSQL — MVCC**: читатели не блокируют писателей и наоборот; читатель видит **снимок** данных на
  момент начала запроса/транзакции. Уровень по умолчанию `READ COMMITTED`. Меньше блокировочных конфликтов на
  чтении, но «раздувание» таблиц мёртвыми версиями → нужен **VACUUM/autovacuum** (зона `1c-dba`).
- **MS SQL**: классически блокировочный (`READ COMMITTED` с разделяемыми блокировками на чтение → читатели
  блокируют писателей). Для 1С включают **версионирование** (`READ_COMMITTED_SNAPSHOT ON` / уровень
  снапшота), чтобы чтение шло по версии и не конфликтовало с записью. Конкретные настройки БД — НАСТРОЙ ПОД
  ПРОЕКТ, выполняет `1c-dba`.
- Уровни изоляции (по возрастанию строгости): `READ UNCOMMITTED` → `READ COMMITTED` → `REPEATABLE READ` →
  `SERIALIZABLE`. Чем строже — тем меньше аномалий (грязное/неповторяющееся чтение, фантомы), но больше
  блокировок и дедлоков. 1С в управляемом режиме держит низкий уровень изоляции СУБД, а целостность
  обеспечивает управляемыми блокировками — это и есть способ снизить блокировочную нагрузку.
- **Короткая транзакция — меньше блокировок.** Не открывать транзакцию заранее, не делать в ней долгих
  запросов/внешних вызовов, выносить подготовку данных до начала транзакции.

## 5. Пошаговый чек-лист расследования конкурентности
1. Включить ТЖ: `TLOCK`, `TTIMEOUT`, `TDEADLOCK` (+ `DBMSSQL`/`DBPOSTGRS` для запросов внутри транзакций).
2. Воспроизвести нагрузку (или дождаться окна проблемы).
3. Дедлоки → алгоритм §3 (порядок захвата: недостаточный уровень vs разный порядок); ожидания → алгоритм §2.
4. Найти **корневую** транзакцию-виновника (держит и не ждёт), а не жертв.
5. Гипотеза: режим блокировок? эскалация до таблицы? долгий запрос в транзакции? длинная транзакция?
   разный порядок захвата? недостаточный уровень?
6. Фикс: управляемый режим + точечные блокировки / упорядочить захват / поднять уровень блокировки заранее /
   ускорить запрос / укоротить транзакцию / отключить эскалацию (осторожно) / retry как страховка.
7. **Повторно замерить**: число TLOCK-ожиданий, TTIMEOUT, TDEADLOCK, `Number of Deadlocks/sec`, APDEX
   операции — снизилось? Иначе вернуться к п.4.

## 6. Типичные провалы
- Лечат **жертву** дедлока, а не корневую транзакцию.
- Отключают эскалацию (1211) без контроля памяти → ошибки нехватки памяти под блокировки.
- Делают только retry, не устранив причину → нагрузка и периодические сбои остаются.
- Считают, что версионирование MS SQL включено «по умолчанию» — проверь фактическую настройку БД.
- Долгий запрос/внешний вызов внутри транзакции → держит блокировки всё время выполнения.
- Путают управляемые (TLOCK, пространства) и автоматические (СУБД, эскалация) блокировки при анализе.
