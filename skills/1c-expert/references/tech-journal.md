# Технологический журнал 1С — настройка, сбор, анализ

Технологический журнал (ТЖ) — основной инструмент расследования проблем производительности и
работоспособности платформы 1С. Источник истины — **сам журнал**, а не предположение о том, что происходит.
Правило прода: собирай **только нужные события на время воспроизведения проблемы**; полный журнал быстро
забивает диск и замедляет систему. Версия платформы — НАСТРОЙ ПОД СВОЙ ПРОЕКТ (поля событий и набор событий
менялись по версиям; неуверенное — [проверить] по справке ИТС своей версии).

## 1. logcfg.xml — где лежит и что делает

Файл `logcfg.xml` настраивает ТЖ и генерацию дампов памяти. Лежит в каталоге конфигурационных файлов
платформы (`conf`), наличие не обязательно. Если файла нет — ТЖ (`<log>`) **выключен**, работает только
`<defaultlog>` (системные события уровня Error, 24 часа), и сохраняются **минимальные** дампы аварийного
завершения (Windows).

Расположение каталога conf — НАСТРОЙ ПОД ПРОЕКТ. Типично:
- Linux: `/opt/1cv8/conf/` или `/opt/1C/v8.3/x86_64/conf/`; журнал по умолчанию — `~/.1cv8/logs`.
- Windows: `C:\Program Files\1cv8\conf\`; журнал по умолчанию — `%LOCALAPPDATA%\1C\1cv8\logs`,
  дампы — `%LOCALAPPDATA%\1C\1cv8\dumps`.

Платформа подхватывает изменения `logcfg.xml` **на лету** (без перезапуска кластера), обычно в течение
~1 минуты. [проверить] точную задержку под свою версию.

### Минимальный каркас
```xml
<?xml version="1.0" encoding="UTF-8"?>
<config xmlns="http://v8.1c.ru/v8/tech-log">
  <log location="C:\LOGS\PERF" history="24">
    <event>
      <eq property="name" value="dbmssql"/>
    </event>
    <property name="all"/>
  </log>
  <dump location="C:\LOGS\DUMPS" create="1" type="2"/>
</config>
```
- `location` — каталог журнала (у `<log>`, `<dump>`, `<defaultlog>` каталоги ДОЛЖНЫ быть разными).
- `history` — сколько **часов** хранить; `0` — не удалять (тогда чистить сторонними средствами, иначе
  переполнение диска).
- `<property name="all"/>` — писать все свойства события. На проде сужай до нужных свойств.
- **Не настраивай > 20 элементов `<log>`** — существенно замедляет систему.

### Полезные атрибуты `<log>`
| Атрибут | Значения | Зачем |
|---|---|---|
| `history` | часы, `0`=без удаления | время жизни файлов |
| `compress` | `none`(умолч.)/`zip` | сжатие устаревших файлов при ротации |
| `format` | `text`(умолч.)/`json` | формат записи |
| `rotation` | `period`(умолч.)/`size` | схема ротации |
| `rotationperiod` | часы (умолч. 1) | период ротации по времени |
| `rotationsize` | МБ (умолч. 100) | размер файла для ротации по размеру |
| `placement` | `folders`(умолч.)/`plain` | подкаталоги по процессам или плоско |

## 2. Отбор событий: `<event>` и `<property>`

- Условия внутри одного `<event>` объединяются **по И**; разные `<event>` — **по ИЛИ**.
- Операторы сравнения: `eq` (=), `ne` (≠), `gt` (>), `ge` (≥), `lt` (<), `le` (≤), `like` (маска).
- `like`: `%` — 0+ любых символов, `_` — 1 любой, `[...]`/`[^...]` — класс символов, `\` — экранирование.
  Регистр в `like` не различается. **`like` медленнее** прочих сравнений — не злоупотребляй на проде.
- `<property name="ИМЯ">` пишет конкретное свойство; без вложенных `<event>` — пишется для всех событий, где
  присутствует. `<property name="all"/>` — все свойства.

**Два класса элементов (частая путаница):**
1. **Управляют формированием события** (без них событие не возникает в принципе): `<dump>`, `<leaks>`,
   `<mem>`, `<plansql>`, `<dbmslocks>`, `<ftextupd>`, `<system>`, `<query>`, `<scriptcircrefs>`,
   `<inputbystring>`. Пример: пока не включён `<plansql>`, отбор по плану запроса ничего не даст.
2. **Фильтруют уже сформированные данные**: `<event>`, `<property>` — могут только **уменьшить** объём.

Типовой отбор «производительность»: писать долгие запросы (> 1 c = `1 000 000` мкс) и блокировки:
```xml
<log location="C:\LOGS\PERF" history="48">
  <event><eq property="name" value="dbmssql"/><ge property="duration" value="1000000"/></event>
  <event><eq property="name" value="dbpostgrs"/><ge property="duration" value="1000000"/></event>
  <event><eq property="name" value="tlock"/></event>
  <event><eq property="name" value="tdeadlock"/></event>
  <event><eq property="name" value="ttimeout"/></event>
  <event><eq property="name" value="qerr"/></event>
  <event><eq property="name" value="excp"/></event>
  <property name="all"/>
</log>
```
ВНИМАНИЕ: `duration` в ТЖ — в **микросекундах**. Длительность события (первое число до `-` в строке) — тоже мкс.

## 3. Ключевые события и что из них извлекать

| Событие | Что это | Что извлекать при расследовании |
|---|---|---|
| `EXCP` | необработанные исключения (риск падения процесса) | контекст, текст ошибки; коррелировать с дампами |
| `EXCPCNTX` | операции, начавшиеся, но не завершённые к моменту сбоя | что выполнялось в момент падения |
| `TLOCK` | управление транзакционными блокировками (управляемый режим) | `Regions` (пространство), `Locks`, `WaitConnections` (кто ждал), `DBPID` |
| `TTIMEOUT` | превышено макс. время ожидания транзакционной блокировки | таймауты блокировок — деградация конкурентности |
| `TDEADLOCK` | взаимоблокировка в управляемом режиме | цикл ожидания, участники, пространства |
| `QERR` | ошибки компиляции запроса / ограничения RLS | проблемные запросы, RLS-конфликты |
| `SDBL` | запросы к модели БД 1С (язык SDBL) | `Func` (Commit/RollbackTransaction и др.), `Context`, длительность |
| `DBMSSQL` | SQL к MS SQL Server | `Sql` (текст), `planSQLText` (при `<plansql>`), `Context`, длительность, `Rows`/`RowsAffected` |
| `DBPOSTGRS` | SQL к PostgreSQL | то же для PostgreSQL |
| `CALL` | входящий серверный вызов | `Context`, длительность, `InBytes/OutBytes`, `Memory` |
| `SCALL` | исходящий серверный вызов | парная сторона CALL |
| `MEM` | рост памяти серверных процессов (ragent/rmngr/rphost) | утечки/всплески памяти |
| `PROC` | старт/останов/аварийное завершение процесса | рестарты rphost, аварии |
| `CONN` | установка/разрыв клиентского соединения | штормы переподключений |

Замечание из справки: `PROC, SCOM, EXCP, CONN, ADMIN` возникают редко и компактны; `SDBL, DBMSSQL,
DBPOSTGRS, DBORACLE, DB2` могут **резко раздуть** журнал — фильтруй по `duration`.

### Сбор планов запросов
```xml
<plansql/>
```
Включает свойство `planSQLText` у событий DBMSSQL/DBPOSTGRS — план как его построила СУБД. Стоит дорого, держи
только на время расследования конкретного запроса.

### Дампы аварийного завершения
```xml
<dump location="C:\LOGS\DUMPS" create="1" type="2"/>
```
`type="1"` — минидамп (стек), `type="2"` — полный дамп (вся память процесса, крупный). Имя дампа:
`rphost_<версия>_<offset>_<YYYYMMDDhhmmss>_<PID>.mdmp`. Группа после версии (`offset` в 0x-представлении
смещения вершины стека) **интерпретируется как тип ошибки** — пара дампов с **одинаковым offset** вероятно
вызвана одной проблемой. Группа `_YYYYMMDDhhmmss_` — **время создания дампа**, последняя группа — **PID**.
Ручной дамп по работающему процессу (Windows): `procdump.exe -ma rphost <dump_имя>` или по PID.

## 4. Анализ ТЖ в Linux (grep/awk/perl) — рабочие конвейеры

ТЖ — текстовый, события могут быть **многострочными** (DBMSSQL с текстом запроса + контекст). Поэтому простой
построчный grep по событию с контекстом неверен — нужно сначала «склеить» событие в одну строку.

Базовые приёмы (из директории с логами, файлы вида `rphost_*/*.log`):
```bash
# топ-10 файлов с "Lock request timeout"
grep -r 'Lock request timeout' | awk -F: '{print $1}' | sort | uniq -c | sort -r | head
# живой поток исключений в работающих rphost
tail -f rphost*/*.log | grep ',EXCP,'
# конкретное поле от событий EXCP
grep -oPh ',EXCP,\K(ClientID=\d+)' rphost*/*.log
```

**Склейка многострочного события в одну строку** (ключевой приём — строка события начинается с `mm:ss.доли`):
```bash
cat rphost*/*.log \
 | perl -n -e 'if (/^\d\d:\d\d\.\d+/){$e=~s/\n/<line>/g; print $e."\n"; $e="";} $e.=$_; END{print $e."\n"};' \
 | perl -pe 's/\xef\xbb\xbf//g' \    # убрать BOM
 | grep "DBMSSQL.*SessionID=2049" \
 | sed 's/<line>/\n/g' | less        # обратно в многострочный вид
```

### Топ-5 наиболее длительных транзакций (по контексту)
```bash
cat rphost*/*.log \
 | perl -n -e 'if (/^\d\d:\d\d\.\d+/){$e=~s/.\n/<line>/g; print $e."\n"; $e="";} $e.=$_; END{print $e};' \
 | perl -pe 's/\xef\xbb\xbf//g' \
 | grep -P "SDBL.*Func=(Commit|Rollback)Transaction.*Context" \
 | perl -pe 's/^\d+:\d+.\d+-//g' \
 | perl -pe 's/,SDBL,.*Context.*<line>[ \t]+/,Context=/g' \
 | perl -pe 's/,SDBL,.*Context=/,Context=/g' | sed 's/<line>//g' \
 | awk -F',Context=' '{sum[$2]+=$1; count[$2]+=1;} END {for(i in sum){print sum[i]" "sum[i]/count[i]" "count[i]" "i}}' \
 | sort -rnb | head -n 5
```
Вывод столбцами: **суммарная длительность · средняя · количество · контекст** — даёт и долгие-редкие, и
короткие-но-частые (по суммарной длительности).

### Топ-5 длительных запросов MS SQL (со сворачиванием текста)
Тексты запросов нормализуют, чтобы сгруппировать одинаковые по структуре: убрать GUID, имена временных
таблиц (`#tt17`), числа:
```bash
... | grep -P "DBMSSQL.*Sql" | grep -v 'Context' \
 | perl -pe 's/,DBMSSQL,.*Sql=/,Sql=/g' \
 | perl -pe 's/\w+-\w+-\w+-\w+-\w+/{GUID}/g' \
 | perl -pe 's/\(\d+\)/({NUM})/g' \
 | perl -pe 's/tt\d+/{TempTable}/g' \
 | awk -F',Sql=' '{sum[$2]+=$1; count[$2]+=1;} END {for(i in sum){print sum[i]" "sum[i]/count[i]" "count[i]" "i}}' \
 | sort -rnb | head -n 5
```

### Топ-5 пространств с ожиданиями на управляемых блокировках
Ожидание было, если поле `WaitConnections` непустое:
```bash
... | grep -P ",TLOCK,.*WaitConnections=\d+,Context" \
 | perl -pe 's/^\d+:\d+.\d+-//g' | grep -P 'WaitConnections=\d+' \
 | perl -pe 's/,Locks=.*$//g' | perl -pe 's/,TLOCK,.*Regions=/,Regions=/g' | sed 's/<line>//g' \
 | awk -F',Regions=' '{sum[$2]+=$1; count[$2]+=1;} END {for(i in sum){printf "%d %d %d %s\n",sum[i],sum[i]/count[i],count[i],i}}' \
 | sort -rnb | head -n 5
```

### Оптимизация и многосерверный сбор
- Отбирай нужный объём **как можно раньше** в конвейере; не гоняй миллионы строк; сортировки на сотнях тысяч
  строк дороги. Засекай `time <конвейер>`; при отладке добавь `head` после первого `|`.
- Когда серверов много:
  `for i in $(seq 1 100); do ssh server$i 'tail -n 1000 /var/log/.../FULL/rphost*/*.log' | grep ",EXCP,"; done`
  — точечно (последние N строк), чтобы не читать терабайты на каждом узле.
- Старые журналы складывай в `zip` и не удаляй сразу — нужны для разбора инцидентов задним числом.

## 5. ЦУП / КИП (графический путь)

Для тех, кто не разбирает ТЖ руками: **ЦУП** (Центр управления производительностью) и **КИП** (Корпоративный
инструментальный пакет) собирают и агрегируют ТЖ + счётчики, показывают долгие запросы, ожидания на
блокировках, дедлоки, APDEX ключевых операций. ЦУП — серверный мониторинг прода: настраивается сбор ТЖ через
встроенную обработку, показатели по информационным базам, динамика APDEX, число взаимоблокировок СУБД.
Поле `Число взаимоблокировок СУБД` в ЦУП — количество взаимоблокировок **во всех базах сервера СУБД**, где
лежит исследуемая база (поэтому ненулевой счётчик при пустом анализе по конкретной базе значит, что дедлоки
были в **других** базах того же сервера). Конкретные версии/настройки ЦУП/КИП — [проверить] под свой проект.

## 6. Типичные провалы и контрприёмы
- **Полный журнал на проде** → диск переполнен, система тормозит. Контр: фильтр по событиям и `duration`,
  ограниченный `history`, отдельный быстрый диск под логи.
- **grep по DBMSSQL с контекстом без склейки** → битые/обрезанные события. Контр: perl-склейка в одну строку.
- **Игнор BOM** → не матчатся регулярки. Контр: `perl -pe 's/\xef\xbb\xbf//g'`.
- **Сортировка top-5 только по средней длительности** → теряешь частые короткие. Контр: смотри и суммарную.
- **`<plansql>`/`<system>` оставлены включёнными** → раздувание журнала. Контр: включать точечно, снимать после.
- **Вывод о причине без замера** → лечение симптома. Контр: сначала данные ТЖ, потом гипотеза (см.
  `investigation-methodology.md`).
