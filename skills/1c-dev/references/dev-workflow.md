# Рабочий цикл реализации: от тех-проекта до передачи кода

Контекст: артефакты приходят УЖЕ описанными (аналитик/архитектор подготовили ЧТЗ и тех-проект, тех-лид дожал
технические вопросы). Разработчик отдаёт их AI. Дальше — этот цикл.

## Разделение ролей в фазе разработки
- **AI (ты):** декомпозиция тех-проекта на атомарные задачи → инструкции разработчику по НЕ-коду → написание кода →
  проверка и тесты → финал prod-ready → передача.
- **Разработчик (человек):** создаёт в IDE (EDT или Конфигуратор) метаданные, формы, макеты, СКД, роли, реквизиты,
  подписки — всё, что НЕ код; применяет код AI; валидирует в IDE и грузит в ИБ; владелец осознанного принятия.
  Структуру меняет только он через IDE — это IDE-гейт.

## Шаг 0. Понять вход и сверить по коду (не домысливать)
- Прочитать ЧТЗ (что/зачем, критерии приёмки) и тех-проект (слои, контракты/DTO, reuse, НФТ).
- Сверить по РЕАЛЬНОМУ коду через MCP (`find_object`→`search`→`read_module`): что уже есть, как устроена типовая,
  какие объекты/модули затронуты, нет ли готового в БСП/расширении (reuse-first).
- Техническая неопределённость/пробел в тех-проекте → вернуть тех-лиду/архитектору КОНКРЕТНЫЙ вопрос, не домысливать.

## Шаг 1. Декомпозиция на атомарные задачи
Разбить тех-проект на УПОРЯДОЧЕННЫЙ список атомов с зависимостями. Каждый атом — ОДНО конкретное проверяемое
действие с критерием готовности. Пометить исполнителя:
- **[Разработчик/IDE]** — создание/изменение метаданных, форм, макетов, СКД, ролей, реквизитов, подписок (не-код).
- **[AI/код]** — написание/правка `*.bsl` (логика).
Порядок: сначала [Разработчик] создаёт объекты, на которые опирается код, затем [AI] пишет код.

## Шаг 2. Инструкции разработчику по НЕ-коду (до написания кода)
Выдать точный список «создай в IDE», потому что код к этим объектам привяжется и без них не скомпилируется:
тип объекта; имя с префиксом расширения; реквизиты (имя, тип, длина/точность, индексируемость); формы и реквизиты
форм; регистры (измерения/ресурсы/периодичность); роли/права/RLS; подписки на события; место в дереве метаданных.

## Шаг 3. Реализация кода
Писать BSL по атомам, сверяя каждый по MCP (сигнатуры/объекты — не по памяти): конвенции проекта
(`conventions-template.md`); reuse-first БСП (`bsp-reuse.md`); производительность/кэш (`performance.md`);
стандарты (`code-standards.md`); prod-ready.
- **EDT-контур:** код кладётся в `*.bsl` расширения в репозитории.
- **Конфигуратор-контур:** код оформляется как готовый ФРАГМЕНТ для вставки в Конфигуратор (модуль/область/место);
  AI не коммитит, переносит разработчик.

## Шаг 4. Проверка и тесты (до передачи)
- **BSL Language Server:** синтаксис + стандарты ИТС. `Error` — блокер; `Vulnerability`/`Security Hotspot` —
  обязательны к разбору; `Code smell` — рекомендация. До 3 циклов автоисправления.
- **Сверка с тех-проектом и критериями приёмки ЧТЗ:** всё ли реализовано, нет ли расхождений с контрактами/DTO.
- **Собственные проверки сложных мест** (без запуска платформы): разбор граничных случаев; проверка логики запроса
  на МОДЕЛЬНЫХ данных; пошаговый разбор алгоритма; транзакции/порядок блокировок. Где есть тест-фреймворк
  (**xUnitFor1C / Vanessa**) — подготовить исполняемые тест-сценарии и отдать разработчику на прогон.
  **Честно:** BSL не исполняется в среде AI — пишутся тесты и проверяется статически/логически; прогон в ИБ — за разработчиком.

## Шаг 5. Финал и передача разработчику
Когда все проверки зелёные и версия production-ready — передать ПАКЕТ:
1. Что создать в IDE — финальный чек-лист объектов.
2. Код — куда вставить (модуль/область/место) или какие `*.bsl` изменены.
3. Как проверить в IDE: проверка конфигурации/синтаксический контроль (Конфигуратор) или валидация EDT.
4. Какие тесты прогнать и ожидаемый результат по критериям приёмки.
5. Как загрузить в ИБ.

## Шаблоны
**Атомарная задача:** `ID · название · исполнитель [Разработчик/IDE]|[AI/код] · зависит от · действие · критерий
готовности · ссылки`.
**Инструкция «создать в IDE»:** `Тип · Имя (префикс) · Реквизиты [имя:тип(длина/точн.), индекс] · Формы/реквизиты
форм · Регистры [измерения/ресурсы] · Роли/права/RLS · Подписки · место в дереве`.
**Пакет передачи:** `чек-лист объектов IDE · модули и места вставки · проверки IDE · тесты+ожидание · загрузка в ИБ`.
