# Правила разработки 1С с Claude Code (наследуются проектом)

Org-agnostic правила. Настрой версии и конвенции под свою конфигурацию (отмечено «НАСТРОЙ»).

## Версионный стек — НАСТРОЙ ПОД СВОЙ ПРОЕКТ
Зафиксируй и строго соблюдай: версия платформы 1С, **режим совместимости**, версия конфигурации, версии библиотек
(БСП и др.). Принцип: справку брать по версии платформы, но НЕ предлагать API, недоступный в твоём режиме
совместимости. Справка БСП — по реальной БСП своей версии.

## Главное правило: спроси инструмент, не угадывай
1С-факты и сигнатуры — только по РЕАЛЬНОМУ коду (MCP) и справке платформы, не по памяти. После правки — прогнать
BSL Language Server. Механизм без сверки помечать **[проверить]**, не «верно».

## Гейт структурных правок (человек/AI)
Структурные правки метаданных (объекты, формы, макеты, СКД) AI выполняет ТОЛЬКО инструментами `onec_metadata`
(`bin/1c-meta`) в пределах реестра покрытия `onec_metadata/catalog.py`, с обязательной гарантией: загрузка в
ТЕСТ-базу платформой + повторная выгрузка + round-trip diff строго целевого изменения (см. skill `1c-metadata`).
Всё вне реестра (или без доступной тест-базы) — создаёт человек в IDE (**1C:EDT** или **Конфигуратор**), как и
раньше. Правки, не прошедшие round-trip, к прод-контуру не допускаются. AI пишет и проверяет КОД (`*.bsl`):
в EDT-контуре правит исходники в репозитории, в Конфигуратор-контуре выдаёт фрагменты для вставки.

## Слои контура — НАСТРОЙ
Контур = конфигурация (основа) + расширение. Искать функционал в ОБОИХ слоях. Правки — в расширение (типовую
read-only). Опиши свои слои/префиксы/точки расширения в `skills/1c-dev/references/conventions-template.md`.

## Скиллы и рабочий цикл
- `1c-analyst` — анализ/архитектура: разбор задачи, ЧТЗ (поведение, не реализация), /clarify.
- `1c-dev` — разработка: рабочий цикл (вход → атомарные задачи [Разработчик/IDE]|[AI/код] → инструкции по не-коду →
  код → проверки/тесты → передача пакетом). Reuse-first БСП, производительность, стандарты `v8-code-style`.
- `1c-admin-devops` — администрирование + DevOps: кластер/`rac`/лицензии, публикация на веб-сервере, обновление
  конфигураций/расширений, CI/CD, контейнеры, мониторинг и бэкап, Linux-сервер. Применять при установке,
  эксплуатации, публикации, обновлении, доставке 1С — сначала снять данные (ТЖ/счётчики/вывод `rac`), потом вывод.
- `1c-dba` — DBA под 1С: тюнинг PostgreSQL/MS SQL, регламентное обслуживание (VACUUM/REINDEX/FREEZE), блокировки на
  уровне СУБД, бэкап с проверкой восстановимости, HA/масштабирование. Применять, когда база тормозит/блокируется на
  уровне СУБД или нужен регламент — сначала измерь (`pg_stat_*`, `EXPLAIN ANALYZE`), потом меняй.
- `1c-expert` — эксперт по техн. вопросам и highload: ТЖ (`logcfg.xml`), план запроса, блокировки/дедлоки, APDEX и
  замеры, кластер и нагрузочное тестирование. Применять для расследования «почему медленно/падает/не масштабируется» —
  диагноз по измерениям; реализацию фикса ведут `1c-dev` (код) и `1c-dba` (СУБД/инфраструктура).

## Безопасность
Токены — только в env/keychain, НИКОГДА в чат/репозиторий. Конфиденциальные/персональные данные в AI не передавать.
