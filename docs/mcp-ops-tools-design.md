# MCP-инструменты для эксплуатационных ролей — ПРОЕКТ (design-only)

> **Дисклеймер.** Это **план/дизайн**, а не реализация. Кода нет, ни один из описанных инструментов в репозитории
> не реализован. Документ задаёт контракт и принципы будущих MCP-серверов для ролей `1c-admin-devops` / `1c-dba` /
> `1c-expert`, в том же духе, что уже существующие `mcp/onec_mcp.py` (чтение кода 1С) и `mcp/bsl_ls_mcp.py` (мост
> над BSL Language Server): тонкая аудируемая обёртка над реальным инструментом, **только чтение**, секреты через
> `env`, кроссплатформенно. Статус: `tech_journal_parse` (§1) — ✅ **реализован** (`mcp/onec_ops_mcp.py`);
> Zabbix perf-слой (§5) — ✅ **реализован 2026-07**; остальные (§2–4) — **design-only, не реализовано**.

## Зачем и общие принципы
Эксплуатационные скиллы требуют «сначала снять данные — потом вывод». Сейчас данные снимаются вручную (вывод `rac`,
ТЖ, `pg_stat_*`). Эти MCP-инструменты дали бы AI-ассистенту **управляемый read-only доступ к измерениям**, чтобы он
работал по факту, а не по памяти — оставаясь в тех же рамках безопасности, что и существующие MCP.

Сквозные правила (применяются ко ВСЕМ инструментам ниже):
- **Только чтение.** Никаких изменений в кластере/СУБД/ФС. Нет `pg_terminate_backend`, нет правок `postgresql.conf`,
  нет рестартов сервисов. Снятие сеансов/блокировок и любые деструктивные операции остаются за человеком.
- **Секреты только через `env`** (`PG*`, путь к `rac`, хост/порт RAS, токен) или `~/.pgpass` (chmod 0600). Никаких
  паролей в аргументах инструмента, в логах MCP или в выводе. В примерах — плейсхолдеры.
- **Персональные данные не наружу.** Выводы ТЖ и планы запросов могут содержать значения параметров/представления —
  инструмент обезличивает или усекает поля со значениями по умолчанию; сырой текст с ПДн в ответ не помещается.
- **Кроссплатформенность и аудируемость** — `os.path`, бинарь из `env`/PATH, разумные таймауты, понятная ошибка при
  отсутствии зависимости (как `bsl_ls_mcp.py` при отсутствии `BSL_JAR`).
- **Зависимости — внешние бинарники**, которые инструмент только вызывает: `rac` (утилита администрирования
  кластера), `psql` (клиент PostgreSQL). MCP их не устанавливает.

## 1. `tech_journal_parse` — разбор технологического журнала
- **Назначение:** распарсить файлы ТЖ (`*.log` из каталога `logcfg.xml`) по событиям и фильтрам, вернуть
  агрегаты/выборки для расследования (роль `1c-expert`, частично `1c-admin-devops`). Заменяет ручные grep/awk-конвейеры.
- **Вход (схема, реализовано):**
  - `roots` — путь, СПИСОК путей или список `{"path","label"}`: каталог ТЖ ИЛИ **группа каталогов разных рабочих
    серверов кластера / нескольких кластеров** (расследование на уровне всей компании). `label` = метка
    сервера/кластера для атрибуции.
  - `events: list[str] = None` — фильтр по событиям (EXCP, TLOCK, TDEADLOCK, QERR, DBMSSQL, DBPOSTGRS, CALL …); пусто = все.
  - `since: str = ""`, `until: str = ""` — окно времени (`YYYY-MM-DD HH:MM`/`YYMMDDHHMM`).
  - `min_duration: int = 0` — порог длительности события (поле ТЖ).
  - `top: int = 50` — сколько самых длительных событий вернуть.
- **Выход:** JSON — `summary {roots, files, events_total, by_event, by_source}` + `events[]` (топ по длительности,
  каждое тегировано **source/process/file/timestamp/props**) + `warnings[]`. Длинные значения (Sql/Context) усекаются.
- **Безопасность:** read-only по файлам; ПДн в тексте запросов усекаются (не отправлять сырой вывод во внешние LLM);
  пути — из доверенных аргументов.
- **Зависимости:** чистый Python (stdlib).
- **Статус:** ✅ **РЕАЛИЗОВАНО** — `mcp/onec_ops_mcp.py` (`tech_journal_parse`), тесты `tests/test_onec_ops_mcp.py`
  (4, зелёные), зарегистрирован в `core/mcp/servers.json` как `onec-ops` (env `ONEC_OPS_MCP`). Мультикорневой обход
  «по всей компании» с атрибуцией источника.

## 2. `cluster_metrics` — метрики кластера через `rac`
- **Назначение:** снять состояние кластера 1С через `rac` — список кластеров, рабочих процессов (`rphost`),
  сеансов, соединений, нагрузки (роль `1c-admin-devops`, частично `1c-expert`).
- **Вход (схема):**
  - `what: str = "sessions"` — что снять: `clusters` | `processes` | `sessions` | `connections` | `locks`.
  - `cluster_id: str = ""` — UUID кластера (если несколько; иначе берётся единственный).
  - `ras_host: str = ""`, `ras_port: int = 0` — адрес RAS (или `env RAS_HOST`/`RAS_PORT`).
  - `limit: int = 100` — ограничение размера выборки.
- **Выход:** JSON — нормализованный разбор табличного вывода `rac` (например, для `processes`:
  `[{host, port, pid, memory, connections, "available": bool}]`; для `sessions`: app-id, user, duration, db-proc-took).
- **Безопасность:** **только чтение** (`rac ... list`), никаких `terminate`/`disconnect`. Учётка администратора
  кластера и пароль — через `env`/`.pgpass`-аналог, не в аргументах. Имена пользователей в сеансах могут быть ПДн —
  обезличивать при выводе наружу.
- **Зависимости:** бинарь `rac` (путь из `env RAC_BIN`/PATH), доступный RAS.
- **Статус:** design-only, не реализовано.

## 3. `pg_health` — снимок здоровья PostgreSQL
- **Назначение:** один снимок ключевых индикаторов СУБД под 1С (роль `1c-dba`): активность, блокировки,
  распухание (bloat), приближение к wraparound, cache hit ratio.
- **Вход (схема):**
  - `section: str = "all"` — `activity` (`pg_stat_activity`: длинные/idle-in-transaction) | `locks` (`pg_locks`:
    кто кого блокирует) | `bloat` (оценка распухания таблиц/индексов) | `freeze` (`age(datfrozenxid)` по БД) |
    `cache` (cache hit ratio) | `all`.
  - `dbname: str = ""` — имя БД (или `env PGDATABASE`).
  - `long_query_sec: int = 60` — порог «длинной» транзакции/запроса.
  - `top: int = 20`.
- **Выход:** JSON по секциям — например `{ activity: [{pid, state, wait_event, duration_sec, query_усечённый}],
  locks: [{blocked_pid, blocking_pid, relation}], bloat: [...], freeze: [{datname, xid_age}], cache: {ratio} }`.
- **Безопасность:** **только `SELECT`** к системным представлениям; никаких `pg_terminate_backend`/`VACUUM`/DDL.
  Подключение по `~/.pgpass`/`env` (`PGHOST`/`PGUSER`/`PGDATABASE`), пароль не в аргументах. Текст запросов
  усекается, значения параметров маскируются (ПДн).
- **Зависимости:** `psql` (или libpq-драйвер) — путь из `env`/PATH.
- **Статус:** design-only, не реализовано.

## 4. `log_grep_perf` — выборка медленного из ТЖ/логов
- **Назначение:** целевая выборка «медленные запросы / ожидания на блокировках / дедлоки» из ТЖ (и, опционально,
  из лога PostgreSQL `log_min_duration_statement`) — быстрый ответ на «что тормозило в окне X» (роли `1c-expert`,
  `1c-dba`).
- **Вход (схема):**
  - `source: str = "tj"` — `tj` (технологический журнал) | `pglog` (лог PostgreSQL).
  - `path: str` — каталог/файл (или соответствующий `env`).
  - `kind: str = "slow_query"` — `slow_query` | `lock_wait` | `deadlock`.
  - `min_duration_ms: int = 1000`, `since/until: str`, `top: int = 20`.
- **Выход:** JSON — топ-N событий с длительностью, усечённым текстом запроса/контекстом, и (для `lock_wait`/
  `deadlock`) участниками ожидания/цикла.
- **Безопасность:** read-only по логам; ПДн в тексте запросов маскируются; путь — из доверенного `env`/аргумента.
- **Зависимости:** чистый Python (+`rg` как ускоритель, опционально).
- **Статус:** design-only, не реализовано.

## Что НЕ входит в дизайн (намеренно)
- Любые **записи/изменения**: снятие сеансов, kill-бэкендов, правки конфигов, рестарты, VACUUM/REINDEX, обновление
  ИБ. Это остаётся ручными операциями человека под бэкап и окно (как требуют SKILL.md ролей).
- Передача **сырых дампов/выгрузок ИБ** и персональных данных во внешние сервисы/LLM.
- Хранение секретов в репозитории или в аргументах инструментов.

## Не изобретать велосипед — готовые опенсорс-реализации (проверено, см. `ops-tools-catalog.md`)
Перед реализацией свериться с готовым:
- **`pg_health` уже существует:** `crystaldba/postgres-mcp` (Python, MIT) — health-анализ (index/connection/
  buffer-cache/vacuum/replication), index-tuning, EXPLAIN, hypothetical indexes, безопасный настраиваемый
  **read-only**. Вариант: **подключить готовый** для роли `1c-dba` вместо своего `pg_health`.
- **Ядро `tech_journal_parse`:** `medigor/tech-log-parser` (Rust, MIT, ТЖ→JSON) или `akpaevj/OneSTools.TechLog`
  (C#, MIT, нормализация + live-режим). Обернуть их вывод MCP-инструментом, не писать парсер ТЖ с нуля.
- **`cluster_metrics` / снятие живых данных ИБ:** образец — `feenlace/mcp-1c` (Go, MIT, read-only: метаданные +
  `execute_query` SELECT + `get_event_log` через HTTP-сервис 1С). Метрики кластера через `rac` уже инкапсулированы
  в `LazarenkoA/prometheus_1C_exporter` (MPL-2.0) — брать список метрик/парсинг вывода `rac` оттуда.
- **Изоляция/доставка MCP:** `docker/mcp-gateway` (официальный, MIT) — запускать чужие MCP в контейнерах.
- ⚠️ НЕ брать официальный `@modelcontextprotocol/server-postgres` — заархивирован, SQL-инъекция в обход read-only.
  Урок: read-only через строковую сборку SQL ненадёжен — нужен безопасный парсинг (как у crystaldba).

## 5. Zabbix perf-слой — ✅ РЕАЛИЗОВАНО (2026-07)
- **Назначение:** анализ производительности, когда данные есть в Zabbix (кейс КЗ): дашборд/хосты → элементы
  данных → история за окно → каноничные метрики → **находки, ранжированные по вкладу в производительность**
  (самое значимое первым) + активные проблемы. Дашборд может меняться — извлечение item'ов не завязано
  на типы виджетов (fields type=4 item, type=6 graph→items).
- **Состав** (`mcp/onec_ops_mcp.py`, тесты `tests/test_onec_ops_zabbix_perf.py`):
  - `zabbix_api(method, params)` — универсальный вызов с **regex-whitelist read-only** (`*.get`,
    `apiinfo.version`, `*.export`); остальным методам — отказ без обращения к серверу (идея mpeirone, GPL → только идея).
  - `zabbix_dashboard_items` / `zabbix_host_items` — items с любого дашборда/хостов (wildcards).
  - `zabbix_item_stats` — history.get (окно ≤48ч) / trend.get (глубже), n/min/avg/max/p95/last (идея grafana-zabbix).
  - `zbx_classify_item` — item → каноничная метрика (Windows perf_counter / Linux-агент / MSSQL-шаблоны;
    cpu_util, cpu_queue, mem_available_pct, swap_pct, PLE, buffer_cache_hit, deadlocks, disk_queue/util/await, net_errors).
  - `rank_findings` — **impact = вес severity × кратность превышения порога × доля времени за порогом**;
    неклассифицированные items не теряются (секция unclassified).
  - `zabbix_perf_report` + `render_perf_report` — сквозной отчёт + markdown; MCP-тулы
    `zabbix_api_tool`/`zabbix_dashboard_items_tool`/`zabbix_perf_report_tool`; CLI `scripts/zabbix_perf.py`.
- **Аутентификация:** токен заголовком `Authorization: Bearer` (Zabbix 6.4+; payload-`auth` deprecated в 7.0,
  удалён в 7.2 — legacy-fallback оставлен). Токен: **read-only пользователь → Профиль → API tokens** → env `ZABBIX_TOKEN`.
- **Безопасность:** строго read-only (whitelist), секреты из env, ПДн в именах хостов наружу не отправлять.
