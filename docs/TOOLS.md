# Каталог инструментов

Что входит в bsl-ai-toolkit, что каждый инструмент делает и как им пользоваться. Так виден объём работы и можно взять
только нужное. Подробности по каждому пункту — в отдельных документах по ссылкам.

---

## Навыки (скиллы) — по ролям
Каждый навык это набор правил и рабочих приёмов для своей роли. Агент подхватывает нужный по описанию задачи.
Лежат в `core/skills/`, формат переносим между агентами.

| Навык | Для кого | Что помогает делать |
|---|---|---|
| `1c-dev` | разработчик | писать и ревьюить код BSL, рабочий цикл реализации, стандарты, повторное использование БСП |
| `1c-analyst` | аналитик, архитектор | разбор задачи, ЧТЗ (поведение и критерии приёмки), карта баз и связей, оценка влияния |
| `1c-metadata` | разработчик | безопасные структурные правки метаданных через движок (реквизиты, поля СКД, колонки печатных форм) |
| `1c-admin-devops` | администратор | кластер и лицензии, публикация, обновление, контейнеры, мониторинг и резервные копии |
| `1c-dba` | администратор БД | настройка PostgreSQL и MS SQL, регламент, блокировки, восстановление, отказоустойчивость |
| `1c-expert` | эксперт по нагрузке | технологический журнал, план запроса, блокировки, APDEX, нагрузочное тестирование |

## Движок правки метаданных — `onec_metadata` (+ `bin/1c-meta`)
Точная правка метаданных на уровне файлов, без ручного редактирования XML. Работает с двумя форматами: выгрузка
Конфигуратора (XML) и 1C:EDT (`.mdo`/`.dcs`). Правка проверяется через round-trip: платформа загружает изменение в
тестовую базу и выгружает обратно, разница должна совпасть ровно с целевой. Что уже поддержано — карта покрытия в
`onec_metadata/README.md` (реквизиты, табличные части, измерения и ресурсы регистров, значения перечислений, поля и
запросы СКД, колонки печатных форм, состав подсистем, права ролей).
- Через агента — навык `1c-metadata`.
- Из командной строки — `bin/1c-meta` (определить формат, добавить реквизит, поле СКД и т.д.).
- Подробнее — `onec_metadata/README.md`, устройство и предохранители — `docs/architecture-review-2026-07.md`.

## Инструменты доступа (MCP-серверы)
Профиль всех серверов — `core/mcp/servers.json`, пути и доступы задаются через переменные окружения (`.env.example`).

- **`onec-code`** — поиск по реальному коду конфигурации и расширения (по локальным клонам, оба слоя контура).
  Основа правила «сверься с кодом, не угадывай». Можно поднять общим на команду по сети (см. раздел «Общий сервер»).
- **`bsl-ls`** — диагностики BSL Language Server после каждой правки. Свободный, идёт в комплекте. Скачивание
  нужных `jar` — `scripts/detect_tools.py --install`.
- **`onec-ops`** — чтение данных эксплуатации: технологический журнал, журнал регистрации, расчёт APDEX, метрики из
  Zabbix и Prometheus. Только чтение. Подробнее — `docs/ops-tools-catalog.md`, `docs/mcp-ops-tools-design.md`.
- **`onec-data`** — данные живой информационной базы, только чтение, под именем исследуемого пользователя (права
  соблюдаются честно, персональные данные маскируются). Работает через OData и служебный сервис `ai_debug`. Нужна
  публикация сервисов администратором. Подробнее — `docs/data-access-architecture.md`.

## Расширение `ai_debug`
Небольшое расширение 1С для сервера `onec-data`: HTTP-сервис, который выполняет запрос или отчёт под правами
пользователя. Обычный режим работает от имени пользователя, привилегированный закрыт отдельной настройкой и по
умолчанию запрещён. Исходники — `extensions/ai_debug/`, описание — там же в `README.md`.

## Встроенные проверки и хуки
- **`scripts/bsl_guard.py`** — находит «запрос в цикле» (частая причина медленной работы). Возвращает ошибку, если
  нашёл. Работает в pre-commit и в конвейере применения правок.
- **Git-хуки** (`scripts/git-hooks/`, ставит `scripts/install_git_hooks.py`): `commit-msg` держит сообщения чистыми,
  `pre-commit` запускает проверку на запрос в цикле.
- Дополнительно движок метаданных защищает от поломки схемы компоновки данных и от случайного сдвига свойств.

## Коннекторы к смежным системам
- **Jira и Confluence** — `scripts/atlassian.py`: прочитать задачу, поиск по JQL, страницы и дерево вики, публикация
  из Markdown в страницу. Домен и токен через переменные окружения, ничего в коде менять не нужно.
- **Zabbix** — `scripts/apdex_to_zabbix.py` отправляет посчитанный APDEX в Zabbix как значения (`1c.apdex[...]`);
  `scripts/zabbix_perf.py` собирает отчёт по производительности.
- Свой коннектор добавляется тонким скриптом в `scripts/` или инструментом в сервере `onec-ops`. Секреты — только
  через переменные окружения.

## Общий сервер на команду
`server/` — способ поднять `onec-code`, `onec-ops` и `onec-data` как общие сервисы по HTTP за авторизацией (Caddy),
чтобы вся команда искала по коду централизованно. Разворачивается через `docker-compose`. Клиент переключается между
локальным и общим режимом командой `scripts/switch_source.py`; токен подключается `scripts/set_token.*`. Подробнее —
`docs/mcp-deploy-and-use.md`, `docs/deploy-center.md`.

## Установка, обновление, проверка
- **`onboard/onboard.sh`** (macOS, Linux) и **`onboard/onboard.ps1`** (Windows) — первичная настройка: навыки,
  сборка конфигов, профиль, хуки, самопроверка.
- **`build.sh`** / **`build.ps1`** — собрать конфиги агентов из `core/` (правила, навыки, профиль инструментов).
- **`scripts/doctor.py`** — проверка окружения: чего не хватает из инструментов и доступов.
- **`scripts/detect_tools.py --install`** — найти платформу и скачать свободные инструменты BSL, прописать переменные.
- **`scripts/update.sh`** / **`update.ps1`** — обновиться после первой установки (подтянуть навыки и правила).

## Проверка и применение на базе
Инструменты, которые доказывают поведение на реальной базе и снимают зависимости от администратора.

- **`scripts/onec_verify.py`** — одна команда от исходников до вердикта по лестнице тестирования: `check` (синтаксический
  контроль платформой) и `smoke` (серверный BSL-сценарий на реальных данных). Транспорт выбирается сам: без `--host`
  платформа берётся на этой машине, с `--host <алиас ssh>` — на сервере рядом с кластером.
- **Автономный сервер** (`onec_metadata/apply/standalone.py`) — поднимает HTTP-доступ к файловой базе (веб-клиент,
  OData, HTTP-сервисы) без веб-сервера и без публикации администратором. Позволяет проверить слой данных `onec-data`
  на файловой копии сразу, а публикацию боевой базы просить потом.
- **Предполёт «защиты от опасных действий»** (`onec_metadata/apply/protection.py`) — заранее и без подключения к базе
  проверяет, не подвесит ли батч-сеанс модальное окно защиты (реальный случай: 55 минут ложного зависания). Проверка
  локальная по файлу настроек платформы.
- **Готовность удалённого хоста** (`onec_metadata/apply/remote.py`) — набор проверок только на чтение перед первым
  прогоном на удалённом сервере, ловит типовые проблемы разом.
- **Сборка и разбор обработок и расширений** (`onec_metadata/apply/external.py`) — внешние обработки и отчёты
  (`.epf`/`.erf`) в обе стороны исходники ↔ бинарник, а также расширение как `.cfe`. Через пакетный режим Конфигуратора
  на машине с платформой. Это же используется при подготовке релизов (см. `docs/RELEASING.md`).

Список действий, которые может сделать только человек (публикация базы, снятие флага защиты, установка платформы),
собран в `docs/setup-actions-required.md` — инструменты сами дают на него ссылку, когда упираются.

## Тестирование
Больше 570 автотестов на движок метаданных, проверки, сервер данных, автономный сервер и удалённые прогоны. Подход
«доказать поведение, а не поверить на глаз» описан в `docs/testing-ladder.md`. Запуск:
`uv run --with mcp --with pytest --with lxml --with openpyxl --with xlrd pytest tests/ -q`.
