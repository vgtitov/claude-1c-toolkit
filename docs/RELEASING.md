# Как выпускать релизы

Релизы нужны, чтобы потребители пинили стабильную версию ядра, а собранные бинарные артефакты 1С (расширение
`ai_debug`) раздавались готовыми, а не собирались каждым вручную.

## Версии (SemVer)
Версия вида `vX.Y.Z`:
- **MAJOR (X)** — несовместимое изменение: переименование, смена структуры `core/`, ломающее правило, несовместимый
  формат движка метаданных.
- **MINOR (Y)** — новый навык, новый инструмент/MCP, новое покрытие метаданных, новый адаптер агента. Совместимо.
- **PATCH (Z)** — исправления, уточнения правил, доки, мелкие фиксы проверок.

## Когда выпускать релиз
- Появился заметный новый функционал (навык, MCP, покрытие метаданных, адаптер) — MINOR.
- Накопились исправления, которые командам стоит забрать — PATCH.
- Изменилось расширение `ai_debug` (BSL или структура) — обязательно релиз с новой сборкой `.cfe` (см. ниже).
- Ломающее изменение структуры или правил — MAJOR, с разделом в CHANGELOG про миграцию.

Не выпускай релиз на каждый коммит. Копи изменения в разделе `[Unreleased]` в `CHANGELOG.md`.

## Как выпустить (по шагам)
1. Обнови `CHANGELOG.md`: перенеси нужное из `[Unreleased]` в новый раздел `[X.Y.Z] - ГГГГ-ММ-ДД`.
2. Проверь, что тесты зелёные и сборка идёт: `uv run --no-project --with mcp --with pytest --with lxml --with openpyxl --with xlrd pytest tests/ -q` и `sh build.sh`.
3. Поставь тег и запушь его:
   ```bash
   git tag -a vX.Y.Z -m "vX.Y.Z"
   git push origin vX.Y.Z
   ```
4. Тег запускает workflow `release.yml`: создаётся GitHub Release, в него кладётся исходник расширения
   (`ai_debug-src.zip`) и автоматический архив исходников репозитория.
5. **Собери и приложи бинарник расширения** (см. следующий раздел) — CI платформы 1С не имеет и `.cfe` собрать не может.

## Сборка расширения `ai_debug` (на машине с платформой 1С)
Расширение `ai_debug` нужно для слоя данных `onec-data` (сервис под правами пользователя, только чтение). Исходник
лежит в `extensions/ai_debug/src` в формате 1C:EDT. Готовый `.cfe` собирается там, где есть платформа и 1C:EDT или
Конфигуратор (обычно Work PC, Windows):
1. Импортируй `extensions/ai_debug/src` как проект расширения в 1C:EDT (или загрузи в Конфигуратор как расширение).
2. Собери файл поставки расширения `ai_debug.cfe`.
3. Приложи `ai_debug.cfe` к соответствующему GitHub Release (кнопка Edit release → Attach binaries), рядом с
   `ai_debug-src.zip`.
4. В описании релиза укажи, на какой версии платформы и режиме совместимости собрано.

Так пользователи получают и исходник для правок, и готовый `.cfe` для установки.

## Git LFS — не используем (обоснование)
Проанализировано: в репозитории только текст (XML, BSL, Python, Markdown), самый крупный файл около 110 КБ, бинарных
файлов нет. Плюсы LFS (компактный клон при больших бинарниках) здесь не нужны, а минусы реальны: LFS усложняет клон и CI,
требует поддержки на всех зеркалах (в том числе корпоративный GitLab-форк) и упирается в квоты. Решение: **бинарные
артефакты (`.cfe`, при появлении `.epf`) держим в GitHub Releases, а не в git**, LFS не подключаем. Вернуться к вопросу
стоит только если понадобится версионировать в самом репозитории крупные бинарные фикстуры.

## Расширения проекта — не забывать
Если в toolkit появляется новое расширение 1С (по образцу `ai_debug`), его нужно вести так же: исходник в
`extensions/<имя>/src`, сборка `.cfe` на машине с платформой, приложение к релизу, строка в `CHANGELOG.md` и, при
необходимости, в `release.yml` (добавить упаковку его исходника в архив).
