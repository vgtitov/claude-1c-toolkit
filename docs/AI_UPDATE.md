# Установка, версионирование и обновление — канон для ИИ (generic)

Как ядро (`claude-1c-toolkit`) доставляется потребителю, как режутся версии и как ИИ (Claude Code)
выполняет установку/обновление БЕЗ ручного запуска скриптов человеком. Организация-специфика (профили,
конкретные репозитории) — в потребительском репо (напр. `claude-1c-team/CLAUDE.md` раздел «Обновление»).

## Модель доставки: пин-тег

- Ядро — **отдельный репозиторий**, версионируется **аннотированными semver-тегами** (`vMAJOR.MINOR.PATCH`).
- Потребитель НЕ тянет `main` ядра. Он **пинит тег** в файле (`onboard/toolkit.ref` = напр. `v1.2.0`).
- Установщик (`onboard`) клонирует/переключает ядро на этот пин и раскладывает его артефакты (движок, скрипты,
  generic-скиллы, конфиг MCP) в рабочее окружение.
- Значит: правки в ядре доходят до команды **только после нового тега + бампа пина** — само с `main` НЕ подтянется.

## Semver: когда какой разряд

- **PATCH** (`vX.Y.Z+1`) — фиксы без новых возможностей/ломающих изменений.
- **MINOR** (`vX.Y+1.0`) — новые операции/инструменты, обратно совместимо.
- **MAJOR** (`vX+1.0.0`) — ломающие изменения (сигнатуры/формат конфигов/удаление операций).

## Релиз новой версии ядра (кто выпускает — по решению владельца)

1. Прогнать тесты ядра — **зелёные** (для этого репо: `python -m pytest tests/onec_metadata/ -q`).
2. Аннотированный тег: `git tag -a vX.Y.Z -m "<что вошло>" <sha>` → `git push origin vX.Y.Z`.
3. В потребительском репо **бампнуть пин** (`onboard/toolkit.ref` → `vX.Y.Z`), закоммитить.
4. Разослать команде «обновитесь» — авто-уведомления о новом пине нет.

## Установка (первый раз)

Клонировать потребительский репо → запустить установщик (`onboard`) с нужным профилем/ролью. Он: ставит ядро
на пин-тег, раскладывает скиллы, конфиг MCP (`.mcp.json`), переменные окружения. Детали шагов и профилей —
в потребительском репо.

## Обновление — ИИ делает САМ (человек скрипты не запускает)

Когда человек просит «подтяни изменения» / «обнови» / «выполни онборд» в каталоге потребительского репо, ИИ
выполняет по шагам, запуская команды сам (запрашивая разрешение при необходимости):
1. `git -C <потребительский репо> pull --ff-only` — свежий пин/правила/профили.
2. Запустить **онборд** (идемпотентный) с профилем пользователя — он переставит ядро на новый пин, перезальёт
   скиллы, `.mcp.json` (в т.ч. НОВЫЕ MCP-серверы) и env. (Лёгкая альтернатива `update` годится, только когда
   менялись лишь скиллы/ядро, но НЕ состав MCP-серверов/env.)
3. Если у пользователя локально есть смежные репозитории (расширения/код) — подтянуть их (`git pull`).
4. **Напомнить перезапустить Claude Code** — MCP-серверы и скиллы грузятся при старте сессии; горячей
   перезагрузки нет, и сессия не может перезапустить сама себя. Это ЕДИНСТВЕННОЕ ручное действие человека.

Если для инструмента нужен **доступ/секрет** (токен, VPN) — ИИ НЕ придумывает его, а проводит человека по
получению: называет ТОЧНО, где взять (профиль/UI), какую переменную окружения задать, и проверяет результат.
Never — секрет в git/чат.

## Принцип

Каждое изменение ядра, которое должно дойти до команды, требует: **тег → бамп пина → онборд (ИИ) → рестарт
(человек)**. ИИ автоматизирует всё, кроме рестарта сессии и выдачи секретов/прав. Так же продолжаем дальше.
