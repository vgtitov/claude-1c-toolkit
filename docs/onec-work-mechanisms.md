# Навигатор: как Claude работает с 1С — какой механизм для чего (канон toolkit)

Единая точка входа. Отвечает на «что уже есть в toolkit и как выбрать инструмент под
задачу». Детали — в канонах, на которые ссылается каждый раздел. **Правило: выбирай по
ВОПРОСУ, который решаешь, а не по механизму — иначе заведёшь дубль.**

## Четыре оси работы с 1С (не путать между собой)

| Ось | Вопрос | Инструменты | Канон |
|---|---|---|---|
| **Код** | «как это написано / где в коде / какая сигнатура» | `onec-code` (ripgrep по клонам), `bsl-ls`, `bsl-platform` | README §Состав |
| **Данные** | «что в данных / что видит пользователь X / какие настройки влияют / отладка RLS» | `onec-data` (OData + ai_debug), браузер | `data-access-architecture.md` |
| **Тестирование** | «разработка ведёт себя правильно (отчёт формируется, проведение отработало, регресс)» | `apply.smoke` (runner-EPF), Vanessa/YAxUnit | `testing-ladder.md` |
| **Эксплуатация** | «почему медленно/падает, что в логах, метрики» | `onec-ops` (ТЖ/ЖР/APDEX/Zabbix), прямой SQL (perf) | `mcp-ops-tools-design.md` |

Оси пересекаются механизмами (СКД, вызов кода, браузер, статика), но **у каждой свой
вопрос и свой транспорт**. Ниже — где перекрытие и как не задвоить.

## Данные vs Тестирование — главное разграничение

Обе имеют «лестницу», нумерация **разная и не взаимозаменяема**: данные — `L0–L4`,
тестирование — `ступени 0–4`.

|  | **ДАННЫЕ** (`onec-data`) | **ТЕСТИРОВАНИЕ** (`apply.smoke`/фреймворки) |
|---|---|---|
| Цель | посмотреть/понять данные, отладить RLS | доказать поведение разработки |
| Транспорт | HTTP-сервис/OData → JSON, интерактивно | batch `/Execute` → файл-вердикт, headless/CI |
| Запись/проведение | **никогда** (read-only) | **можно** (в тест-базе — это предмет теста) |
| RLS | честный per-user (Basic-auth исследуемого) | под `/N /P` пользователя запуска |
| Вердикт | выборка данных (маскирование ПДн) | `__SMOKE_OK__/FAIL` + артефакт |
| Требует **публикации** бази | **ДА** (веб-сервер) | **НЕТ** (SSH к серверу 1С) |

**Перекрытие механизма (осознанное, не дубль).** `data_report`/`data_call` формируют СКД /
исполняют серверный код тем же API, что `smoke_runner`. Выбор:

- доказать поведение · CI · нет веб-публикации · нужна запись/проведение → **тестирование**
  (`apply.smoke`);
- интерактивно увидеть данные/настройки под пользователем · отладка RLS · строго read-only
  → **данные** (`onec-data`);
- «пользователь увидит и нажмёт» (UI, динамические списки, доступность команд) → **браузер**
  = L4 данных = ступень 3 тестирования (один слой, не дублировать).

## Требования доступности — что нужно включить, и КТО это делает

Многое из «данных» не заработает без действий администратора/DevOps (роль `1c-admin-devops`).
Claude этого не делает — он потребитель уже опубликованного.

> **Сначала автономный сервер, потом заявка админу.** Для ФАЙЛОВОЙ базы веб-сервер не
> нужен вовсе: `scripts/onec_verify.py serve --base <каталог> --port 8315` поднимает
> веб-клиент, OData и HTTP-сервисы силами самой платформы (`ibsrv`) за ~10 секунд.
> Строки «должен включить админ» ниже относятся к клиент-серверным базам и к общему
> доступу команды. Детали и грабли — `data-access-architecture.md`.

| Механизм | Что должен включить админ | Без этого |
|---|---|---|
| `onec-data` L2 (OData) | для файловой базы — ничего, хватает автономного сервера; для клиент-серверной — **опубликовать ИБ на веб-сервере** + состав стандартного интерфейса OData. В любом случае роль `УдаленныйДоступOData` исследуемым пользователям (в конфигурациях на БСП — через профиль групп доступа) | OData отвечает 404 без публикации и 401 без роли |
| `onec-data` L3 (ai_debug) | публикация **с HTTP-сервисами** (для файловой базы даёт автономный сервер) + загрузить расширение `ai_debug` + настроить `aidbg_Настройки` | L3 не работает |
| `onec-data` L4 / тест ступень 3 | веб-клиент: для файловой базы — автономный сервер, для клиент-серверной — публикация | UI-проверка недоступна (заменяется толстым клиентом) |
| Тестирование ступени 0–2 | **НЕ требует публикации** — платформа 1С + учётка конфигуратора. Транспорт: **SSH к серверу** (`Runner`) ИЛИ **локально** (`LocalRunner`, если платформа на своей машине — файловая или клиент-серверная база); фабрика `make_runner(host)`, `apply.smoke` работает обоими | — |
| `onec-code` / `bsl-ls` (код) | ничего (локальные клоны/выгрузка) | — |
| `onec-ops` (логи/метрики) | доступ к каталогам ТЖ/ЖР (+ опц. Zabbix/Prometheus) | — |

**Практический вывод для внедрения:** доступ к ДАННЫМ (L2–L4) появляется только после того,
как базу **опубликовали** (OData + HTTP-сервисы) и загрузили расширение `ai_debug`. Пока
публикации нет — доступны код (onec-code), логи (onec-ops) и **доказательство поведения +
разовые данные под пользователем через тестирование ступеней 0–2** (по SSH ИЛИ локально,
публикация не нужна). Это часто и есть первый практический канал, пока идёт согласование публикации.

### Кто может публиковать (это НЕ роль в 1С)

Публикация на веб-сервере (Apache/IIS) — это операция на уровне ОС веб-сервера, не «роль
администратора 1С»:
- Конфигуратор → «Администрирование → Публикация на веб-сервере» создаёт `default.vrd` в
  каталоге публикации и правит конфиг веб-сервера (`httpd.conf`), затем нужен **reload/restart
  веб-сервера**. Требуются: права записи в каталог публикации + правка конфига + рестарт службы.
- **Программист МОЖЕТ публиковать сам** — если у него есть эти права на КОНКРЕТНОМ (dev/тест)
  веб-сервере (свой локальный Apache, dev-стенд). На общем/прод-сервере обычно админ/DevOps.
  Роль в 1С тут ни при чём — вопрос в доступе к ОС и рестарту веб-сервера.
- **Отдельный сервер Apache** (не на машине сервера 1С) — нормальная топология. На нём должен
  стоять **модуль веб-сервера 1С** (`wsap24.dll`, та же версия/битность платформы), а строка
  соединения в `default.vrd` указывает на кластер 1С (`Srvr=…;Ref=…`) или файловую базу по сети.
- **Claude сам публиковать не может** на Mac (нет платформы/`wsap`); через Windows-узел
  (Work PC/Server PC) — технически возможно (положить `default.vrd` + прописать Location в
  `httpd.conf` + reload), но это изменение инфраструктуры → делать только с явным подтверждением.

## Режимы развёртывания onec-data (напоминание)

- **Локальный** — `mcp/dev.mcp.json`, stdio; клиент ходит в опубликованную базу сам.
- **Центральный** — docker + Caddy (bearer) на общем узле; секреты учёток 1С только на
  сервере. Детали и профили — `data-access-architecture.md` §7.
