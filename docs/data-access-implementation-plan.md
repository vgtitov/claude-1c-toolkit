## Карта сценариев (S1–S10) → инструмент → режим

| # | Сценарий (наши задачи доработок) | Инструмент | Режим |
|---|---|---|---|
| S1 | «Что реально легло в регистр/справочник/документ после проведения» — проверка разработки по данным | `data_query` (сложное) / `data_read` (плоское) | `user`; `privileged` — когда RLS мешает, а вопрос не про права |
| S2 | «Почему отчёт пустой / цифра неверная» — выполнить ТОТ ЖЕ вариант отчёта СКД с теми же отборами | `data_report` + контрольный `data_query` по регистру | `user` под пользователем из инцидента; эталон — `privileged` |
| S3 | «Что видит пользователь X» — отладка RLS/групп доступа | `data_rls_probe` | строго `user` (privileged тут бессмыслен) |
| S4 | «Какие настройки влияют» — константы, ФО (в т.ч. с параметрами), настройки пользователей, роли/группы | `data_settings` | `user`/`privileged` (чтение чужих настроек требует прав) |
| S5 | «Проверить API-метод» — дёрнуть экспортную функцию (цена, заполнение, расчёт) с контрольными параметрами | `data_call` (whitelist) | по методу; тест-база |
| S6 | «Понять структуру» — состав объектов, реквизиты, типы | `data_metadata` (+ onec-code для кода) | — |
| S7 | «Эталон против пользователя» — diff выборки privileged vs user = эффект RLS в цифрах | `data_query` ×2 / `data_rls_probe` | оба |
| S8 | «Остатки/обороты/срез быстро, без расширения» | `data_virtual` (OData-функции) | `user` |
| S9 | «Дежурные проверки» — преднастроенные выборки/отчёты под конвейер (после релиза, при инциденте) | `data_preset` | пин в пресете |
| S10 | «Под кем я и что доступно» — самопроверка канала | `data_health` | — |

Сценарии за пределами слоя данных: поведение UI/динамических списков — L4 (браузер);
медленно под пользователем — onec-ops/1c-expert (ТЖ, планы); история изменений — ЖР (onec-ops).
