# Архитектурный аудит onec_metadata (2026-07)

Проверка: действительно ли наш подход к программному редактированию метаданных
1С — лучший из доступных. Сравнение с отраслью и передовыми методами; выводы и
задачи. Обезличено.

## Наши ключевые решения

1. Манипуляция ИСХОДНЫМИ файлами выгрузки (Конфигуратор XML + EDT `.mdo`) через
   lxml.
2. Паттерн «deepcopy реального элемента-образца + подстановка», без моделирования
   матрицы свойств по версии/совместимости.
3. byte-perfect минимальный дифф.
4. Применение батч-DESIGNER в ТЕСТ-базу + round-trip verify как гейт (платформа =
   источник истины).
5. Оба формата с автоопределением.
6. Реестр покрытия `catalog.SUPPORTED`.

## Вердикт: подход обоснован и совпадает с best practice по фундаменту

Подтверждено (со ссылками — в задаче-исследователе, ниже тезисно):

- **File-first оправдан фундаментально.** Встроенный язык 1С даёт доступ к
  метаданным ТОЛЬКО на чтение; создавать/менять объекты можно лишь в дизайн-тайме
  (Конфигуратор/EDT) или правкой исходных файлов + загрузкой платформой (ITS
  metod8dev/2318). Рантайм-API создания метаданных не существует — значит наш путь
  (правка файлов, headless, без Java/Eclipse) — самый автоматизируемый из легальных.
- **«Платформа = источник истины» + round-trip verify** — ровно то, что делает
  весь git-workflow отрасли (gitsync, 1С:GitConverter): человек/инструмент меняет
  → платформа выгружает/проверяет.
- **deepcopy образца, а не сборка с нуля** — совпадает с отраслевой утилитой
  программного создания ролей (infostart 2099929) и обосновано тем, что порядок
  свойств и версия сериализации официально НЕ документированы.
- **byte-perfect минимальный дифф — НАШЕ преимущество.** Отраслевой пайплайн
  (gitsync) полагается на платформенную перевыгрузку и чистого минимального диффа
  ввода не даёт. Наш чистый PR-дифф — плюс к ревью, которого у отрасли нет.

## Проверено запуском (наши факты)

- `set-property` корректно правит реальный справочник УТ (реальные флаги,
  byte-perfect, BOM+CRLF). Text-подход на реальных данных держится.
- Платформа НЕ поставляет XSD-схемы формата в `bin` → дешёвой офлайн-валидации
  «по схеме» нет; round-trip через платформу остаётся основным гейтом.
- Реестр `catalog.SUPPORTED` НЕ энфорсится в рантайме (используется только в
  `catalog.py` и тестах) — заявленный в CLAUDE.md «предохранитель» пока
  документационный.

## Где отрасль сильнее и что перенять

1. **Каскадный рефакторинг (главное отставание).** Передовые EDT-плагин-MCP
   (DitriXNew/EDT-MCP, MCP:RSV) правят метаданные через внутреннюю EMF-модель EDT
   и при переименовании/удалении объекта чинят ссылки во ВСЁМ проекте (BSL +
   формы + метаданные). Наша text-манипуляция этого не даёт даром.
   **Решение (архитектурная граница, см. ниже):** наш инструмент сознательно
   ограничивается БЕЗОПАСНЫМИ аддитивными и скалярными правками; деструктив с
   каскадом (rename/delete/move) — вне scope, отдаётся EDT-плагину или человеку.
2. **Пересчёт ConfigDumpInfo платформой.** Финальный артефакт для git должен быть
   платформенно-нормализован (версии/порядок/ConfigDumpInfo — платформой, не
   руками). Наш round-trip уже выгружает платформенное дерево — надо явно
   зафиксировать, что коммитим ИМЕННО его, а не пред-apply правку.
3. **`1cedtcli` как стандартный конвертер/валидатор** для `.mdo` (headless, но
   тянет Java/Eclipse) — оценить, не дешевле ли он самодельных обёрток для
   EDT-формата.

## Архитектурная граница (ключевое решение аудита)

Инструмент делает и будет делать ТОЛЬКО правки, безопасные без анализа ссылок:

- **В scope (безопасно):** добавить реквизит/ТЧ/команду/измерение/ресурс/значение,
  колонку печатной формы, поле/запрос СКД, включить в подсистему, выдать право,
  изменить существующее СКАЛЯРНОЕ свойство. Ни одна из них не ломает внешние ссылки.
- **Вне scope (нужен каскад):** переименование, удаление, перемещение объекта,
  смена вида объекта. Требуют правки ссылок во всём проекте — это надёжно делает
  только EDT internal-model (плагин), не text-манипуляция. Такие операции —
  человеку в IDE или через EDT-плагин-MCP.

Это не слабость, а осознанная граница: аддитивные+скалярные правки закрывают
подавляющее большинство рутинных доработок с гарантией round-trip, а рискованный
каскад не имитируется вручную.
