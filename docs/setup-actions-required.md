# Что должен сделать ЧЕЛОВЕК — Claude этого не может

Короткий список действий, без которых механизмы toolkit не заработают, и по которым
инструменты сами выдают подсказку со ссылкой сюда. Каждый пункт: **симптом → что сделать →
как проверить**. Всё остальное Claude делает сам.

---

## 1. Защита от опасных действий — иначе батч-прогоны ВИСНУТ

**Симптом.** `onec_verify.py smoke` (ступень 2 лестницы) не возвращается и падает по таймауту;
файл-результат не появляется, `/Out`-лог пустой. На экране пользователя при этом всплывает окно
1С «Разрешить открытие файла?» / предупреждение безопасности. Скрипт этого окна не видит —
снаружи выглядит как зависание.

**Почему.** Раннер-обработка пишет результат в файл. Платформа с 8.3.9 спрашивает подтверждение
на такие действия («Защита от опасных действий»). В батч-сеансе (`/Execute`) диалог всё равно
модальный и ждёт человека вечно. **Ключа командной строки, отключающего защиту, у платформы
НЕТ** — проверено по бинарникам 8.3.27 (все ключи запуска лежат в `1cv8.exe`, а
`DisableUnsafeActionProtection` — только в `core83.dll`, рядом с механизмом защиты) и по
открытым источникам.

**Что сделать — любой из двух способов.**

*Способ A (точечно, рекомендуется): снять флаг у пользователя ИБ.*
Конфигуратор/предприятие → Администрирование → Пользователи → нужный пользователь →
снять «Защита от опасных действий». Действует только на этого пользователя.

*Способ B (по маске баз): `conf.cfg` платформы.*
Файл `<каталог установки 1С>\conf\conf.cfg` (Windows обычно `C:\Program Files\1cv8\conf\conf.cfg`;
Linux — `/opt/1cv8/conf/conf.cfg`). Дописать строку с маской строк соединения, например только
для тест-баз:
```
DisableUnsafeActionProtection=*Franchise*Test*;*bases\erp_test*
```
`*.*` отключает защиту для ВСЕХ баз — на рабочей машине так не делать. Правится и на клиенте,
и на сервере 1С. Защита считается выключенной, если снят флаг у пользователя ИЛИ строка
соединения попала под маску.

**Как проверить.** `python scripts/onec_verify.py smoke --base <база> --scenario <сценарий>`
возвращает `__SMOKE_OK__` и текст отчёта, а не таймаут.

---

### Как снять — командой toolkit

```
python scripts/onec_verify.py protection --status                       # что сейчас
python scripts/onec_verify.py protection --disable --mask "*Test*"      # маска поуже
python scripts/onec_verify.py protection --disable --mask "*.*"         # все базы машины
python scripts/onec_verify.py protection --enable                       # вернуть защиту
```
Команда правит `conf.cfg` платформы, делает резервную копию рядом и идемпотентна.
`conf.cfg` лежит в каталоге установки, поэтому **нужны права администратора** — из
обычной сессии команда честно скажет об этом и ничего не тронет.

## 2. Публикация базы на веб-сервере — нужна НЕ всегда

**Симптом.** Инструменты `data_*` не отвечают; `data_health` не достучаться. Порт 80/443 у
сервера приложений может быть открыт — это ничего не значит, HTTP там может не отвечать.

**Сначала проверь, нужна ли она вообще.** Для ФАЙЛОВОЙ базы публикация не требуется:
`python scripts/onec_verify.py serve --base <каталог файловой ИБ> --port 8315` поднимает
автономный сервер платформы с веб-клиентом, OData и HTTP-сервисами за ~10 секунд, без
администратора. Отлаживай на файловой копии, а публикацию проси только для
клиент-серверной базы или общего доступа команды.

**Почему нужна для остальных.** L2–L4 лестницы доступа
(`docs/data-access-architecture.md`) ходят по HTTP: OData и HTTP-сервисы расширения
`ai_debug`. Публикация клиент-серверной базы — операция на веб-сервере (Apache/IIS),
а не роль в 1С; Claude её не выполняет.

**Что сделать (администратор / DevOps).**
1. Опубликовать ТЕСТ-базу на веб-сервере (Конфигуратор → Администрирование → Публикация на
   веб-сервере), **с галкой HTTP-сервисов** (нужен `aidbg_API`).
2. Включить нужные объекты в состав стандартного интерфейса OData.
3. Выдать исследуемым пользователям роль `УдаленныйДоступOData`.
4. ⚠ Регистр имени базы в URL публикации должен совпадать точно — иначе редирект превращает
   POST в GET и запросы ломаются молча.

**Про роль `УдаленныйДоступOData`.** Отдельного флажка в карточке пользователя нет — в
конфигурациях на БСП (ERP, УТ, КА) роли выдаются **через профили групп доступа**:
Администрирование → Настройки пользователей и прав → Профили групп доступа → создать
профиль с этой ролью → включить в него нужного пользователя. Поэтому в интерфейсе роль
и «не находится». Без неё OData отвечает **401**, а не пустой выборкой — по коду ответа
это отличается от «нет публикации» (там 404).

**Как проверить.** `curl -i http://<веб-сервер>/<публикация>/hs/aidbg/health` под учёткой
пользователя отдаёт JSON с `session_user`.

Пока публикации нет — данные и серверный код смотрим ступенью 2 лестницы тестирования
(она публикации не требует), см. `docs/testing-ladder.md`.

## 3. Учётка ИБ для ступеней 1–2

`ONEC_IB_USER` и `ONEC_IB_PASS[_<КЛЮЧ>]` в локальном `.env` (шаблон — `.env.example`).
КЛЮЧ = имя базы в ВЕРХНЕМ регистре, передаётся как `--contour`. Пароли в репозиторий и в
чат не попадают.

**Симптом при отсутствии:** Конфигуратор завершается с ненулевым кодом, лог пуст —
типичная картина при неверной учётке ИБ или занятой базе.

## 4. Токен центрального MCP

`set_token` записывает `ONEC_MCP_URL`/`ONEC_MCP_TOKEN` в **пользовательские** переменные
окружения — они видны только процессам, запущенным ПОСЛЕ этого. Отсюда типичная путаница:
в Claude Code канал работает, а `doctor.py` из старой консоли показывает провал авторизации.

**Что сделать.** Перезапустить консоль (и Claude Code) после `set_token`. Если провал остался —
токен на сервере сменился, взять актуальный у тимлида и прогнать `set_token` заново.

**Чего это НЕ значит.** Отсутствие маршрута `/health` у MCP-сервера провалом НЕ считается:
`doctor` трактует любой ответ 4xx как «сервер жив и авторизация прошла». Если `doctor` показал
КРАСНЫЙ по onec-code — дело в авторизации или в сети, а не в `/health`.
