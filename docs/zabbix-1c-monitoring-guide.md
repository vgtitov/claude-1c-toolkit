# Мониторинг 1С:Предприятие через Zabbix — референс-архитектура (обезличенно)

Как силами Zabbix получить полную картину производительности контура 1С — железо, кластер, СУБД,
события технологического журнала с **контекстами кода** — и отдать её AI-анализу через `mcp/onec_ops_mcp.py`
(`zabbix_perf_report_tool` / `scripts/zabbix_perf.py`). Собрано по методикам ИТС и живым open-source
шаблонам (лицензии проверены, см. таблицу). Проверено на реальном контуре: Zabbix 7.0, Windows-сервер 1С,
PostgreSQL-сервер на Linux.

## Три канала данных

| Канал | Транспорт | Модель |
| --- | --- | --- |
| 1. Метрики ОС/железа | штатный Zabbix agent (perf_counter на Windows, стандартные item на Linux) | pull |
| 2. Метрики кластера 1С (сессии, лицензии, rphost) | UserParameter-скрипт поверх `rac` (нужен запущенный RAS) | pull |
| 3. События ТЖ (EXCP/TLOCK/TTIMEOUT/CALL + контексты кода) | разбор ТЖ по cron → `zabbix_sender` в trapper-item'ы | push |

Канал 3 — ключевой для «профессионального отчёта»: помимо СЧЁТЧИКОВ событий (сколько EXCP/TLOCK за
интервал) выгружайте ТЕКСТОВЫЕ item'ы с топом контекстов CALL (RAM/CPU/длительность + строка кода 1С)
и информацией о самой долгой удерживаемой сессии. `onec_ops_mcp.py` распознаёт их
(`1c.tj.call_context`, `1c.sessions.held_info` — см. `_ZBX_TEXT_EVIDENCE`) и кладёт в отчёт секцией
«Указатели на места кода».

## Пороги железа (методика 1С, ИТС metod8dev/5838 — пересказ, не копия)

Экспресс-оценка «где болит железо» (в `diagnose_metrics` уже вшито):

| Показатель | Порог методики | На что указывает превышение |
| --- | --- | --- |
| Memory Pages/sec | ~0, максимум 20 | пейджинг / нехватка ОЗУ (проверять вместе с Available и свопом) |
| CPU % Processor Time (_Total) | ≤70% штатно | нехватка процессорной мощности |
| System Processor Queue Length | ≤2 × ядер | дефицит CPU |
| PhysicalDisk Avg. Disk Queue Length | ≤2 × дисков массива | дисковая подсистема не справляется |
| Network Bytes Total/sec | ≤65% пропускной способности | узкая сеть |

Дополнительно собирать: Available Mbytes, % Free Space, Avg. Disk sec/Read|Write, Context Switches/sec,
Threads/Processes, % User Time. Интервал сбора по методике — 15 сек (при инцидентах чаще).
Первоисточники (нужна подписка ИТС): its.1c.ru/db/metod8dev → статьи 5838 (загруженность оборудования)
и 6019 (анализ ТЖ: официальный стек «ТЖ в JSON → fluentbit → OpenSearch», ЖР через ibcmd, метрики через RAS).

## Шаги настройки (референс)

1. **RAS** на сервере 1С: Linux `ras cluster --daemon --port=1545`, Windows — служба «1C:Enterprise RAS».
   Проверка: `rac cluster list`.
2. **Zabbix agent** на серверах 1С и СУБД; для СУБД — официальный шаблон (PG: agent2 + `{$PG.*}` макросы).
3. **UserParameter** для кластера: обёртка над rac (bash/PowerShell), item'ы сессий/лицензий/фоновых.
   Грабли: таймаут в скрипте обязателен (rac без RAS висит), парсить вывод по ключам (локализация),
   на Windows — кодировка OEM; спецсимволы → экранирование, `UnsafeUserParameters=1` только осознанно.
4. **ТЖ**: `logcfg.xml` на рабочем сервере — ТОЛЬКО нужные события (EXCP, TLOCK/TTIMEOUT/TDEADLOCK,
   CALL с фильтром по длительности), отдельный каталог, ограниченный `history`. «Всё подряд» на прод не включать.
5. **Парсер ТЖ** по cron (1–5 мин): инкрементальный разбор (курсор позиции, пережить часовую ротацию),
   агрегаты → `zabbix_sender`: счётчики (`1c.tj.excp`, `1c.tj.tlock`, `1c.tj.ttimeout`, `1c.tj.call`) +
   текстовые топы (`1c.tj.call_context` — RAM/CPU/Duration/COUNT + контекст 1С; `1c.sessions.held_info`).
6. **Триггеры/макросы**: пороги из таблицы выше + `{$EXCP_THRESHOLD}`, `{$MAX_LOCK_WAIT}`, память rphost.
7. **Секреты**: логин/пароль админа кластера — secret-макросы хоста, не в скриптах.
8. **Доступ для AI-анализа**: read-only пользователь Zabbix, в роли — «Доступ к API» (Allow list `*.get`)
   и «Управление API-токенами»; токен → env `ZABBIX_TOKEN` (см. mcp-ops-tools-design.md §5).

## Анализ ТЖ: с Zabbix и без, в одном или разных периодах

Zabbix-канал 3 даёт **агрегаты и last-снимки** (счётчики EXCP/TLOCK, топ контекстов на «сейчас»).
Для глубокого разбора этого мало — нужен сам ТЖ. Toolkit поддерживает оба пути, и они стыкуются:

- **Без Zabbix (или параллельно с ним):** `tech_journal_parse` (`mcp/onec_ops_mcp.py`) читает каталоги
  ТЖ напрямую — один сервер или группа (`roots=[{path,label},…]` — все рабочие серверы кластера),
  фильтры `events=["TLOCK","TDEADLOCK","EXCP","CALL","DBMSSQL","DBPOSTGRS"]`, `min_duration`, `top`.
- **Один период:** окно `since/until` у `tech_journal_parse` задаётся тем же интервалом, что
  `window_hours` у `zabbix_perf_report` — Zabbix говорит «ГДЕ и КОГДА болит» (метрика, хост, % времени),
  ТЖ отвечает «ЧТО именно» (запрос, контекст модуля, участники блокировки).
- **Разные периоды:** два прогона одного инструмента с разными `since/until` (или два
  `zabbix_perf_report` с разными окнами) → сравнение «до/после» оптимизации, поиск деградации
  между релизами. Apdex-инструменты (`apdex_tool`, `apdex_by_operation_tool`) — та же логика
  до/после по временам операций.

Рабочий цикл: `zabbix_perf_report` (ранжированные находки за окно) → по топ-находке
`tech_journal_parse` за то же окно с целевым фильтром событий → конкретный код/запрос/виновник.

## Чего чаще всего не хватает (чек-лист полноты)

- [ ] Apdex по ключевым операциям (замер времени операций БСП «Оценка производительности» → item'ы) — без него нет «ощущения пользователя»
- [ ] TDEADLOCK отдельным счётчиком (не только TLOCK/TTIMEOUT)
- [ ] Долгие серверные вызовы: не только счётчик CALL, но и топ с контекстом (текстовый item)
- [ ] СУБД: для PG — pg_stat_statements (топ запросов по total_time) хотя бы текстовым item'ом; temp_files/temp_bytes; для MSSQL — PLE/Buffer cache/Deadlocks
- [ ] Память по процессам кластера раздельно (ragent/rmngr/rphost) и рестарты rphost
- [ ] Доступная производительность рабочих процессов (rac `available_performance`)
- [ ] Сетевые счётчики к СУБД (Bytes Total/sec против ёмкости канала)
- [ ] История текстовых evidence-item'ов (хранить, не только lastvalue — иначе после инцидента контекст потерян)
