# Лестница тестирования 1С — какой механизм для чего (канон toolkit)

Приоритет: **всегда самая нижняя ступень, которой достаточно** для проверяемого
утверждения. Выше — только когда нижняя не может дать доказательство.
Каждая ступень = «что проверяет / что НЕ проверяет / что нужно из доступов».

> **Ось этой лестницы — доказать ПОВЕДЕНИЕ** (отчёт формируется, проведение
> отработало), вердикт по артефакту, headless/CI. Чтобы **посмотреть ДАННЫЕ**
> под пользователем (что в регистре, что видит пользователь X, какие настройки
> влияют, отладка RLS) — это другая ось: `docs/data-access-architecture.md`
> (MCP `onec-data`, read-only, интерактивно). Не путать нумерацию: ступени 0–4
> здесь ≠ уровни L0–L4 там. Перекрытие механизмов (СКД, вызов кода) и правило
> выбора — в §4.1 того документа.

## Ступень 0 — статика (без базы, секунды)
- BSL Language Server (диагностики), `scripts/bsl_guard.py` (запрос-в-цикле,
  межпроцедурно), `1c-meta scd validate` (целостность СКД), `1c-meta bsl-gate
  check` (ratchet против легаси).
- Проверяет: синтаксис, анти-паттерны, целостность XML. НЕ проверяет: поведение.
- Доступ: только исходники. **Обязательна всегда** (встроена в pre-commit и в
  `upload_tree`).

## Ступень 1 — платформенный batch DESIGNER (минуты)
- `LoadConfigFromFiles → UpdateDBCfg -Extension → DumpConfigToFiles` +
  round-trip diff (`apply.dumpload`, `scope_guard`); синтакс-контроль —
  `apply.external.check_config` (`/CheckConfig` во ВСЕХ контекстах компиляции:
  тонкий/веб/сервер/внешнее соединение/толстый управляемый/модули поставки).
- Проверяет: метаданные валидны для платформы, применение прошло, дрейфа
  свойств нет, модули компилируются под каждый контекст. НЕ проверяет:
  исполнение кода на данных.
- Доступ: 1cv8 на сервере + учётка с правами конфигуратора.
- `check_config` ловит то, чего не видят ни BSL LS, ни статика: несуществующий
  метод/тип, вызов, недоступный в контексте, битую ссылку метаданных. Ложные
  диагностики линтера тоже отсеиваются здесь — вердикт платформы главнее.

## Ступень 2 — batch ENTERPRISE + runner-EPF (минуты) ← «дымовой прогон»
- Внешняя обработка-раннер, запуск `1cv8 ENTERPRISE /Execute runner.epf
  /C"<параметры>"`; раннер programmatically вызывает проверяемое (печать через
  `УправлениеПечатью.СформироватьПечатныеФормы`, компоновку СКД через
  `КомпоновщикМакетаКомпоновкиДанных`), пишет результат/ошибку В ФАЙЛ и
  завершает сеанс (`ЗавершитьРаботуСистемы(Ложь)`).
- Сборка раннера из XML-исходников — `apply.external.build_external`; запуск —
  `apply.external.run_enterprise` (см. ниже). Работает headless по SSH.
- Проверяет: серверная логика реально исполняется на реальных данных
  (печатные формы, СКД-отчёты, проведение, API). НЕ проверяет: UI/формы.
- Доступ: как ступень 1. **Это основной уровень для контуров, куда нельзя
  ставить сторонние фреймворки** (клиентские прод-серверы).
- Пока база НЕ опубликована на веб-сервере, ступень 2 — ещё и единственный
  способ посмотреть данные и проверить серверный код: `onec-data` (L2–L4)
  без публикации не включается вовсе. Ограничение: под `/N /P` пользователя
  запуска, а объект `HTTPСервисЗапрос` из кода не конструируется — HTTP-контракт
  сервисов так не проверить.
- Вердикт ТОЛЬКО по файлу-результату (маркер OK/FAIL + артефакт, напр. mxl):
  вывод в консоль/ЖР не считается доказательством (боевой урок: ложная
  верификация).

## Ступень 3 — UI-смоук через веб-клиент (минуты—десятки минут)
- Веб-клиент + агент/человек проходит сценарий в браузере (открыть отчёт, нажать
  печать, увидеть колонку).
- Проверяет: команды видны, формы открываются, вариант отчёта настраивается —
  то, что batch не покажет. Проверка публикации засчитывается только по
  `access.log` со статусом 200 (либо по факту открывшейся формы).
- Откуда берётся веб-клиент:
  - **файловая база — автономный сервер, публикация НЕ нужна**:
    `scripts/onec_verify.py serve --base <каталог> --port 8315`, дальше `http://localhost:8315/`;
  - клиент-серверная база — публикация на веб-сервере (vrd + конфиг веб-сервера).
- Чем управлять браузером — три равноправных способа, выбор по стеку и по задаче:

| Способ | Когда он лучший | Чего не умеет |
|---|---|---|
| **Браузерное расширение агента** (у Claude — Chrome-расширение) | живая отладка в СВОЁМ браузере: сессия и доступы уже есть, видно консоль и сеть, скриншот в отчёт | не воспроизводится в CI, привязан к конкретному агенту |
| **Playwright / Chrome MCP** | повторяемый сценарий, headless, CI, а также **любой другой AI-агент** — способ не привязан к вендору | сессию/логин надо готовить отдельно (storage state) |
| **Управление экраном и приложениями** | UI не веб: толстый клиент, Конфигуратор, окна платформы | дороже и медленнее, хрупко к разрешению и локализации |

  Правило выбора: веб-интерфейс и разовая проверка → расширение; веб-интерфейс и
  повторяемость/CI/другой агент → Playwright; не-веб интерфейс → управление экраном.
  Дублировать проверку несколькими способами не нужно.
- ⚠ Модальные окна платформы блокируют автоматизацию так же, как батч-сеанс: диалог
  ждёт человека, а агент видит «зависание». Ступень 0–2 к этому чувствительна через
  «Защиту от опасных действий» (`docs/setup-actions-required.md` §1).

## Ступень 4 — фреймворки автотестов (часы на настройку; для команд/CI)
Выбор по задаче:
- **Vanessa Automation** (github.com/Pr-Mex/vanessa-automation, BDD/Gherkin,
  сценарные и UI-тесты через платформенный механизм автоматизированного
  тестирования `/TESTMANAGER + /TESTCLIENT`; запуск в CI —
  vanessa-opensource/vanessa-runner на OneScript). Корп-стандарт Аскона
  (см. OneDrive `DIT-1С/1С-Инфраструктура/CI CD`): smoke-набор на каждый MR.
- **YAxUnit** (github.com/bia-technologies/yaxunit, расширение .cfe) — юнит-тесты
  серверного кода, плагин EDT, JUnit-отчёты для CI. Лучший выбор для модульных
  тестов общих модулей.
- **Тестер** (github.com/grumagargler/tester) — сценарные тесты кодом 1С;
  альтернатива VA, когда Gherkin не нужен.
- **1С:Сценарное тестирование** (ИТС) — коробка от вендора, если у клиента
  куплен КИП.
- Проверяет: регресс, сквозные сценарии, юниты. Требует: право ставить
  расширение/обработки фреймворка в тест-базу + (для UI) машину с клиентом.

## Как запустить (ступени 1–2)
```
python scripts/onec_verify.py check --base "срв\ERP_Test" --ext МоёРасширение
python scripts/onec_verify.py smoke --base "D:\bases\erp" --scenario scen.bsl --deploy
```
Без `--host` платформа берётся на этой машине, с `--host <алиас ssh>` — на сервере
1С; один и тот же код работает обоими транспортами. Ступень 0 живёт в pre-commit
и в `upload_tree`, отдельного запуска не требует.

## Быстрый выбор
| Задача | Ступень |
|---|---|
| «код без анти-паттернов, СКД цел» | 0 |
| «расширение применяется, ничего не задето» | 1 |
| «печатная форма/отчёт реально формируется» | 2 |
| «пользователь это увидит и нажмёт» | 3 |
| «регресс на каждый MR / юнит-тесты» | 4 |

## Ограничения доступности (реальные кейсы)
- Клиентский прод-сервер, ставить фреймворки нельзя → потолок: ступени 0–3
  (раннер-EPF в ТЕСТ-базу — допустимо: это данные, не конфигурация).
- Нет публикации/веб-сервера → ступень 3 заменяется толстым/тонким клиентом
  на машине с доступом, либо пропускается с фиксацией «UI не проверен».
- Полноценный CI (Аскона): ступени 0–2 на каждый коммит, 4 (VA smoke) на MR,
  регресс — по релизу (дорожная карта DIT).
