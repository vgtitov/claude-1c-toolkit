# Установка инструментов Клодом — гайд и грабли (для ИИ)

Как Claude Code проводит установку/настройку инструментов у члена команды БЕЗ затыков. Составлено по
разбору реальной установочной встречи — тут собраны конкретные грабли и КАК их избежать. Организация-специфика
(конкретные группы GitLab, адрес центра, версии платформы) — в потребительском репо; здесь — процедура и гарды.

## Принцип

Клод ведёт установку сам: запускает команды, проверяет результат, ловит типовые ошибки заранее. Человек делает
руками ТОЛЬКО то, что Клод физически не может: **рестарт Claude Code**, **установка платформы/IDE-плагинов**
(внешние дистрибутивы), **выдача секретов/прав** (токены, доступы в системах). Всё это Клод не «делает за
кадром», а называет ТОЧНО: что скачать/нажать, куда положить, какую переменную задать.

## 0. Preflight — Клод проверяет ПЕРЕД установкой (не молча)

Проверить наличие и версии, при провале — печатать точную команду доустановки (не общее «поставь»):
- `git`, `uv` (Python-раннер), `java`, `claude` (Claude Code CLI), `rg` (ripgrep).
- **PowerShell ≥ 5.1** (Windows): `$PSVersionTable.PSVersion`. Грабля встречи: «установите последнюю версию
  powershell… а не отработала» — авто-подсказка не сработала; Клод должен САМ проверить и дать команду.
- Определить **cwd/каталог**: Клод в начале проверяет, в том ли каталоге запущен (см. §5). Грабля: люди
  запускали Claude не в том репозитории — Клод должен сам сказать «перейди в <каталог>».

## 1. Токены и доступы — через скрипт-сеттер, НЕ правкой .env руками

Грабли встречи (самые частые): токен нигде не персистился и его не было в `.env.example`; Клод не знал про
скрипт установки токена и полез править `.env` вручную; перепутали переменные (GitLab-PAT вставили в MCP-токен);
при ручной перезаписи `.env` **сломалась кодировка**.

Правила для Клода:
- **Секреты вводить готовым скриптом-сеттером** (в этом стеке — `scripts/set_token`), а НЕ редактированием
  `.env` вручную. Скрипт кладёт значение в нужную переменную под фиксированным именем — нельзя перепутать.
  Если для инструмента сеттера нет — дописывать в `.env` ТОЧЕЧНО одну строку, не перезаписывая файл целиком.
- **`.env` НИКОГДА не перезаписывать целиком** и не пересохранять «блокнотом» — только точечная правка,
  кодировка **UTF-8 без BOM**. Если кодировка уже сломана — чинить программным пересохранением в UTF-8, не руками.
- **`.env.example` — полный**: в нём должны быть ВСЕ переменные с однострочным комментарием «что это и где взять»
  (в т.ч. MCP-токен центра, GitLab-PAT, Jira/Confluence/Zabbix). Если чего-то нет — добавить в `.env.example` СРАЗУ.
- **Проверять валидность токена** тестовым вызовом и говорить «этот токен read-only / не тот scope» ДО того, как
  человек упрётся в ошибку. Секреты — только в env/.env, НИКОГДА в git/чат.
- Токены, которые раздаёт тим-лид out-of-band (напр. токен центра) — Клод не выдумывает, а говорит «возьми у тим-лида».

## 2. git / GitHub / GitLab — предупреждать про доступ и политику ЗАРАНЕЕ

Грабли: у новичка нет учётки GitLab (блокер); логин = доменный AD-пароль (не понимали); `git clone` в непустой
каталог падал; не хватало прав в группе → 403 на clone/push; токен со scope только read → push невозможен;
main защищён и прямой push запрещён, MR у проекта отключён → «ссылка на MR не открывается».

Правила для Клода:
- Перед клонированием — **проверить, что целевой каталог пуст**; при непустом не падать, а сказать «удали/
  переклонируй <путь>». Различать `git clone` и «ввести токен» (человек путал).
- Ловить **403** и печатать причину: «нет прав в группе репозитория ИЛИ токен read-only — нужно: <добавить в
  группу ролью Maintainer/Developer> и <PAT со scope api/write>».
- **Заранее предупреждать про protected main**: не пытаться пушить в `main`, а СОЗДАВАТЬ ветку и MR — это штатная
  политика, не сюрприз. Если у проекта MR-фичи отключены — сказать об этом, а не давать неоткрывающуюся ссылку.
- Правки в generic-ядро (toolkit на GitHub) — через **fork + pull request**, не прямой push.
- Чек-лист доступа ДО установки (специфика — в потребительском репо): учётка есть; добавлен в нужную группу
  ролью Maintainer (кто пушит); создан PAT со scope `api` (не read-only).

## 3. MCP-серверы — рестарт + healthcheck, не «на глаз»

Грабли: после прописки токена MCP не подхватывается без рестарта Claude; проверяли «работает ли MCP» только
вручную по просьбе; локальный сервер не поднялся (нет платформы/клон не качается).

Правила для Клода:
- После установки токенов/серверов — **сказать «перезапусти Claude Code»** (MCP грузятся при старте) и дать
  команду проверки: `claude mcp list`, затем прогнать по каждому серверу лёгкий вызов.
- **Doctor — один вызов «проверь всё»:** `uv run "$ONEC_TOOLKIT_DIR/scripts/doctor.py"` (из рабочего каталога
  с `.mcp.json`). Печатает по каждому пункту ЗЕЛЁНЫЙ/жёлтый/КРАСНЫЙ с причиной: пререквизиты (git/uv/java/claude/rg,
  PowerShell), каждый MCP из `.mcp.json` (onec-code локаль/центр, bsl-platform, bsl-ls, edt:8765 — localhost мимо
  прокси), `.mcp.json` найден, `ONEC_SRC_DIR`. Код возврата 1 при красных. Клод запускает doctor после установки/
  обновления и разбирает красные пункты по подсказкам.

## 4. Платформа 1С / внешние дистрибутивы — Клод их НЕ ставит, а ведёт человека

Самый большой затык встречи (растянулся надолго): у человека стоял только тонкий клиент; Клод пытался
«поставить платформу», хотя доступа к дистрибутиву у него нет; ставили не ту разрядность (x86 вместо x64);
заводили лишнего пользователя; путались, нужен «автономный сервер, не служба» + консоль администрирования;
версии по контурам независимы (франч 8.3.27.2214 / СНГ 8.3.27.1606, режим совместимости 8.3.21).

Правила для Клода:
- **Сразу честно:** «дистрибутив платформы/IDE-плагин я скачать и установить не могу — вот ЧТО скачать, откуда
  (ИТС-портал/логин у тим-лида) и куда положить». Не имитировать установку.
- Дать точную спецификацию: **полный дистрибутив (не только тонкий клиент)**, **x64**, при необходимости сервер
  **в автономном режиме (НЕ служба) + консоль администрирования**, версия — **по контуру** (версии независимы,
  не переносить смену одного контура на другой).

## 5. Где запускать Claude — карта каталогов + самопроверка cwd

Грабля: постоянная путаница «в каком репозитории/каталоге запускать». Тим-лид много раз объяснял модель вручную.

Правила для Клода:
- В начале установки/работы **проверить cwd** и, если не тот, сказать «перейди в <каталог>» (не ждать, пока
  подскажет человек). Карта «контур → каталог → что делает Клод» — в потребительском репо (`CLAUDE.md`).
- Пути (корень репозиториев, каталог кода) — **из env, не хардкод `C:`** (диск D тоже; кросс-платформенность).
  `onboard`/установщик спрашивает корень и записывает, `.env` создаёт из `.env.example` сам, если его нет.

## 6. Мета: идемпотентность и знание вспомогательных скриптов

Грабли: онбординг переспрашивал профиль на каждом под-скрипте; Клод не знал про скрипт-сеттер токена и пошёл
длинным ручным путём; `/config` язык/модель искали руками.

Правила:
- Установщик — **идемпотентный**, профиль **запоминается** (env/состояние), не переспрашивается.
- Клод должен **знать про вспомогательные скрипты** стека (сеттер токена, doctor, установщик) и предлагать их,
  а не изобретать ручной путь. `/config`: язык — `Russian`, модель — по указанию (напр. Opus) — называть дословно.
- Сеть/VPN (если мешает): дать готовый рецепт «Claude мимо VPN» (маршруты/прокси) — в потребительском репо.

## Итоговый порядок (happy path)

preflight (§0) → токены через сеттер (§1) → доступ git/GitLab проверен (§2) → onboard (ядро@пин, скиллы,
`.mcp.json`, env) → внешнее человеку: платформа/плагин (§4) → рестарт Claude Code → doctor: все MCP зелёные (§3).
Каждый ручной шаг человека Клод называет ТОЧНОЙ командой/действием и каталогом, не общими словами.
