# Интеграция с 1C:EDT (живой доступ к рабочему пространству)

Наши инструменты `onec-code` и `bsl-ls` работают по файлам исходников. Для тех, кто разрабатывает в **1C:EDT**, есть
отдельный MCP-сервер-плагин, который даёт агенту доступ к **живому рабочему пространству** EDT: ошибки проекта,
подсказки, метаданные, формы, отладка, журнал регистрации, xUnitFor1C. Это дополняет toolkit, а не заменяет: поиск по
коду и правки метаданных остаются на нашей стороне, а EDT MCP добавляет то, что видит только сама IDE.

Плагин необязателен. Ставить его стоит, если вы работаете в EDT и хотите, чтобы агент видел состояние проекта в реальном
времени. Без него toolkit полностью работает на файлах.

## Какой плагин
- **`fedukhin-sys/1C_EDT_MCP_PUBLIC`** — MCP-сервер внутри 1C:EDT, около 97 инструментов (проекты, модули BSL,
  информационные базы и деплой, запуск клиента, отладка, проверки качества, редактор метаданных для 11 видов объектов,
  формы, схемы компоновки данных, журнал регистрации, xUnitFor1C, обезличивание персональных данных по 152-ФЗ).
  Лицензия Apache 2.0, ставится из готового update site. Рекомендуется 1C:EDT 2026.x.
- **`DitriXNew/EDT-MCP`** — альтернативный MCP для 1C:EDT (тот же класс задач). Можно рассмотреть как замену.

## Установка плагина (в 1C:EDT)
1. Установите 1C:EDT (edt.1c.ru), версия 2026.x рекомендуется.
2. В EDT: **Help → Install New Software → Add…**, в поле Location вставьте адрес update site и нажмите Add:
   `https://fedukhin-sys.github.io/1C_EDT_MCP_PUBLIC/`
3. Отметьте **EDT MCP** → Next → примите лицензию → Finish → перезапустите IDE.
   Офлайн-вариант: скачать ZIP из Releases репозитория и поставить через **Add → Archive…**.
4. **Window → Preferences → EDT MCP**: задайте порт (по умолчанию 3001), при желании включите автозапуск, нажмите
   **Regenerate token** и сохраните токен в безопасное место (например `~/.edt-mcp-token`).
5. Запустите сервер: **Window → EDT MCP → Start**. В статус-баре появится `MCP :3001`.

## Подключение агента (Claude Code, Cursor и другие)
Сервер отдаёт HTTP+SSE с защитой по токену. Токен держите в переменной окружения, не в файле репозитория. Добавьте в
свой локальный `.mcp.json` отдельный сервер (в дополнение к нашему профилю, не вместо него):

```json
{
  "mcpServers": {
    "edt": {
      "type": "sse",
      "url": "http://127.0.0.1:3001/mcp/sse",
      "headers": { "Authorization": "Bearer ${EDT_MCP_TOKEN}" }
    }
  }
}
```

Задайте `EDT_MCP_TOKEN` в окружении (значение из шага 4). Проверка вручную:
```bash
curl -N -H "Authorization: Bearer $EDT_MCP_TOKEN" http://127.0.0.1:3001/mcp/sse
```
Должен пойти SSE-ответ. После этого перезапустите агента и подтвердите сервер `edt`.

## Как это сочетается с toolkit
- **`onec-code`** — поиск по исходникам конфигурации и расширения. Основной способ «сверься с кодом».
- **`bsl-ls`** — диагностики BSL по файлам после правки.
- **`edt` (плагин)** — живое состояние проекта в IDE: текущие ошибки, подсказки, метаданные, формы, отладка. Полезно,
  когда важно то, что видит именно EDT, а не только файлы на диске.
- Правки метаданных остаются на движке `onec_metadata` (byte-perfect, проверка в тестовой базе). EDT MCP тут не нужен.

Безопасность: токен EDT MCP только в окружении. Слой обезличивания персональных данных в плагине не отменяет наших
правил — данные информационной базы в модель без необходимости не отправляем.
