# Доступ AI к ДАННЫМ 1С — архитектура и методика (design)

> **Статус:** архитектура + первая реализация готовы (2026-07-22). MCP `onec-data`
> (10 инструментов) и расширение `ai_debug` написаны и покрыты тестами (17 зелёных,
> регресс toolkit 525). Round-trip и синтакс-контроль расширения на живой базе
> (ERP WE 2.5.14.82 / 8.3.27.2214) **выполнены 23.07** — «Ошибок не обнаружено».
> Остаток: публикация базы на веб-сервере, без неё L2–L4 не включаются.
> План и статусы: `data-access-implementation-plan.md`. Ресёрч: разбор
> `vladimir-kharin/1c_mcp`, обзор экосистемы GitHub/openyellow/Infostart (~50 решений),
> сравнительный анализ всех механизмов доступа к данным платформы 8.3 (включая 8.3.25–8.3.27).
> Документ org-agnostic; данные конкретной организации — в её слое локализации.

Toolkit уже покрывает три слоя: **код** (`mcp/onec_mcp.py` — чтение клонов конфигураций),
**синтаксис/диагностики** (`mcp/bsl_ls_mcp.py`, bsl-platform), **эксплуатацию**
(`mcp/onec_ops_mcp.py` — ТЖ/ЖР/APDEX/Zabbix/Prometheus). Этот документ проектирует
четвёртый, недостающий слой — **данные живой ИБ**: AI видит не только код, но и данные,
настройки, права — для отладки, проверки разработок, поиска багов и написания/отладки RLS.

---

## 1. Задача и сценарии

1. **Отладка и проверка разработок** — «выполни запрос, покажи что реально лежит в регистре
   после проведения», «почему отчёт пустой» → нужен произвольный запрос к живой базе.
2. **Написание кода с пониманием данных** — какие значения у констант и функциональных опций,
   какие настройки пользователей, какие режимы включены → код пишется под реальные данные,
   а не под предположения.
3. **Разработка и отладка RLS** — «что видит пользователь X» → запрос должен выполняться
   ОТ ИМЕНИ исследуемого пользователя, с его ролями и группами доступа.
4. **Поиск багов** — данные + ЖР + код вместе: увидеть расхождение «код ожидает — данные содержат».

Требования, унаследованные от принципов toolkit: тонкая аудируемая обёртка над реальным
механизмом; **read-only по умолчанию**; секреты только в env; маскирование ПДн; работа в двух
режимах — локальный stdio и центральный docker (как `server/docker-compose.yml`).

## 2. Механизмы доступа — сравнительная матрица

| Механизм | Произвольный запрос | RLS от имени пользователя | Лицензия 1С | Доработка | Вердикт |
|---|---|---|---|---|---|
| **OData** (`odata/standard.odata`) | ❌ только $filter/$expand + функции вирт. таблиц (Balance/Turnovers/SliceLast) | ✅ честно, под Basic-auth пользователя (роль `УдаленныйДоступOData` прав не добавляет) | не расходует клиентскую | нет (публикация + состав) | **Ступень 1**: чтение без доработок |
| **HTTP-сервис в расширении** | ✅ полный язык запросов (вирт. таблицы, соединения, ВТ) | ✅ сеанс пользователя из Basic-auth; RLS честный, ЕСЛИ нет привилегированного режима | не расходует клиентскую | да, но в расширении | **Ступень 2**: главный рабочий инструмент |
| SOAP | ✅ | ✅ | не расходует | да (XDTO/WSDL) | проигрывает HTTP+JSON по всем статьям — только уже существующие операции |
| COM (`V83.COMConnector`) | ✅ | ⚠️ да, но легко случайно работать под админом | **расходует клиентскую на соединение** | нет | только Windows; резерв через Windows-узлы |
| Веб-клиент + Playwright/Chrome | ❌ | ✅✅ эталон: буквально глазами пользователя (динамические списки, доступность команд) | расходует сеанс+лицензию | нет | **Ступень 3**: финальная верификация, не канал данных |
| Прямой SQL к СУБД (read-only) | ✅ SQL по внутренним именам | ❌ **RLS обходится полностью** | вне лицензий; запись = нарушение ЛС 1С | нет | только расследование производительности (слой ops) |
| `1cv8 ENTERPRISE /Execute` | ✅ | ✅ (`/N /P`) | расходует сеанс+лицензию | нет | **уже реализовано** как ступень 2 тестирования (`apply.smoke`); для доступа к данным — не дублировать, см. §4.1 |

Ключевые факты (проверено по открытым источникам, 2026-07):
- OData у платформы так и остался **v3**; в 8.3.25–8.3.27 нового REST/OData-канала не появилось.
  Из полезного: OpenMetrics кластера (8.3.25 → стыкуется с onec-ops), «копии БД» для читающих
  запросов, WebSocket-клиент (8.3.27) для событийного пуша.
- **Нативного MCP у фирмы «1С» нет и не анонсирован** (ни 8.3.27, ни 8.5). 1С:Напарник — код/EDT
  и ИТС, доступа к данным ИБ не даёт, внешние MCP не поддерживает. Доступ LLM к данным —
  целиком территория сторонних/своих решений.
- ⚠️ Типовые на БСП часто настраивают OData под **служебного пользователя** — для отладки RLS
  так нельзя: аутентифицироваться нужно под ИССЛЕДУЕМЫМ пользователем.
- ⚠️ `УстановитьПривилегированныйРежим(Истина)` и галка «Привилегированный режим» на
  HTTP-сервисе отключают RLS — в отладочном сервисе их быть не должно.

### Вывод build-vs-adopt

**Ответ на вопрос «через 1c_mcp, браузер или OData»: ни одно из трёх не является основой в
одиночку.** Архитектура — лестница, где:

- **основной путь — OData (чтение) + свой отладочный HTTP-сервис в расширении (запросы/RLS)**,
  обёрнутые СВОИМ тонким MCP `onec-data` в духе остальных MCP toolkit;
- `1c_mcp` **не берём как основу**: из коробки пуст (2 инструмента метаданных), бизнес-логику
  всё равно писать самим на BSL, а протокольную обвязку нам даёт собственный python-MCP,
  который проще аудировать и который уже вписан в local/central-инфраструктуру toolkit.
  Заимствуем у него паттерны: OAuth2-прокси с per-user кредами, контейнерная модель;
- **браузер — не канал данных**, а ступень финальной верификации RLS/UI (динамические списки
  и доступность команд запросом не проверяются);
- **быстрый пилот** до написания своего — `feenlace/mcp-1c` (MIT, один бинарник): проверить
  ценность слоя данных на живой базе за один вечер, не принимая архитектурных обязательств.

## 4. Архитектура: лестница уровней доступа

Принцип: **каждая задача решается на самой низкой достаточной ступени**. Чем выше ступень,
тем больше возможностей — и тем жёстче контур безопасности.

```
L4  Браузер (веб-клиент, Playwright/Chrome MCP)     ── верификация «глазами пользователя», скриншоты
L3  Отладочный HTTP-сервис в расширении (onec-data) ── произвольные запросы от имени пользователя, настройки
L2  OData read-only (onec-data)                     ── чтение данных/констант без доработок, первая линия RLS
L1  Метаданные живой базы (onec-data / выгрузка)    ── структура: объекты, реквизиты, роли, права
L0  Код и логи (УЖЕ ЕСТЬ: onec-code, bsl-ls, onec-ops) ── статика, диагностики, ТЖ/ЖР, perf
```

| Ступень | Что даёт | Механизм | Риск | Когда достаточно |
|---|---|---|---|---|
| **L0** | код, диагностики, ТЖ/ЖР, метрики | существующие MCP | — | вопрос отвечается статикой/логами |
| **L1** | структура: состав объектов, типы, роли, состав прав | `$metadata` OData / выгрузка конфигурации / `Метаданные.*` через L3 | низкий | «какие поля у регистра», «в какие роли входит право» |
| **L2** | чтение данных: справочники, документы, регистры (вкл. остатки/обороты/срезы функциями), константы | стандартный OData под учёткой исследуемого пользователя | низкий (RO, честный RLS) | «что лежит в таблице», простые проверки RLS |
| **L3** | произвольный запрос (соединения, вирт. таблицы с параметрами, ВТ) от имени пользователя; настройки: ФО с параметрами, хранилища настроек, группы доступа | свой HTTP-сервис в расширении | средний → жёсткий гейт | отладка RLS-шаблонов, сложные проверки данных, «почему запрос отчёта пуст» |
| **L4** | поведение UI: видимость в динамических списках, доступность команд, условное оформление; скриншот-доказательство | веб-клиент + браузерный MCP | сеанс+лицензия, дорого по токенам | финальная верификация RLS/бага перед выводом «готово» |

Прямой SQL в лестницу данных **не входит** — он остаётся в слое L0/ops исключительно для
производительности (планы, pg_stat, блокировки): RLS он обходит, интерпретация внутренних
таблиц ненадёжна, запись запрещена лицензионным соглашением 1С.

> ⚠️ **Нумерация L0–L4 (доступ к данным) — НЕ то же, что ступени 0–4 в
> `docs/testing-ladder.md` (тестирование).** Это две разные оси, см. §4.1.

### 4.1. Связь с лестницей тестирования (`docs/testing-ladder.md`)

В toolkit **две пересекающиеся лестницы, это не дубль-конфликт, а разные оси** — держи их
раздельно и выбирай по вопросу, а не по механизму:

| | Лестница ДОСТУПА (этот док, L0–L4) | Лестница ТЕСТИРОВАНИЯ (`testing-ladder.md`, 0–4) |
|---|---|---|
| Отвечает на | «что в данных / что видит пользователь / какие настройки влияют» | «разработка ведёт себя правильно (отчёт формируется, проведение отработало)» |
| Транспорт | HTTP-сервис/OData → JSON, **интерактивный** живой диалог | batch `/Execute` runner-EPF → **файл-вердикт** (headless, CI, где нет HTTP) |
| Запись/проведение | **никогда** (read-only by design) | **можно** (в тест-базе — это и есть предмет теста) |
| Реализация | `mcp/onec_data_mcp.py` + `extensions/ai_debug/` | `onec_metadata/apply/{smoke,external,runner}.py` + `templates/smoke_runner/` |
| RLS | честный per-user (Basic-auth исследуемого) | под `/N /P` пользователя запуска |

**Перекрытие по механизму (осознанное).** `data_report` (СКД) и `data_call` (вызов метода)
формируют СКД / исполняют серверный код тем же платформенным API
(`КомпоновщикМакетаКомпоновкиДанных`, `Выполнить`), что и `smoke_runner` ступени 2. Правило
выбора, чтобы **не плодить третий путь**:

- Нужно **доказательство поведения** (вердикт OK/FAIL + артефакт), CI-прогон, контур **без
  HTTP-публикации**, или требуется **запись/проведение** → **ступень 2 тестирования**
  (`apply.smoke`), не onec-data.
- Нужно **интерактивно посмотреть данные/отчёт под конкретным пользователем** (отладка RLS,
  «почему у пользователя пусто», понимание настроек), строго read-only → **onec-data**
  (`data_report`/`data_query`/`data_call`).
- **L4 доступа = ступень 3 тестирования** (оба — веб-клиент/браузер): один механизм, разные
  цели (увидеть данные глазами пользователя vs доказать, что UI работает). Дублировать
  инструментарий не нужно — переиспользуй браузерный слой.

Прямой SQL (perf) и статика кода (ступень 0 / L0) уже согласованы: живут в onec-ops/onec-code
и не дублируются ни одной из лестниц.

## 5. Компонент `mcp/onec_data_mcp.py` — контракт (design)

Тонкий python-MCP (httpx, stdlib-стиль как onec_ops_mcp), регистрируется как `onec-data`.
Оба бэкенда — за одним MCP: OData (L1–L2) и отладочный сервис (L3), чтобы агент видел одну
модель инструментов, а сервер сам выбирал канал.

Инструменты (все read-only) — РЕАЛИЗОВАНО, `mcp/onec_data_mcp.py`:

1. `data_health(user)` — самопроверка L3: версия, под кем сеанс, разрешён ли privileged.
2. `data_metadata(mask, entity, user)` — список сущностей OData / свойства объекта (L1).
3. `data_read(entity, filter, select, top, skip, order, expand, user)` — чтение через
   OData c лимитом строк и маскированием (L2).
4. `data_virtual(register, function, params, select, top, user)` — Balance/Turnovers/
   BalanceAndTurnovers/SliceLast/SliceFirst функциями OData (L2).
5. `data_query(text, params, user, mode, top)` — произвольный текст запроса через
   отладочный сервис; выполняется в сеансе `user`; `mode` = user|privileged (L3).
6. `data_report(report, variant, params, filters, user, mode, top)` — СКД-отчёт как его
   видит пользователь: вариант настроек + параметры + отборы → плоская таблица (L3).
7. `data_settings(what, names, params, target_user, user, mode)` — константы, ФО (в т.ч.
   с параметрами), хранилища настроек, роли/группы доступа пользователя (L3).
8. `data_call(method, args, user, mode)` — вызов экспортного метода получения данных из
   whitelist расширения (L3).
9. `data_rls_probe(users[], query_text | entity+filter, params, top)` — RLS-отладка:
   один запрос под несколькими пользователями (строго mode=user), матрица «кто что видит».
10. `data_preset(name, params)` — преднастроенные выборки/отчёты/вызовы из
    `ONEC_DATA_PRESETS`, режим и пользователь запинены, подставляются только
    params_allowed (S9 «дежурные проверки»).

Сквозные правила:
- **Учётки per-user:** креды исследуемых пользователей — из env/локального secrets-файла
  (`ONEC_DATA_USERS=path`), никогда в аргументах инструментов и никогда в выводе.
- **Лимиты:** максимум строк/байт на ответ (env), усечение длинных строк — LLM не нужны
  мегабайты; всегда сообщать «показано N из M».
- **Маскирование ПДн:** конфигурируемый список полей-масок (ФИО, телефоны, ИНН…) —
  паттерн ROCTUP/RSV Data; включено по умолчанию, отключается явным env только для тест-баз.
- **Никакой записи и никакого `Выполнить()`** — в контракте отсутствуют write-инструменты
  принципиально (не «выключены», а не существуют). Тестовое проведение в откатываемой
  транзакции — возможное будущее расширение L3, отдельным решением.
- **Аудит:** каждый вызов L3 логируется (кто, какой запрос, под кем) в локальный журнал MCP.

## 6. Отладочное расширение (L3) — контракт HTTP-сервиса

Минимальное расширение `ai_debug` (не трогает конфигурацию поставщика): HTTP-сервис
с Basic-auth, БЕЗ привилегированного режима, три операции:

- `POST /query` — `{text, params}` → выполняет `Запрос` в сеансе аутентифицированного
  пользователя, JSON-ответ с лимитом строк. RLS применяется честно by construction.
- `GET /settings?what=constants|fo|user_settings|access` — чтение настроек (см. `data_settings`).
- `GET /health` — версия расширения + имя текущего пользователя сеанса (самопроверка «под кем я»).

Гейт развёртывания: **только тест-контуры** (копии прод-баз) либо, для прод-чтения,
белый список пользователей/IP на веб-сервере + отдельная публикация. Запрос-executor на
проде без гейта = дыра — это осознанное ограничение, не дефолт.

Реализация BSL — своя (~200–300 строк) ЛИБО адаптация контейнера под движок `1c_mcp`,
если организация уже развернула его; контракт HTTP при этом сохраняется.

## 7. Режимы работы: локальный и центральный

Тот же паттерн, что у `onec-code`/`onec-ops` (см. `server/docker-compose.yml`):

- **Локальный (по умолчанию):** `onec-data` в `mcp/dev.mcp.json` как stdio; env — URL
  публикации, путь к secrets-файлу учёток. Работает с любой машины разработчика,
  которой сетево доступна публикация ИБ.
- **Центральный (команда):** сервис `onec-data` в compose рядом с `onec-code`/`onec-ops`,
  наружу — через тот же Caddy c bearer-токеном; секреты учёток 1С живут ТОЛЬКО на сервере
  (клиенты знают bearer, но не пароли 1С). Опционально — OAuth2-фасад per-user (паттерн
  `1c_mcp`), когда команде нужно ходить каждому под своими кредами.

Docker-песочница 1С (паттерн `SteelMorgan/1c-ai-sandbox`) — рекомендуемое будущее для
центрального тест-контура: изолированная копия базы, которую агенту не страшно нагружать.

## 8. Безопасность — железные правила слоя данных

1. Read-only by design: write-инструментов в MCP нет; отладочное расширение не пишет.
2. RLS-выводы — только с каналов, где RLS честный (L2/L3/L4). Из SQL — никогда.
3. Привилегированный режим в отладочном сервисе запрещён (ревью расширения обязано это проверять).
4. ПДн-маскирование включено по умолчанию; сырые данные прод-базы во внешние сервисы не отдаются.
5. Учётки/токены — env/secrets-файлы вне репозитория; в аргументах и выводе инструментов их нет.
6. L3 на проде — только за белым списком; дефолтная среда — тест-копия.
7. Аудит вызовов L3 обязателен.
8. Лицензии: OData/HTTP-сервисы клиентских лицензий не тратят; браузерная ступень (L4) и
   /Execute тратят сеанс+лицензию — использовать точечно и закрывать сеансы.

## 9. Методика отладки RLS (рабочий цикл)

1. **Статика (L0):** прочитать шаблоны ограничений роли и код заполнения групп доступа
   (onec-code); зафиксировать ОЖИДАЕМОЕ поведение письменно.
2. **Контекст пользователя (L3 `data_settings` / L1):** роли, группы доступа, значения ФО —
   подтвердить, что пользователь сконфигурирован так, как предполагает гипотеза.
3. **Данные (L2/L3):** выполнить контрольный запрос под админом (эталон «всего») и под
   исследуемым пользователем (`data_rls_probe`) — расхождение и есть эффект RLS; сравнить
   с ожидаемым из шага 1.
4. **Сложные случаи (L3 `data_query`):** воспроизвести ровно тот запрос, что в отчёте/коде,
   от имени пользователя — так ловятся эффекты соединений с ограниченными таблицами.
5. **Верификация UI (L4):** войти веб-клиентом под пользователем, убедиться в видимости/
   невидимости в динамическом списке и доступности команд; скриншот — в отчёт о проверке.
6. **Производительность RLS (L0):** если под пользователем медленно — ТЖ/план запроса через
   onec-ops/1c-expert (RLS-условия раздувают SQL — это отдельный класс багов).

Тот же цикл без шагов RLS — общая методика «проверить разработку по данным»: ожидание из
кода → настройки → фактические данные → при расхождении ЖР/ТЖ.

## 10. План внедрения

| Этап | Содержание | Критерий готовности |
|---|---|---|
| 0. Пилот (1 вечер) | `feenlace/mcp-1c` на тест-базе: подтвердить ценность слоя данных на 2–3 реальных задачах отладки | субъективный вердикт + список нужных инструментов уточнён |
| 1. OData-ступень | публикация тест-базы + состав OData + роль `УдаленныйДоступOData`; `onec_data_mcp.py` c инструментами L1–L2 + тесты | data_read/data_virtual/data_rls_probe работают на тест-базе; лимиты и маскирование покрыты тестами |
| 2. L3-ступень | расширение `ai_debug` (query/settings/health) + `data_query`/`data_settings` в MCP; round-trip на тест-базе | RLS-цикл §9 проходит целиком на реальном кейсе; аудит пишется |
| 3. Центральный режим | сервис в compose + Caddy; secrets на сервере | команда подключается по bearer, локальный режим не ломается |
