# Развёртывание центра onec-code «под ключ» (runbook для Claude)

Этот файл — **исполняемая инструкция для ассистента**: Claude читает её и сам разворачивает центральный
`onec-code` MCP от начала до проверки. Org-agnostic — без имён организаций, хостов, репозиториев (подставляй из env/ввода).
Справочник по стеку (что внутри, безопасность, тюнинг) — `docs/docker.md`; подключение клиента к готовому центру —
`docs/connect.md`.

---

## ШАГ 0 — развилка: разворачивать центр или нет (СНАЧАЛА реши это)

Центр нужен **не каждому**. Сначала определи роль человека:

- **Разработчик/аналитик присоединяется к команде** → центр, скорее всего, УЖЕ поднят. **НЕ разворачивай.**
  Иди в `docs/connect.md`: `set_token` (URL+токен у владельца центра) → `switch_source --mode central`. Конец.
- **Один разработчик, оффлайн / нет сервера** → центр не нужен вовсе. Локальный stdio-режим: `onboard` без `--central`
  склонирует код в `ONEC_SRC_DIR`, источник `onec-code` = local. Конец.
- **Ты — владелец/админ и центра ещё нет** (или поднимаешь свой одно-хостовый центр) → разворачивай. Продолжай ниже.

> Если не уверен, к какой ветке относится запрос, — **спроси пользователя один вопрос**: «подключиться к существующему
> центру или развернуть новый?» Не разворачивай центр по умолчанию — это инфраструктурное действие.

Когда пользователь подтвердил «разворачиваем» — пройди шаги 1→6 по порядку, не пропуская проверок.

---

## ШАГ 1 — предусловия (проверь, не угадывай)

Выполни и убедись, что всё на месте на ЦЕЛЕВОМ хосте (локальном или удалённом по ssh):

```bash
docker --version && docker compose version     # Docker + plugin compose v2
git --version                                  # для клонирования/обновления кода 1С
openssl version                                # для генерации токена (или любой генератор случайных строк)
```

Чего не хватает — доложи пользователю и останови развёртывание (не ставь Docker молча на чужой сервер).
Если хост удалённый — все команды ниже выполняй на нём (ssh), а не на машине разработчика.

Узнай у пользователя / определи:
- **каталог исходников кода 1С** на хосте → станет `ONEC_SRC_DIR` (абсолютный путь);
- **порт наружу** → `ONEC_PUBLIC_PORT` (по умолчанию 8000; на занятом хосте выбери свободный);
- **доступ к git-репозиториям** конфигураций/расширений (что клонировать в каталог исходников).

---

## ШАГ 2 — код 1С в каталог исходников (read-only вход центра)

Центр отдаёт ровно то, что лежит в `ONEC_SRC_DIR` (монтируется в контейнер **read-only**). Положи туда клоны
всех нужных слоёв (типовая/конфигурации/расширения/библиотеки) — каждый отдельным подкаталогом:

```bash
mkdir -p "$ONEC_SRC_DIR"
cd "$ONEC_SRC_DIR"
git clone <repo-1> ; git clone <repo-2> ; ...     # перечень репозиториев — у владельца кода
```

- Имена репозиториев/организация — НЕ в этом файле: бери из ввода пользователя или локализации.
- Этот каталог центр только читает. Правки кода тут не делаются — это вход поиска, не рабочая копия.
- Для автообновления кода (шаг 6) клонируй по **read-only** доступу (deploy-key без push), а не по своему write-токену.

---

## ШАГ 3 — токен и server/.env (секрет — мимо git и чата)

1. Сгенерируй общий секрет авторизации и **сразу запиши его в `.env`**, не печатая в чат:

```bash
cd server
cp .env.example .env
# подставь значения; токен сгенерируй на месте и НЕ выводи в ответ пользователю:
TOKEN="$(openssl rand -hex 32)"
```

2. Заполни `server/.env` (правкой файла, не echo секрета в терминал):
   - `ONEC_SRC_DIR=` — абсолютный путь из шага 1;
   - `ONEC_BEARER_TOKEN=` — сгенерированный токен;
   - `ONEC_PUBLIC_PORT=` — порт наружу;
   - (опц.) `ONEC_LAYERS_CONFIG=` — свой конфиг слоёв, если он есть (иначе чистое auto-discovery).

Безопасность токена (тот же принцип, что у клиента):
- `server/.env` **не коммитится** (он в `.gitignore`) — проверь, что так и есть.
- Токен НЕ выводи в чат и НЕ клади в git. Разработчикам отдашь его **вне репозитория** (шаг 6, out-of-band).
- Никаких других секретов в образ/compose — центр читает только код.

---

## ШАГ 4 — поднять сервис

```bash
cd server
docker compose up -d --build
docker compose ps
```

Ожидаемо: `proxy` (Caddy) слушает `:${ONEC_PUBLIC_PORT}` наружу; `onec-code` — только во внутренней сети
(`expose`, не `ports`); `/src` смонтирован read-only. Если `proxy` не стартует с ошибкой про `ONEC_BEARER_TOKEN` —
токен не попал в `.env` (вернись к шагу 3).

## ШАГ 5 — проверка «центр живой и закрыт» (без этого не объявляй готовым)

```bash
# 1) без токена — 401 (центр не открыт наружу без авторизации):
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:${ONEC_PUBLIC_PORT}/sse           # ждём 401

# 2) с токеном — проход и реальный ответ MCP по коду:
ONEC_BEARER_TOKEN="<токен из .env>" uv run --with mcp tests/smoke_http_client.py \
    http://127.0.0.1:${ONEC_PUBLIC_PORT}/sse
```

`smoke_http_client.py` подключается по SSE, перечисляет инструменты и зовёт `find_object`/`search_1c`/`list_modules`
на реальном наборе. Должны вернуться слои с тегами. **401 без токена + осмысленный ответ с токеном = центр готов.**
Любой иной исход — разбирайся (`docker compose logs onec-code` / `... proxy`), не объявляй «готово».

## ШАГ 6 — раздать подключение и (опц.) автообновление

**Раздать разработчикам** — out-of-band (не в git, не в общий чат пачкой): URL `http://<хост>:<порт>/sse` + токен.
Каждый у себя делает `scripts/set_token.*` (см. `docs/connect.md`) — токен ляжет в env пользователя, в `.mcp.json`
попадёт лишь плейсхолдер.

**Автообновление кода (по желанию).** Чтобы центр не отставал от репозиториев — периодический `git pull` каталога
исходников + при изменениях пересборка. Принципы, чтобы не выстрелить себе в ногу:
- доступ к репозиториям — **read-only** (deploy-key без push), write-токен на общий сервер не класть никогда;
- пересборка с **fail-safe**: старый контейнер продолжает работать, если новая сборка упала;
- расписание — `cron`/таймер на хосте; код монтируется read-only, контейнер перечитывает корни на рестарте
  (`docker compose restart onec-code` после `git pull`, либо `up -d --build` при изменении стека).

Пример сторожевого скрипта-обёртки (псевдо, адаптируй под хост — пути и список репо подставь свои):
```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$ONEC_SRC_DIR" && for d in */; do (cd "$d" && git pull --ff-only) ; done
cd "$SERVER_DIR" && docker compose up -d --build   # пересоберёт только при изменениях; старый жив при ошибке
```

## Ротация токена / снятие центра

- Сменить токен: поправь `ONEC_BEARER_TOKEN` в `server/.env` → `docker compose up -d` → раздай новый токен заново.
- Остановить: `cd server && docker compose down` (данные не теряются — код лежит на хосте).
- Выставление за пределы доверенной сети — только с TLS (Caddy умеет автоматически по доменному имени); см. `docs/docker.md`.
