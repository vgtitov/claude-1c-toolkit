# Производительность и кэширование кода 1С (BSL)

Prod-ready = ещё и производительный. Сигнатуры БСП — по 3.1.x, сверяй по своей версии. Если режим совместимости
ниже 8.3.18: async/`Ждать` НЕ использовать (фоновость — через подсистему `ДлительныеОперации`, не языковой async).
Правило: **сначала измерь узкое место (см. §5), потом оптимизируй именно его** — не кэшируй «на всякий случай».
Запросы — отдельный файл `references/code-standards.md` (топ-20 правил производительных запросов).

## 1. Кэширование

### ПовтИсп (Повторное использование возвращаемых значений) — главный механизм
Свойство общего модуля. Платформа кэширует результат ЭКСПОРТНОЙ ФУНКЦИИ по значениям её параметров.
Требования: только функции; параметры и результат сериализуемы; функция чистая (без побочных эффектов).
- **На время вызова** — кэш до конца серверного вызова. Безопасно. Для дорогого расчёта, многократного в одной операции.
- **На время сеанса** — до конца сеанса / до сброса. Для редко меняющегося (настройки, константы, параметры учёта).

```bsl
// Общий модуль НастройкиУчетаПовтИсп: Сервер=Истина; ПовторноеИспользование=НаВремяСеанса
Функция ВалютаУправленческогоУчета() Экспорт
    Возврат Константы.ВалютаУправленческогоУчета.Получить();
КонецФункции
```

**Подводные камни (обязательно учитывать):**
- Кэш «На время сеанса» сам НЕ сбрасывается. После изменения закэшированных данных — `ОбновитьПовторноИспользуемыеЗначения()`
  (глобальный, сбрасывает весь ПовтИсп-кэш сеанса), например в `ПриЗаписи` константы/настройки.
- Параметры = ключ кэша. НЕ передавать `ТекущаяДата()` или объекты с UUID — кэш всегда промахивается и растёт.
- НЕ возвращать мутабельные объекты, которые вызывающий будет менять (испортит кэш для всех). Отдавать через
  `ОбщегоНазначения.ФиксированныеДанные(Данные)` (сверено: `(Данные, ВызыватьИсключение=Истина)`) или `Новый ФиксированныйСтруктура(...)`.
- Не кэшировать данные под RLS, если ключ не включает пользователя/организацию.

### Прочие уровни кэша
- **Параметры сеанса** — ленивая инициализация в `УстановкаПараметровСеанса`; сброс `ПараметрыСеанса.X = Неопределено`.
- **Реквизиты формы** — дорогой расчёт один раз в `ПриСозданииНаСервере`, дальше читать из реквизита (без серверных вызовов).
- **Приватные переменные модуля** (`Перем`) — ленивое поле в пределах объекта/обработки.
- **Хранилища настроек** (персистентный кэш между сеансами): `ОбщегоНазначения.ХранилищеОбщихНастроекСохранить/`
  `Загрузить/Удалить/СохранитьМассив(...)` — с обработкой прав/длинных ключей (лучше прямого `ХранилищеОбщихНастроек.Сохранить`).

### Lazy init
```bsl
Функция КурсВалюты()
    Если _КэшКурса = Неопределено Тогда _КэшКурса = РассчитатьКурс(); КонецЕсли;  // приватная Перем
    Возврат _КэшКурса;
КонецФункции
```
Эквивалент через ПовтИсп «На время вызова» предпочтительнее, когда расчёт чистый (платформа сама кэширует и сбрасывает).

### Когда кэш ВРЕДЕН
Часто меняющиеся данные (остатки/цены/статусы) → максимум «На время вызова». RLS без пользователя в ключе.
Мутабельные объекты в многопользовательском режиме. Большой вариативный ключ (нет попаданий, ест память).

## 2. Минимизация накладных расходов
- **Меньше клиент→сервер вызовов.** В цикле на клиенте серверных вызовов нет — собрать и сделать ОДИН вызов.
- **`&НаСервереБезКонтекста`** где не нужны данные формы — вызов в разы дешевле (не гоняется контекст формы).
- **Чтение пачкой**, не в цикле через точку — `ОбщегоНазначения.ЗначенияРеквизитовОбъектов(Ссылки, Реквизиты)`
  (см. `references/bsp-top-methods.md`).
- **Привилегированный режим** для служебных операций: `УстановитьПривилегированныйРежим(Истина)` … `(Ложь)` —
  не гонять RLS (быстрее и корректнее для технической логики; НЕ как обход прав в прикладном коде).
- **Меньше лишней работы:** не перепроводить ради пересчёта (писать набор записей точечно); не плодить запросы в
  цикле (пакетный запрос + `МенеджерВременныхТаблиц`); создание `Запрос`/`СКД`/форматных строк — до цикла, в цикле
  менять только параметры; не поднимать объект целиком (`.ПолучитьОбъект()`) ради реквизитов — читать запросом.
  **ENFORCED-детектор** (не только это правило): `python scripts/bsl_guard.py <.bsl|каталог>` (exit 1 = находки,
  вывод `файл:строка — [код] что переделать`); он же на коммите — pre-commit хук на staged `*.bsl`. Ловит
  Запрос…Выполнить()/`Новый Запрос`/`.ПолучитьОбъект()`/`.Ссылка.<Реквизит>` в теле `Пока/Для`.

## 3. Длительные операции (подсистема БСП `ДлительныеОперации`)
Длинную обработку — в фон, не в синхронный серверный вызов. Сигнатуры сверены с исходником:
- `ДлительныеОперации.ВыполнитьФункцию(ПараметрыВыполнения, ИмяФункции, П1…П7)` → структура с `Статус`,
  `ИдентификаторЗадания`, `АдресРезультата`, `КраткоеПредставлениеОшибки`, `ПодробноеПредставлениеОшибки`, `Сообщения`.
- `ВыполнитьПроцедуру(...)`, `ПараметрыВыполненияФункции(ИдентификаторФормы)`, `ПараметрыВыполненияПроцедуры()`,
  `ВыполнитьФункциюВНесколькоПотоков(ИмяФункции, ПараметрыВыполнения, НаборПараметров)`, `ВыполнитьВФоне(...)` (низкоуровневый).
- Прогресс (сверено): `ДлительныеОперации.СообщитьПрогресс(Процент=Неопределено, Текст=Неопределено, ДопПараметры=Неопределено)`
  внутри фоновой функции; `ПрочитатьПрогресс(ИдентификаторЗадания)` снаружи.
- Клиент: `ДлительныеОперацииКлиент.ПараметрыОжидания(Форма)`, `ОжидатьЗавершение(Операция, Оповещение, ПараметрыОжидания)`.

```bsl
&НаСервере
Функция НачатьВыполнениеНаСервере()
    ПВ = ДлительныеОперации.ПараметрыВыполненияФункции(УникальныйИдентификатор);
    Возврат ДлительныеОперации.ВыполнитьФункцию(ПВ, "Обработка.МояОбработка.РассчитатьЗначение", П1, П2);
КонецФункции
```
Параметры/результат — сериализуемые; параметры — `Знач`. `КлючФоновогоЗадания` — чтобы не плодить дубли активного задания.

## 4. Порционная обработка больших выборок
Не грузить всё в память и не держать одну гигантскую транзакцию — порциями, граница по ссылке (keyset, не СМЕЩЕНИЕ):
```bsl
Запрос.Текст = "ВЫБРАТЬ ПЕРВЫЕ 1000 Ссылка ИЗ Документ.РеализацияТоваровУслуг
|ГДЕ Ссылка > &ПоследняяСсылка И НЕ Проведен УПОРЯДОЧИТЬ ПО Ссылка";
Пока Истина Цикл
    Запрос.УстановитьПараметр("ПоследняяСсылка", ПоследняяСсылка);
    Выборка = Запрос.Выполнить().Выбрать();
    Если Выборка.Количество() = 0 Тогда Прервать; КонецЕсли;
    Пока Выборка.Следующий() Цикл ОбработатьДокумент(Выборка.Ссылка); ПоследняяСсылка = Выборка.Ссылка; КонецЦикла;
КонецЦикла;
```
Короткие транзакции (меньше блокировок), ограниченная память, возобновляемость. Очень большие объёмы — многопоточно.

## 5. Паттерны проектирования на 1С (приземлённые)
- **Facade** — общий модуль-фасад с `ПрограммныйИнтерфейс`, детали в `*Служебный`. Так устроена вся БСП.
- **Strategy** — выбор алгоритма по данным (диспетчер по перечислению / имя модуля+`Выполнить`).
- **Registry/Реестр** — регистр сведений или соответствие обработчиков, наполняется при старте.
- **Singleton** — ПовтИсп «На время сеанса» (один закэшированный экземпляр); параметры сеанса.
- **Lazy Load** — ленивое поле / ПовтИсп «На время вызова».
- **Observer** — подписки на события (`ПодпискиНаСобытия`) + определяемые `*Переопределяемый` события БСП.
- **Adapter** — общий модуль-коннектор интеграции (наружу единый метод, внутри маппинг под чужой API; имя с префиксом твоего расширения).
- **Repository** — общий модуль `Получить…/Записать…`, инкапсулирует запросы; прикладной код не пишет запросы напрямую.

## 6. Измерение (мерить, а не гадать)
- Замер производительности в Конфигураторе/EDT (построчно) — первый инструмент локально.
- Технологический журнал (`logcfg.xml`): `SDBL`/`DBMSSQL` (долгие/частые запросы), `CALL`, `EXCP`, ожидания на блокировках.
- АПДЕКС/целевые времена ключевых операций; ЦУП/ЦКК — серверный мониторинг прода (долгие запросы, дедлоки).
- План запроса для конкретного тяжёлого запроса (сканы вместо индексов, соединения с подзапросами).
- (БСП-подсистема замера АПДЕКС из кода — имя уточнить через onec-code; не подтверждено как `ОценкаПроизводительности`.)
