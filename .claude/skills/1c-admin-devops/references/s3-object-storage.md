# S3-совместимое объектное хранилище и 1С

Правило раздела: **секреты S3 (access key / secret key) — только в env или защищённом хранилище (keychain, vault,
параметры тома в защищённой части ИБ), НИКОГДА в код, конфиг репозитория или чат.** Все endpoint/bucket/ключи ниже —
плейсхолдеры. Версии платформы — **[проверить под свою версию]** по официальной документации (см. источники в конце).

S3 (Simple Storage Service) — протокол объектного хранилища: данные кладутся как *объекты* (ключ + содержимое +
метаданные) в *бакеты* (buckets), доступ по HTTP(S) REST API с подписью. Совместимы Amazon S3, MinIO, Ceph RGW,
Яндекс Object Storage, VK Cloud, Mail.ru (hb.bizmrg.com), Cloud.ru, CROC, Timeweb и др. В 1С S3 встречается в двух
ролях: **(1) нативно** — платформа сама хранит двоичные данные/файлы в S3; **(2) программно** — BSL-код ходит в S3
API напрямую (своя интеграция, выгрузка отчётов, обмен). Ниже обе стороны.

## 1. Нативно платформой 1С (хранилище двоичных данных / тома файлов)

### 1.1 Хранилище двоичных данных по протоколу S3 — с версии 8.3.24

Начиная с платформы **8.3.24** в качестве хранилища двоичных данных (то, где платформа держит содержимое реквизитов
типа «ХранилищеЗначения», присоединённых файлов и т. п.) **можно подключать внешние хранилища по протоколу S3**.
Заявлено официально в «Новое в платформе 8.3.24» (v8.1c.ru) и анонсе consulting.1c.ru. **[проверить под свою версию
платформы — формулировки и доступность могут отличаться по релизам и редакциям лицензии]**

Ключевое из релиза 8.3.24:
- Внешнее S3-хранилище двоичных данных — данные выносятся из основной СУБД в объектное хранилище.
- **Дедупликация** двух видов: *на записи* (объекты с полным побайтовым совпадением не дублируются) и *фоновая*
  (применяется к уже записанным данным в фоне).
- **Администрирование хранилища:** статистика (объём, число объектов, время последней очистки), оценка
  дедуплицированных объёмов, явный запуск очистки удалённых объектов.
- **Ограничение:** работает в **клиент-серверном** режиме; файловые ИБ не поддерживаются. **[проверить]**

Развитие в последующих релизах (фиксируй под свою версию):
- **8.3.27** — изменены механизмы хранения двоичных данных (в т. ч. в сторону более прозрачной работы с S3);
  на этой версии построены практические разборы (almaz-sharipov.ru, форум Инфостарт по «методам платформы 8.3.27»).
- **8.3.28** — расширенные функции управления двоичными данными. **[проверить под свою версию]**

> Важно различать слои. «Хранилище двоичных данных» — это **платформенный** механизм (настраивается администратором
> кластера/ИБ, прозрачен для прикладного кода). Отдельно есть **прикладной** механизм БСП «Работа с файлами» — справочник
> **«Тома хранения файлов»**; в ряде сборок/расширений у тома появляется **тип/настройка S3** (endpoint, бакет, ключи).
> Доступность S3-тома зависит от **версии БСП и конфигурации**, не только от платформы — **[проверить по своей БСП]**.

### 1.2 Настройка (типовые параметры)

Состав параметров подключения к S3 (имена реквизитов зависят от механизма/версии — сверь по документации):

| Параметр | Назначение | Плейсхолдер / пример |
| --- | --- | --- |
| Endpoint (адрес сервера) | URL S3-сервиса | `https://s3.example-provider.tld` или `http://minio-host:9000` |
| Bucket (бакет/корзина) | имя контейнера объектов | `<your-bucket>` |
| Access Key ID | идентификатор ключа доступа | `<ACCESS_KEY>` — из env/секретницы |
| Secret Access Key | секретный ключ | `<SECRET_KEY>` — **только из защищённого хранилища** |
| Region (регион) | регион подписи | `us-east-1` (дефолт MinIO), `ru-central1` (Яндекс) и т. п. |
| Path-style / Virtual-hosted | формат адресации бакета | path-style для MinIO/он-прем (см. §3) |

Порядок настройки тома хранения файлов (механизм БСП «Работа с файлами», обобщённо):
1. Включить хранение файлов в томах (НСИ → Администрирование → «Хранить файлы в томах на диске»).
2. Завести элемент справочника **«Тома хранения файлов»**, у тома выбрать тип/режим **S3** (если поддерживается).
3. Заполнить endpoint, бакет, регион, ключи доступа; задать путь/префикс ключей объектов.
4. Проверить запись/чтение тестовым файлом; включить регламент проверки целостности тома.

### 1.3 Ограничения и нюансы нативного механизма

- Клиент-серверный режим (файловые ИБ — нет). **[проверить]**
- Хранение ключей: в части реализаций реквизиты подключения лежат в служебных таблицах ИБ (напр.
  `binarydatastoragecontent`) **в незашифрованном виде** — учитывай при модели угроз; ограничивай доступ к СУБД/ИБ.
- При удалении настройки хранилища ссылки на двоичные данные могут «повиснуть» → риск потери данных; меняй
  конфигурацию хранилища осознанно, с бэкапом.
- Часть сценариев не поддерживает перенос ссылок на двоичные данные между базами и параллельную запись в разные
  хранилища — проверяй ограничения своей версии перед проектированием. **[проверить]**
- Совместимость и доступность функций зависят от **режима совместимости** конфигурации и **редакции лицензии** —
  фиксируй под свой стек (см. корневой CLAUDE.md «Версионный стек»).

## 2. Программно из BSL (S3 API + AWS Signature V4)

Когда нужна не «прозрачная» выгрузка платформой, а **управляемая** работа с объектами (свой бакет для интеграций,
выгрузка отчётов/архивов, обмен через S3, presigned-ссылки), 1С ходит в S3 REST API **сама** через
`HTTPСоединение`/`HTTPЗапрос`/`HTTPОтвет`. Реализуемо на **чистом BSL** без внешних компонент (загрузка, скачивание,
удаление, листинг). Главная сложность — корректная **подпись AWS Signature V4**.

### 2.1 Операции (REST-маппинг)

| Операция | HTTP | URI (path-style) |
| --- | --- | --- |
| PutObject (загрузить) | `PUT` | `/<bucket>/<key>` + тело-байты |
| GetObject (скачать) | `GET` | `/<bucket>/<key>` |
| DeleteObject (удалить) | `DELETE` | `/<bucket>/<key>` |
| ListObjects(V2) (листинг) | `GET` | `/<bucket>?list-type=2&prefix=...` |
| HeadObject (метаданные) | `HEAD` | `/<bucket>/<key>` |

### 2.2 Подпись AWS Signature V4 — шаги

Алгоритм подписи — `AWS4-HMAC-SHA256`. Все хэши/HMAC по SHA-256 (в 1С — `ХешированиеДанных` с
`ХешФункция.SHA256` и `HMAC`), итоговая подпись — **строчный hex** (важно: оборачивать в `НРег()`/`Lower()`).

**Шаг 1. Canonical Request** (порядок строк, разделитель — перевод строки `\n`):
```
<HTTP-МЕТОД>\n
<CanonicalURI: путь, URL-энкод, '/' не кодируется>\n
<CanonicalQueryString: параметры, отсортированы по имени, URL-энкод>\n
<CanonicalHeaders: 'имя:значение\n', имена в нижнем регистре, отсортированы>\n
<SignedHeaders: имена через ';', нижний регистр, отсортированы>\n
<PayloadHash: SHA256(тело) в hex, или 'UNSIGNED-PAYLOAD'>
```
Обязательны к подписи `host` и все `x-amz-*` заголовки. Минимальный набор: `host`, `x-amz-date`,
`x-amz-content-sha256`.

**Шаг 2. String to Sign:**
```
AWS4-HMAC-SHA256\n
<X-Amz-Date: формат 20240115T093000Z (UTC, ISO8601 basic)>\n
<CredentialScope: <yyyymmdd>/<region>/s3/aws4_request>\n
<HEX(SHA256(CanonicalRequest))>
```

**Шаг 3. Signing Key** (вложенные HMAC-SHA256):
```
kDate    = HMAC("AWS4" + SecretKey, <yyyymmdd>)
kRegion  = HMAC(kDate,   <region>)        // напр. us-east-1
kService = HMAC(kRegion, "s3")
kSigning = HMAC(kService,"aws4_request")
Signature = HEX( HMAC(kSigning, StringToSign) )   // строчный hex
```

**Шаг 4. Заголовки запроса:**
- `X-Amz-Date: 20240115T093000Z`
- `X-Amz-Content-SHA256: <hex sha256 тела>` (или `UNSIGNED-PAYLOAD`)
- `Authorization: AWS4-HMAC-SHA256 Credential=<AccessKey>/<yyyymmdd>/<region>/s3/aws4_request, SignedHeaders=<...>, Signature=<...>`

### 2.3 Presigned URL

Альтернатива header-based: подпись передаётся в query-string (`X-Amz-Algorithm`, `X-Amz-Credential`, `X-Amz-Date`,
`X-Amz-Expires`, `X-Amz-SignedHeaders`, `X-Amz-Signature`). Даёт временную ссылку на объект без передачи ключей —
удобно отдать пользователю/внешней системе ссылку на скачивание/загрузку. У многих провайдеров срок жизни ограничен
(напр. до 7 суток у AWS; у конкретного провайдера — **[проверить]**).

### 2.4 Готовые наработки и объекты 1С

- **Объекты:** `HTTPСоединение` (соединение + SSL), `HTTPЗапрос` (метод, URI, заголовки, тело), `HTTPОтвет`.
  Для TLS — `ЗащищённоеСоединениеOpenSSL`/`OpenSSLСертификатыКлиента`/`...УдостоверяющихЦентров`.
- **github BlizD/S3_SSL** — расширение к БСП «Работа с файлами»: хранит файлы тома в S3 (MinIO, Amazon S3, Mail.ru
  Cloud). Использует библиотеку `vbondarevsky/Connector` (HTTP-клиент). Нюанс: требует **отключённого безопасного
  режима** расширения; сверь версии БСП/платформы под свой стек. **[проверить]**
- **infostart 2033497** «Из 1С в S3 и обратно» — разбор PutObject/GetObject/DeleteObject/ListObjects и SigV4 на
  чистом BSL (на примере Яндекс Object Storage, path-style).
- **habr (VK) 892824** — практический кейс адаптации механизма «Тома хранения файлов» под S3 (AWS4-HMAC-SHA256).

### 2.5 Типичные ошибки

| Ошибка / симптом | Причина | Что проверить |
| --- | --- | --- |
| `SignatureDoesNotMatch` | расхождение в canonical request / ключе подписи | порядок и нижний регистр заголовков; сортировка query; **точные байты тела** в payload hash; `region`/service в scope |
| `RequestTimeTooSkewed` | время клиента ≠ времени сервера >15 мин | синхронизировать часы (NTP); `X-Amz-Date` строго в **UTC**, формат `yyyymmddThhmmssZ` |
| `SignatureDoesNotMatch` (только знак) | подпись в ВЕРХНЕМ регистре | hex-подпись должна быть **строчной** (`НРег()`) |
| `NoSuchBucket` / DNS-ошибка | virtual-hosted адресация к он-прем | использовать **path-style** (`/bucket/key`), см. §3 |
| `AccessDenied` | неверный регион в scope | для MinIO дефолт `us-east-1`; у провайдера — свой регион |
| 301/`PermanentRedirect` | бакет в другом регионе (AWS) | подписывать под фактический регион бакета |

Частые грабли: подписывать **не те** заголовки, что реально отправляются; забыть `host` в SignedHeaders; кодировать
`/` в пути; считать payload hash от изменённого тела (gzip/кодировка). Для отладки — `UNSIGNED-PAYLOAD` временно
снимает зависимость от хэша тела (по HTTPS).

## 3. MinIO / self-hosted S3

**MinIO** — популярное self-hosted S3-совместимое хранилище (он-прем, для тестов и прод-он-прем). API-порт по
умолчанию **9000**, веб-консоль **9001**. Запуск (Docker, dev-режим, креды-плейсхолдеры):

```bash
docker run -d --name minio -p 9000:9000 -p 9001:9001 \
  -e MINIO_ROOT_USER="$MINIO_ROOT_USER" \
  -e MINIO_ROOT_PASSWORD="$MINIO_ROOT_PASSWORD" \
  -v /srv/minio/data:/data \
  quay.io/minio/minio server /data --console-address ":9001"
```
(креды — через переменные окружения, не литералами; для прод — за reverse-proxy с TLS.)

Нюансы под 1С:
- **Регион:** MinIO по умолчанию **`us-east-1`** — именно его указывай в CredentialScope подписи (иначе
  `SignatureDoesNotMatch`/`AccessDenied`). Можно переопределить (`MINIO_REGION`).
- **Только path-style:** MinIO **не** поддерживает virtual-hosted (`bucket.host`) без отдельной DNS-настройки →
  обращайся **`endpoint/bucket/key`** (`forcePathStyle`/`addressing_style=path` в SDK-терминах).
- **Endpoint:** `http://minio-host:9000` (dev) / `https://s3.internal.example` (прод за TLS). HTTP только в
  доверенной сети; наружу — обязательно HTTPS.
- **Совместимость:** MinIO/Ceph/прочие реализуют **подмножество** S3 API — проверяй наличие нужных операций
  (ListObjectsV2, multipart, presigned). **[проверить у своего сервера]**
- Бакет/политики создаются заранее (`mc` CLI или веб-консоль 9001); для 1С-доступа — отдельный сервис-аккаунт с
  минимальными правами на конкретный бакет.

MinIO удобен как **локальный полигон**: отлаживаешь SigV4 на нём (`us-east-1`, path-style), затем меняешь только
endpoint/region/ключи под боевого провайдера.

## 4. Эксплуатация (бэкап, доступы, нюансы)

**Доступы и секреты.**
- Ключи S3 — в **env/секретнице/защищённой части ИБ**, не в коде и не в репозитории. Для интеграций заводи
  **отдельный сервис-аккаунт** с правами только на нужный бакет/префикс (least privilege), а не root-креды MinIO.
- Ротация ключей: предусмотри смену access/secret без простоя (две пары ключей на период ротации).
- Сетевой контур: S3-трафик 1С↔хранилище — внутри доверенной сети или по TLS; наружу — только HTTPS.

**Бэкап и сохранность.**
- Нативное S3-хранилище двоичных данных: бэкап ИБ **+** бэкап/репликация бакета — это **две** разные задачи. Содержимое
  объектов в основной дамп СУБД может не попадать → отдельно бэкапь/версионируй бакет (versioning, lifecycle, репликация
  на второй сайт). Проверяй восстановимость **связки** «ИБ ↔ бакет», а не по отдельности.
- Версионирование бакета (bucket versioning) спасает от случайного перезаписи/удаления объекта; lifecycle-правила —
  для очистки старых версий/временных объектов.
- Дедупликация (8.3.24+) экономит место, но усложняет ручной разбор бакета — управляй очисткой удалённых объектов
  штатными средствами платформы, не руками в бакете.

**Нюансы.**
- Latency: S3 — это сеть, а не локальный диск. Для часто читаемых мелких файлов оцени задержки; тяжёлые операции
  выноси в фон.
- Согласованность: современный S3 (AWS, большинство) — read-after-write consistency, но у конкретного провайдера
  поведение листинга/удаления **[проверить]**.
- Мониторинг: следи за размером бакета, числом объектов, ошибками 4xx/5xx от S3 (логи веб-сервиса/провайдера) —
  встроить в общий мониторинг (см. `monitoring-and-backup.md`).
- Часы серверов 1С синхронизируй по NTP — иначе плавающие `RequestTimeTooSkewed` в программной интеграции.
