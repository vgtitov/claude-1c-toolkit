# MS SQL Server для 1С (кратко)

Для DBA, привыкшего к PostgreSQL/Postgres Pro: что отличается в MS SQL под 1С и что обязательно держать в
регламенте. Версии/редакции SQL Server — **НАСТРОЙ ПОД СВОЙ ПРОЕКТ**, поведение под конкретную версию
помечай **[проверить]**.

---

## 1. Ключевые отличия от PostgreSQL (модель хранения)

- MS SQL — **не версионник** (по умолчанию): данные лежат согласно **кластеризованному индексу** (больше
  «порядка» на диске), нет мёртвых версий строк как в PostgreSQL → нет VACUUM. Вместо борьбы с мёртвыми
  строками — борьба с **фрагментацией** индексов.
  - (При включённом RCSI/SNAPSHOT MS SQL ведёт версии строк в `tempdb` — отдельная тема, **[проверить]**
    под профиль 1С.)
- Регламент здоровья в MS SQL: **INDEX REBUILD/REORGANIZE** + **UPDATE STATISTICS** (аналог
  REINDEX + ANALYZE в PostgreSQL).
- 1С работает поверх обеих СУБД; принципы «держи порядок» одинаковы, инструменты разные.

---

## 2. Модель восстановления

- Для продуктивной 1С — модель восстановления **FULL** (полная): даёт восстановление на момент времени за
  счёт бэкапа лога транзакций (аналог PITR через WAL в PostgreSQL).
- Регламент бэкапа: полный бэкап + регулярный **бэкап журнала транзакций** (иначе лог растёт без границ).
  Без бэкапа лога в FULL-модели файл лога разрастётся и заполнит диск.
- SIMPLE-модель — только для баз, где допустима потеря с момента последнего полного/разностного бэкапа
  (обычно НЕ продуктивная учётная 1С).

---

## 3. tempdb

- **tempdb** — общая системная база под временные объекты (а 1С активно создаёт временные таблицы) →
  частое узкое место. Рекомендации: несколько равных файлов данных tempdb (по числу ядер, с разумным
  пределом), размещение на быстрых дисках, преднастроенный размер (autogrowth не на каждый чих).
- Аналогия с PostgreSQL: где там `temp_buffers`/`work_mem`/временные таблицы сеанса — здесь
  производительность tempdb. Точные рекомендации по числу файлов — по версии SQL Server, **[проверить]**.

---

## 4. Обслуживание индексов и статистики

- **Фрагментация индексов:** при фрагментации **> 30%** — `INDEX REBUILD` (не `REORGANIZE`: реорганизация
  при сильной фрагментации порядок не наведёт). Онлайн-перестроение — `REBUILD WITH (ONLINE = ON)` (аналог
  `REINDEX CONCURRENTLY` в PostgreSQL; доступность ONLINE зависит от редакции — **[проверить]**).
- **Статистика:** автообновление (`AUTO_UPDATE_STATISTICS`) инкрементально и его **недостаточно** для 1С.
  По рекомендации ИТС — обновлять с **FULLSCAN** (сканировать всю таблицу):
  `UPDATE STATISTICS <table> WITH FULLSCAN;` или насильно `sp_updatestats` (тоже инкрементальный — учесть).
  Это прямой аналог агрессивного `autovacuum_analyze_scale_factor`/ручного ANALYZE в PostgreSQL.
- Регламент обычно ночной (REBUILD + UPDATE STATISTICS); на больших инсталляциях — отдельное частое
  обновление статистики горячих таблиц (каждый час/несколько часов), как и в PostgreSQL отдельный
  ANALYZE-крон.

---

## 5. Управление кэшем («утренние тормоза»)

Особенность MS SQL, которой нет в PostgreSQL: ночной REBUILD/UPDATE STATISTICS поднимает в память данные,
которые утром пользователям не нужны, и держит их (MS SQL «съедает всю отданную RAM»). Утром на больших
базах — картина «100% нагрузка на диск при 10 МБ/с» на быстрой СХД, пока кэш не вытеснится полезными
данными.
- `DBCC DROPCLEANBUFFERS` — выгрузить из кэша всё, что сейчас не нужно (по-хорошему после ночных
  регламентов).
- `DBCC FREEPROCCACHE` — очистить кэш планов (1С рекомендует «при проблемах»; на практике многие делают
  планово).
- В **PostgreSQL этого делать НЕ нужно** — он сам вытесняет ненужные страницы (счётчик «сколько транзакций
  требуют поднятые в кэш данные»; цикл контрольных точек). «Утренних тормозов» от ночного регламента в
  PostgreSQL нет.
- Прогрев данных перед рабочим днём (эмуляция пользовательской нагрузки, напр. Тест-центр 1С) — отдельная
  методика, актуальная скорее для MS SQL; применять только если проблема реально наблюдается.

## 6. Флаги трассировки (trace flags) для 1С

- **1224** — отключает эскалацию блокировок по числу блокировок (эскалация остаётся по памяти). Под 1С
  снижает риск массовой эскалации до блокировки таблицы при большом числе строчных блокировок. Включать
  осознанно, понимая влияние на память.
- **4199** — включает накопленные исправления оптимизатора запросов (query optimizer hotfixes). Влияет на
  планы; в новых версиях SQL Server поведение частично управляется через уровень совместимости БД —
  **[проверить]** под свою версию, тестировать влияние на нагрузку 1С.
- Прочие флаги (напр. по tempdb) зависят от версии и в новых редакциях заменены настройками по умолчанию
  — **[проверить]**, не включать по памяти.

> Флаги трассировки и параметры — выставлять по официальным рекомендациям 1С/Microsoft под конкретную
> версию и проверять на предпроде, как и любые изменения СУБД.

## 7. Сводка соответствий PostgreSQL ↔ MS SQL

| Задача | PostgreSQL | MS SQL |
|---|---|---|
| Убрать «мусор»/порядок строк | VACUUM (мёртвые версии) | нет (кластерный индекс); REBUILD от фрагментации |
| Перестроить индекс онлайн | REINDEX INDEX CONCURRENTLY / pg_repack | INDEX REBUILD WITH (ONLINE = ON) |
| Обновить статистику | ANALYZE / autovacuum analyze | UPDATE STATISTICS WITH FULLSCAN / sp_updatestats |
| Восстановление на момент | архив WAL + PITR | модель FULL + бэкап лога транзакций |
| Временные объекты сеанса | temp_buffers/work_mem, временные таблицы | tempdb (несколько файлов, быстрые диски) |
| Кэш утром | сам вытесняет, ничего не делать | DBCC DROPCLEANBUFFERS / FREEPROCCACHE |
| Защита от переполнения счётчика | VACUUM FREEZE (TXID wraparound) | не применимо |

Подробности по PostgreSQL — в остальных reference этого скилла; здесь только сжатый контур MS SQL для
смешанных ландшафтов.
