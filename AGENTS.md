# AGENTS.md — правила разработки на 1С:Предприятие с AI-агентом

Единый свод правил, **общий для всех AI-агентов** — не привязан к конкретному (по открытому стандарту [agents.md](https://agents.md/)).
Из этого файла генерируются конфиги под конкретные агенты (Claude Code, Cursor, Gemini CLI, Copilot, Cline, Codex…) —
см. `adapters/` и `build.sh`. Claude Code не читает AGENTS.md нативно → подключается через `@AGENTS.md` в `CLAUDE.md`.

Область: **1С:Предприятие 8.3 (язык BSL)** — конфигурации и расширения (типовые ERP/УТ/БП и отраслевые). Не 1С-Битрикс
(это отдельный продукт — CMS на PHP), не Битрикс24. Правила **org-agnostic**: версии и конвенции настраиваются под свою
конфигурацию (отмечено «НАСТРОЙ»).

---

## Правило №0: перед работой — синхронизация
`git pull` toolkit и репозитория проекта (EDT-исходники / выгрузка конфигуратора). Обновить локальные клоны кода, по
которым ищет MCP (`onec-code`). Не работать на устаревшем клоне — MCP покажет старый код, выводы будут неверны.

## Версионный стек — НАСТРОЙ ПОД СВОЙ ПРОЕКТ
Зафиксируй и строго соблюдай: версия платформы 1С, **режим совместимости**, версия конфигурации, версии библиотек
(БСП и др.). Принцип: справку брать по версии платформы, но НЕ предлагать API, недоступный в твоём режиме
совместимости. Справка БСП — по реальной БСП своей версии. Опиши стек в `core/skills/1c-dev/references/conventions-template.md`.

## Главное правило: спроси инструмент, не угадывай
Модель врёт в деталях 1С — сигнатурах методов, составе типовой, поведении платформы. Источник истины — реальный код и справка:
- **1С-факты и сигнатуры — только по РЕАЛЬНОМУ коду** (MCP `onec-code`: ripgrep по локальным клонам конфигурации и
  расширения) и по справке платформы, **не по памяти**. Искать в ОБОИХ слоях контура (конфигурация + расширение).
- **После правки** — прогнать **BSL Language Server** (MCP `bsl-ls` в комплекте, свободный). Механизм без сверки
  помечать **[проверить]**, не «верно».
- Нет поведения ни в коде, ни в справке — так и сказать.

## Слои контура — НАСТРОЙ
Контур = **конфигурация** (основа, типовая — read-only) + **расширение** (свои доработки). Искать функционал в ОБОИХ
слоях. Правки — в расширение, типовую конфигурацию не менять (перезаписывается при обновлении). Опиши свои
слои/префиксы/точки расширения в `core/skills/1c-dev/references/conventions-template.md`, карту баз/контуров — в
`config/contours.example.md`.

## Гейт структурных правок (человек / AI)
Структурные правки метаданных (объекты, формы, макеты, СКД) AI выполняет **ТОЛЬКО** инструментами `onec_metadata`
(CLI `bin/1c-meta`, скилл `1c-metadata`). Два настоящих предохранителя (архитектурный аудит `docs/architecture-review-2026-07.md`):
1. **round-trip verify** — загрузка в ТЕСТ-базу платформой + повторная выгрузка + diff строго целевого изменения;
   платформа отвергает невалидное. Правки, не прошедшие round-trip, к прод-контуру не допускаются.
2. **Scope-граница** — только БЕЗОПАСНЫЕ правки без анализа ссылок:
   - **в scope** — добавить реквизит/ТЧ/команду/измерение/ресурс/значение перечисления, колонку печатной формы,
     поле/запрос СКД, включить объект в подсистему, выдать право, изменить существующее СКАЛЯРНОЕ свойство;
   - **вне scope** — переименование/удаление/перемещение/смена вида объекта (нужен каскадный ремонт ссылок во всём
     проекте — надёжно делает только EDT internal-model, не text-манипуляция).

`onec_metadata/catalog.py` — карта ПРОВЕРЕННОГО покрытия (`is_supported`), не рантайм-гейт. Деструктив с каскадом и
всё без доступной тест-базы — человек в IDE (**1C:EDT** / **Конфигуратор**) или EDT-плагин. AI пишет и проверяет КОД
(`*.bsl`): в EDT-контуре правит исходники в репозитории, в Конфигуратор-контуре выдаёт фрагменты для вставки.

## Производительность (кратко; глубоко — скиллы `1c-expert` / `1c-dba`)
1С тормозит предсказуемо. Обязательный минимум при написании кода:
- **Никогда — запрос в цикле** (собирай отбор по массиву, один запрос → выборка). Есть зашитый детектор
  `scripts/bsl_guard.py` (межпроцедурный) в pre-commit/конвейере применения.
- **Запросы:** явные поля в `ВЫБРАТЬ`, не `*`; отборы по индексируемым полям; временные таблицы для тяжёлых соединений.
- **Транзакции/блокировки:** управляемые блокировки, минимальная длительность транзакции; не держать транзакцию на время диалога.
Диагностика «почему медленно/падает» — скилл `1c-expert` (ТЖ, план запроса, APDEX), СУБД — `1c-dba`.

## Безопасность
- Токены/пароли/строки соединения — только в env/keychain, **НИКОГДА** в чат/репозиторий/код.
- Конфиденциальные и персональные данные (ПДн) в AI не передавать. Слой доступа к данным живой ИБ — только read-only
  под исследуемым пользователем (честный RLS), с маскированием ПДн (см. `docs/data-access-architecture.md`).
- Права ролей — по принципу минимальных привилегий; привилегированный режим `ai_debug` — за явным гейтом (fail-closed).

## Скиллы toolkit (формат SKILL.md — переносимый между агентами)
- `1c-analyst` — анализ/архитектура: разбор задачи, ЧТЗ (поведение, не реализация), `/clarify`, влияние доработки, карта контуров/баз.
- `1c-dev` — разработка/ревью BSL: рабочий цикл (вход → атомарные задачи [Разработчик/IDE] | [AI/код] → инструкции по
  не-коду → код → проверки/тесты → передача пакетом). Reuse-first БСП, производительность, стандарты `v8-code-style`.
- `1c-metadata` — программные структурные правки метаданных (модуль `onec_metadata`, CLI `bin/1c-meta`): колонка в
  печатную форму, поле/запрос СКД, реквизит/ТЧ/команда/измерение/ресурс/значение — оба формата (Конфигуратор-XML и
  EDT `.mdo`/`.dcs`), byte-perfect, применение в ТЕСТ-базу с round-trip verify.
- `1c-admin-devops` — администрирование + DevOps: кластер/`rac`/лицензии, публикация на веб-сервере, обновление
  конфигураций/расширений, CI/CD, контейнеры, мониторинг и бэкап, Linux-сервер.
- `1c-dba` — DBA под 1С: тюнинг PostgreSQL/MS SQL, регламент (VACUUM/REINDEX/FREEZE), блокировки на уровне СУБД, бэкап
  с проверкой восстановимости, HA/масштабирование. Сначала измерь (`pg_stat_*`, `EXPLAIN ANALYZE`), потом меняй.
- `1c-expert` — эксперт по техн. вопросам и highload: ТЖ (`logcfg.xml`), план запроса, блокировки/дедлоки, APDEX и
  замеры, кластер и нагрузочное тестирование. Диагноз по измерениям; реализацию фикса ведут `1c-dev` (код) и `1c-dba` (СУБД).

## Инструменты (MCP — единый профиль `core/mcp/servers.json`)
- `onec-code` — чтение реального кода 1С (ripgrep по локальным клонам конфигурации/расширения, авто-обнаружение слоёв).
- `bsl-ls` — мост над BSL Language Server (диагностики BSL, в комплекте, свободный).
- `onec-ops` — эксплуатация read-only (ТЖ/ЖР/APDEX/Zabbix/Prometheus).
- `onec-data` — слой ДАННЫХ живой ИБ read-only (OData + отладочный сервис `ai_debug`), под исследуемым пользователем,
  за гейтом; требует публикации сервисов админом (см. `docs/data-access-architecture.md`).
- `edt` (опционально, для тех, кто в 1C:EDT) — MCP-плагин внутри IDE: живой доступ к рабочему пространству (ошибки,
  метаданные, формы, отладка). Отдельный плагин и токен; установка и подключение — `docs/EDT_SETUP.md`.

## Что агент НЕ делает без человека
Структурный деструктив с каскадом ссылок (переименование/удаление/смена вида объекта); правки в типовой конфигурации;
любое изменение без доступной ТЕСТ-базы для round-trip; передачу ПДн в модель; применение к ПРОД-контуру без явного
подтверждения. Любую правку человек принимает после проверки (в т.ч. в 1C:EDT).
