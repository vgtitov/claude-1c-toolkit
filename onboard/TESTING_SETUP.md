# Настройка тестирования 1С (выполняет Claude) — по лестнице docs/testing-ladder.md

Инструкция для агента: как развернуть и использовать ступени тестирования в
НОВОМ контуре (клиент/организация). Предпосылки: ssh-алиас на сервер 1С с
платформой (или платформа на своей машине), учётка ИБ с правами конфигуратора,
пароли в локальном файле организации (НЕ в чат/репо).

> **Сначала — `docs/setup-actions-required.md`.** Там то, что Claude сделать НЕ может и
> о чём обязан попросить человека: снятая «Защита от опасных действий» (иначе батч-прогоны
> ступени 2 виснут на модальном окне), публикация базы для слоя данных, учётка ИБ, токен
> центра. `scripts/doctor.py` проверяет это и печатает конкретную инструкцию.

## Ступень 0 — статика (всегда, ничего разворачивать не нужно)
1. `python scripts/install_git_hooks.py` в репо исходников — pre-commit с
   `bsl_guard` (staged *.bsl).
2. Легаси-долг: `bin/1c-meta bsl-gate baseline <корень расширения>` — фиксирует
   существующие находки; коммить `.bsl_guard_baseline.json` в репо исходников.
   Дальше гейт пропускает только «не хуже» (ratchet), и он же стоит внутри
   `upload_tree` (загрузка в базу с НОВЫМ запросом-в-цикле невозможна).
3. СКД после правок: `bin/1c-meta scd validate <Template.xml>`.

## Ступени 1–2 одной командой: `scripts/onec_verify.py`
```
# ступень 1 — синтакс-контроль платформой (все контексты компиляции)
python scripts/onec_verify.py check --base "срв\ERP_Test" --ext МоёРасширение --contour ERP_TEST
# ступень 2 — серверный сценарий на реальных данных (--deploy пересоберёт раннер)
python scripts/onec_verify.py smoke --base "D:\bases\erp" --scenario scen.bsl --deploy
```
Транспорт выбирается сам: без `--host` платформа берётся на ЭТОЙ машине,
с `--host <алиас ssh>` — на сервере 1С. Учётки — `ONEC_IB_USER` /
`ONEC_IB_PASS[_<КЛЮЧ>]` в локальном `.env` организации. Ниже — те же ступени
через библиотеку, когда нужен нестандартный сценарий.

## Ступень 1 — batch DESIGNER (round-trip)
Настроить env организации: `ONEC_1CV8_BIN`, ssh-алиас; использовать
`apply.dumpload`: `upload_tree → load_extension (Load+UpdateDBCfg -Extension) →
dump_extension → fetch_tree → diff` (NFC-нормализация имён при сравнении
Mac↔Windows!). Канонический эталон для git — платформенная перевыгрузка.
Синтакс-контроль — `apply.external.check_config`; убрать расширение за собой в
чужой базе — `apply.external.delete_extension`.
Локально (платформа на своей машине) `upload_tree`/`fetch_tree` не нужны —
`load_extension`/`dump_extension` работают прямо по локальным путям.

## Ступень 2 — смоук-раннер (ГЛАВНОЕ для контуров без фреймворков)
1. Разово: `apply.smoke.deploy_smoke_runner(runner, ib, user, pwd,
   workdir=r"D:\...\smoke")` — зальёт шаблон `templates/smoke_runner` и соберёт
   `СмоукРаннер.epf` платформой.
2. Прогон: `apply.smoke.run_smoke(..., scenario_text=<серверный BSL>,
   epf=..., workdir=...)` → текст отчёта; `__SMOKE_FAIL__` = ApplyError.
3. Сценарии храним В РЕПО КОНТУРА организации (клиентские данные/имена НЕ в
   toolkit), там же журнал прогонов. Контракт сценария: переменная `Отчет`.
4. Грабли (проверено в бою): компоновка отчётов — только штатным
   `Объект.СкомпоноватьРезультат`; поля для группировок требуют totalField;
   `ВнешниеОтчеты.Подключить` на сервере — через временное хранилище;
   запись справочников — `ОбменДанными.Загрузка = Истина`; таймаут >10 мин —
   запускать фоном.
5. Первый ENTERPRISE-сеанс к клиент-серверной базе по VPN может идти ДЕСЯТКИ
   минут (платформа тянет кэш конфигурации), при том что DESIGNER на той же базе
   отвечает за секунды. Не считать это зависанием: ставить `--timeout` с запасом
   и запускать фоном.

## Ступень 3 — веб-смоук
Публикация тест-базы (vrd + конфиг веб-сервера) → сценарий в браузере.
Публикация «работает» ТОЛЬКО при показанной строке `200` в access.log.

## Ступень 4 — фреймворки (по решению организации)
Vanessa Automation (+vanessa-runner, корп-стандарт Аскона для smoke на MR),
YAxUnit (юнит-тесты, EDT-плагин), Тестер. Ставятся в тест-базы организаций,
в чужие прод-контуры НЕ ставим. Подробности и выбор — docs/testing-ladder.md.
